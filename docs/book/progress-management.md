# 项目进度管理规范（Progress Management）

> 适用范围：`docs/book/` 下围绕书稿推进的状态管理与协作交接。

## 1. 先看哪里：当前状态 vs 历史快照

### 1.1 当前状态（Single Source of Truth）

优先看这两个文件：

1. `docs/book/project-status.md`
   - 作用：项目“当前全局状态”的唯一总览。
   - 适合回答：
     - 现在处于哪个阶段？
     - 已完成了哪些块？
     - 当前协作模式、主要风险、明日优先级是什么？

2. `docs/book/roadmap.md`
   - 作用：短周期（默认 24h）目标与执行节奏。
   - 适合回答：
     - 当前窗口期重点是什么？
     - 这轮要交付什么？
     - 执行阶段和节奏规则是什么？

> 结论：
> - 想知道“现在项目在哪儿”→ 看 `project-status.md`
> - 想知道“接下来 24h 怎么推进”→ 看 `roadmap.md`

### 1.2 历史快照（Timeline / Checkpoints）

`docs/book/status-*.md` 是**时间点快照日志**，不是当前真相源。

- 文件命名：`status-YYYY-MM-DD-HHMM.md`
- 内容特征：当时刻完成了什么、做了哪些自检、最小增量是什么。
- 使用场景：
  - 回溯某次决策背景
  - 对齐“为什么当时这样做”
  - 复盘节奏、定位偏航点

> 规则：
> - **当前判断**以 `project-status.md` + `roadmap.md` 为准；
> - `status-*.md` 用于补充上下文证据，不直接替代当前状态文档。

## 2. 状态更新规则（How to Update）

## 2.1 更新触发条件

发生以下任一情况时，必须更新状态文档：

1. 阶段变化（例如从“初步梳理”进入“体系化推进”）
2. 关键交付完成（章节里程碑、核心 notes 完成、关键脚本落地）
3. 优先级变化（P0/P1 重排）
4. 风险显著变化（新增关键风险或风险解除）
5. 协作方式变化（分工、交接模板、轮动节奏调整）

## 2.2 更新职责分工

- `project-status.md`：只保留“当前仍成立”的内容，持续覆盖更新。
- `roadmap.md`：维护下一窗口期目标与执行相位，滚动更新。
- `status-*.md`：每次小步完成后按时间追加快照，不回写历史文件。

## 2.3 建议更新流程（最小闭环）

1. 完成一个可验证小块（分析、注释、章节、索引、脚本等）
2. 新增一条 `status-*.md`（记录“做了什么 + 最小增量 + 是否偏航”）
3. 若影响当前真相，同步更新：
   - `project-status.md`（全局状态）
   - `roadmap.md`（短期计划）
4. `git add` / `commit` / `push`，保证远端可见

## 2.4 内容约束（写状态时必须包含）

至少覆盖以下 4 项：

1. 本轮完成项（事实）
2. 当前判断（是否偏航、是否满足验收）
3. 最小增量落盘位置（具体文件）
4. 下一步动作（可立即执行）

## 3. 协作 Agent 快速恢复上下文（Fast Context Recovery）

新接手 Agent 建议按下面顺序读取，目标是在 5~10 分钟内恢复可执行上下文。

## 3.1 五步读取顺序

1. `docs/book/project-status.md`（现在在哪）
2. `docs/book/roadmap.md`（下一步做什么）
3. 最近 3~5 个 `docs/book/status-*.md`（近期轨迹与节奏）
4. `docs/book/collab-split.md`（当前分工边界）
5. 与任务直接相关的章节或 `architecture-notes/` 文件

## 3.2 快速判断模板（接手后立即给出）

建议接手 Agent 用 4 行输出：

1. 当前阶段：
2. 本轮目标（24h）：
3. 我负责的最小闭环：
4. 完成后将更新哪些状态文件：

## 3.3 交接最小包（handoff package）

完成一轮后，至少保证以下信息可被下一位直接接住：

- 一句话结论（本轮推进价值）
- 变更文件清单
- 关键证据锚点（源码或文档位置）
- 未完成但已识别的下一动作
- 对 `project-status.md` / `roadmap.md` 是否需要同步更新的判断

---

## 4. 约束与反模式

## 4.1 必守约束

1. 不用 `status-*.md` 代替 `project-status.md`
2. 不在历史快照里回写“当前真相”
3. 任何阶段/优先级变化必须落盘
4. 小步提交，远端持续可见

## 4.2 常见反模式

1. 只写了 status 快照，没更新 project-status（导致当前状态漂移）
2. roadmap 长时间不滚动（导致执行与计划脱节）
3. 快照只写“做了很多”，没有具体落盘文件与下一步
4. 协作交接靠聊天，不写入仓库文档

---

## 5. 一页版执行清单（Checklist）

每完成一轮工作，按此检查：

- [ ] 本轮是否形成可验证最小增量
- [ ] 是否新增了对应 `status-*.md`
- [ ] 当前真相是否需要更新 `project-status.md`
- [ ] 24h 计划是否需要更新 `roadmap.md`
- [ ] 是否明确写出下一步动作
- [ ] 是否已 commit + push

该清单通过后，才算本轮“进度管理”完成。
