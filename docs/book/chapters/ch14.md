# 第14章 工具作为手脚

模型负责思考，工具负责执行。这句话说起来简单，但工具系统的设计质量，直接决定 Agent 能不能在真实场景里稳定工作。

## 14.1 工具是什么

在 Claude Code 里，工具是一个有统一接口的执行单元。`src/Tool.ts` 定义了这个接口：工具有名称、描述、输入格式（JSON Schema）、执行函数，还有几个约束位：是否只读、是否可并发、权限检查函数。

这些约束位不是装饰，而是系统在执行前用来做决策的依据。权限检查决定这次调用是否需要用户确认，只读标记决定是否可以在受限模式下运行，并发标记决定多个工具调用能不能同时执行。

`buildTool()` 函数在构建工具时会用 `TOOL_DEFAULTS` 补齐未实现的字段。默认值是保守的：默认不可并发、默认非只读。这意味着新工具如果没有明确声明，会被当作"可能有副作用、不能并发"来处理，而不是反过来。

## 14.2 工具池的组装方式

`src/tools.ts` 管理工具池。流程是：`getAllBaseTools()` 给出所有内建工具，`filterToolsByDenyRules()` 按权限规则裁剪，`assembleToolPool()` 把内建工具和 MCP 工具合并，处理重名冲突。

这个"先全集、后过滤"的顺序有一个实际好处：权限规则集中在一处管理，不散落在各个工具的实现里。如果某个工具在某种模式下不应该可用，只需要在过滤规则里加一条，不需要修改工具本身。

MCP（Model Context Protocol）工具是外部工具，通过标准协议接入。`assembleToolPool()` 在合并时会处理内建工具和 MCP 工具重名的情况，保证工具名称在整个工具池里唯一。

## 14.3 工具执行的实际流程

工具调用发生在 `query.ts` 的主循环里。模型返回 `tool_use` 块，循环收集这些块，然后执行对应的工具函数，把结果作为 `tool_result` 注入下一轮上下文。

执行前会做权限检查。如果工具的 `needsPermissions()` 返回需要确认，系统会暂停执行，等待用户授权。这个检查在每次调用时都会执行，不是一次性授权。

工具执行的结果有两种形态：成功返回内容，或者返回错误信息。两种情况都会作为 `tool_result` 注入上下文，让模型知道执行结果。如果工具执行失败，模型可以根据错误信息决定下一步：重试、换一种方式，或者告知用户。

## 14.4 自己做工具时的关键决策

**接口统一比功能丰富更重要**。如果每个工具有不同的调用方式、不同的错误格式、不同的权限处理，系统会越来越难维护。统一接口的代价是前期多写一些约束代码，收益是后期加工具时不容易出问题。

**默认值要保守**。一个工具如果不确定是否有副作用，就标记为非只读。如果不确定能不能并发，就标记为不可并发。保守默认值在出问题时更容易排查，因为行为是可预期的。

**错误要可读**。工具执行失败时，错误信息会进入模型的上下文。如果错误信息是"Error: undefined is not a function"，模型很难据此做出有用的决策。如果是"文件 /path/to/file 不存在，请检查路径是否正确"，模型就有了可以处理的信息。

**权限粒度要合理**。不是所有工具都需要用户确认，但有副作用的操作（写文件、执行命令、发送请求）应该有明确的权限标记。权限检查的目的不只是安全，还有可预期性：用户知道哪些操作会被执行，哪些需要确认。

## 14.5 工具数量的边界

工具越多，模型在选择时的负担越重。Claude Code 的工具池在不同运行模式下会过滤到不同的子集，不是把所有工具都暴露给模型。

对业务 Agent 来说，工具数量控制在 10-15 个以内通常是合理的起点。每个工具的描述要清晰，让模型能准确判断什么时候该用哪个工具。描述不清晰的工具，比没有工具更糟糕，因为它会导致模型做出错误的工具选择。

## 14.6 本章小结

工具系统的核心是统一接口和集中过滤。`src/Tool.ts` 定义接口和保守默认值，`src/tools.ts` 管理工具池的组装和过滤。工具执行发生在主循环里，结果作为上下文注入下一轮。

自己做工具时，接口统一、默认值保守、错误可读、权限粒度合理，这四点比工具数量更重要。下一章看工具的另一面：提示词和任务指令如何控制模型的行为。
