# Chapter 2

## Claude Code 为什么强

很多人看 Claude Code，会先看到表面的能力。
它能读文件，改文件，跑命令，还能接很多工具。
如果只看这些，很容易把它理解成一个能力更多的聊天工具。

这个理解不够准确。
Claude Code 的价值，主要不在功能清单有多长，而在这些功能是怎样被组织起来的。
同样的模型，同样的几个工具，换一套粗糙的组织方式，系统表现会差很多。原因很简单。工作不是一次回答，工作是一个连续过程。只要过程一长，系统就会遇到上下文增长、任务切换、工具调用、失败恢复、权限控制这些问题。谁能把这些问题处理得更稳定，谁才更接近真正可用的 Agent。

这一章想说明一件事。Claude Code 的强，不是单点能力的强，而是整体组织能力的强。

## 先看表面，再看骨架

从表面看，Claude Code 做了几件很容易被人理解的事。
它接收任务。
它读环境。
它决定下一步。
它调用工具。
它把结果继续接回自己的工作过程。
如果任务没结束，它还会继续推进。

这些动作单独看都不稀奇。
真正有难度的是把这些动作连成一条稳定的工作链路。
在这条链路里，系统要知道当前做到哪里，为什么继续，什么时候停，失败之后怎么接上，哪些动作可以做，哪些动作不能做。
这已经不是提示词写得漂不漂亮的问题了，而是运行时架构的问题。

从仓库结构看，这条骨架大致可以分成几层。
入口层负责接住请求，把不同运行模式分开。
能力装配层负责整理命令、工具、插件和可见能力。
会话编排层负责组织一次会话怎么开始、怎么持续、怎么留下记录。
单轮执行层负责处理一轮任务里的状态变化、工具调用和继续执行。
任务、历史、恢复、压缩、记忆这些部分，再共同保证长任务可以走下去。

这也是为什么我前面一直强调，Claude Code 的重点在系统结构。
你可以把单个工具抄过去，也可以把某段提示词抄过去，但如果没有这一层组织能力，系统很快就会露出问题。

## 强在连续工作能力

普通聊天系统最擅长的是一问一答。
你问，它答。答完这一轮，事情基本结束。
Claude Code 处理的不是这种短关系。它处理的是持续推进。

持续推进意味着什么。
意味着系统不能只会生成文字，还要会管理过程。
它要能记住现在在处理什么任务。
它要能判断自己是否刚刚调用了工具，是否还需要继续下一轮。
它要能把中间结果写进历史里，方便后面恢复。
上下文太长时，它还要能压缩，但压缩以后不能把真正关键的事实弄丢。

这些能力放在一起，构成了一个更接近工作系统的东西。
所以当我们说 Claude Code 强，很多时候说的并不是它每一句回答都更漂亮，而是它在一个较长的任务过程中，出错更少，断掉以后更容易接上，动作边界更清楚。

## 强在统一的工具契约

很多人第一次接触 Agent，会把工具理解成外挂。
这种理解容易把问题想简单。
对于 Claude Code 这类系统，工具更像一组被统一管理的动作接口。
系统要知道一个工具能做什么，不能做什么，默认风险是什么，是否允许并发，调用结果怎样回到主流程里。

这件事在源码里很清楚。
工具不是零散地贴在系统边上，而是在统一契约里被整理、过滤和调度。
这样做的好处很直接。
第一，系统行为更一致。
第二，权限边界更清楚。
第三，后续增加新工具时，不需要每次都重新发明一套接入规则。

对于想自己做 Agent 的读者，这里有一个很重要的启发。
先不要急着堆工具。
先把工具边界讲清楚，先让系统知道什么时候能调用、调用后怎么接回主流程。这个顺序比工具数量重要得多。

## 强在状态和恢复

真实工作里，任务很少一口气完成。
中断是常态。
上下文变长是常态。
任务换人接手也是常态。

所以，一个系统是否可靠，常常取决于它断掉以后还能不能继续。
Claude Code 在这方面下了很大功夫。
从 history、conversation recovery、compact、session memory 这些结构可以看出来，系统一直在处理同一个核心问题：怎样让任务在更长时间尺度上继续成立。

这里要注意一件事。
恢复不是把过去的话原样拼回来就结束。
恢复分成至少两层。
第一层是事实恢复，也就是把做过什么、产生了什么结果、当前任务到哪里了这些信息找回来。
第二层是语义修复，也就是把这些事实重新组织成系统还能继续理解和推进的上下文。

这一点很关键。因为它说明 Claude Code 关心的不是聊天记录本身，而是工作连续性。

## 强在任务系统与并发一致性

很多产品也会有任务列表，但真正进入执行时，问题就复杂了。
谁来认领任务。
同一个任务会不会被重复执行。
一个任务失败以后，状态怎样更新。
多个执行体同时工作时，怎样避免互相覆盖。

Claude Code 相关代码里可以看到，它在任务系统上花了很多心思。任务不是一个装饰性的列表，而是运行过程的一部分。任务有标识，有生命周期，有认领逻辑，也有并发下的一致性保护。

对普通读者来说，不需要一开始就做这么复杂。
但至少要学会一个判断标准。一个 Agent 系统如果要用于工作，它就不能只有一个输入框。它得知道任务当前在哪里，谁在处理，结果写到哪里，失败以后怎样继续。

## 强在它更像系统，而不是功能集合

现在很多 Agent 产品给人的印象是功能很多。
看起来什么都能接，什么都能做。
但功能多，不等于系统成熟。

Claude Code 值得认真研究，是因为它提供了一个比较清楚的样本。它让我们看到，一个真正有工作能力的 Agent，大致需要哪些骨架：
入口，能力装配，会话编排，单轮执行，任务系统，历史，恢复，压缩，权限。
这些东西彼此配合，才构成了稳定性。

所以，如果把这一章压成一句话，可以这样说。
Claude Code 强，主要强在它把模型和工具放进了一套能持续工作的运行结构里。

## 对读者最有用的结论

如果你想从 Claude Code 学东西，先学这几件事。
先学怎样组织状态。
先学怎样接工具。
先学怎样记录历史。
先学怎样处理恢复。
先学怎样让任务可以被追踪。

这些东西会直接决定你的 Agent 能不能进入真实工作。
等这些骨架站稳了，再去考虑更多工具、更复杂的自动化、更大的系统规模，顺序会更稳。

下一章我们就按这个思路继续往下走，直接拆 Claude Code 的主干结构，看它从入口到执行，是怎么一层层接起来的。
