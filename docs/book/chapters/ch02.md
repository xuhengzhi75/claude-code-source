# 第2章（提纲）Claude Code 为什么强：能力来源的系统分解

> 状态：提纲版（待扩写）

## 2.1 先给结论
- Claude Code 的优势主要来自系统级组合，而非单点模型性能。
- “模型能力”与“系统能力”需分开评估。

## 2.2 五层能力来源（建议作为全章主骨架）
1. 推理层：模型在任务理解、拆解、修正中的作用
2. 工具层：文件/命令/网络/浏览器等可执行接口
3. 编排层：何时思考、何时调用工具、何时收敛
4. 状态层：上下文记忆、阶段进度、约束保持
5. 恢复层：失败检测、重试策略、回退路径

## 2.3 “强”具体强在哪（读者可感知指标）
- 能连续完成多步任务，而非只回答单问句
- 能在真实环境执行动作，不停留在建议层
- 能在中断后继续推进，不轻易任务崩盘
- 能输出结构化结果，便于复核与追踪

## 2.4 常见误判
- 误判1：把“回答像专家”当成“系统可交付”
- 误判2：只堆提示词，不补工具与状态机制
- 误判3：忽略恢复设计，导致一错全停

## 2.5 小实验（章节实践草案）
- 实验A：仅用聊天模型完成任务（记录中断点）
- 实验B：加入工具与状态后复做同任务
- 对比维度：完成率、耗时、返工次数、可追溯性

## 2.6 给普通读者的迁移策略
- 先搭最小可用链路：输入 → 拆解 → 工具执行 → 结果校验
- 再逐步补：状态缓存、失败恢复、模板化工作流
- 避免一开始追求“全功能 Claude Code 复刻”
