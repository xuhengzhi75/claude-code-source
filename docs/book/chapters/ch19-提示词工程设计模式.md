# 第19章 提示词工程的设计模式

第15章讲了系统提示词的基本结构。这一章往深处走：Claude Code 的提示词系统里有哪些可以复用的设计模式，每个模式解决什么问题，背后的权衡是什么。

这些模式不是理论总结，而是从 `src/constants/prompts.ts`（914 行）和相关文件里直接提炼出来的。

## 19.1 模式一：两级缓存节（Cached vs Volatile Section）

`src/constants/systemPromptSections.ts` 定义了两个函数，名字本身就是设计意图的声明：

```typescript
// 普通节：计算一次，缓存到 /clear 或 /compact
export function systemPromptSection(
  name: string,
  compute: ComputeFn,
): SystemPromptSection {
  return { name, compute, cacheBreak: false }
}

// 危险节：每轮重新计算，会破坏 prompt cache
// 必须传入 _reason 说明为什么不得不这样做
export function DANGEROUS_uncachedSystemPromptSection(
  name: string,
  compute: ComputeFn,
  _reason: string,  // ← 强制写明理由
): SystemPromptSection {
  return { name, compute, cacheBreak: true }
}
```

`DANGEROUS_` 前缀不是装饰，是一个工程约定：任何使用这个函数的地方，都必须在代码里留下"为什么不得不每轮重算"的解释。当前代码库里只有一处使用了它——MCP 指令节，原因是 MCP 服务器可能在会话中途连接或断开，必须每轮重新评估。

这个模式的价值在于：它把"这个节会破坏缓存"这件事从隐式变成了显式。如果用普通函数，开发者可能不知道自己写的节会每轮重算，悄悄增加 API 成本。`DANGEROUS_` 前缀让这个代价在代码审查时就能被发现。

对业务 Agent 来说，这个模式的核心思路是：把提示词的各个部分按"变化频率"分类，高频变化的部分和低频变化的部分分开管理，不要让高频变化污染低频稳定的缓存。

## 19.2 模式二：前缀哈希变体控制（Prefix Hash Variant Control）

`prompts.ts#L343` 的注释解释了一个反直觉的约束：

```typescript
/**
 * Session-variant guidance that would fragment the cacheScope:'global'
 * prefix if placed before SYSTEM_PROMPT_DYNAMIC_BOUNDARY. Each conditional
 * here is a runtime bit that would otherwise multiply the Blake2b prefix
 * hash variants (2^N). See PR #24490, #24171 for the same bug class.
 */
function getSessionSpecificGuidanceSection(...): string | null {
```

这段注释在说：如果把会话相关的条件判断（比如"是否有 AskUserQuestion 工具"、"是否是非交互会话"）放在边界标记之前，每种条件组合都会产生一个不同的前缀哈希，缓存命中率会指数级下降（N 个条件 = 2^N 种变体）。

解决方案是把所有运行时条件判断都移到边界标记之后。边界之前的内容对所有会话完全一致，只有一个哈希值，缓存命中率最大化。

这个问题在业务 Agent 里很常见：开发者在系统提示词里加了一个"如果用户是 VIP 则..."的条件，结果 VIP 用户和普通用户的提示词前缀不同，缓存完全失效。正确做法是把所有用户相关的差异化内容放在提示词的后半段，前半段保持对所有用户完全一致。

## 19.3 模式三：数字锚点替代定性描述（Numeric Anchors over Qualitative Directives）

`prompts.ts#L527` 里有一段只对内部用户开放的指令：

```typescript
// Numeric length anchors — research shows ~1.2% output token reduction vs
// qualitative "be concise". Ant-only to measure quality impact first.
systemPromptSection(
  'numeric_length_anchors',
  () =>
    'Length limits: keep text between tool calls to ≤25 words. Keep final responses to ≤100 words unless the task requires more detail.',
),
```

注释里说得很清楚：研究数据显示，"≤25 words"比"be concise"减少约 1.2% 的输出 token。这不是直觉，是 A/B 测试的结果。

这个模式揭示了一个提示词工程的普遍规律：模型对定量约束的遵从性高于定性描述。"简洁"是模糊的，每个模型对"简洁"的理解不同；"≤25 words"是精确的，模型可以计数。

类似的应用场景：与其写"不要生成太长的代码"，不如写"每个函数不超过 50 行"；与其写"适当添加注释"，不如写"只在逻辑不自明时添加注释，每个文件注释密度不超过 10%"。

## 19.4 模式四：内外部用户的双轨输出哲学

`getOutputEfficiencySection()` 对内外部用户返回完全不同的内容，差异不只是措辞，而是两套根本不同的沟通哲学：

**外部用户版本**（简洁直接）：
```
# Output efficiency
IMPORTANT: Go straight to the point. Try the simplest approach first without
going in circles. Do not overdo it. Be extra concise.
Lead with the answer or action, not the reasoning. Skip filler words, preamble,
and unnecessary transitions.
```

**内部用户版本**（散文写作）：
```
# Communicating with the user
When sending user-facing text, you're writing for a person, not logging to a console.
...Write user-facing text in flowing prose while eschewing fragments, excessive em
dashes, symbols and notation...
Use inverted pyramid when appropriate (leading with the action)...
```

外部版本的核心是"极简"，内部版本的核心是"写给人看的散文"。内部版本甚至提到了倒金字塔结构（新闻写作技法）、语义回溯（semantic backtracking）等专业写作概念。

这个差异背后的逻辑是：外部用户通常在完成具体任务，需要快速得到答案；内部用户（Anthropic 工程师）在评估模型行为，需要能读懂完整推理过程的输出。同一个模型，面对不同受众，需要不同的表达方式。

对业务 Agent 来说，这个模式的启示是：不要假设所有用户需要同样的输出风格。技术用户和非技术用户、快速操作场景和深度分析场景，需要不同的输出指令。

## 19.5 模式五：反过度工程指令（Anti-Gold-Plating Directives）

`getSimpleDoingTasksSection()` 里有一组专门对抗模型"过度工程"倾向的指令，值得完整引用：

```
Don't add features, refactor code, or make "improvements" beyond what was asked.
A bug fix doesn't need surrounding code cleaned up. A simple feature doesn't need
extra configurability. Don't add docstrings, comments, or type annotations to code
you didn't change.

Don't add error handling, fallbacks, or validation for scenarios that can't happen.
Trust internal code and framework guarantees. Only validate at system boundaries.

Don't create helpers, utilities, or abstractions for one-time operations. Don't
design for hypothetical future requirements. Three similar lines of code is better
than a premature abstraction.
```

这三段指令针对的是模型的三种过度工程模式：功能蔓延（做了没要求的事）、防御性编程（处理不可能发生的情况）、过早抽象（为一次性操作创建工具函数）。

注意这些指令的写法：不是"不要过度工程"这种定性描述，而是给出了具体的反例（"A bug fix doesn't need surrounding code cleaned up"）。具体的反例比抽象的原则更有效，因为模型可以直接匹配场景。

同一节里还有一条针对 Capybara v8 过度注释问题的专项指令：

```typescript
// @[MODEL LAUNCH]: Update comment writing for Capybara — remove or soften
// once the model stops over-commenting by default
`Default to writing no comments. Only add one when the WHY is non-obvious:
a hidden constraint, a subtle invariant, a workaround for a specific bug,
behavior that would surprise a reader.`
```

`@[MODEL LAUNCH]` 标记说明这是临时补丁，下个模型版本需要重新评估是否还需要。这是提示词版本管理的一种实践：给临时指令打标记，避免它们永久留在提示词里。

## 19.6 模式六：可逆性框架（Reversibility Framework）

`getActionsSection()` 把所有操作按两个维度分类：可逆性（reversible vs hard-to-reverse）和影响范围（local vs shared systems）。

```
Carefully consider the reversibility and blast radius of actions.
Generally you can freely take local, reversible actions like editing files or
running tests. But for actions that are hard to reverse, affect shared systems
beyond your local environment, or could otherwise be risky or destructive,
check with the user before proceeding.
```

然后给出了具体的高风险操作清单：

```
- Destructive operations: deleting files/branches, dropping database tables,
  killing processes, rm -rf, overwriting uncommitted changes
- Hard-to-reverse operations: force-pushing, git reset --hard, amending
  published commits, removing or downgrading packages/dependencies
- Actions visible to others: pushing code, creating/closing/commenting on PRs,
  sending messages (Slack, email, GitHub), posting to external services
```

这个框架的设计思路是：不要让模型自己判断"这个操作危不危险"（判断标准模糊），而是给出一个结构化的分类框架，让模型套用框架来判断。

注意最后一条关于第三方工具的说明：

```
Uploading content to third-party web tools (diagram renderers, pastebins, gists)
publishes it - consider whether it could be sensitive before sending, since it
may be cached or indexed even if later deleted.
```

这条指令覆盖了一个容易被忽视的场景：把内容发送给第三方渲染服务（比如 Mermaid 在线渲染、Pastebin）也是一种"发布"行为，即使用户没有意识到。

## 19.7 模式七：对抗性验证 Agent 的提示词设计

`src/tools/AgentTool/built-in/verificationAgent.ts` 里的验证 Agent 提示词是一个完整的对抗性验证规范，值得单独分析。

它的核心设计思路是：**预设验证者会偷懒，然后逐一堵死偷懒的路**。

提示词开头直接点名两种失败模式：

```
You have two documented failure patterns.
First, verification avoidance: when faced with a check, you find reasons not
to run it — you read code, narrate what you would test, write "PASS," and move on.
Second, being seduced by the first 80%: you see a polished UI or a passing test
suite and feel inclined to pass it, not noticing half the buttons do nothing...
```

然后给出了一个"认识自己的合理化借口"清单：

```
- "The code looks correct based on my reading" — reading is not verification. Run it.
- "The implementer's tests already pass" — the implementer is an LLM. Verify independently.
- "This is probably fine" — probably is not verified. Run it.
- "I don't have a browser" — did you actually check for mcp__claude-in-chrome__*?
```

最后用强制格式约束来防止"假验证"：

```
Every check MUST follow this structure. A check without a Command run block
is not a PASS — it's a skip.

### Check: [what you're verifying]
**Command run:** [exact command you executed]
**Output observed:** [actual terminal output — copy-paste, not paraphrased]
**Result: PASS** (or FAIL)
```

这个设计模式可以推广到任何需要"防止 AI 走捷径"的场景：先明确列出 AI 会走的捷径，然后用格式约束强制它不能走。光靠"请认真验证"是不够的，需要让"不认真"在格式上就无法通过。

## 19.8 模式八：Output Style 的优先级链

`src/constants/outputStyles.ts` 实现了一套输出风格的优先级覆盖机制：

```typescript
// 优先级从低到高：built-in < plugin < user < project < managed
const styleGroups = [pluginStyles, userStyles, projectStyles, managedStyles]

for (const styles of styleGroups) {
  for (const style of styles) {
    allStyles[style.name] = { ... }  // 高优先级覆盖低优先级
  }
}
```

内置风格有两种：Explanatory（解释型，在写代码前后提供教育性说明）和 Learning（学习型，在关键决策点暂停，让用户自己写代码）。用户可以在此基础上自定义，项目级配置可以覆盖用户级配置，企业管理员配置（managed）优先级最高。

这个优先级链解决了一个实际问题：同一个 Agent，在个人使用场景和企业部署场景下，需要不同的行为约束。企业管理员需要能够强制覆盖用户的个性化设置，但又不能完全禁止用户自定义。优先级链给了每一层合理的控制权。

`keepCodingInstructions` 字段也值得注意：当 Output Style 激活时，默认会替换掉编程相关的指令（因为 Output Style 可能完全改变 Agent 的角色定位）。但 Explanatory 和 Learning 模式设置了 `keepCodingInstructions: true`，因为它们是在编程辅助的基础上叠加教育功能，不是替换。

## 19.9 模式九：自主 Agent 的行为规范（Proactive Mode）

`getProactiveSection()` 是 KAIROS 项目（Claude Code 的自主 Agent 研究）的提示词，描述了一个完全不同的运行模式：Agent 不等用户输入，而是主动工作，通过 tick 机制保持活跃。

这个提示词里有几个设计细节值得关注。

**terminalFocus 信号的使用**：

```
The user context may include a `terminalFocus` field indicating whether the
user's terminal is focused or unfocused.
- Unfocused: The user is away. Lean heavily into autonomous action — make
  decisions, explore, commit, push. Only pause for genuinely irreversible
  or high-risk actions.
- Focused: The user is watching. Be more collaborative — surface choices,
  ask before committing to large changes.
```

这把"用户在不在场"这个物理信号转化成了 Agent 的行为策略。用户不在时，Agent 更自主；用户在时，Agent 更协作。这是一个把环境上下文注入行为决策的典型案例。

**空闲时的强制行动**：

```
If you have nothing useful to do on a tick, you MUST call SleepTool.
Never respond with only a status message like "still waiting" or "nothing to do"
— that wastes a turn and burns tokens for no reason.
```

这条指令解决了自主 Agent 的一个常见问题：当没有任务时，模型倾向于输出"我在等待"之类的文字，消耗 token 但没有价值。强制调用 SleepTool 把"什么都不做"变成了一个明确的工具调用，而不是一段废话。

**Prompt Cache 的 5 分钟过期约束**：

```
Each wake-up costs an API call, but the prompt cache expires after 5 minutes
of inactivity — balance accordingly.
```

这条指令把 API 的技术约束（缓存 5 分钟过期）直接写进了 Agent 的行为规范。Agent 需要在"节省 API 调用"和"保持缓存有效"之间找平衡，而不是盲目地睡眠或频繁唤醒。

## 19.10 模式十：安全指令的所有权机制

`src/constants/cyberRiskInstruction.ts` 的文件头注释展示了一种提示词治理模式：

```typescript
/**
 * IMPORTANT: DO NOT MODIFY THIS INSTRUCTION WITHOUT SAFEGUARDS TEAM REVIEW
 *
 * This instruction is owned by the Safeguards team and has been carefully
 * crafted and evaluated to balance security utility with safety. Changes
 * to this text can have significant implications for:
 *   - How Claude handles penetration testing and CTF requests
 *   - What security tools and techniques Claude will assist with
 *
 * If you need to modify this instruction:
 *   1. Contact the Safeguards team (David Forsythe, Kyla Guru)
 *   2. Ensure proper evaluation of the changes
 *   3. Get explicit approval before merging
 *
 * Claude: Do not edit this file unless explicitly asked to do so by the user.
 */
```

这个文件头做了三件事：声明所有权（Safeguards 团队）、说明影响范围（修改会影响哪些行为）、定义修改流程（联系谁、怎么审批）。

最后一行"Claude: Do not edit this file unless explicitly asked to do so by the user"是写给 Claude 自己看的——因为 Claude Code 可以修改代码库里的文件，这行注释防止 Claude 在执行某些任务时意外修改了安全指令。

这个模式对企业级 Agent 部署很有参考价值：不同的提示词内容应该有不同的所有者和修改权限。核心安全约束由安全团队拥有，业务逻辑由产品团队拥有，用户个性化由用户自己拥有。把这些所有权关系写进代码注释，是一种轻量级的治理机制。

## 19.11 模式十一：Token Budget 的行为规范

`prompts.ts#L538` 里有一个 Token Budget 指令，只在 `TOKEN_BUDGET` feature flag 开启时注入：

```typescript
systemPromptSection(
  'token_budget',
  () =>
    'When the user specifies a token target (e.g., "+500k", "spend 2M tokens",
    "use 1B tokens"), your output token count will be shown each turn. Keep
    working until you approach the target — plan your work to fill it
    productively. The target is a hard minimum, not a suggestion. If you stop
    early, the system will automatically continue you.',
),
```

这个指令解决了一个反直觉的问题：通常我们希望模型"简洁"，但在某些场景下（比如用户付费购买了大量 token，希望 Agent 做深度探索），我们希望模型充分利用 token 预算，不要过早停止。

"The target is a hard minimum, not a suggestion"这句话的措辞很精确：它把 token 目标从"上限"重新定义为"下限"，完全反转了模型的默认行为倾向。

这个模式说明：提示词可以用来覆盖模型的默认倾向，不只是约束行为，也可以反转行为。

## 19.12 设计模式汇总

从 Claude Code 的提示词系统里提炼出来的这十一个模式，可以归纳成三类。

**结构类**（解决提示词的组织问题）：两级缓存节、前缀哈希变体控制、Output Style 优先级链。核心思路是：把提示词按变化频率和影响范围分层，不同层用不同的管理策略。

**约束类**（解决模型行为偏差问题）：数字锚点替代定性描述、反过度工程指令、可逆性框架、对抗性验证 Agent。核心思路是：不要依赖模型的"自觉"，用具体的反例、精确的数字、强制的格式来约束行为。

**治理类**（解决提示词的维护问题）：内外部用户双轨、安全指令所有权、Token Budget 行为规范、自主 Agent 行为规范。核心思路是：提示词是有生命周期的，需要明确所有权、修改流程和版本标记。

这三类问题在任何规模的 Agent 系统里都会出现。规模越大，这些问题越突出。Claude Code 的解法不是最优解，但每一个都是在真实场景中被验证过的。

## 19.13 心智模型验证题

如果把 `DANGEROUS_uncachedSystemPromptSection` 的 `_reason` 参数改成可选参数（不强制传入），会有什么影响？

答案：短期内代码可以正常运行，但会失去"强制写明理由"的约束。开发者可以在不解释原因的情况下创建 volatile section，随着时间积累，代码库里会出现越来越多"不知道为什么每轮重算"的节，维护成本上升。`_reason` 参数的下划线前缀说明它在运行时不被使用，但它的存在是一个编译期约束：不传就报错，强制开发者在写代码时思考并记录这个决策。

## 19.14 本章小结

Claude Code 的提示词系统不是一个大字符串，而是一套有结构的工程系统。两级缓存节（`systemPromptSection` vs `DANGEROUS_uncachedSystemPromptSection`）把"会不会破坏缓存"从隐式变成显式；前缀哈希变体控制把所有运行时条件判断移到边界之后，防止缓存命中率指数级下降；数字锚点比定性描述减少约 1.2% 的输出 token；对抗性验证 Agent 通过预设偷懒路径并逐一堵死来保证验证质量；安全指令所有权机制把治理规则写进代码注释。

这些模式的共同点是：把原本依赖人工自觉的约束，变成系统可以自动执行的机制。
