# 第19章 提示词工程的设计模式

提示词工程最常被忽视的代价是：成本是不可见的，行为偏差是滞后的，结构腐化是渐进的。

这一章从 `src/constants/prompts.ts`（914 行）和相关文件里直接提炼出 11 个可复用的设计模式，每一个都有源码支撑，每一个都解决了一个用"提示词是字符串"这个假设无法解决的问题。

## 19.1 模式一：两级缓存节

`src/constants/systemPromptSections.ts` 定义了两个函数：

```typescript
export function systemPromptSection(
  name: string,
  compute: ComputeFn,
): SystemPromptSection {
  return { name, compute, cacheBreak: false }
}

export function DANGEROUS_uncachedSystemPromptSection(
  name: string,
  compute: ComputeFn,
  _reason: string,
): SystemPromptSection {
  return { name, compute, cacheBreak: true }
}
```

`DANGEROUS_` 前缀是一个工程约定：任何使用这个函数的地方，都必须在 `_reason` 参数里写明"为什么这个节不得不每轮重算"。`_reason` 以下划线开头，运行时不被使用——它是编译期约束，强制开发者在写代码时记录决策。

当前代码库里只有一处使用了 `DANGEROUS_` 函数——MCP 指令节（`prompts.ts#L508-L520`）。原因：MCP 服务器可能在会话中途连接或断开，这一节必须每轮重新评估。

把 `_reason` 参数改成可选不影响运行，但会让"为什么每轮重算"变成代码库里的隐藏成本——随着时间积累，会出现越来越多"不知道为什么 volatile"的节。

## 19.2 模式二：前缀哈希变体控制

`prompts.ts#L343` 的注释解释了一个反直觉的约束：

```typescript
/**
 * Session-variant guidance that would fragment the cacheScope:'global'
 * prefix if placed before SYSTEM_PROMPT_DYNAMIC_BOUNDARY. Each conditional
 * here is a runtime bit that would otherwise multiply the Blake2b prefix
 * hash variants (2^N). See PR #24490, #24171 for the same bug class.
 */
function getSessionSpecificGuidanceSection(...): string | null {
```

如果把任何运行时条件判断（"是否有 AskUserQuestion 工具"、"是否是非交互会话"）放在边界标记之前，每种条件组合都会产生一个不同的前缀哈希，缓存命中率指数级下降（N 个条件 = 2^N 种变体）。

PR #24490 和 #24171 是两次真实踩坑记录。解决方案是把所有运行时条件判断移到边界标记之后，边界之前的内容对所有会话完全一致，只有一个哈希值。

这个模式的业务场景版本：把"VIP 用户和普通用户的差异化内容"放在提示词前缀，等于为每种用户类型维护一套独立缓存，缓存命中率按用户类型分裂。正确做法是把所有用户相关的差异化内容放在提示词后段。

## 19.3 模式三：数字锚点替代定性描述

`prompts.ts#L527` 里有一段只对内部用户开放的指令：

```typescript
// Numeric length anchors — research shows ~1.2% output token reduction vs
// qualitative "be concise". Ant-only to measure quality impact first.
systemPromptSection(
  'numeric_length_anchors',
  () =>
    'Length limits: keep text between tool calls to ≤25 words. Keep final responses to ≤100 words unless the task requires more detail.',
),
```

注释里有实测数据：数字锚点（"≤25 words"）比定性描述（"be concise"）减少约 1.2% 的输出 token。这是 A/B 测试结果，不是直觉。

模型对定量约束的遵从性高于定性描述："`be concise`"是模糊的，每个模型对"简洁"的理解不同；"`≤25 words`"是精确的，模型可以计数。类似的应用：用"每个函数不超过 50 行"替代"不要生成太长的代码"，用"只在逻辑不自明时添加注释"替代"适当添加注释"。

这条模式只对内部用户开放，目的是先测量对输出质量的影响，说明 Anthropic 在用 A/B 测试管理提示词变更，而不是直接全量上线。

## 19.4 模式四：内外部用户双轨输出哲学

`getOutputEfficiencySection()` 对内外部用户返回两套不同的指令。

外部用户版本的核心是"极简"：直接给答案，跳过推理，不要铺垫，不要过渡。

内部用户版本的核心是"写给人看的散文"：倒金字塔结构（新闻写作技法）、语义回溯（semantic backtracking）、避免过度使用破折号和符号。

差异不是风格偏好，是服务不同目的的功能性差异：外部用户在完成具体任务，需要快速得到答案；内部用户在评估模型行为，需要能读懂完整推理过程的输出。

这个模式的迁移版本：不要假设所有用户需要同样的输出风格。技术用户和非技术用户、快速操作场景和深度分析场景，应该走不同的输出指令路径。

## 19.5 模式五：反过度工程指令

`getSimpleDoingTasksSection()` 里有一组专门对抗模型"过度工程"倾向的指令：

```
Don't add features, refactor code, or make "improvements" beyond what was asked.
A bug fix doesn't need surrounding code cleaned up.

Don't add error handling, fallbacks, or validation for scenarios that can't happen.
Trust internal code and framework guarantees. Only validate at system boundaries.

Don't create helpers, utilities, or abstractions for one-time operations.
Three similar lines of code is better than a premature abstraction.
```

这三段指令针对的是三种过度工程模式：功能蔓延、防御性编程、过早抽象。

注意写法：不是"不要过度工程"这种定性描述，而是给出具体的反例（"A bug fix doesn't need surrounding code cleaned up"）。具体的反例比抽象的原则更有效，因为模型可以直接匹配场景，不需要自己解释"什么叫过度工程"。

同一节里还有针对 Capybara v8 过度注释问题的专项指令（`@[MODEL LAUNCH]` 标记，用于标识随模型版本发布临时注入、后续需要清理的补丁指令），这是版本补丁管理在提示词工程里的具体应用。

## 19.6 模式六：可逆性框架

`getActionsSection()` 把所有操作按两个维度分类：可逆性（reversible vs hard-to-reverse）和影响范围（local vs shared systems）。

```
Generally you can freely take local, reversible actions like editing files or
running tests. But for actions that are hard to reverse, affect shared systems
beyond your local environment, or could otherwise be risky or destructive,
check with the user before proceeding.
```

然后给出了具体的高风险操作清单，包括：删除文件/分支、force-push、发送消息（Slack、email、GitHub）、向第三方服务推送内容。

最后一条值得单独关注：

```
Uploading content to third-party web tools (diagram renderers, pastebins, gists)
publishes it - consider whether it could be sensitive before sending, since it
may be cached or indexed even if later deleted.
```

把内容发送给第三方渲染服务（比如 Mermaid 在线渲染）也是一种"发布"行为，即使用户没有意识到。这条指令覆盖了一个容易被忽视的场景——模型帮用户生成了一个 Mermaid 图，顺手渲染了一下，结果把内部架构信息发给了第三方服务。

这个框架的设计思路：不让模型自己判断"这个操作危不危险"（模型的判断标准不一致），而是给出结构化的分类框架，让模型套用框架判断。

## 19.7 模式七：对抗性验证 Agent 的提示词设计

`src/tools/AgentTool/built-in/verificationAgent.ts` 里的验证 Agent 提示词是对抗性验证规范的完整实现。

它的核心设计思路：预设验证者会偷懒，然后逐一堵死偷懒的路。

提示词开头直接点名两种失败模式：

```
You have two documented failure patterns.
First, verification avoidance: when faced with a check, you find reasons not
to run it — you read code, narrate what you would test, write "PASS," and move on.
Second, being seduced by the first 80%: you see a polished UI or a passing test
suite and feel inclined to pass it, not noticing half the buttons do nothing.
```

然后给出"认识自己的合理化借口"清单：

```
- "The code looks correct based on my reading" — reading is not verification. Run it.
- "The implementer's tests already pass" — the implementer is an LLM. Verify independently.
- "This is probably fine" — probably is not verified. Run it.
```

最后用强制格式约束防止"假验证"：

```
Every check MUST follow this structure. A check without a Command run block
is not a PASS — it's a skip.

### Check: [what you're verifying]
**Command run:** [exact command you executed]
**Output observed:** [actual terminal output — copy-paste, not paraphrased]
**Result: PASS** (or FAIL)
```

这个设计模式可以推广到任何需要"防止 AI 走捷径"的场景：先明确列出 AI 会走的捷径，然后用格式约束强制它不能走。光靠"请认真验证"不够——模型不知道自己在走捷径，需要显式地被告知"'代码看起来是对的'不算验证"。

## 19.8 模式八：Output Style 优先级链

`src/constants/outputStyles.ts` 实现了一套输出风格的优先级覆盖机制：

```typescript
// 优先级从低到高：built-in < plugin < user < project < managed
const styleGroups = [pluginStyles, userStyles, projectStyles, managedStyles]

for (const styles of styleGroups) {
  for (const style of styles) {
    allStyles[style.name] = { ... }  // 高优先级覆盖低优先级
  }
}
```

内置风格有两种：Explanatory（解释型，在写代码前后提供教育性说明）和 Learning（学习型，在关键决策点暂停，让用户自己写代码）。企业管理员配置（managed）优先级最高，可以强制覆盖用户的个性化设置，但不能完全禁止用户自定义低于 managed 层的内容。

`keepCodingInstructions` 字段说明了一个设计细节：当 Output Style 激活时，默认会替换掉编程相关的指令（因为 Output Style 可能完全改变 Agent 的角色定位）。但 Explanatory 和 Learning 模式设置了 `keepCodingInstructions: true`——它们是在编程辅助的基础上叠加教育功能，不是替换。激活 style 时是替换还是叠加，是两种不同的语义，需要在 schema 层面显式区分。

## 19.9 模式九：自主 Agent 的行为规范

`getProactiveSection()` [verified: prompts.ts#L860] 是 KAIROS 项目（Claude Code 的自主 Agent 研究）的提示词，描述了一个完全不同的运行模式：Agent 不等用户输入，通过 tick 机制保持活跃。

`terminalFocus` 信号的使用是其中最值得关注的设计细节：

```
- Unfocused: The user is away. Lean heavily into autonomous action — make
  decisions, explore, commit, push. Only pause for genuinely irreversible
  or high-risk actions.
- Focused: The user is watching. Be more collaborative — surface choices,
  ask before committing to large changes.
```

把"用户在不在场"这个物理信号转化成行为策略，是把环境上下文注入行为决策的典型案例。用户不在时，Agent 更自主；用户在时，Agent 更协作。这不是两种不同的 Agent，是同一个 Agent 根据上下文调整自主程度。

还有一条针对空闲时的强制行动：

```
If you have nothing useful to do on a tick, you MUST call SleepTool.
Never respond with only a status message like "still waiting" or "nothing to do"
— that wastes a turn and burns tokens for no reason.
```

自主 Agent 的常见问题是"没有任务时输出废话"——模型倾向于输出"我在等待"之类的文字，消耗 token 但没有价值。强制调用 SleepTool 把"什么都不做"变成一个明确的工具调用，而不是一段文字，可以在日志里被追踪和统计。

提示词里还嵌入了一条 API 技术约束：

```
Each wake-up costs an API call, but the prompt cache expires after 5 minutes
of inactivity — balance accordingly.
```

把"缓存 5 分钟过期"这个 API 限制直接写进 Agent 的行为规范，让 Agent 在调度决策时考虑这个约束，而不是把 API 成本管理完全交给基础设施层。

## 19.10 模式十：安全指令的所有权机制

`src/constants/cyberRiskInstruction.ts` 的文件头注释展示了一种提示词治理模式：

```typescript
/**
 * IMPORTANT: DO NOT MODIFY THIS INSTRUCTION WITHOUT SAFEGUARDS TEAM REVIEW
 *
 * This instruction is owned by the Safeguards team and has been carefully
 * crafted and evaluated...
 * If you need to modify this instruction:
 *   1. Contact the Safeguards team (David Forsythe, Kyla Guru)
 *   2. Ensure proper evaluation of the changes
 *   3. Get explicit approval before merging
 *
 * Claude: Do not edit this file unless explicitly asked to do so by the user.
 */
```

这个文件头做了三件事：声明所有权（Safeguards 团队）、说明影响范围（修改会影响哪些行为）、定义修改流程。

最后一行写给 Claude 自己——因为 Claude Code 可以修改代码库里的文件，这行注释防止 Claude 在执行某些任务时意外修改了安全指令。这是 Agent 自修改能力和安全约束之间的张力在实现层的解决方式：不是从技术上阻止 Claude 修改文件，而是在文件里声明意图，依靠 Claude 遵守这个约束。

对企业级 Agent 部署的迁移价值：不同的提示词内容应该有不同的所有者和修改权限。核心安全约束由安全团队拥有，业务逻辑由产品团队拥有，用户个性化由用户自己拥有。把所有权关系写进代码注释，是一种轻量级的治理机制，不需要引入复杂的权限管理系统。

## 19.11 模式十一：Token Budget 行为规范

`prompts.ts#L538` 里有一个 Token Budget 指令，只在 `TOKEN_BUDGET` feature flag 开启时注入：

```typescript
systemPromptSection(
  'token_budget',
  () =>
    'When the user specifies a token target (e.g., "+500k", "spend 2M tokens",
    "use 1B tokens"), your output token count will be shown each turn. Keep
    working until you approach the target — plan your work to fill it
    productively. The target is a hard minimum, not a suggestion. If you stop
    early, the system will automatically continue you.',
),
```

"The target is a hard minimum, not a suggestion"这句话的措辞是精确的：它把 token 目标从上限重新定义为下限，完全反转了模型的默认行为倾向。通常我们希望模型简洁，但在某些场景下（用户付费购买了大量 token，希望 Agent 做深度探索），我们希望模型充分利用 token 预算，不要过早停止。

这个模式说明了一件事：提示词可以覆盖模型的默认倾向，不只是约束行为，也可以反转行为。"模型的默认倾向"不是固定的，它是可以被提示词改写的——只要措辞足够精确。

## 19.12 三类模式的归纳

这 11 个模式可以归纳成三类。

**结构类**（解决提示词的组织问题）：两级缓存节、前缀哈希变体控制、Output Style 优先级链。核心思路：按变化频率和影响范围分层管理，不同层用不同的管理策略，高频变化的部分不污染低频稳定的缓存。

**约束类**（解决模型行为偏差问题）：数字锚点替代定性描述、反过度工程指令、可逆性框架、对抗性验证 Agent 提示词。核心思路：不依赖模型的"自觉"，用具体的反例、精确的数字、强制的格式来约束行为。定量约束优于定性描述，具体反例优于抽象原则。

**治理类**（解决提示词的维护问题）：内外部用户双轨、安全指令所有权、Token Budget 行为规范、自主 Agent 行为规范。核心思路：提示词是有生命周期的，需要明确所有权、修改流程和版本标记。不同的内容由不同的人拥有，临时补丁必须打标记，API 技术约束要显式地注入行为规范。

这三类问题在任何规模的 Agent 系统里都会出现。规模越大，这些问题越突出——团队越大，"提示词就是字符串"这个假设导致的问题越多。

## 19.13 可迁移原则

把原本依赖人工自觉的约束，变成系统可以自动执行的机制——这是这 11 个模式的共同思路。

`DANGEROUS_` 前缀是一个低成本的工程约定，不需要专用工具，只需要命名规范。数字锚点只需要把定性词替换成数字。对抗性验证 Agent 的格式约束只需要在提示词里定义必填格式。安全指令所有权只需要在文件头加注释。

代价都很低。触发条件是：团队里有超过一个人修改提示词，或者系统在真实场景中运行了超过几个月。

## 19.14 本章不覆盖项

这 11 个模式都从 Claude Code 的当前代码库里提炼，不覆盖模型训练层面的提示词优化（RLHF、Constitutional AI 等）。跨版本提示词测试框架不在本书范围内。多模型系统的提示词适配（同一系统调用不同厂商的模型时如何管理提示词差异）不在本书范围内。
