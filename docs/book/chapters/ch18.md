# 第18章 恢复是玩具和真实系统的分水岭

一个 Agent 能在演示里跑通，不代表它能在真实工作中稳定运行。两者之间最关键的差距，是恢复能力。

## 18.1 为什么恢复是分水岭

演示场景的特点是：任务短、不中断、失败了重来。真实场景的特点是：任务长、随时可能中断、失败了需要从中间状态继续。

没有恢复机制的 Agent，在真实场景里的表现是：执行到一半进程崩溃，所有进度丢失，需要从头开始。用户等了十分钟，什么都没有。这种体验在演示里不会出现，但在真实使用中会频繁发生。

Claude Code 的恢复机制是一个分层流水线，不是单一的"保存和加载"。每一层解决不同类型的问题。

## 18.2 第一层：事实恢复

`src/utils/conversationRecovery.ts` 的 `loadConversationForResume()` 从持久化存储里加载历史会话。它支持三种加载方式：最近会话、指定 sessionId、指定 jsonl 文件。

加载后，它会沿着消息链找到主链，恢复消息列表、快照、元数据。这一层解决的是"进程重启后能找回历史"的问题。

## 18.3 第二层：语义修复

加载历史消息后，`deserializeMessagesWithInterruptDetection()` 做语义层面的修复。这一步处理的是"历史消息存在，但结构不合法"的情况。

具体包括：迁移旧版本的 attachment 类型（`migrateLegacyAttachmentTypes()`）、清理非法的 permissionMode、过滤未配对的工具调用和孤立的 thinking 块、检测中断 turn 并补充 sentinel 消息。

最后一步很关键：如果上次执行在 assistant 消息中途被中断，消息链的结构对 API 来说是不合法的。`deserializeMessagesWithInterruptDetection()` 会检测这种情况，补充必要的 sentinel 消息，让恢复后的消息链对 API 合法。

## 18.4 第三层：压缩后连续性

长会话的上下文会超过模型的处理上限。`src/services/compact/compact.ts` 在压缩时不是简单截断，而是按固定协议重建最小上下文：边界消息、摘要、可选的保留段、文件和技能附件、hooks。

`compactMetadata.preservedSegment` 记录保留段的重连信息。这意味着压缩后的上下文有元数据记录从哪里接续，resume 时能找到正确的接续点。

这一层解决的是"上下文太长，但不能丢失关键信息"的问题。压缩是有损的，但损失的是细节，保留的是结构和关键事实。

## 18.5 第四层：长期记忆

`src/services/SessionMemory/sessionMemory.ts` 在会话积累到一定长度后，提取关键信息存入 memory 文件。下次会话开始时，这些信息会作为背景知识注入上下文。

这一层解决的是"跨会话的连续性"问题。用户今天告诉 Agent 的偏好，明天开新会话时 Agent 还记得。

记忆更新用受限子代理执行，只允许访问 memory 文件。这个隔离设计防止了记忆更新操作影响当前执行流程。

## 18.6 主循环里的恢复分支

除了上面四层，`query.ts` 的主循环里还有几个内联的恢复分支：

上下文过长时，先尝试 context collapse（删除不重要的历史），再尝试 reactive compact（触发压缩）。输出被截断时（`max_output_tokens`），注入"继续输出"的 meta 消息重试，有重试次数上限。工具调用结果缺失时，`yieldMissingToolResultBlocks()` 补齐缺失的 `tool_result` 块，防止消息链结构损坏。

这些分支处理的是执行过程中的实时异常，不是事后恢复。

## 18.7 最小恢复实现

对业务 Agent 来说，不需要一次性实现 Claude Code 的全部恢复机制。最小可用的恢复实现只需要两件事：

第一，在开始执行任务时，把任务状态写入文件（任务 ID、描述、当前步骤）。执行完成后更新状态为"完成"。进程崩溃后，下次启动时读取这个文件，知道上次执行到哪里。

第二，在上下文接近上限时，触发一次摘要压缩：让模型把当前对话的关键信息总结成几段文字，用这个摘要替换历史对话，继续执行。

这两件事能覆盖大多数真实场景里的恢复需求。更复杂的恢复机制（语义修复、保留段元数据、长期记忆）可以在系统稳定后逐步添加。

## 18.8 本章小结

恢复机制是玩具 Agent 和真实系统的分水岭。Claude Code 的恢复是四层流水线：事实恢复（加载历史）、语义修复（修复不合法结构）、压缩后连续性（保留关键信息）、长期记忆（跨会话连续性）。

对业务 Agent 来说，最小恢复实现是：任务状态落盘 + 上下文超限时摘要压缩。这两件事能覆盖大多数真实场景，其余机制可以逐步补充。
