# 第12章 该学什么、先别急着抄什么

**本章核心主张**：Claude Code 的架构是在特定约束下做出的一组设计决策。直接抄，会把约束一起抄进来。值得学的是约束背后的原则，不是约束本身的产物。

---

架构是在特定约束下做出的设计决策集合。Claude Code 的约束包括：本地运行、单机、TypeScript 生态、Claude API 消息格式、特定的上下文窗口大小。

如果你的系统约束不同，直接抄架构会把这些约束一起带进来，而它们在你的场景下可能是错误的。

## 12.1 文件系统锁：不是死锁，是串行化

`src/utils/tasks.ts` 用文件系统存储任务状态，用 `.lock` 文件做并发控制。这在简单描述下听起来很脆弱——实际上是经过设计的。锁配置在 L94–L108：

```typescript
// Budget sized for ~10+ concurrent swarm agents: each critical section does
// readdir + N×readFile + writeFile (~50-100ms on slow disks), so the last
// caller in a 10-way race needs ~900ms. retries=30 gives ~2.6s total wait.
const LOCK_OPTIONS = {
  retries: { retries: 30, minTimeout: 5, maxTimeout: 100 },
}
```

注释里埋着两个重要信息。第一，锁的容量是按"10+ 个并发 agent"预算的，不是拍脑袋的数字。第二，注释里还写了一句："The sync lockSync API blocked the event loop"——说明他们最初用的是同步锁，后来因为阻塞了事件循环才改成异步轮询。这是典型的"先能用、后优化"演进，不是一开始就设计对的。

另一个细节：`updateTask()` 在加锁前先做一次无锁读（L382–L384），如果任务不存在就直接返回 null。`proper-lockfile` 对不存在的文件会抛异常，所以这个无锁读不只是性能优化，也是防御性的前置检查。

**隐含假设**：这个方案假设系统是单机运行的，任务队列不需要跨机共享。10 个并发 agent 争同一把锁时，最后一个需要等约 900ms——注释里明确说了这是设计范围内的代价。如果你的系统需要 50 个并发 agent，这里的等待会变成 4-5 秒，接近超时阈值。这不是"抄了会出 bug"的问题，是"抄了会在 scale 时出性能问题"的问题。

**替代方案**：Redis 的 `SETNX`，或 PostgreSQL 的 `SELECT FOR UPDATE SKIP LOCKED`，可以做到真正的并发任务认领。代价是引入外部依赖，部署复杂度提高。Claude Code 是本地运行，外部数据库不在选项里。

## 12.2 压缩阈值不是拍脑袋的数字

ch11 提到主动压缩，这里看具体数字是怎么算的。`src/services/compact/autoCompact.ts` L28–L91：

```typescript
// Reserve this many tokens for output during compaction
// Based on p99.99 of compact summary output being 17,387 tokens.
const MAX_OUTPUT_TOKENS_FOR_SUMMARY = 20_000

export const AUTOCOMPACT_BUFFER_TOKENS = 13_000      // 触发缓冲
export const WARNING_THRESHOLD_BUFFER_TOKENS = 20_000  // 警告缓冲
export const AUTOCOMPACT_MANUAL_COMPACT_BUFFER_TOKENS = 3_000  // 手动压缩缓冲

export function getAutoCompactThreshold(model: string): number {
  const effectiveContextWindow = getEffectiveContextWindowSize(model)
  return effectiveContextWindow - AUTOCOMPACT_BUFFER_TOKENS
}
```

阈值不是固定数字，是基于模型上下文窗口动态计算的。`20_000` 来自 p99.99 的 compact summary 输出为 17,387 tokens——这不是估算，是从实际数据推出来的。`13_000` 的触发缓冲给了压缩操作本身的运行空间。

以 claude-3-5-sonnet（200k 上下文窗口，8k max output）为例：`effectiveContextWindow` = 200,000 - 8,000 = 192,000，`autoCompactThreshold` = 192,000 - 13,000 = **179,000 tokens**，约 89.5% 使用率触发。

还有一个环境变量覆盖机制：`CLAUDE_AUTOCOMPACT_PCT_OVERRIDE` 可以按百分比覆盖阈值，用于测试。这个 escape hatch 的存在说明：即使数字是经过计算的，设计者也知道它在某些测试场景下需要被覆盖。

**如果你直接把 `13_000` 和 `20_000` 这两个数字抄到自己系统里**：如果你用的模型上下文窗口是 8k（很多本地部署模型），这两个 buffer 加起来已经超过上下文窗口的40%，触发阈值会变成负数，逻辑直接失效。

## 12.3 assistant sentinel：Claude API 的格式约束

`src/utils/conversationRecovery.ts` L226 附近的 assistant sentinel 补丁，是为了应对一个 Claude API 的具体约束：**对话必须以 assistant 消息结尾才能发起新请求**（user/assistant 交替规则）。

恢复后对话可能以 user 消息结尾（比如用户发了消息但进程在 API 响应前被杀）。这时如果直接发给 API，会报格式错误。补丁在 L229–L248 处理：

```typescript
// Append a synthetic assistant sentinel after the last user message so
// the conversation is API-valid if no resume action is taken.
if (lastRelevantIdx !== -1 && filteredMessages[lastRelevantIdx]!.type === 'user') {
  filteredMessages.splice(
    lastRelevantIdx + 1, 0,
    createAssistantMessage({ content: NO_RESPONSE_REQUESTED }),  // = 'No response requested.'
  )
}
```

注意 `splice(idx, 2)` 的设计：sentinel 被插在 user 消息的正后方（不是末尾），这样 `removeInterruptedMessage` 可以用 `splice(idx, 2)` 同时删除 user 消息和 sentinel，保持操作的原子性。如果 sentinel 插在末尾，删除时需要找到配对的 user 消息，逻辑复杂很多。

**如果你用的是其他模型 API**：这个补丁完全不需要。不同 API 对对话格式的约束不同，有的允许以 user 消息结尾，有的有不同的格式要求。把这段代码抄过去，可能既不起作用，又引入了对 Claude API 格式假设的依赖。

## 12.4 工具列表和系统提示缓存的隐藏耦合

`src/tools.ts` L191 有一条注释："MUST stay in sync with Statsig config"。

这条注释的背景：`getAllBaseTools()` 返回的工具列表会被序列化进 system prompt（工具定义部分）。Anthropic 在 Statsig 上维护了一个动态配置 `claude_code_global_system_caching`，用于跨用户共享 system prompt 的 prompt cache。如果工具列表和 Statsig 配置不同步，生成的 system prompt 内容不同，cache key 不匹配，prompt cache 失效，每次请求都要重新处理完整 system prompt。

这是一个"看起来是工具配置，实际上影响成本"的隐藏耦合。如果你把工具列表抄过来但不维护这个 Statsig 配置，你不会报错，也不会崩溃，只是 prompt cache 效率会下降，API 调用延迟增加，成本升高——而且你很可能不知道为什么。

## 12.5 值得学的设计原则

上面四节说的都是"不建议直接抄的"。这里是值得抽象出来的原则：

**分层错误处理**：不同类型的错误对应不同的处理策略，这个原则和具体实现无关。用户主动中止、外部信号中断、资源超限，处理方式本来就不同，混在一起处理是设计问题。

**集中可变状态**：`src/query.ts#L282` 的 `State` 对象把所有跨迭代的可变状态集中在一处。好处是可观察、可测试、行为可预测。代价是需要刻意约束对象的边界，否则会膨胀成垃圾桶——注释里有告诫，但告诫不等于约束。

**最小权限副作用**：把副作用操作限制在最小范围内，防止意外蔓延。`createMemoryFileCanUseTool()` 用严格相等而不是前缀匹配，不是专门防路径遍历，是严格限制副作用范围的自然结果。

**主动管理资源上限**：在接近上限之前主动处理，而不是等到超限后被动响应。被动响应的失败路径是连锁的（见 11.2 节），主动管理的失败路径是可控的。

**正确性优先于性能**：`src/utils/conversationRecovery.ts#L164` 在恢复时做了多项检查，确保恢复后的状态对 API 合法。这些检查有性能开销，但如果恢复后的状态不合法，后续所有操作都会失败，性能优化没有意义。

## 12.7 心智模型验证题

你把 Claude Code 的文件系统任务队列抄到一个需要支持 50 个并发 agent 的系统里。在没有明显 bug 的情况下，你会观察到什么现象？如何确认问题来源？

答案：不会有 bug，但会观察到任务认领延迟增加，吞吐量达到瓶颈。50 个 agent 争一把锁，每个临界区约 100ms，最后一个 agent 需要等约 5 秒（超过了 `proper-lockfile` 的 2.6 秒超时配置），开始出现锁超时错误。确认来源的方法：监控任务认领时间的 p99，对比 agent 数量变化时的时间分布。解决方案是换成支持原子操作的后端（Redis SETNX、PostgreSQL SELECT FOR UPDATE SKIP LOCKED）。

## 12.8 本章小结

从 Claude Code 架构里，不建议直接抄的：

- 文件系统任务队列（[`tasks.ts#L94`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/utils/tasks.ts#L94)）：假设单机运行，10 个并发是设计上限
- 压缩阈值常量（`autoCompact.ts`）：基于 Claude 模型和实际数据统计，换模型需要重新校准
- assistant sentinel 逻辑（[`conversationRecovery.ts#L226`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/utils/conversationRecovery.ts#L226)）：针对 Claude API 对话格式约束，换 API 则不适用
- 工具列表和 Statsig 的同步约束（`tools.ts#L191`）：隐藏在注释里，不抄注释不知道

值得学的原则：分层错误处理、集中可变状态、最小权限副作用、主动管理资源上限、正确性优先于性能。这些原则和具体实现无关，适用于任何需要在真实场景稳定运行的 Agent 系统。

抄代码容易，抄原则难。原则需要理解背后的约束，才能在不同场景下正确应用。
