# 第12章 该学什么、先别急着抄什么

**核心主张**：Claude Code 的每一个具体实现都绑定了它的运行约束。直接复制代码，会把那些约束一起带进你的场景，而它们在你的场景下可能根本不成立。

---

菜谱是在特定灶台、特定食材、特定气候下试出来的。同一道菜换到高原上做，水沸腾温度不同，时间就得调。你要学的不是"大火 20 分钟"这个数字，是"用沸腾判断熟度"这条原理。原理在高原上仍然成立，数字不一定。

Claude Code 的约束是：本地单机运行、TypeScript 生态、Claude API 的消息格式、特定大小的上下文窗口。你的系统约束可能有一两条完全不同。从"原理"层抽象，换场景依然有效；从"解法"层复制，约束跟着来，你就得替它背锅。

## 12.1 文件锁方案对高并发不是有 bug，是约束边界到了

`src/utils/tasks.ts` 用文件系统存储任务状态，用 `.lock` 文件做并发控制。锁配置在 L94–L108：

```typescript
// Budget sized for ~10+ concurrent swarm agents: each critical section does
// readdir + N×readFile + writeFile (~50-100ms on slow disks), so the last
// caller in a 10-way race needs ~900ms. retries=30 gives ~2.6s total wait.
const LOCK_OPTIONS = {
  retries: { retries: 30, minTimeout: 5, maxTimeout: 100 },
}
```

注释给出了两个事实：锁的容量是按"10+ 个并发 agent"预算的，不是估算；`retries: 30` 对应约 2.6 秒总等待时间。同一段注释里还有一句话没有出现在上面这个代码块里：`The sync lockSync API blocked the event loop`，说明他们最初用同步锁，后来因为阻塞事件循环改成了异步轮询。这是演进的结果，不是一开始就设计对了。

`updateTask()` 在加锁前先做一次无锁读（L382–L384）：如果任务不存在就直接返回 null。`proper-lockfile` 对不存在的文件会抛异常，这个无锁读是防御性的前置检查，同时也避免了不必要的锁竞争。

把这套方案移植到 50 个并发 agent 的场景里，不会报错，也不会崩溃。你会观察到任务认领的 p99 延迟持续升高：50 个 agent 争一把锁，每个临界区约 100ms，最后一个 agent 需要等约 5 秒，超过了 2.6 秒的超时配置，开始出现锁超时错误，吞吐量触顶。

替代方案是 Redis 的 `SETNX` 或 PostgreSQL 的 `SELECT FOR UPDATE SKIP LOCKED`，两者都支持真正的原子任务认领。代价是引入外部依赖，部署复杂度提高。Claude Code 是本地运行，外部数据库根本不在选项里，所以这套文件锁方案在其约束下是合理的。你的约束不同，合理的解法也不同。

## 12.2 压缩阈值绑定了模型的上下文窗口，换模型就得重算

`src/services/compact/autoCompact.ts` L28–L91：

```typescript
// Reserve this many tokens for output during compaction
// Based on p99.99 of compact summary output being 17,387 tokens.
const MAX_OUTPUT_TOKENS_FOR_SUMMARY = 20_000

export const AUTOCOMPACT_BUFFER_TOKENS = 13_000
export const WARNING_THRESHOLD_BUFFER_TOKENS = 20_000
export const AUTOCOMPACT_MANUAL_COMPACT_BUFFER_TOKENS = 3_000

export function getAutoCompactThreshold(model: string): number {
  const effectiveContextWindow = getEffectiveContextWindowSize(model)
  return effectiveContextWindow - AUTOCOMPACT_BUFFER_TOKENS
}
```

`20_000` 来自 p99.99 的压缩摘要输出为 17,387 tokens，这不是估算，是从实际数据推导出来的。阈值本身是动态计算的：以 claude-3-5-sonnet（200k 上下文窗口）为例，`effectiveContextWindow` = 192,000，`autoCompactThreshold` = 179,000 tokens，约 89.5% 使用率触发。

环境变量 `CLAUDE_AUTOCOMPACT_PCT_OVERRIDE` 可以按百分比覆盖阈值，用于测试。这个 escape hatch（允许在运行时绕过默认行为的覆盖机制）的存在说明：即使数字是从数据推导的，设计者也知道它在某些场景下需要被覆盖。

把 `13_000` 和 `20_000` 这两个数字直接带进一个上下文窗口为 8k 的本地部署模型，这两个 buffer 加起来已超过上下文窗口的 40%，触发阈值会变成负数，整个压缩逻辑直接失效，且不报错。问题不在数字本身，在于这两个数字是针对 200k 窗口模型校准的，它们是结论，不是参数。

## 12.3 Assistant sentinel 是一个 API 格式补丁，换 API 就失效

`src/utils/conversationRecovery.ts` L226 附近的 assistant sentinel 处理，是为了应对 Claude API 的一个格式约束：对话必须以 assistant 消息结尾才能发起新请求（user/assistant 交替规则）。

进程被杀时，对话可能以 user 消息结尾——用户发了消息，但进程在 API 响应前退出了。直接用这个状态发起请求会报格式错误。L229–L248 的处理方式：

```typescript
// Append a synthetic assistant sentinel after the last user message so
// the conversation is API-valid if no resume action is taken.
if (lastRelevantIdx !== -1 && filteredMessages[lastRelevantIdx]!.type === 'user') {
  filteredMessages.splice(
    lastRelevantIdx + 1, 0,
    createAssistantMessage({ content: NO_RESPONSE_REQUESTED }),
  )
}
```

sentinel（哨兵消息，用于占位的合成 assistant 消息）被插在 user 消息的正后方，不是末尾。`removeInterruptedMessage` 用 `splice(idx, 2)` 同时删除 user 消息和 sentinel，操作是原子的。如果 sentinel 插在末尾，删除时需要找到配对的 user 消息，逻辑复杂三倍。

其他模型的 API 对对话格式的约束各不相同。有的允许以 user 消息结尾，有的有不同的交替规则。把这段代码抄过去，可能既不起作用，又引入了对 Claude API 格式假设的隐式依赖。

## 12.4 工具列表和 Statsig 配置不同步，不报错只让缓存悄悄失效

`src/tools.ts` L191 有一条注释："MUST stay in sync with Statsig config"。

`getAllBaseTools()` 返回的工具列表被序列化进 system prompt 的工具定义部分。Anthropic 在 Statsig（一个功能开关和远端配置服务）上维护了 `claude_code_global_system_caching` 配置，用于跨用户共享 system prompt 的 prompt cache。工具列表和 Statsig 配置不同步时，生成的 system prompt 内容不同，cache key 不匹配，prompt cache 失效，每次请求都要重新处理完整 system prompt。

这是一个"看起来是工具配置，实际上影响成本和延迟"的隐藏耦合。把工具列表抄过来但不维护 Statsig 配置，不会报错，不会崩溃，只是 prompt cache 效率下降，API 调用延迟升高，成本升高，而且原因对排查者完全不透明。

## 12.5 五条原则在约束之外仍然成立

前四节说的是不建议直接复制的具体实现。这里是从中可以抽象出来、在不同约束下依然有效的原则。

**分层错误处理**：用户主动中止、外部信号中断、资源超限，三种情况的处理逻辑本来就不同。把它们混在一个错误处理分支里是设计问题，和具体的运行时或框架无关。

**集中可变状态**：`src/query.ts#L282` 的 `State` 对象把所有跨迭代的可变状态集中在一处，好处是可观察、可测试、行为可预测。代价是需要刻意维护对象边界，否则会膨胀。代码里有告诫注释，但告诫不等于约束（见第 11 章）。

**最小权限副作用**：把副作用操作限制在最小范围内，防止意外蔓延。`createMemoryFileCanUseTool()` 用严格相等判断路径，不是专门防路径遍历，是严格限制副作用范围的自然结果。这个原则的适用条件是你有能力在设计阶段划清副作用边界，反例是遗留系统中副作用已经扩散到处处，补救成本高。

**主动管理资源上限**：在接近上限之前主动处理，被动响应的失败路径是连锁的，主动管理的失败路径是可控的（见第 11 章）。适用条件：系统能在运行时感知资源用量，且压缩或截断操作不会丢失关键信息。如果运行时无法感知上限，这条原则缺乏前提，强行实施只会引入误判。

**正确性优先于性能**：`src/utils/conversationRecovery.ts#L164` 在恢复时做了多项格式合法性检查。这些检查有性能开销，但恢复后的状态如果不合法，后续所有操作都会失败，性能优化没有意义。

原则是工具，不是教条。五条原则各有适用条件，在本章已一一指出。把原则当检查清单用，而不是当规范照搬，才是学架构该有的姿势。

## 12.6 心智模型验证题

你把 Claude Code 的文件系统任务队列抄到一个需要支持 50 个并发 agent 的系统里。在没有明显 bug 的情况下，你会观察到什么现象？如何确认问题来源？

答案：不会有报错，但任务认领的 p99 延迟会持续爬升。50 个 agent 争一把锁，每个临界区约 100ms，最后一个 agent 需要等约 5 秒，超过 `proper-lockfile` 的 2.6 秒超时配置，开始出现锁超时错误，吞吐量触顶。确认来源：监控任务认领时间的 p99，对比 agent 数量变化时的时间分布；缩减并发数后 p99 恢复正常即可定位。解决路径：换成支持原子操作的后端，Redis `SETNX` 或 PostgreSQL `SELECT FOR UPDATE SKIP LOCKED`。
