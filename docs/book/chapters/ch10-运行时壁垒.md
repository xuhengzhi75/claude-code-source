# 第10章 运行时壁垒

**核心主张（可被反驳）：** 边界情况处理积累的是故障历史，不是算法。复制代码拿不走这段历史，因为没有经历那次故障，代码就没有意义。

---

想象一位老厨师的灶台：上面有一块焦痕，旁边贴着一张纸条——"大火不超过三分钟，否则锅底焦"。新来的厨师可以抄这条规矩，但不知道为什么三分钟，也不知道曾经烧焦过几次锅。规矩本身没有迁移走经验。

`migrateLegacyAttachmentTypes()`（[`conversationRecovery.ts#L77`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/utils/conversationRecovery.ts#L77)）就是那张纸条。竞争对手可以把这个函数复制过去，但函数存在的原因是：Claude Code 在某次版本升级时改变了 attachment 的类型字段名，读取旧数据需要迁移。竞争对手的系统没有这段升级历史，这个函数对他们没有意义。

本章分析三组壁垒：中断恢复的正确性、压缩协议的连续性、并发安全与权限隔离。每一组都指向同一个结构——设计决策背后有具体的故障数据，光读代码看不到这些数据。

---

## 10.1 中断状态的识别依赖对合法终态的精确定义

中断恢复的真正难点不是"如何恢复"，而是"如何判断是否发生了中断"。

`transcript`（持久化到磁盘的对话记录文件）的结构可能因两种原因显得"不完整"：一种是进程被杀、网络断开导致的真中断；另一种是最后一条消息形态特殊，看起来像中间状态但其实是合法的结束。误判方向有两个：把已完成的任务重新执行，或把真正的中断当作完成跳过。

[`conversationRecovery.ts#L272`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/utils/conversationRecovery.ts#L272) 的 `detectTurnInterruption()` 区分三种状态：`interrupted_turn`（模型回复到一半被中断）、`interrupted_prompt`（用户消息发出后 API 还没响应就中断）、正常结束。检测到中断后，[`conversationRecovery.ts#L226`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/utils/conversationRecovery.ts#L226) 补一个 assistant sentinel 消息，保证恢复后的对话结构对 API 合法。

Brief 模式（脚本调用场景，区别于交互模式）下有一个非直觉边界：tool_result 作为 turn 结尾是合法的终态，但结构上看起来像一个未完成的中间状态。`isTerminalToolResult` 专门识别这类情况，防止把合法完成的会话误判为中断。这个规则无法从 API 文档推导出来，它是在具体模式下试出来的边界。

当前实现的已知限制：推断中断类型的前提是 transcript 结构完整。如果写入中途进程崩溃导致文件内容损坏，推断可能失败。这个场景当前没有处理。

## 10.2 压缩协议的连续性是隐式约定，格式变化就断链

上下文压缩（compact：在会话超长时生成摘要替换历史消息，缩短 token 的机制）不只是"缩短会话"，压缩后的结构必须能被恢复系统识别，resume 才能在正确位置接续。

[`compact.ts#L328`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/services/compact/compact.ts#L328) 的注释明确了压缩后重建上下文的固定顺序：`boundaryMarker, summaryMessages, messagesToKeep, attachments, hookResults`。这个顺序是协议，恢复系统按这个顺序找接续点。顺序变了，resume 链路就断了。

[`compact.ts#L360`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/services/compact/compact.ts#L360) 的 `compactMetadata.preservedSegment` 维护保留段的重连信息：记录 head/anchor/tail，让 resume 知道从哪里接续，而不是简单截断（[`annotateBoundaryWithPreservedSegment`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/services/compact/compact.ts)）。这套机制需要和恢复系统、历史系统、附件系统同时对齐，任何一个环节的格式变化都会导致链路断裂，维护成本是持续的。

替代方案是滑动窗口截断：只保留最近 N 条消息，实现简单，但可能丢失早期的任务目标和约束条件。压缩的代价是实现复杂，摘要质量依赖模型能力；换来的是在极长会话中仍能续上关键信息。

### 25 万次 API 调用的教训固化为一个常量

自动压缩系统的代码里有一段注释：

```
// BQ 2026-03-10: 1,279 sessions had 50+ consecutive failures (up to 3,272)
// in a single session, wasting ~250K API calls/day globally.
```

1279 个会话里，压缩连续失败 50 次以上，最极端的一个会话失败了 3272 次，全球每天因此浪费约 25 万次 API 调用。解决方案只有一行：`MAX_CONSECUTIVE_AUTOCOMPACT_FAILURES = 3`。

注释里记录了发现时间（2026-03-10）、问题规模（1279 个会话、最多 3272 次）、全局影响（25 万次/天）。下一个读到这里的人不需要翻 issue 历史，就能理解为什么是 3 而不是 10 或无限。这是一种工程规范：把触发修复的数据写在修复旁边，代码本身成为可读的故障记录。

这个规范有适用条件：有具体的可量化数据支撑（1279 个会话、25 万次调用）。如果修复是预防性的、没有具体数据，加这种注释反而制造噪音。

## 10.3 并发安全和权限隔离要求把操作语义嵌入锁的边界

TOCTOU（Time-of-Check-to-Time-of-Use，检查和使用之间存在时间窗口的竞态条件）竞态不会在低并发演示里出现，只有在真实的多 Agent 场景下才暴露。

[`tasks.ts#L621`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/utils/tasks.ts#L621) 的 `claimTaskWithBusyCheck()` 把检查和认领放在同一把 list-level lock 内（[`tasks.ts#L532`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/utils/tasks.ts#L532) 的注释明确写了：使用 task-list-level lock 防止 TOCTOU 竞态）。把检查和认领拆成两个独立调用的后果是：Agent A 检查任务 T 未被认领，Agent B 同时也检查 T 未被认领，两者都认领成功，任务被执行两遍。单元测试很难覆盖这类 bug，因为需要精确的时间窗口。

文件系统锁（`.lock` 文件，[`tasks.ts#L508`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/utils/tasks.ts#L508)）的优点是简单、可观察、零外部依赖。已知部署约束：NFS 不可靠地实现 POSIX 锁语义，部署在 NFS 上时文件锁失效，TOCTOU 竞态重新出现。

### 最小权限通过权限函数而不是调用约定落地

[`sessionMemory.ts#L460`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/services/SessionMemory/sessionMemory.ts#L460) 的 `createMemoryFileCanUseTool()` 创建了一个只允许 Edit 操作且只允许访问特定 memory 文件的权限函数：`only ${FILE_EDIT_TOOL_NAME} on ${memoryPath} is allowed`。

如果记忆更新逻辑和主对话共享上下文，更新操作可能意外影响主流程的状态。权限函数把副作用限制在 memory 文件本身，这是最小权限在 Agent 系统里的具体形态：子代理的工具集单独配置、上下文单独隔离、执行结果有明确的归并方式。缺任何一件，最小权限就只是一个说法。

## 10.4 边界与可迁移原则

### 架构层面的已知约束

这三组壁垒都依赖单机文件系统。规模化到分布式后，文件锁不可靠，JSONL append 的原子性假设不成立，目录扫描成为瓶颈。单机场景内这些方案足够可靠，这是已知边界，不是设计缺陷。

transcript 损坏的情况（写入中途进程崩溃导致文件内容损坏）当前没有处理，是已知的技术债。

`restoreSkillStateFromMessages`（[`conversationRecovery.ts`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/utils/conversationRecovery.ts)）在反序列化前从 attachment 回放技能状态，原因是：如果技能状态在 compact 之前没有回放，下次 compact 会遗忘技能。这个约束是 compact 和恢复机制之间的隐式协议，没有在接口层声明，只能读代码发现。隐式协议是跨版本数据兼容的持续成本来源。

### 修复事故时把事故数据写进注释

这条原则来自 `MAX_CONSECUTIVE_AUTOCOMPACT_FAILURES = 3` 旁边那段注释的写法。适用条件：修复有具体的可量化触发数据。预防性修复没有数据，加数据注释会制造对后继者的误导。

原则是工具，不是教条。没有故障数据时，把修复原因写清楚比伪造数据更有价值。

## 10.5 本章不覆盖项

本章聚焦边界情况处理机制本身，不覆盖任务系统的对象模型（见第8章），也不覆盖护城河的商业和战略分析（见第9章）。跨版本数据兼容的具体迁移格式（`new_file` → `file`，`new_directory` → `directory`）是第7章的内容，本章只说明这类持续成本的结构性原因。
