# 第10章 难以复制的运行时壁垒

假设你完整复制了 Claude Code 的代码，部署起来，开始跑真实任务。第一天，一切正常。第三天，有用户反映 `--resume` 偶尔失败。第七天，两个并发 Agent 把同一个任务执行了两遍。第十天，某个旧版本保存的会话文件加载时报错。

这些问题在演示里不会出现，在代码审查里也看不出来。它们只在真实使用中、在特定条件组合下才会触发。修复每一个都需要先理解它的触发条件，而触发条件往往是多个因素的组合。修复一个可能引入新的。

这就是运行时壁垒的本质：不是某个单一的技术难点，而是边界情况处理的积累——每一条都来自真实踩坑，不是设计文档里能预先写全的。

## 10.1 中断恢复的正确性问题

恢复听起来简单，实际上有一个坑：真正的中断（进程被杀、网络断开、用户强制退出），和看起来像中断但其实已经完成的情况（最后一轮输出形态特殊，导致状态看起来不完整），这两类情况必须区分。

如果混淆，就会出现两种错误：把已完成的任务当成中断重新执行，或者把真正的中断当成完成跳过。

`src/utils/conversationRecovery.ts#L272` 的 `detectTurnInterruption()` 专门处理这个区分。它检测三种情况：`interrupted_turn`（模型回复到一半被中断）、`interrupted_prompt`（用户消息发出后 API 还没响应就被中断）、正常结束。`conversationRecovery.ts#L226` 在检测到中断后，会补一个 assistant sentinel 消息，保证恢复后的对话结构对 API 合法。

这个设计假设中断类型可以从 transcript 的结构推断出来。如果 transcript 本身损坏（比如写入中途进程崩溃），推断可能失败。当前实现没有处理 transcript 损坏的情况——这是一个已知的边界条件。

## 10.2 压缩后语义不丢失

上下文压缩（compact）的目标是缩短 token 长度，同时保留足够的信息让任务能继续。这两个目标之间有张力：压缩越激进，token 越少，但丢失关键信息的风险越高。

`src/services/compact/compact.ts#L328` 的注释说明了压缩后重建上下文的顺序：`boundaryMarker, summaryMessages, messagesToKeep, attachments, hookResults`。这个固定顺序保证了压缩后的上下文有固定结构，resume 时能找到接续点。

`compact.ts#L360` 的 `compactMetadata.preservedSegment` 维护保留段的重连信息。这意味着压缩不是简单截断，而是有元数据记录哪些内容被保留、从哪里接续。

压缩协议需要和恢复系统、历史系统、附件系统同时对齐。任何一个环节的格式变化，都可能导致 resume 时链路断裂。维护这套协议的一致性，需要持续的工程投入——这不是一次性的设计工作，而是随着系统演进持续存在的维护成本。

另一种做法是不压缩，而是用滑动窗口截断历史（只保留最近 N 条消息）。滑动窗口的优势是实现简单，劣势是可能丢失早期的关键上下文（比如任务目标、约束条件）。压缩的优势是能保留语义摘要，劣势是实现复杂，摘要质量依赖模型能力。

## 10.3 压缩失败的熔断：25 万次 API 调用的教训

自动压缩系统里有一段注释，是整个代码库里最真实的工程记录之一：

```
// BQ 2026-03-10: 1,279 sessions had 50+ consecutive failures (up to 3,272)
// in a single session, wasting ~250K API calls/day globally.
```

1279 个会话，每个会话里压缩连续失败 50 次以上，最极端的一个会话失败了 3272 次。全球每天因此浪费约 25 万次 API 调用。

解决方案只有一行：`MAX_CONSECUTIVE_AUTOCOMPACT_FAILURES = 3`。连续失败三次，停止继续尝试。

这个注释值得仔细看。它记录了发现时间（2026-03-10）、问题规模（1279 个会话、最多 3272 次）、全局影响（25 万次/天）。这不是随手写的注释，而是一次真实事故的复盘记录，直接嵌在修复代码旁边。

这种写法有一个好处：下一个读到这段代码的人，不需要去翻 issue 或 commit 历史，就能理解为什么这里要有一个"3"，而不是"10"或者"无限"。数字背后的故事就在代码里。

## 10.4 并发下的原子认领

`src/utils/tasks.ts#L621` 的 `claimTaskWithBusyCheck()` 把检查 agent 是否忙和认领任务这两步放在同一把 list-level lock 内完成（[`tasks.ts#L532`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/utils/tasks.ts#L532) 的注释明确写了：使用 task-list-level lock 防止 TOCTOU 竞态）。

这解决的是 TOCTOU（time-of-check to time-of-use）竞态：如果检查和认领之间有间隙，两个 agent 可能同时通过检查，然后同时认领同一个任务。

文件系统锁（`.lock` 文件，[`tasks.ts#L508`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/utils/tasks.ts#L508)）是这里的实现基础。它的优点是简单、可观察、不依赖外部服务。缺点是在极高并发下会有性能瓶颈，但对大多数业务 Agent 场景来说，这个方案足够可靠。文件系统锁在 NFS（网络文件系统）上不可靠——如果系统部署在 NFS 上，文件锁可能失效，导致 TOCTOU 竞态重新出现。

## 10.5 记忆更新的最小权限

`src/services/SessionMemory/sessionMemory.ts#L460` 的 `createMemoryFileCanUseTool()` 创建了一个只允许 Edit 操作且只允许访问特定 memory 文件的权限函数。注释写得很直接：`only ${FILE_EDIT_TOOL_NAME} on ${memoryPath} is allowed`。

这个设计解决的是副作用蔓延问题：如果记忆更新逻辑和主对话共享上下文，更新操作可能意外影响主流程的状态。隔离上下文后，记忆更新的影响范围被限制在 memory 文件本身。

在 Agent 系统里，最小权限的落地方式是：子代理的工具集要单独配置，上下文要单独隔离，执行结果要有明确的归并方式。这三件事缺任何一件，最小权限就无法真正实现。

## 10.6 跨版本数据兼容

`src/utils/conversationRecovery.ts#L77` 的 `migrateLegacyAttachmentTypes()` 把旧版本的 `new_file` 类型迁移成当前的 `file` 类型（[`L88`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/utils/conversationRecovery.ts#L88)），把 `new_directory` 迁移成 `directory`（[`L99`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/utils/conversationRecovery.ts#L99)）。

这类迁移代码的存在，说明系统在演进过程中改变过数据格式，但需要保持对旧数据的兼容。每次格式变化都需要写迁移逻辑，并在恢复路径上测试。

对于一个持续迭代的系统，跨版本兼容是持续成本，不是一次性工作。

## 10.7 心智模型验证题

如果去掉 `claimTaskWithBusyCheck()` 里的文件锁，把检查和认领分成两个独立的函数调用，会在什么场景下出现问题？

答案：在多个 agent 并发运行时。Agent A 调用检查函数，发现任务 T 未被认领。在 Agent A 调用认领函数之前，Agent B 也调用检查函数，也发现任务 T 未被认领。然后 Agent A 和 Agent B 都调用认领函数，都成功认领了任务 T。结果是两个 agent 同时执行同一个任务，产生重复执行。

## 10.8 本章小结

Claude Code 的运行时壁垒主要体现在 5 个方面：中断恢复的正确性（[`conversationRecovery.ts#L272`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/utils/conversationRecovery.ts#L272)）、压缩后语义不丢失（[`compact.ts#L360`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/services/compact/compact.ts#L360)）、压缩失败的熔断（`MAX_CONSECUTIVE_AUTOCOMPACT_FAILURES = 3`）、并发下的原子认领（[`tasks.ts#L621`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/utils/tasks.ts#L621)）、记忆更新的最小权限（[`sessionMemory.ts#L460`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/services/SessionMemory/sessionMemory.ts#L460)）。

这些壁垒的共同特点是：需要在真实使用中积累，不是一次性设计能完成的。每一条都有一个故事，每个故事都是一次真实的失败。
