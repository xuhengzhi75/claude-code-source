# 第10章 难以复制的运行时壁垒

> 核心主张：Claude Code 的运行时壁垒不是某个单一的技术难点，而是五类边界情况处理的叠加——每一类都来自真实踩坑，不是设计文档里能预先写全的，这让"复制架构"和"复制能力"之间存在巨大的实践鸿沟。

上一章说了护城河的三个来源。这一章具体看：这些壁垒在技术层面是什么，为什么难以在短时间内复制。

## 10.1 中断恢复的正确性问题

恢复听起来简单，实际上有两类情况需要区分：真正的中断（进程被杀、网络断开、用户强制退出），和看起来像中断但其实已经完成的情况（最后一轮输出形态特殊，导致状态看起来不完整）。

如果把这两类情况混淆，就会出现两种错误：把已完成的任务当成中断重新执行，或者把真正的中断当成完成跳过。

`src/utils/conversationRecovery.ts#L272` 的 `detectTurnInterruption()` 专门处理这个区分。它检测三种情况：`interrupted_turn`（模型回复到一半被中断）、`interrupted_prompt`（用户消息发出后 API 还没响应就被中断）、正常结束。`conversationRecovery.ts#L226` 在检测到中断后，会补一个 assistant sentinel 消息，保证恢复后的对话结构对 API 合法。

这里的难点不是代码量，而是需要知道哪些情况会导致结构不合法，以及每种情况对应的修复方式。这些知识来自真实踩坑，不是设计文档里能预先写全的。

## 10.2 压缩后语义不丢失

上下文压缩（compact）的目标是缩短 token 长度，同时保留足够的信息让任务能继续。这两个目标之间有张力：压缩越激进，token 越少，但丢失关键信息的风险越高。

`src/services/compact/compact.ts#L328` 的注释说明了压缩后重建上下文的顺序：`boundaryMarker, summaryMessages, messagesToKeep, attachments, hookResults`。这个固定顺序保证了压缩后的上下文有固定结构，resume 时能找到接续点。

`compact.ts#L360` 的 `compactMetadata.preservedSegment` 维护保留段的重连信息。这意味着压缩不是简单截断，而是有元数据记录哪些内容被保留、从哪里接续。

难以复制的原因是：这个协议需要和恢复系统、历史系统、附件系统同时对齐。任何一个环节的格式变化，都可能导致 resume 时链路断裂。维护这套协议的一致性，需要持续的工程投入。

## 10.3 并发下的原子认领

`src/utils/tasks.ts#L621` 的 `claimTaskWithBusyCheck()` 把检查 agent 是否忙和认领任务这两步放在同一把 list-level lock 内完成（[`tasks.ts#L532`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/utils/tasks.ts#L532) 的注释明确写了：使用 task-list-level lock 防止 TOCTOU 竞态）。

这解决的是 TOCTOU（time-of-check to time-of-use）竞态：如果检查和认领之间有间隙，两个 agent 可能同时通过检查，然后同时认领同一个任务。

文件系统锁（`.lock` 文件，[`tasks.ts#L508`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/utils/tasks.ts#L508)）是这里的实现基础。它的优点是简单、可观察、不依赖外部服务。缺点是在极高并发下会有性能瓶颈，但对大多数业务 Agent 场景来说，这个方案足够可靠。

这类并发 bug 的特点是：在测试环境和低负载下几乎不出现，在生产高压场景下才暴露。修复后还需要回归测试覆盖，否则容易在后续改动中重新引入。

## 10.4 记忆更新的最小权限

`src/services/SessionMemory/sessionMemory.ts#L460` 的 `createMemoryFileCanUseTool()` 创建了一个只允许 Edit 操作且只允许访问特定 memory 文件的权限函数。注释写得很直接：`only ${FILE_EDIT_TOOL_NAME} on ${memoryPath} is allowed`。

这个设计解决的是副作用蔓延问题：如果记忆更新逻辑和主对话共享上下文，更新操作可能意外影响主流程的状态。隔离上下文后，记忆更新的影响范围被限制在 memory 文件本身。

最小权限原则在安全领域是基础概念，但在 Agent 系统里落地需要额外设计：子代理的工具集要单独配置，上下文要单独隔离，执行结果要有明确的归并方式。

## 10.5 跨版本数据兼容

`src/utils/conversationRecovery.ts#L77` 的 `migrateLegacyAttachmentTypes()` 把旧版本的 `new_file` 类型迁移成当前的 `file` 类型（[`L88`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/utils/conversationRecovery.ts#L88)），把 `new_directory` 迁移成 `directory`（[`L99`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/utils/conversationRecovery.ts#L99)）。

这类迁移代码的存在，说明系统在演进过程中改变过数据格式，但需要保持对旧数据的兼容。每次格式变化都需要写迁移逻辑，并在恢复路径上测试。

对于一个持续迭代的系统，跨版本兼容是持续成本，不是一次性工作。

## 10.6 这些壁垒的共同特点

上面 5 类壁垒有一个共同特点：它们都不是在初始设计阶段能完整预见的，而是在真实使用中逐步发现和补齐的。

这意味着复制这些壁垒，不只是复制代码，还需要经历同样的发现过程。从零开始做一个 Agent 系统，即使参考了 Claude Code 的架构，也需要在真实场景中积累自己的边界情况处理。

## 10.7 本章小结

Claude Code 的运行时壁垒主要体现在 5 个方面：中断恢复的正确性（[`conversationRecovery.ts#L272`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/utils/conversationRecovery.ts#L272)）、压缩后语义不丢失（[`compact.ts#L360`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/services/compact/compact.ts#L360)）、并发下的原子认领（[`tasks.ts#L621`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/utils/tasks.ts#L621)）、记忆更新的最小权限（[`sessionMemory.ts#L460`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/services/SessionMemory/sessionMemory.ts#L460)）、跨版本数据兼容（[`conversationRecovery.ts#L77`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/utils/conversationRecovery.ts#L77)）。

这些壁垒的共同特点是：需要在真实使用中积累，不是一次性设计能完成的。下一章从另一个角度看这个问题：为什么很多 Agent 看起来功能完整，但在真实工作中表现不稳定。
