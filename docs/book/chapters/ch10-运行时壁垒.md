# 第10章 运行时壁垒

**核心主张（可被反驳）：** 运行时壁垒的本质是边界情况处理的积累，每一条都来自真实踩坑，无法通过读一遍源码重现——这才是竞争对手复制代码后在第 7 天遇到奇怪 bug 的原因。

## 10.1 第一性问题：为什么边界情况处理很难抄

复制代码很容易，复制踩坑历史很难。

`migrateLegacyAttachmentTypes()`（[`conversationRecovery.ts#L77`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/utils/conversationRecovery.ts#L77)）这个函数，竞争对手可以复制过去。但它存在的原因是：系统在某次版本升级时改变了 attachment 的类型字段名，需要在读取旧数据时做迁移。竞争对手的系统没有这段历史，复制过去的这个函数对他们没有意义。他们需要在自己系统演进时，积累自己的迁移函数。

这就是为什么"抄代码"无法抄走护城河。边界情况处理对应的是真实发生过的故障，没有故障就没有对应的处理代码的意义。

## 10.2 中断恢复的正确性：区分真中断和看起来像中断的完成状态

中断恢复有一个容易忽视的问题：如何区分真正的中断（进程被杀、网络断开）和看起来像中断但其实已完成的情况（最后一轮输出形态特殊，导致状态看起来不完整）。

混淆的后果是两种：把已完成的任务当成中断重新执行，或者把真正的中断当成完成跳过。

`src/utils/conversationRecovery.ts#L272` 的 `detectTurnInterruption()` 专门处理这个区分。它检测三种情况：`interrupted_turn`（模型回复到一半被中断）、`interrupted_prompt`（用户消息发出后 API 还没响应就中断）、正常结束。[`conversationRecovery.ts#L226`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/utils/conversationRecovery.ts#L226) 在检测到中断后，补一个 assistant sentinel 消息，保证恢复后的对话结构对 API 合法。

Brief 模式下有一种特殊情况：tool_result 作为 turn 结尾是合法的终态，但看起来像一个未完成的中间状态。[`isTerminalToolResult`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/utils/conversationRecovery.ts) 专门识别这类情况，防止把合法完成的会话误判为中断。

这个设计假设中断类型可以从 transcript 结构推断出来。如果 transcript 本身损坏（写入中途进程崩溃），推断可能失败。当前实现不处理 transcript 损坏的情况，这是已知的边界条件。

## 10.3 压缩后连续性：顺序即协议

上下文压缩（compact）需要在缩短 token 和保留语义之间做权衡：压缩越激进，token 越少，丢失关键信息的风险越高。

`src/services/compact/compact.ts#L328` 的注释说明了压缩后重建上下文的固定顺序：`boundaryMarker, summaryMessages, messagesToKeep, attachments, hookResults`。这个顺序是协议，不是实现细节。resume 时按这个顺序找接续点，如果顺序变了，resume 链路就断了。

[`compact.ts#L360`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/services/compact/compact.ts#L360) 的 `compactMetadata.preservedSegment` 维护保留段的重连信息：记录 head/anchor/tail，让 resume 知道从哪里接续，而不是简单截断（[`annotateBoundaryWithPreservedSegment`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/services/compact/compact.ts)）。

压缩协议需要和恢复系统、历史系统、附件系统同时对齐。任何一个环节的格式变化都可能导致 resume 链路断裂。维护这套协议的一致性是持续的工程投入，不是一次性设计工作。

另一种做法是滑动窗口截断（只保留最近 N 条消息）。实现简单，但可能丢失早期的关键上下文（任务目标、约束条件）。压缩的代价是实现复杂，摘要质量依赖模型能力；收益是保留语义摘要，在极长会话中仍能续上关键信息。

## 10.4 压缩失败熔断：25 万次 API 调用的教训

自动压缩系统的代码里有一段注释，是整个代码库里最有价值的工程记录之一：

```
// BQ 2026-03-10: 1,279 sessions had 50+ consecutive failures (up to 3,272)
// in a single session, wasting ~250K API calls/day globally.
```

1279 个会话里，压缩连续失败 50 次以上，最极端的一个会话失败了 3272 次，全球每天因此浪费约 25 万次 API 调用。

解决方案只有一行：`MAX_CONSECUTIVE_AUTOCOMPACT_FAILURES = 3`。连续失败三次，停止继续尝试。

这段注释记录了发现时间（2026-03-10）、问题规模（1279 个会话、最多 3272 次）、全局影响（25 万次/天）。下一个读到这段代码的人，不需要翻 issue 历史，就能理解为什么这里是 3 而不是 10 或无限。数字背后的故事就在代码旁边。

这种写法体现了一个工程规范：修复事故的同时，把事故的来龙去脉写进注释。这让代码本身成为可读的事故记录，而不依赖外部的文档系统。

## 10.5 并发认领：检查和认领必须在同一把锁里

`src/utils/tasks.ts#L621` 的 `claimTaskWithBusyCheck()` 把检查和认领放在同一把 list-level lock 内（[`tasks.ts#L532`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/utils/tasks.ts#L532) 的注释明确写了：使用 task-list-level lock 防止 TOCTOU 竞态）。

文件系统锁（`.lock` 文件，[`tasks.ts#L508`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/utils/tasks.ts#L508)）的优点是简单、可观察、零外部依赖。在 NFS 上不可靠——NFS 不可靠地实现 POSIX 锁语义，部署在 NFS 上时文件锁失效，TOCTOU 竞态重新出现。这是已知的部署约束。

## 10.6 记忆更新最小权限：隔离副作用

`src/services/SessionMemory/sessionMemory.ts#L460` 的 `createMemoryFileCanUseTool()` 创建了一个只允许 Edit 操作且只允许访问特定 memory 文件的权限函数：`only ${FILE_EDIT_TOOL_NAME} on ${memoryPath} is allowed`。

这个设计解决的是副作用蔓延问题。如果记忆更新逻辑和主对话共享上下文，更新操作可能意外影响主流程的状态。隔离上下文后，记忆更新的影响范围被限制在 memory 文件本身。

在 Agent 系统里，最小权限的落地需要三件事：子代理的工具集单独配置、上下文单独隔离、执行结果有明确的归并方式。缺任何一件，最小权限就无法真正实现。

## 10.7 跨版本数据兼容：持续成本

[`migrateLegacyAttachmentTypes()`（`conversationRecovery.ts#L77`）](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/utils/conversationRecovery.ts#L77) 把旧版的 `new_file` 迁移成 `file`（[`L88`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/utils/conversationRecovery.ts#L88)），把 `new_directory` 迁移成 `directory`（[`L99`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/utils/conversationRecovery.ts#L99)）。

用文件系统存数据的代价在这里体现：不能像数据库那样做 schema migration，只能在读取时处理所有历史格式。每次格式变化都需要写迁移逻辑，并在恢复路径上测试。这是持续成本，不是一次性工作。

技能状态也有类似的问题。[`restoreSkillStateFromMessages`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/utils/conversationRecovery.ts) 在反序列化前从 attachment 回放技能状态，原因是：如果技能状态在 compact 之前没有回放，下次 compact 会遗忘技能。这个约束是 compact 和恢复机制之间的隐式协议，没有在接口层声明，需要读代码才能发现。

## 10.8 边界与技术债

这六个壁垒都依赖单机文件系统。规模化到分布式后，文件锁不可靠，JSONL append 的原子性假设不成立，目录扫描成为瓶颈。这是架构层面的已知约束，单机场景内这些方案足够可靠。

transcript 损坏的情况（写入中途进程崩溃导致文件内容损坏）当前没有处理，是已知的边界条件。

## 10.9 可迁移原则

事故修复代码里加事故记录注释。适用条件：有一个具体的、可量化的问题触发了这次修复（1279 个会话、25 万次 API 调用）。反例是：如果修复是预防性的（"理论上可能出现这个问题"），没有具体的数据，加这种注释反而制造噪音。

## 10.10 本章不覆盖项

本章聚焦边界情况处理机制本身，不覆盖任务系统的对象模型（见第8章），也不覆盖护城河的商业和战略分析（见第9章）。这六个壁垒描述的是"在真实使用中才暴露的问题"，设计文档里无法提前写全。

**心智模型验证：** 去掉 `claimTaskWithBusyCheck()` 里的文件锁，把检查和认领分成两个独立的函数调用，哪种场景会出问题？多 Agent 并发时：Agent A 检查 T 未被认领，Agent B 同时也检查 T 未被认领，两者都认领成功，任务被执行两遍。低并发时完全正常，高并发时必现。这类 bug 在演示里不出现，在单元测试里也难覆盖，只有在真实的多 Agent 场景下才暴露。
