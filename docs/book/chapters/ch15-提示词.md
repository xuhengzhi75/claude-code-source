# 第15章 提示词是工程产物，结构决定成本

想象一家连锁餐厅的操作手册。有些内容对所有门店永远成立：食品安全规范、品牌话术、服务流程。有些内容每家门店不同：本地食材、今日特价、当班员工名单。如果把这两类内容混在同一个文件里，总部每次更新"今日特价"，就要把整本手册重印一遍。明智的做法是分册：通用手册印一次长期用，动态信息另发。

Claude Code 的系统提示词做了同一件事。提示词不是"描述 Agent 应该做什么"的说明书，而是一套有结构成本、有版本补丁、按运行上下文动态注入约束的工程产物。提示词的组织方式决定 API 缓存命中率，提示词里的版本补丁记录了系统在真实运行中踩过的坑，提示词根据用户类型注入完全不同的输出策略。

这一章要回答一个问题：**提示词的结构是怎么设计的，以及这些设计决策的代价是什么。**

## 15.1 静态/动态分割让缓存命中，但边界耦合是已知技术债

`src/constants/prompts.ts#L106` 定义了 `SYSTEM_PROMPT_DYNAMIC_BOUNDARY`（系统提示词的动态边界标记）。边界之前的内容可以使用 `scope: 'global'` 全局缓存；边界之后包含用户特定信息，每次会话重新计算。注释明确警告：不要在不同步更新 `src/utils/api.ts` 的 `splitSysPromptPrefix` 和 `src/services/api/claude.ts` 的 `buildSystemPromptBlocks` 的情况下移动这个边界。

这个耦合是故意的。边界标记本身只是一行注释，类型系统无法感知它。移动了边界标记但忘了更新两处下游，缓存会静默失效——API 调用不报错，成本悄悄增加。发现这个问题需要专门监控缓存命中率，不是跑测试。

静态前缀包含工具描述、基础行为约束、通用安全规则，这些内容在不同用户、不同会话之间完全一致，命中全局缓存。动态后缀包含当前工具列表、运行模式说明、权限边界、当前环境信息，每次会话重新计算，不走缓存。替代方案是把所有内容压进单一字符串，代码更简单，代价是每次 API 调用都重算全部 token，高频场景下成本显著上升。Claude Code 选择了结构复杂性换取 API 经济性。

### 边界前的运行时条件判断使缓存变体指数膨胀

`prompts.ts#L343` 的注释说明了一个反直觉的约束：

```
// Session-variant guidance that would fragment the cacheScope:'global'
// prefix if placed before SYSTEM_PROMPT_DYNAMIC_BOUNDARY. Each conditional
// here is a runtime bit that would otherwise multiply the Blake2b prefix
// hash variants (2^N). See PR #24490, #24171 for the same bug class.
```

在边界标记之前加一个"是否有 AskUserQuestion 工具"的条件判断，有/没有这个工具会产生两个不同的前缀哈希（Blake2b 哈希，用于缓存键的唯一标识）。再加一个"是否是非交互会话"的判断，变成四种变体。N 个条件产生 2^N 种缓存变体，全局缓存命中率指数级下降。

PR #24490 和 #24171 是两次实际踩坑记录——系统在真实运行中发现了这个问题，然后把相关条件全部移到边界之后。这类 bug 不触发报错，只让缓存命中率下降，需要专门分析成本数据才能发现。对业务 Agent 来说，同样的坑藏在"VIP 用户和普通用户走不同提示词"的设计里：如果差异化内容放在前缀，缓存按用户类型分裂，等于没有缓存。

## 15.2 提示词可以携带版本补丁，但不带标记的补丁是维护陷阱

`src/constants/prompts.ts` 里有一段带 `@[MODEL LAUNCH]` 标记的注释：

```
// Internal data shows Capybara v8 has a 29–30% incorrect-statement rate
// (vs 16.7% for v4). Explicit prohibitions:
// - Do not claim tests pass when they fail
// - Do not hide failing checks to manufacture success
// - Do not describe incomplete work as complete
```

29-30% 不是"偶尔出错"，是三次对话里有一次会说错。在这个基线上不加专项指令的系统不可用。这段指令的存在不是角色设定，是针对特定模型版本行为缺陷的工程补丁。同一章节里还有一个针对 Capybara 过度添加注释倾向的指令，明确要求只有当 WHY is non-obvious 时才允许添加注释。

`@[MODEL LAUNCH]` 标记说明这是临时补丁，下个模型版本需要重新评估是否还需要。没有版本标记的代价是补丁永久留在提示词里，没人知道它是否还有必要，清理时需要逐段判断。随着版本迭代，积累的补丁越来越多，每次模型升级都要做一次系统性的补丁审计——这不是假设，是所有依赖 LLM 的长期项目的共同命运。

### MCP 指令每轮重算，DANGEROUS_ 约定强制记录原因

`systemPromptSections.ts` 定义了两种节函数：

```typescript
// 普通节：计算一次，缓存到 /clear 或 /compact
export function systemPromptSection(
  name: string,
  compute: ComputeFn,
): SystemPromptSection {
  return { name, compute, cacheBreak: false }
}

// 危险节：每轮重新计算，会破坏 prompt cache
// 必须传入 _reason 说明为什么不得不这样做
export function DANGEROUS_uncachedSystemPromptSection(
  name: string,
  compute: ComputeFn,
  _reason: string,
): SystemPromptSection {
  return { name, compute, cacheBreak: true }
}
```

当前代码库里只有一处使用了 `DANGEROUS_` 函数：MCP 指令节（`prompts.ts#L508-L520`）。MCP（Model Context Protocol）服务器可能在会话中途连接或断开，因此这一节必须每轮重新评估。把 MCP 指令放进 system prompt 的静态前缀，每次 MCP 连接状态变化都会使整个前缀缓存失效；移到 message 层则 system prompt 保持不变，缓存命中率不受 MCP 状态影响。

`_reason` 参数以下划线开头，运行时不被使用。它是一个编译期约束，强制开发者在写代码时记录决策。改成可选参数不影响运行，但会让"为什么每轮重算"变成代码库里的隐藏成本。这是一个典型的文档即约束设计：代码强制记录设计原因，而不是依赖工程师之后补注释。

## 15.3 提示词根据运行上下文注入不同约束，用户类型驱动输出策略

相同的模型，面对不同上下文，需要完全不同的行为约束。Claude Code 通过两个机制实现这一点：卧底模式根据仓库上下文动态注入保密规则，双轨输出哲学根据用户类型切换表达规范。

### 仓库上下文触发身份约束注入

`utils/undercover.ts` 实现了卧底模式。触发条件：当前用户类型是 `ant`（Anthropic 内部员工），且 git 远程仓库不在内部白名单里。满足这两个条件，卧底模式自动开启，没有强制关闭选项。

开启后，系统提示词注入一段标题为 `UNDERCOVER MODE: CRITICAL` 的指令，禁止在 commit 信息、PR 标题和 PR 内容里包含：内部模型代号（Capybara、Tengu 等）、未发布的模型版本号、内部仓库名称、Slack 频道名称、"Claude Code"这一表述，以及 `Co-Authored-By` 等署名信息。约束的触发逻辑是：同一个内部用户在内部仓库里工作时不需要隐藏身份，在外部仓库里贡献代码时需要。环境上下文决定行为策略，参数无法直接控制。

### 内外部用户获得根本不同的输出指令

`getOutputEfficiencySection()` 对内外部用户返回两套不同的指令。外部用户版本的核心词是"极简"：直接给答案，跳过推理，不要铺垫，不要过渡。内部用户版本的核心是"写给人看的散文"：倒金字塔结构（新闻写作技法，结论在最前）、语义回溯（semantic backtracking，主动标记推理路径）、避免过度使用破折号和符号。

外部用户在完成具体任务，需要快速得到答案；内部用户（Anthropic 工程师）在评估模型行为，需要能读懂完整推理过程的输出。[inference] 这个区分不是风格偏好，是服务不同使用目的的功能性差异：同一个模型，用提示词切换受众模式，而不是训练两个模型。

情绪信号检测是这个机制的延伸。`utils/userPromptKeywords.ts` 只有 26 行，在每条用户输入发送到 API 之前，用两组正则表达式检测用户是否在骂人。这不是内容审核，是用来判断用户体验是否良好的质量信号。[inference: based on code context] 用户开始骂人，往往意味着 Agent 反复出错、任务卡住。直接问"你满意吗"，大多数人不认真回答；骂人是真实反应，不需要用户主动配合。

## 15.4 可迁移原则

提示词按变化频率分层管理的前提是"有全局缓存"。高频变化（会话特定信息）和低频稳定（基础规范）分开，高频部分不污染低频缓存。反例：单用户本地工具，每次会话 token 预算充足，分层带来的工程复杂度超过收益，不必分层。

运行时条件判断必须移到缓存边界之后。每增加一个条件，缓存变体翻倍；达到阈值后缓存实际上失效。反例：每次请求都有用户特定的 API key，不存在全局缓存，条件放在哪里都不影响命中率。

临时补丁必须打版本标记。`@[MODEL LAUNCH]` 类型的标记让补丁在日后可见、可复评。反例：补丁针对的行为缺陷已经在基础训练里修复，与模型版本无关，不需要版本标记，直接作为永久约束处理。

原则是工具，不是教条。提示词工程的每一条规则背后都有具体的成本假设，当假设不成立，规则随之失效。

## 15.5 本章不覆盖项

提示词版本管理的最简实践：`constants/prompts.ts#L444` 的 `getSystemPrompt()` 通过代码构建系统提示词，自然纳入 git 版本管理。业务 Agent 里"改了提示词，行为变了，但不知道改了什么"的问题，根因是提示词存在数据库或配置表里，没有纳入 git 历史。最简迁移做法是把提示词存成独立文件，和代码一起提交。本章不展开，视为最佳实践直接采用。

第19章从这些结构里提炼出可复用的设计模式，包括：两级缓存节的完整实现细节、对抗性验证 Agent 的提示词设计、Output Style 优先级链、Token Budget 行为规范。`getProactiveSection()`（KAIROS 自主 Agent 模式）的设计也在第19章讨论。本章只覆盖提示词的基本结构机制。
