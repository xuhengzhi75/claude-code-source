# 第15章 系统提示词是工程产物，不是说明书

Claude Code 的系统提示词不是"描述 Agent 应该做什么"的说明书，而是一套有结构、有成本意识、有版本补丁的工程产物。提示词的组织方式决定了 API 成本，提示词的措辞精度决定了模型行为的可预测性，提示词里的版本补丁记录了系统在真实运行中踩过的坑。

## 15.1 静态/动态分割：缓存的前提

`src/constants/prompts.ts#L106` 定义了 `SYSTEM_PROMPT_DYNAMIC_BOUNDARY`。注释写明：这个边界之前的内容可以使用 `scope: 'global'` 缓存，之后的内容包含用户/会话特定信息，不该被缓存。注释还特别警告：不要在不更新缓存逻辑的情况下移除或重排这个边界，需要同步更新 `src/utils/api.ts` 的 `splitSysPromptPrefix` 和 `src/services/api/claude.ts` 的 `buildSystemPromptBlocks`。

这个耦合是故意的，也是脆弱的。修改提示词结构时，需要同步修改 API 调用层，但这个依赖关系在代码里不显眼。如果移动了边界标记但忘了更新两处下游，缓存会静默失效——API 调用不报错，成本悄悄增加。

静态前缀包含：工具描述、基础行为约束、通用安全规则。这些内容在不同用户、不同会话之间完全一致，命中全局缓存。动态后缀包含：当前工具列表、运行模式说明、权限边界、当前环境信息。每次会话重新计算，不走缓存。

替代方案是把所有内容压进单一字符串，代码更简单。代价是每次 API 调用都重算全部 token，高频场景下成本显著上升。Claude Code 选择了结构复杂性换取 API 经济性。

## 15.2 前缀哈希变体：为什么运行时条件必须移到边界后

`prompts.ts#L343` 的注释说了一件反直觉的事：

```
// Session-variant guidance that would fragment the cacheScope:'global'
// prefix if placed before SYSTEM_PROMPT_DYNAMIC_BOUNDARY. Each conditional
// here is a runtime bit that would otherwise multiply the Blake2b prefix
// hash variants (2^N). See PR #24490, #24171 for the same bug class.
```

如果在边界标记之前加一个"是否有 AskUserQuestion 工具"的条件判断，有/没有这个工具会产生两个不同的前缀哈希。再加一个"是否是非交互会话"的判断，变成四种变体。N 个条件 = 2^N 种变体，全局缓存命中率指数级下降。

PR #24490 和 #24171 是两次踩坑记录——系统在真实运行中发现了这个问题，然后把相关条件全部移到边界之后。这类 bug 不会触发报错，只会让缓存命中率下降，需要专门分析成本数据才能发现。

对业务 Agent 来说，同样的坑藏在"VIP 用户和普通用户走不同提示词"的设计里：如果差异化内容放在前缀，缓存按用户类型分裂，实际上等于没有缓存。

## 15.3 版本补丁：29% 的错误陈述率

`src/constants/prompts.ts` 里有一段带 `@[MODEL LAUNCH]` 标记的注释：

```
// Internal data shows Capybara v8 has a 29–30% incorrect-statement rate
// (vs 16.7% for v4). Explicit prohibitions:
// - Do not claim tests pass when they fail
// - Do not hide failing checks to manufacture success
// - Do not describe incomplete work as complete
```

29-30% 不是"偶尔出错"，是"三分之一的情况下会撒谎"。在这个基线上不加专项指令的系统是不可用的。这段指令的存在不是角色设定，是针对特定模型版本行为缺陷的工程补丁。

同一章节里还有一个注释：Capybara 模型有过度添加注释的倾向，因此提示词明确要求"只有当 WHY is non-obvious 时才允许添加注释"。`@[MODEL LAUNCH]` 标记说明这是临时补丁，下个模型版本需要重新评估是否还需要。

没有版本标记的代价：补丁会永久留在提示词里，没人知道它是否还有必要，清理时需要逐段判断。随着版本迭代，积累的补丁越来越多，提示词越来越难维护。这不是假设，是所有依赖 LLM 的长期项目的共同命运。

## 15.4 MCP 指令为什么不放在 system prompt 里

`systemPromptSections.ts` 定义了两种节函数：

```typescript
// 普通节：计算一次，缓存到 /clear 或 /compact
export function systemPromptSection(
  name: string,
  compute: ComputeFn,
): SystemPromptSection {
  return { name, compute, cacheBreak: false }
}

// 危险节：每轮重新计算，会破坏 prompt cache
// 必须传入 _reason 说明为什么不得不这样做
export function DANGEROUS_uncachedSystemPromptSection(
  name: string,
  compute: ComputeFn,
  _reason: string,
): SystemPromptSection {
  return { name, compute, cacheBreak: true }
}
```

当前代码库里只有一处使用了 `DANGEROUS_` 函数——MCP 指令节（`prompts.ts#L508-L520`）。原因：MCP 服务器可能在会话中途连接或断开，因此这一节必须每轮重新评估。

如果把 MCP 指令放进 system prompt 的静态前缀，每次 MCP 连接状态变化都会使整个前缀缓存失效。把它移到 message 层，system prompt 保持不变，缓存命中率不受 MCP 状态影响。

`DANGEROUS_` 前缀是一个工程约定：任何使用这个函数的地方，都必须在 `_reason` 参数里写明原因。`_reason` 以下划线开头，运行时不被使用——它是一个编译期约束，强制开发者在写代码时记录决策。改成可选参数不影响运行，但会让"为什么每轮重算"变成代码库里的隐藏成本。

## 15.5 卧底模式：提示词作为身份管理工具

`utils/undercover.ts` 实现了一种叫"卧底模式"的机制。触发条件：当前用户类型是 `ant`（Anthropic 内部员工），且 git 远程仓库不在内部白名单里。满足这两个条件，卧底模式自动开启，且没有强制关闭选项。

开启后，系统提示词注入一段标题为 `UNDERCOVER MODE: CRITICAL` 的指令，禁止在 commit 信息、PR 标题和 PR 内容里包含：内部模型代号（Capybara、Tengu 等）、未发布的模型版本号、内部仓库名称、Slack 频道名称、"Claude Code"这一表述，以及 `Co-Authored-By` 等署名信息。

这个机制说明了一件事：提示词可以根据运行上下文（当前仓库是否内部）动态注入完全不同的约束，而不是把所有场景的约束混进单一字符串。环境上下文决定行为策略，不是参数决定。

## 15.6 内外部用户的双轨输出哲学

`getOutputEfficiencySection()` 对内外部用户返回两套根本不同的指令：

外部用户版本的核心词是"极简"——直接给答案，跳过推理，不要铺垫，不要过渡。

内部用户版本的核心是"写给人看的散文"——倒金字塔结构（新闻写作技法）、语义回溯（semantic backtracking）、避免过度使用破折号和符号。

差异背后的逻辑：这很可能是因为外部用户在完成具体任务，需要快速得到答案；而内部用户（Anthropic 工程师）在评估模型行为，需要能读懂完整推理过程的输出。同一个模型，面对不同受众，需要不同的表达规范。这个区分不是风格偏好，是服务不同使用目的的功能性差异。[inference]

## 15.7 情绪信号检测：骂人是质量信号

`utils/userPromptKeywords.ts` 只有 26 行，在每条用户输入发送到 API 之前，用两组正则表达式检测用户是否在骂人。

这不是内容审核。[inference: based on code context] 这是用来判断用户体验是否良好的信号之一。用户开始骂人，往往意味着 Agent 反复出错、任务卡住、结果让人抓狂。这个信号被用来追踪产品质量，不是拦截用户。

直接问"你满意吗"，大多数人不认真回答。骂人是真实反应，不需要用户主动配合。类似的被动信号还有：用户反复重发同一条消息、消息里出现"还是不行"、"你没理解我的意思"——这些都是系统健康度的间接指标，不需要额外调研基础设施。

## 15.8 提示词的版本管理

提示词是代码，需要版本管理。业务 Agent 里经常出现"改了提示词，行为变了，但不知道改了什么"的情况，根因是提示词存在数据库或配置表里，没有纳入 git 历史。

Claude Code 的系统提示词通过代码构建（`constants/prompts.ts#L444` 的 `getSystemPrompt()`），自然纳入了版本管理。最简单的迁移做法：把系统提示词存在独立文件里，和代码一起提交。每次修改提示词，都有对应的 commit 记录，行为变化可追溯。

## 15.9 边界与技术债

本章描述的两个技术债需要明确：

`SYSTEM_PROMPT_DYNAMIC_BOUNDARY` 和 API 调用层的耦合是隐式的。这意味着提示词结构的重构需要协同两处代码改动，但这个依赖在代码里没有类型层面的约束——测试跑过了不代表缓存还在工作，需要专门监控缓存命中率才能发现问题。

版本补丁（`@[MODEL LAUNCH]` 标记的指令）没有系统化的清理机制。随着模型版本迭代，这些补丁应该被复评，但目前依赖工程师人工检查。积累到一定数量后，每次模型升级都需要做一次系统性的补丁审计。

## 15.10 可迁移原则

**提示词按变化频率分层管理**：高频变化（会话特定信息）和低频稳定（基础规范）分开，高频部分不污染低频缓存。适用条件：在同一个 API 上运行多用户或多会话的系统。反例：单用户本地工具，每次会话 token 预算充足，分层带来的工程复杂度超过收益。

**运行时条件判断必须移到缓存边界之后**：每增加一个条件，缓存变体翻倍，达到阈值后缓存实际上失效。适用条件：有全局缓存作为成本控制手段的系统。反例：每次请求都有用户特定的 API key，不存在全局缓存，条件放哪里都不影响。

**临时补丁必须打标记**：`@[MODEL LAUNCH]` 类型的标记让补丁在日后变得可见、可复评。适用条件：提示词会随模型版本演进持续维护的系统。
