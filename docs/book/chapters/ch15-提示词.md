# 第15章 提示词与任务指令

`src/constants/prompts.ts` 里有一段注释，带着 `@[MODEL LAUNCH]` 标记：

```
// Internal data shows Capybara v8 has a 29–30% incorrect-statement rate
// (vs 16.7% for v4). Explicit prohibitions:
// - Do not claim tests pass when they fail
// - Do not hide failing checks to manufacture success
// - Do not describe incomplete work as complete
```

29-30% 的错误陈述率不是"模型有时会撒谎"，而是"模型在近三分之一的情况下会撒谎"。在这个基线上，不加专项指令约束的系统是不可用的。

这就是为什么系统提示词里需要"如实报告结果"的强制性指令，而不是依赖模型的自觉。

## 15.1 系统提示词的静态/动态分割

系统提示词的内容可以分成两类：在不同会话之间不变的内容（工具描述、基础约束、通用行为规范），和在不同会话之间可能变化的内容（当前工具列表、运行模式说明、权限边界、当前环境信息）。

如果把这两类内容混在一起，每次会话都需要重新计算整个提示词的 token，无法利用 API 的缓存机制。如果把它们分开，静态部分可以跨会话缓存，只有动态部分需要每次重新计算。

`src/constants/prompts.ts#L106` 定义了 `SYSTEM_PROMPT_DYNAMIC_BOUNDARY`，注释写得很直接：这个边界标记之前的内容可以使用 `scope: 'global'` 缓存，之后的内容包含用户/会话特定信息，不应该被缓存。

注释还特别警告：不要在不更新缓存逻辑的情况下移除或重排这个边界标记（需要同步更新 `src/utils/api.ts` 的 `splitSysPromptPrefix` 和 `src/services/api/claude.ts` 的 `buildSystemPromptBlocks`）。这说明静态/动态分割不只是提示词内部的约定，而是和 API 调用层深度耦合的设计。

这个耦合是一个隐式约束——修改提示词结构时，需要同时修改 API 调用层，但这个依赖关系在代码里不显眼。如果未来需要修改分割点，容易遗漏 API 调用层的同步修改。

## 15.2 MCP 指令为什么不放在 system prompt 里

MCP 服务器相关的指令没有放在 system prompt 里，而是通过 message 的增量附加（delta attachments）传递。

原因是：如果把 MCP 指令放在 system prompt 里，每次有新的 MCP 服务器连接，system prompt 就会变化，导致整个静态前缀的缓存失效。把 MCP 指令移到 message 层，system prompt 保持不变，缓存命中率不受 MCP 连接状态影响。

这是一个为了保护缓存而做的架构决策，不是随意的实现选择。

## 15.3 系统提示词里的模型版本补丁

系统提示词不只是角色设定，还包含针对特定模型版本行为问题的专项指令。

除了 29-30% 错误陈述率的补丁，还有一个关于注释的补丁：Capybara 模型有过度添加注释的倾向，因此提示词明确要求"只有当 WHY is non-obvious 时才允许添加注释"。

这和 ch07 里的 `yieldMissingToolResultBlocks()` 是同一类问题的两种处理层次：`yieldMissingToolResultBlocks()` 在消息构建层用代码修复模型的格式问题，系统提示词里的版本补丁在指令层用自然语言修复模型的行为问题。两者都是"针对特定模型版本的临时修复"，都会随着模型版本演进而需要清理。

版本专属补丁散落在系统提示词里，没有统一的标注格式说明"这段指令是针对哪个模型版本的临时修复"。随着版本迭代，这类补丁会积累，清理时需要逐段判断是否仍然必要。没有标注的代价，在模型版本更替时才会显现。

## 15.4 内外部用户的提示词差异

Claude Code 对 Anthropic 内部用户（通过 `isAnthropicInternalUser()` 判断）和外部用户注入的系统提示词内容不完全相同。内部用户可能获得额外的调试指令、更宽松的行为约束，或者针对内部工具的专项说明。

这个差异不是功能差异，而是信任边界的体现：内部用户被假设理解系统的工作方式，不需要某些面向外部用户的保护性约束。这个设计假设"内部用户"和"外部用户"的边界是清晰的，可以通过账号属性可靠判断。如果这个判断出错（比如内部账号被外部人员使用），内部用户的宽松约束会被错误地应用。

## 15.5 任务指令和系统提示词的区别

系统提示词是角色设定，任务指令是具体任务。前者在会话开始时设定，后者在每次用户提交时给出。

一个常见的错误是把任务指令塞进系统提示词。比如"你是一个数据分析助手，你的任务是分析用户上传的 CSV 文件，找出异常数据，生成报告"。这里"你是一个数据分析助手"是角色设定，应该在系统提示词里；"分析用户上传的 CSV 文件，找出异常数据，生成报告"是具体任务，应该在用户消息里。

把任务指令放在系统提示词里的问题是：系统提示词在整个会话里是固定的，但任务会变。用户这次要分析 CSV，下次可能要分析 JSON，如果任务写死在系统提示词里，灵活性就没了。系统提示词回答"这个 Agent 是什么"，任务指令回答"这次要做什么"。两者的生命周期不同，不应该混用。

## 15.6 命令系统的设计

`src/commands.ts#L481` 的 `getCommands(cwd)` 管理的不只是工具，还有斜杠命令（slash commands）。斜杠命令是用户可以直接输入的快捷指令，比如 `/compact`、`/clear`、`/help`。

斜杠命令和工具的区别在于：工具是模型调用的，斜杠命令是用户直接触发的。`commands.ts#L420` 的 `meetsAvailabilityRequirement()` 和 `isCommandEnabled()` 在每次 `getCommands()` 调用时重新评估，所以登录状态变化后，命令列表可以及时变化。

## 15.7 提示词的版本管理

提示词是代码，需要版本管理。这一点在业务 Agent 里经常被忽视，导致"改了提示词，行为变了，但不知道改了什么"。

Claude Code 的系统提示词通过代码构建（[`constants/prompts.ts#L444`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/constants/prompts.ts#L444) 的 `getSystemPrompt()`），自然纳入了版本管理。对业务 Agent 来说，最简单的做法是把系统提示词存在一个独立文件里，和代码一起提交。每次修改提示词，都有对应的 commit 记录，出问题时可以回溯。

## 15.8 心智模型验证题

如果把 `SYSTEM_PROMPT_DYNAMIC_BOUNDARY` 删掉，把静态前缀和动态后缀合并成一个字符串，会发生什么？

答案：API 调用层无法找到分割点，无法把静态前缀标记为可缓存。每次会话都需要重新计算整个系统提示词的 token，无法利用缓存。对于频繁调用的场景，这会显著增加 API 成本和延迟。

## 15.9 本章小结

系统提示词被明确分成静态前缀（可缓存）和动态后缀（不可缓存）两段（[`prompts.ts#L106`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/constants/prompts.ts#L106)），这个分割和 API 调用层深度耦合，不是随意的代码组织。系统提示词里的版本补丁（Capybara v8 的 29-30% 错误陈述率）说明提示词不只是角色设定，还是针对特定模型版本行为问题的修复层。系统提示词设定角色和约束，任务指令描述具体任务，两者不要混用。
