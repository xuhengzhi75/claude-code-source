# 第7章 Query Loop：真正让系统跑起来的运行时引擎

结论先说：`src/query.ts` 不是普通工具文件，而是 Claude Code 的运行时状态机。模型调用、工具执行、继续判断、压缩、恢复和预算控制，都在这里汇合。

如果没有这层循环，系统只能完成一次回答。一旦任务跨多轮推进，就必须回答这些问题：何时继续、何时结束、何时进入恢复分支、何时因为预算或上下文长度收口。`query.ts` 处理的就是这套运行时控制问题。

## 7.1 为什么它不是简单的 while 循环

从 `src/query.ts` 可以直接看到两个结构锚点：`type State` 和 `while (true)` 主循环。

这说明这里不是零散分支集合，而是显式维护状态的循环内核。`State` 里保存的不只是消息，还包括 compact/recovery 相关状态、turn 计数、pending summary，以及上一轮为什么继续的 `transition` 字段。

## 7.2 继续条件为什么更看 `tool_use`

这一章最关键的实现点，是系统判断是否继续下一轮时，不只看 `stop_reason`，而是更看真实流里有没有出现 `tool_use`。

这个选择背后的逻辑很硬：不同 provider 和 SDK 对 stop reason 的语义并不完全稳定，但 `tool_use` 是否真的出现，是更可观察的事实。

所以 Claude Code 在这里优先相信发生了什么，而不是完全相信模型怎么描述自己停下来的原因。

## 7.3 为什么压缩、恢复和预算控制都挂在主循环里

从 `query.ts` 能直接看到多种运行时机制都在主循环内部接入，例如：auto compact、reactive compact、context collapse、max output token recovery、task budget / token budget。

这说明 Claude Code 没把上下文太长、输出被截断、预算快耗尽当成边缘异常，而是把它们当成长流程中的常见情况。于是相关分支不是事后补丁，而是主循环的一部分。

## 7.4 它为什么直接决定系统稳定性

主循环的设计会直接影响这些结果：工具调用后能不能稳定回到下一轮，prompt 太长时能不能继续而不是直接崩掉，输出打满时能不能进入恢复分支，到达终态时能不能给出清楚的结束 reason。

从代码结构看，这里的终态也不是单一成功/失败，而是多种 `return { reason: ... }` 分支。系统不是只会跑完或报错，而是在不同终止面之间明确分流。

## 7.5 对自己做 Agent 最值得学什么

如果你要做一个缩小版 Agent，这一层最值得学的是三条原则：继续条件必须写成可观察规则，不能只靠模型口头说做完了；异常要按常态设计，尤其是超长上下文、输出截断、预算限制；终态要有明确 reason，而不是统一糊成一个失败或结束。

这三条比工具接得多不多更决定系统能不能稳定工作。

## 7.6 本章小结

`src/query.ts` 的核心价值，在于它把模型、工具、压缩、恢复和预算控制组织成了一套可持续推进的状态机循环。

理解这一层后，下一章看任务系统会更顺，因为长任务并不是绕开主循环独立存在，而是要和主循环的继续、恢复和终态逻辑接起来。
