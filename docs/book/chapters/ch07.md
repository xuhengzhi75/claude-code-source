# 第7章 Query Loop：真正让系统跑起来的运行时引擎

前一章讲 `QueryEngine` 是会话编排核心。
这一章要看更靠内核的位置：`src/query.ts`。

如果把 Claude Code 当成一台工作机器，那么 `QueryEngine` 负责把任务送上轨道，`query.ts` 负责让这台机器一轮一轮真正跑下去。

这也是为什么很多复杂行为最后都会汇聚到这里：
模型调用、工具执行、继续判断、压缩、恢复、预算控制、异常分支，都会在这层碰头。

## 7.1 为什么真正的 Agent 一定要有运行时循环

一次模型回答，并不等于一次任务完成。

真实任务通常要经历很多轮：

- 先读用户输入
- 看上下文
- 选择工具
- 执行工具
- 吃回工具结果
- 再决定下一步
- 直到任务结束

如果系统没有一个明确的运行时循环，这些步骤就会散落在各处，最后很难维护。

所以成熟 Agent 的核心通常不是“会不会调模型”，而是：
**有没有一套稳定的循环结构，把模型、工具和状态组织成连续工作。**

## 7.2 `query.ts` 在系统中的角色

从仓库结构看，`src/query.ts` 不是普通工具函数文件，而是高密度控制流文件。

前面 `QueryEngine.submitMessage()` 会把执行权交给这里。
进入这层后，系统开始处理一轮又一轮的执行推进。

可以把它理解成一个状态机式循环：

- 准备当前轮的上下文
- 发起模型请求
- 读取流式输出
- 识别工具调用
- 执行工具
- 回写工具结果
- 判断是否继续
- 必要时进入压缩、恢复或预算控制分支

这就是 Claude Code 真正的运行时引擎。

## 7.3 它为什么比普通“while 循环”复杂得多

很多 demo 也会写一个 while 循环。
但产品级系统和 demo 的区别，在于它要处理很多失败模式和边界条件。

从 `src/query.ts` 可以看到，这里不只是“循环执行”，还要同时面对：

- prompt 太长
- output token 打满
- 历史需要裁剪
- 内容需要 compact
- 工具结果可能很多
- provider 行为存在差异
- 某些错误需要恢复而不是直接终止
- 预算和 turn 数需要控制

因此这里更像“运行时调度内核”，而不是简单循环体。

## 7.4 一个非常关键的设计：看真实 `tool_use`，不只看 stop reason

这一层里有一个很值得记住的设计点。

Claude Code 判断是否需要继续下一轮时，并不单纯依赖 `stop_reason`，而是更看重真实流里有没有出现 `tool_use`。

这背后的逻辑很务实：
不同 provider、不同 SDK 实现，对 stop reason 的表现可能不完全稳定。
如果系统过度依赖这些外部语义，行为就容易飘。

所以 Claude Code 更倾向于根据“真实发生了什么动作”来判断，而不是只根据模型给出的结束标签判断。

这是一种典型的工程思路：
**优先相信可观察事实，不优先相信外部语义承诺。**

这类细节，往往就是系统稳定性的来源。

## 7.5 为什么压缩、恢复、预算控制都挂在这里

从代码结构看，`query.ts` 同时接入了很多 compact 与 recovery 相关机制，例如：

- auto compact
- reactive compact
- history snip
- context collapse
- max output token recovery
- task budget / token budget 控制

这说明 Claude Code 没把“上下文过长”“中途中断”“输出被截断”当成例外，而是当成运行时的一部分来处理。

这很重要。
因为真正长流程里，这些问题不是偶发，而是常态。

一个成熟系统不会假设“每轮都顺顺利利”。
它会假设：

- 有些轮次会超长
- 有些响应会被截断
- 有些上下文必须被收缩
- 有些状态必须被修复后才能继续

所以这些分支必须直接放进主循环里，而不是事后打补丁。

## 7.6 运行时循环为什么决定系统上限

很多人理解 Agent 时，容易把上限归因于模型能力。
其实当任务变长后，真正决定体验上限的往往是运行时循环。

因为它决定：

- 工具调用后能不能稳定接回主流程
- 错误发生后是直接崩掉还是能恢复
- 长上下文时是变慢、变乱，还是还能继续跑
- 一轮结束后能不能清楚判断要不要继续

从这个角度看，`query.ts` 不是“细节文件”，而是 Claude Code 的发动机舱。

模型像发动机燃料的一部分，但真正把燃料变成连续运动的，是这套运行时机构。

## 7.7 对自己做业务 Agent 的启发

如果你要做一个简化版业务 Agent，这一层最值得学的不是所有复杂机制，而是三件基础原则。

### 第一，把“继续条件”写清楚

系统不能只靠模型说“我做完了”。
你需要明确：

- 什么情况下继续
- 什么情况下结束
- 什么情况下进入恢复分支

### 第二，把异常当常态

不要把超长上下文、失败重试、输出截断都当成边缘问题。
只要任务流程足够长，它们迟早会出现。

### 第三，主循环尽量围绕可观察状态运转

尽量根据：

- 是否出现工具调用
- 是否收到工具结果
- 是否命中预算限制
- 是否进入恢复边界

这类事实状态做判断，而不是只靠模型口头语义。

## 7.8 本章小结

这一章的核心结论是：

**`src/query.ts` 是 Claude Code 的运行时引擎，它把模型、工具、恢复、压缩和预算控制组织成一个可持续推进的状态机循环。**

这也是 Claude Code 和很多“只会调一次模型”的简单 Agent 最大的区别之一。

理解这一层之后，下一章再看任务系统，就更容易明白：为什么长任务不是靠一轮循环硬扛，而要有更上层的任务与持续执行结构来接住。
