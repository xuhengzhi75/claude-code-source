# 第3章 从入口到执行：主干是怎么跑起来的

`claude --version` 为什么几乎瞬间返回？`claude ps` 为什么不初始化 Agent 系统？入口层的设计回答这两个问题。

## 3.1 --version 快，是因为提前退出了

[`cli.tsx#L40`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/entrypoints/cli.tsx#L40) 的代码：

```typescript
// Fast-path for --version/-v: zero module loading needed
if (args.length === 1 && (args[0] === '--version' || args[0] === '-v')) {
  console.log(`${MACRO.VERSION} (Claude Code)`)
  return
}
```

`MACRO.VERSION` 是构建时内联的字符串常量，不是运行时读取的。这个分支执行完，进程退出。没有配置文件读取，没有模块加载，没有 Keychain 操作。

完整主流程的对比：初始化配置、加载插件列表、建立 GrowthBook 连接（GrowthBook 是特性开关服务，用于运行时控制功能可见性），从毫秒变成秒级。`--version` 之所以快，是因为它在任何重操作发生之前就已经返回了。

## 3.2 走哪条路，只加载哪条路的模块

`cli.tsx` 的 `main()` 是一棵从上到下的 if-else 路由树。每个分支用 `await import(...)` 动态加载当前路径需要的模块。没有命中的分支，对应模块不被加载。

```mermaid
flowchart TD
    A([用户执行 claude ...]) --> B{"--version / -v?"}
    B -->|是| C["打印版本号 · 零模块加载 · 直接退出"]
    B -->|否| D{"--dump-system-prompt?"}
    D -->|是| E["加载配置 + prompt 系统 · 打印后退出"]
    D -->|否| F{"Chrome 集成?"}
    F -->|是| G["Chrome 集成路径"]
    F -->|否| H{"--daemon-worker?"}
    H -->|是| I["启动 Worker 进程 · 性能敏感路径"]
    H -->|否| J{"daemon 子命令?"}
    J -->|是| K["启动长期 Supervisor 进程"]
    J -->|否| L{"remote-control / bridge?"}
    L -->|是| M["远程控制模式"]
    L -->|否| N{"ps / logs / attach / --bg?"}
    N -->|是| O["后台会话管理 · 不进入 Agent 系统"]
    N -->|否| P["完整主流程 · main.tsx 初始化装配"]
    P --> Q["launchRepl 或 runHeadless"]
    Q --> R["QueryEngine.submitMessage"]
    R --> S["query.ts queryLoop"]
```

`claude ps` 命中 [`cli.tsx#L189`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/entrypoints/cli.tsx#L189) 的 fast-path，直接处理后返回。不初始化 Agent 系统，不加载工具列表，不建立 API 连接。路由的作用是让每条请求路径的加载范围与自身职责对齐。

## 3.3 顺序颠倒，进程身份就错了

[`cli.tsx#L99`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/entrypoints/cli.tsx#L99) 的注释：

```typescript
// Fast-path for `--daemon-worker=<kind>` (internal — supervisor spawns this).
// Must come before the daemon subcommand check: spawned per-worker, so
// perf-sensitive.
if (feature('DAEMON') && args[0] === '--daemon-worker') { ... }

// Fast-path for `claude daemon [subcommand]`: long-running supervisor.
if (feature('DAEMON') && args[0] === 'daemon') { ... }
```

`--daemon-worker` 是 supervisor 内部生成的参数，用来启动每个 worker 进程。如果把这两个检查的顺序交换，worker 进程启动时会先命中 `daemon` 分支，被当成 supervisor 来处理，整个 daemon 系统的进程身份就错了。

路由树的分支顺序不是代码整洁问题，每个位置对应一个具体的运行时约束。注释记录了这个约束的原因，这类注释的作用不是解释代码做了什么，而是解释为什么不能改成另一种写法。

## 3.4 提取成变量，死代码消除就失效了

路由树里到处有 `feature('DAEMON')`、`feature('BRIDGE_MODE')` 这样的检查，没有提取成共享变量。[`cli.tsx#L114`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/entrypoints/cli.tsx#L114) 的注释：

```typescript
// feature() must stay inline for build-time dead code elimination
if (feature('BRIDGE_MODE') && ...) { ... }
```

`feature()` 是构建时开关。编译器遇到 `if (feature('BRIDGE_MODE') && ...)` 时，如果 `BRIDGE_MODE` 在构建配置里关闭，整个 `if` 块连同其中的 `import` 语句一起从产物里删除。

如果改写成 `const bridgeEnabled = feature('BRIDGE_MODE')`，再用 `if (bridgeEnabled && ...)`，编译器在静态分析阶段无法推断 `bridgeEnabled` 在这个 `if` 里的值，死代码消除失效。被关闭的功能代码仍然出现在产物里，增加包体积，并且把未发布功能的实现暴露在可访问的代码中。

这是"代码看起来可以重构，但重构会破坏构建期优化"的情况。代码重复本身不是问题，破坏构建时约束才是。

## 3.5 所有路径最终都汇到同一个入口

没有命中任何 fast-path 的请求进入完整主流程。`main.tsx` 完成初始化和装配，然后按模式分流：交互模式走 `launchRepl`，非交互模式走 `runHeadless`，两条路都到 [`QueryEngine.ts#L213`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/QueryEngine.ts#L213) 的 `submitMessage()`，再进入 [`query.ts#L243`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/query.ts#L243) 的 `queryLoop()`。

`main.tsx` 的装配细节在第4章。`QueryEngine` 的会话编排在第6章。`queryLoop()` 的状态机逻辑在第7章。本章覆盖范围到"路径如何收敛到 `submitMessage()`"为止。

## 3.6 路由前置不是万能的，有个前提

把路由放在最前面，轻操作不为重操作付出启动成本。这个结论有一个前提：轻操作在路由阶段就能获得自己需要的全部信息。

如果轻操作需要读取某个只有在完整初始化后才存在的状态，路由前移的前提就不成立了。这时候要么把所需状态提前到路由层可访问的位置，要么接受轻操作也要走完整初始化。两种方案都有代价，不存在自动成立的最优解。[inference]

原则是工具，不是教条。
