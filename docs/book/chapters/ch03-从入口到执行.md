# 第3章 从入口到执行：Claude Code 主干是怎么跑起来的

> **核心主张：Claude Code 的主干不是一条直线，而是一个先分流、再按需加载的路由树——这个设计解决的第一性问题是"启动成本与功能完整性的矛盾"：轻命令不应该为重功能付出启动代价。**

## 3.1 第一性问题：启动成本与功能完整性的矛盾

一个功能完整的 Agent 系统，启动时需要加载大量模块：工具系统、权限系统、会话管理、历史记录、压缩系统……但用户有时只是想查看版本号（`--version`）或者列出后台任务（`ps`）。

这产生了一个矛盾：**功能完整性要求加载所有模块，启动速度要求只加载必要模块。**

解决这个矛盾有两种思路：

**思路一：懒加载（lazy loading）。** 所有模块都在需要时才加载，不需要时不加载。问题是：懒加载需要在每个使用点判断模块是否已加载，增加了代码复杂度，也可能导致首次使用时有延迟。

**思路二：路由分流（routing）。** 在入口处判断请求类型，不同类型走不同路径，每条路径只加载自己需要的模块。问题是：需要在入口处维护路由逻辑，路由逻辑本身可能变得复杂。

Claude Code 选择了路由分流，并且把路由逻辑集中在 `src/entrypoints/cli.tsx` 的 `main()` 函数里。

## 3.2 主干路径：从参数到执行循环

把主干压成一条线：

1. `src/entrypoints/cli.tsx#L33` 的 `main()` 接住命令行参数，先判断是否命中 fast-path。
2. 没命中 fast-path 才进入完整主程序 `src/main.tsx`。
3. `main.tsx` 完成初始化、参数解析、命令与工具装配。
4. 根据模式进入交互或非交互路径（`launchRepl` 或 `runHeadless`）。
5. 具体执行由 `QueryEngine.submitMessage()`（[`QueryEngine.ts#L213`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/QueryEngine.ts#L213)）驱动。
6. `QueryEngine` 调用 `query()`，进入 `src/query.ts#L243` 的 `queryLoop()` 状态机。
7. 循环里反复执行"调用模型 → 执行工具 → 决定继续或收尾"，直到返回结果。

这条线的关键不是步骤数量，而是每一步的职责边界：`cli.tsx` 只做路由，`main.tsx` 只做装配，`QueryEngine` 只做会话编排，`query.ts` 只做循环推进。

## 3.3 fast-path 的隐含假设

`cli.tsx` 的 fast-path 分支（`--version` 在 [`L41`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/entrypoints/cli.tsx#L41)、`daemon` 在 [`L169`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/entrypoints/cli.tsx#L169)、`--bg` 在 [`L189`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/entrypoints/cli.tsx#L189)）有一个隐含假设：**这些路径的需求是稳定的，不会随着系统功能的增加而改变。**

如果这个假设不成立（比如 `--version` 命令需要查询远程服务器获取最新版本），fast-path 就需要加载更多模块，路径隔离的优势就会减弱。

**反直觉的地方：** `--daemon-worker` 的检查必须在 `daemon` 子命令检查之前（[`cli.tsx#L104`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/entrypoints/cli.tsx#L104)）。原因是 `--daemon-worker` 是 supervisor 内部生成的参数，用于启动每个 worker 进程；如果顺序反了，worker 进程会被误判为 daemon 子命令。这说明路由逻辑的分支顺序不是随意的，每个分支的位置都有对应的约束。

## 3.4 命令动态生成的代价

`src/commands.ts#L481` 的 `getCommands(cwd)` 每次调用都会重新做可用性判断，而不是在启动时生成一次固定的命令列表。

**优势：** 登录状态变化后，命令列表可以及时变化。用户登录后立刻能看到需要登录才能使用的命令。

**代价：** 每次需要命令列表时都要重新计算，有一定的性能开销。Claude Code 用 `memoize`（[`commands.ts#L351`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/commands.ts#L351)）缓解了这个问题，但缓存失效的时机需要仔细管理。

**技术债：** 如果命令来源越来越多（内建命令、skill 命令、插件命令、workflow 命令），`getCommands()` 的计算成本会线性增长。当前的 memoize 策略在命令来源稳定时有效，在命令来源频繁变化时会频繁失效。

## 3.5 继续条件：为什么不依赖 `stop_reason`

`src/query.ts#L561` 的注释解释了一个关键设计决策：`needsFollowUp` 由是否出现 `tool_use` 块驱动，而不是依赖 `stop_reason`。

**为什么不用 `stop_reason`？** 不同 provider 和 SDK 版本的 `stop_reason` 细节不总一致。有些 provider 在流式输出时延迟发送 `stop_reason`，有些 provider 的 `stop_reason` 值和文档描述不完全一致。

**用 `tool_use` 的优势：** `tool_use` 是模型输出的内容块，不是元数据。它的存在是事实，不是声明。"模型产生了工具调用"比"模型说它还没完成"更可靠。

**可迁移原则：** 用事实（输出内容）而不是声明（元数据）来判断系统状态。声明可能不准确，事实不会撒谎。

## 3.6 职责边界的价值

从源码结构看，Claude Code 的稳定性靠这条主干把职责拆开：

- `cli.tsx`：路由，不做业务
- `main.tsx`：装配，不做执行
- `QueryEngine.ts`：会话编排，不做循环推进
- `query.ts`：循环推进，不做会话管理

这种职责分离让每一层可以独立测试、独立修改。修改路由逻辑不需要理解执行循环，修改执行循环不需要理解会话管理。

**边界条件：** 职责分离的前提是层与层之间的接口稳定。如果 `QueryEngine` 和 `query.ts` 之间的接口频繁变化，职责分离反而会增加维护成本（每次接口变化都需要同时修改两层）。

## 3.7 心智模型验证题

读完这章，你应该能回答：用户运行 `claude ps` 查看后台任务列表，这个请求会走到 `query.ts` 的 `queryLoop()` 吗？

答案：不会。`claude ps` 命中了 `cli.tsx#L189` 的 fast-path，直接处理后返回，不进入完整主流程，更不会进入 `queryLoop()`。

## 3.8 本章小结

Claude Code 的主干是先分流（[`cli.tsx#L33`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/entrypoints/cli.tsx#L33)）、再装配（`main.tsx`）、后循环执行（[`QueryEngine.ts#L213`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/QueryEngine.ts#L213) + [`query.ts#L243`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/query.ts#L243)）。这个设计解决的是启动成本与功能完整性的矛盾，代价是路由逻辑需要集中维护，分支顺序有约束。

需要记住的核心概念：**路由分流让每条路径只为自己的需求付出成本，这是系统可扩展的前提。**
