# 第3章 从入口到执行：Claude Code 主干是怎么跑起来的

这一章只做一件事：把 Claude Code 的主干路径讲清楚。

我们关心的问题很简单：
当你在命令行里输入一次请求后，系统到底按什么顺序把它变成一次可执行的任务循环。

为了可读性，下面会尽量少讲细节变量名，但每个结论都对应源码位置。

## 先看总路线

把主干压成一条线，大致是：

1. `src/entrypoints/cli.tsx` 接住命令行参数，先判断是否命中 fast-path。
2. 没命中 fast-path 才进入完整主程序 `src/main.tsx`。
3. `main.tsx` 完成初始化、参数解析、命令与工具装配。
4. 根据模式进入交互或非交互路径（`launchRepl` 或 `runHeadless`）。
5. 具体执行由 `QueryEngine.submitMessage()` 驱动。
6. `QueryEngine` 调用 `query()`，进入 `src/query.ts` 的状态机循环。
7. 循环里反复执行“调用模型 → 执行工具 → 决定继续或收尾”，直到返回结果。

这条线就是“从入口到执行”的主干。

## 3.1 入口层先做分流

在 `src/entrypoints/cli.tsx`，`main()` 一开始不会立刻加载整个系统。
它先检查参数是否属于快速路径，例如 `--version`、`daemon`、`remote-control`、`--bg` 等。

源码里可以看到，这些分支大量使用动态 `import(...)`，只有在需要时才加载对应模块。
如果都不命中，才会走到最后的 `import('../main.js')`。

对读者来说，这一步的意义是：
- 轻操作响应更快。
- 特殊模式和日常会话分开，互不拖累。

## 3.2 进入 main.tsx 后，先“把场地搭好”

`src/main.tsx` 的职责很多，但可以按顺序理解：

第一类是启动与安全相关准备，例如：
- 初始化告警处理。
- 判断交互/非交互模式（如 `-p/--print`）。
- 提前处理部分特殊参数改写。

第二类是命令与能力装配：
- 通过 `getCommands(cwd)` 生成当前可见命令集。
- 通过 `initializeToolPermissionContext(...)` 和 `getTools(...)` 生成工具能力。

第三类是把会话送到正确执行通道：
- 非交互模式进入 `runHeadless`。
- 交互模式进入 `launchRepl`。

这意味着 `main.tsx` 更像“装配与路由中心”，而不是最终执行循环本体。

## 3.3 命令是动态生成的，不是固定清单

`src/commands.ts` 里，命令来源会被合并：
- 内建命令
- skill 目录命令
- 插件命令
- workflow 命令
- 动态发现的 skill 命令

随后再做过滤：
- `meetsAvailabilityRequirement(cmd)`：根据账号/服务条件过滤。
- `isCommandEnabled(cmd)`：根据启用状态过滤。

`getCommands(cwd)` 每次调用都会重新做可用性判断，所以登录状态变化后，命令列表可以及时变化。

这和“启动时写死一份命令表”是不同的。

## 3.4 工具也走“先全集、再过滤”

`src/tools.ts` 把工具管理分成两步：

1. `getAllBaseTools()` 给出理论可用全集。
2. `getTools(permissionContext)` 按模式和权限过滤，得到本次可用集合。

关键点有两个：
- `filterToolsByDenyRules(...)` 会先剔除被拒绝规则命中的工具。
- `assembleToolPool(...)` 把内建工具和 MCP 工具合并，并在重名时保持内建优先。

这让工具能力既可扩展，又能保持权限边界清晰。

## 3.5 真正执行从 QueryEngine 开始

在执行阶段，核心入口是 `src/QueryEngine.ts` 的 `submitMessage(...)`。

它做的事情可以概括为：

- 准备上下文：system prompt、user/system context、权限包装。
- 处理输入：先把用户输入及相关消息并入 `mutableMessages`。
- 做持久化：在进入主循环前就写 transcript，减少中断丢失。
- 调用 `query(...)`：把执行权交给循环状态机。
- 消费循环输出：把 assistant/user/progress/attachment 等消息转成 SDK 输出，并持续记账（usage、cost、stop reason）。

这里可以看到一条很清楚的边界：
- QueryEngine 负责会话级编排、持久化和结果汇总。
- `query.ts` 负责一轮轮推进。

## 3.6 query.ts 的循环怎么决定“继续还是结束”

`src/query.ts` 中的 `queryLoop(...)` 是一个 `while (true)` 状态机。

每次迭代都做类似流程：

1. 准备当前消息与上下文（包括压缩/裁剪相关处理）。
2. 调模型流式输出。
3. 收集本轮 `tool_use`。
4. 执行工具并写回结果。
5. 判断是否继续下一轮。

其中一个非常关键的实现细节是：
`needsFollowUp` 主要由是否出现 `tool_use` 块驱动，而不是仅靠 `stop_reason`。

这点在代码注释里写得很直接：这样可以降低不同 provider/SDK 下 stop_reason 差异带来的不稳定。

所以主干判断逻辑更接近“看事实动作是否发生”，而不是“听模型口头说结束了没有”。

## 3.7 为什么这条主干对稳定性很重要

从源码结构看，Claude Code 的稳定性并不是靠单点技巧，而是靠这条主干把职责拆开：

- 入口分流（`cli.tsx`）控制启动成本。
- 装配层（`main.tsx`、`commands.ts`、`tools.ts`）控制可见能力。
- 执行层（`QueryEngine.ts`、`query.ts`）控制循环推进与收尾。

这让系统在长流程里更容易做到三件事：
- 不中断时持续推进。
- 中断后能恢复到可继续状态。
- 权限和工具边界可追踪。

## 3.8 本章小结

你可以把这一章记成一句话：

Claude Code 的主干是“先分流，再装配，后循环执行”。

从源码上看，入口在 `cli.tsx`，执行核心在 `QueryEngine.ts + query.ts`，中间由 `main.tsx / commands.ts / tools.ts` 把命令和工具组织好。

理解这条主干后，后面看任务系统、恢复机制、上下文压缩就会更容易，因为它们都挂在这条主链上。
