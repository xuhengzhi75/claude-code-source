# 第16章 状态和上下文是两件不同的事

Claude Code 对这两者的区分很严格。状态是系统在某个时刻的内部情况；上下文是模型在当前轮次能看到的信息。两者的混淆导致系统在中断恢复时行为不可预测——恢复时状态加载了，但上下文没有正确重建，模型看不到它需要看到的信息。

## 16.1 集中状态对象：为什么不用多个局部变量

[`src/query.ts#L282`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/query.ts#L282) 的 `queryLoop()` 初始化了一个 `State` 对象，把所有跨迭代的可变状态集中在一处：`messages`、`toolUseContext`、`maxOutputTokensOverride`、`hasAttemptedReactiveCompact`、`turnCount`。

注释里解释了为什么用单一对象：每个恢复/继续分支可以原子替换所有循环变量，避免新增字段时遗漏更新。

这是反直觉的。通常我们认为局部变量更清晰，因为每个变量只在需要的地方存在。但在有多个恢复路径的系统里，局部变量意味着每条恢复路径都需要逐个修改每个变量。加一个新字段时，需要记得在所有恢复路径里同步处理。集中状态对象让恢复路径可以整体替换状态，不会因为漏掉某个字段而产生部分恢复的不一致状态。

代价是：随着系统演进，`State` 对象会越来越大。每次加字段都有充分的局部理由，但积累到一定程度，`State` 变成什么都往里放的容器，失去集中管理的意义。这类膨胀没有触发点，只能靠定期专项清理。

## 16.2 上下文的三层结构

每一轮执行开始时，`QueryEngine` 组装当前轮次的上下文。上下文不是把所有信息平铺进去，而是有层次的。

第一层是系统上下文，一启动就注入，整个会话期间不变：当前 Git 仓库状态、今天的日期、用户的 CLAUDE.md 配置文件内容。`context.ts` 的 `getSystemContext()` 和 `getUserContext()` 都用了 `memoize`，这些上下文只计算一次，之后直接从缓存取。

第二层是对话历史，每一轮用户输入、AI 回复、工具调用结果都在这里。增长最快，对话越长占用越多，是压缩管理的主要目标。

第三层是记忆文件，把重要信息存到文件里，需要时再读取。这是 Claude Code 跨会话保留信息的机制。

这三层的注入顺序对模型行为有影响。系统提示词在最前面，权重最高；最近的对话在最后面，模型对近期内容的注意力更强 [inference: consistent with general LLM attention patterns]。上下文的顺序和格式，是提示词工程里最常被忽视的变量之一。

## 16.3 hasAttemptedReactiveCompact：一个 flag 背后的无限循环 bug

[`src/query.ts`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/query.ts) 的 `State` 对象里有一个 `hasAttemptedReactiveCompact` flag（[`query.ts#L1089`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/query.ts#L1089)）。它的作用：防止压缩触发无限循环。

真实的失败场景：上下文过长，触发反应式压缩，压缩后仍然过长，再次触发压缩，再压缩还是太长…循环不停，API 调用不停，token 烧完。

`hasAttemptedReactiveCompact` 在第一次触发压缩时设为 true，之后再遇到过长错误，不再尝试压缩，而是让错误浮出水面。这是"fail-closed"的变体：与其陷入无限循环，不如让系统明显地失败。

这个 flag 是集中状态对象的一个具体价值体现：如果状态是局部变量，在某个恢复路径里重置这个 flag 是很自然的操作（"恢复了就清状态嘛"），但这会把系统带回无限循环的陷阱。把 flag 放在 `State` 对象里，恢复路径原子替换状态时，这个 flag 的值作为状态的一部分被正确传递。

## 16.4 会话记忆：为什么更新用受限子代理

[`src/services/SessionMemory/sessionMemory.ts#L460`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/services/SessionMemory/sessionMemory.ts#L460) 的 `createMemoryFileCanUseTool()` 声明了记忆更新子代理的权限边界：

```typescript
return {
  behavior: 'deny' as const,
  message: `only ${FILE_EDIT_TOOL_NAME} on ${memoryPath} is allowed`,
}
```

记忆更新子代理只能做一件事：用 `FileEdit` 工具编辑指定的 memory 文件。任何其他工具调用都被拒绝。

这个限制不是随意的。记忆更新是在后台异步运行的（[`sessionMemory.ts#L318`，`runForkedAgent`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/services/SessionMemory/sessionMemory.ts#L318)），如果它能访问任意工具，就可能意外触发文件写入、命令执行、网络请求，污染主会话状态或产生副作用。最小权限隔离把"记忆更新"和"执行任务"这两个关注点分开。

触发条件（[`sessionMemory.ts#L135-L166`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/services/SessionMemory/sessionMemory.ts#L135-L166)）是双重阈值：token 增量必须超过 `minimumTokensBetweenUpdate`，同时工具调用次数也必须超过 `toolCallsBetweenUpdates`。注释强调：token 阈值是必要条件，即使工具调用次数满足，也必须等 token 阈值满足才触发。单一阈值的问题是短对话里频繁触发，在什么都没发生多少的情况下就开始更新记忆；双重阈值要求对话必须有一定实质内容才会触发。

## 16.5 历史记录系统的大内容处理

`history.ts` 的 `addToHistory()` 对大内容有专门处理：小于 1024 字节直接存原文，大于 1024 字节存 Hash 值，原文另存到 `paste-store`。

这个模式解决了历史文件膨胀问题：用户粘贴了一段几千行的代码，历史文件不会存整段代码，只存哈希，需要时再从 paste-store 还原。

这个模式也是 Agent 系统里通用的状态压缩策略：大内容做摘要或哈希，只在需要时还原全量数据。对应到上下文管理：不是所有历史工具调用结果都要保留完整内容，超过一定长度的结果可以存摘要，完整内容按需加载。

历史记录本身跳过了一类特殊会话（`CLAUDE_CODE_SKIP_PROMPT_HISTORY` 为 true 时），这类会话是 Claude 自动创建的验证/测试会话，不应该污染用户的真实命令历史 [inference]。系统内部产生的操作记录和用户操作记录需要隔离。

## 16.6 状态持久化的时机

[`QueryEngine.ts#L443`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/QueryEngine.ts#L443) 在进入主循环之前，先把用户输入写入 transcript。

这个顺序是整个恢复机制能工作的前提：先落盘，再执行。"执行完成后保存"的系统在进程中途崩溃时，这一步的输入已经丢失，恢复时不知道上次执行到哪里。"执行前先落盘"的系统在进程崩溃后，输入已在磁盘上，`--resume` 知道从哪个点继续。

这不是"最终一致性"，是"执行前落盘"：副作用不可撤销的操作，必须在执行之前确保状态已被记录，而不是执行之后。

## 16.7 边界：什么情况下这些设计会退化

集中状态对象假设恢复路径可以正确实现"整体替换"。如果某条恢复路径只替换了部分字段（开发者忘了某个新加的字段），系统仍然会产生部分恢复的不一致状态，集中对象并不能在类型层面阻止这个错误。

记忆更新的双重阈值假设 token 计数是可靠的。如果 API 返回的 token 计数不准确（流式响应中途的计数可能有偏差），触发条件可能在错误的时机满足或不满足。

文件系统存储状态的方案在单机上工作良好，多机部署时失效。`sessionMemory.ts` 的 `runForkedAgent` 假设 fork 出去的子进程能访问同一个文件系统，在分布式场景下这个假设不成立。

## 16.8 可迁移原则

**集中状态对象优于分散局部变量**，适用条件：系统有多条恢复路径，且这些路径需要重置系统到一致的已知状态。反例：单条执行路径且不需要恢复的工具，局部变量的可读性更好。

**副作用操作在执行前落盘**，适用条件：任何需要支持中断恢复的系统。反例：纯内存操作、不需要持久化的临时会话。

**记忆更新用最小权限隔离执行**，适用条件：后台异步操作可能意外影响主执行流程的系统。反例：操作本身没有副作用、不可能污染主流程的情况。
