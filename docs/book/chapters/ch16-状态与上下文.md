# 第16章 状态和上下文是两件不同的事

**核心主张：状态集中管理和上下文分层注入，决定了中断恢复是否可靠。**

餐厅后厨有两类东西：备菜台上的食材（执行任务用的当下信息），和冷藏库里的存货（跨班次流转的持久状态）。厨师换班时，备菜台清空了，冷藏库还在。Claude Code 的设计沿着同样的边界切：上下文是每一轮执行时模型能看到的信息，状态是跨轮次跨会话持续存在的内部情况。两者混淆，中断恢复时就会出现"食材还在但厨师忘了做到哪步"的问题。

本章覆盖：状态集中管理策略、上下文三层结构、执行前落盘机制、记忆更新隔离设计，以及这些设计在哪些条件下会退化。

上下文窗口超限时的压缩处理（反应式压缩、上下文折叠）在第18章覆盖。任务系统的状态管理（任务 ID、阻塞图、认领原子化）在第17章覆盖。

## 集中状态对象让恢复路径原子替换

**单一 `State` 对象是 `queryLoop()` 的设计约束，不是编码风格选择。**

[`src/query.ts#L282`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/query.ts#L282) 的 `queryLoop()` 初始化了一个 `State` 对象，把所有跨迭代的可变状态集中在一处：`messages`、`toolUseContext`、`maxOutputTokensOverride`、`hasAttemptedReactiveCompact`、`turnCount`（turn：从用户发消息到 AI 回复完毕算一个执行单元）。

注释解释了为什么不用局部变量：每个恢复或继续分支可以原子替换所有循环变量，避免新增字段时遗漏更新。系统有多条恢复路径（中断重连、上下文压缩后继续、`--resume` 恢复），每条路径都需要把系统带回一致的已知状态。用局部变量，每条路径要逐字段修改；加一个新字段就要在所有路径里同步处理，漏掉一处就会产生部分恢复的不一致状态。集中对象让恢复路径整体替换状态，从结构上消除了"漏字段"的机会。

代价是对象随系统演进会膨胀。每次加字段都有充分的局部理由，但积累之后 `State` 变成什么都往里放的容器，失去集中管理的意义。这类膨胀没有触发点，只能靠定期专项清理。

### `hasAttemptedReactiveCompact` 是防无限循环的最后一道闸

这个 flag 的存在说明集中对象的价值超过了理论分析。

失败场景如下：上下文过长触发反应式压缩，压缩后仍然过长，再次触发压缩，循环不停，API 调用不停，token 烧完。[`src/query.ts#L1089`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/query.ts#L1089) 的 `hasAttemptedReactiveCompact` 在第一次触发压缩时置为 true，之后再遇到过长错误，系统不再尝试压缩，而是让错误浮出水面。这是 fail-closed（宁可明显失败也不陷入循环）的具体实现。

如果状态是局部变量，在某条恢复路径里重置这个 flag 是很自然的操作，但这会把系统带回无限循环的陷阱。集中对象让恢复路径整体替换状态时，这个 flag 的值作为状态的一部分被正确传递，不会被意外清零。

## 上下文三层结构决定模型每轮能看到什么

**`QueryEngine` 每轮组装上下文时，三层信息的注入顺序和缓存策略都是确定的设计，不是随机的实现细节。**

第一层是系统上下文，一启动就注入，整个会话期间不变：当前 Git 仓库状态、今天的日期、用户的 CLAUDE.md 配置文件内容。`context.ts` 的 `getSystemContext()` 和 `getUserContext()` 都用了 `memoize`（一个返回值缓存装饰器：相同输入只计算一次，后续直接返回缓存结果），这些上下文只计算一次，之后直接从缓存取。

第二层是对话历史（transcript），每一轮用户输入、AI 回复、工具调用结果都追加在这里。增长最快，对话越长占用越多，是压缩管理的主要目标。

第三层是记忆文件，把重要信息存到文件里，需要时再读取。这是 Claude Code 跨会话保留信息的机制，不依赖对话历史的连续性。

三层的注入顺序影响模型行为。系统提示词在最前面，权重最高；最近的对话在最后面，模型对近期内容的注意力更强 [inference: consistent with general LLM attention patterns]。上下文的顺序和格式，是提示词工程里最常被忽视的变量之一。

## 执行前落盘和最小权限记忆更新，把不可逆操作的风险隔离

**状态管理的正确性不只取决于集中对象的设计，还取决于写入时机和权限边界。**

### 先写 transcript，再进主循环

[`QueryEngine.ts#L443`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/QueryEngine.ts#L443) 在进入主循环之前，先把用户输入写入 transcript（持久化到磁盘的对话记录文件）。

这个顺序是整个恢复机制能工作的前提。"执行完成后保存"的系统在进程中途崩溃时，这一步的输入已经丢失，恢复时不知道上次执行到哪里。"执行前先落盘"的系统在进程崩溃后，输入已在磁盘上，`--resume` 知道从哪个点继续。这是"执行前落盘"原则：副作用不可撤销的操作，必须在执行之前确保状态已被记录。

### 记忆更新子代理只能编辑一个文件

[`src/services/SessionMemory/sessionMemory.ts#L460`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/services/SessionMemory/sessionMemory.ts#L460) 的 `createMemoryFileCanUseTool()` 声明了记忆更新子代理的权限边界：

```typescript
return {
  behavior: 'deny' as const,
  message: `only ${FILE_EDIT_TOOL_NAME} on ${memoryPath} is allowed`,
}
```

记忆更新子代理（runForkedAgent，从主进程 fork 出去异步运行）只能用 `FileEdit` 工具编辑指定的 memory 文件，任何其他工具调用都被拒绝。如果它能访问任意工具，就可能意外触发文件写入、命令执行、网络请求，污染主会话状态或产生副作用。最小权限把"记忆更新"和"执行任务"这两个关注点分开 [`sessionMemory.ts#L318`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/services/SessionMemory/sessionMemory.ts#L318)。

触发条件是双重阈值（[`sessionMemory.ts#L135-L166`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/services/SessionMemory/sessionMemory.ts#L135-L166)）：token 增量必须超过 `minimumTokensBetweenUpdate`，同时工具调用次数也必须超过 `toolCallsBetweenUpdates`。注释强调 token 阈值是必要条件，即使工具调用次数满足，也必须等 token 阈值满足才触发。单一阈值的问题是短对话里频繁触发，在什么都没发生多少的情况下就开始更新记忆；双重阈值要求对话必须有一定实质内容才会触发。

### 大内容哈希化防历史文件膨胀

`history.ts` 的 `addToHistory()` 对大内容有专门处理：小于 1024 字节直接存原文，大于 1024 字节存哈希值，原文另存到 `paste-store`。用户粘贴了几千行代码，历史文件不会存整段代码，只存哈希，需要时再从 paste-store 还原。

这个模式是 Agent 系统里通用的状态压缩策略：大内容做摘要或哈希，只在需要时还原全量数据。历史记录本身跳过了一类特殊会话（`CLAUDE_CODE_SKIP_PROMPT_HISTORY` 为 true 时），这类会话是 Claude 自动创建的验证或测试会话，不应污染用户的真实命令历史 [inference]。

## 边界：这些设计在哪些条件下退化

集中状态对象假设恢复路径可以正确实现"整体替换"。如果某条恢复路径只替换了部分字段（开发者忘了某个新加的字段），系统仍然会产生部分恢复的不一致状态，集中对象并不能在类型层面阻止这个错误。

记忆更新的双重阈值假设 token 计数是可靠的。如果 API 返回的 token 计数不准确（流式响应中途的计数可能有偏差），触发条件可能在错误的时机满足或不满足。

文件系统存储状态的方案在单机上工作良好，多机部署时失效。`sessionMemory.ts` 的 `runForkedAgent` 假设 fork 出去的子进程能访问同一个文件系统，在分布式场景下这个假设不成立。

## 可迁移原则

三条原则，适用条件比结论更重要：

**集中状态对象优于分散局部变量**，适用条件：系统有多条恢复路径，且这些路径需要重置系统到一致的已知状态。单条执行路径且不需要恢复的工具，局部变量的可读性更好。

**副作用操作在执行前落盘**，适用条件：任何需要支持中断恢复的系统。纯内存操作、不需要持久化的临时会话不在此列。

**后台异步操作用最小权限隔离**，适用条件：后台异步操作可能意外影响主执行流程的系统。操作本身没有副作用、不可能污染主流程的情况下，隔离是多余的。

原则是工具，不是教条。
