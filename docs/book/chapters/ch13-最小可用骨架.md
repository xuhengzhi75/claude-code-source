# 第13章 最小可用 Agent 骨架

**核心主张**：最小可用 Agent 骨架必须三层齐备，任何一层的缺失不会让系统"功能减少"，而是让系统在整类场景下进入未定义行为。

---

搭一个真正能干活的 Agent，最少需要什么？

这是个"地基问题"。不是问砖头够不够用，是问打地基的时候少了哪种土，楼会从哪个方向倒。

这三层是：入口层、能力层、执行层。它们解决的不是同一个问题：入口层决定轻操作的响应代价，能力层决定安全边界在哪，执行层决定多步任务能不能跑完。缺哪层，系统在对应那类场景下的行为就会变成未定义——不是出错提示，是系统安静地做错。

## 13.1 入口层缺失，所有路径的代价等同于最慢的那条

`src/entrypoints/cli.tsx` 是一个纯分流器。它本身不做任何 Agent 逻辑，只决定哪条请求走哪条加载路径。整个文件里所有 `import` 都是动态的（`await import(...)`），模块在进入对应分支时才加载。

这让快速路径真正做到零开销：

| 触发条件 | 加载的模块 |
|---------|-----------|
| `--version` / `-v` | 零（`MACRO.VERSION` 在构建期内联为常量） |
| `--dump-system-prompt` | config + model + prompts |
| `ps/logs/attach/kill/--bg` | config + cli/bg |
| 完整主流程（兜底） | `main.js`（全量） |

`--version` 是唯一零模块加载路径，直接打印编译期内联的版本号后退出。完整主流程是最重的路径，其他快速路径各自只加载自己需要的部分。

`feature()` 调用在 `if` 条件里内联，不提取为变量（`cli.tsx#L114–L115`）。这不是代码风格选择，是 Bun 构建工具做编译期死代码消除（DCE，dead code elimination）的前提条件。`DAEMON`、`BRIDGE_MODE` 等 feature flag 在外部构建版本中为 `false`，对应分支在构建期整块被裁掉，不出现在最终产物里。把 `feature()` 提取成变量，DCE 就失效了，产物里会留下死代码。

没有入口层会发生什么：所有请求走同一条路径，在主程序里判断模式。代码更简单，但 `--version` 这类轻操作也要等完整运行时加载。在 SSD 上约 4ms，在磁盘竞争时约 30ms。CLI 工具的 30ms 额外延迟是可感知的，用户会觉得"这个工具反应慢"，即使实际功能完全正常。

## 13.2 能力层权限逻辑集中才能被审计，分散则漏洞不可避免

`getAllBaseTools()`（`src/tools.ts#L193`）先构建理论上可用的工具全集，`filterToolsByDenyRules()`（`tools.ts#L265`）再按 deny 规则过滤出本次会话实际可用的工具。顺序是固定的：先全集，后过滤。

这个顺序的价值在于权限逻辑的集中性。如果反过来，在每个工具注册时各自判断是否应该加入，权限逻辑就会散落在每个工具的注册代码里。新增一个工具时，需要记得在每个相关地方添加权限检查。这是容易遗漏的操作，遗漏会产生安全漏洞，而且这类漏洞不会报错，会安静地放行本应拒绝的工具调用。

`TOOL_DEFAULTS`（`Tool.ts#L763`）和 `buildTool()`（`Tool.ts#L789`）保证每个工具即使只实现最小接口，也会被安全默认值补齐：

```typescript
const TOOL_DEFAULTS = {
  isReadOnly: () => false,           // 默认假设有写操作（保守）
  isConcurrencyEnabled: () => false, // 默认不允许并发（保守）
  toAutoClassifierInput: () => '',   // 默认跳过安全分类（需显式覆盖）
}
```

`isReadOnly` 和 `isConcurrencyEnabled` 两个默认值都是 `false`。保守方向是对的：忘记实现时，系统假设"有副作用、不允许并发"，拒绝潜在危险操作。如果默认值是 `true`，忘记实现时系统会放行。

`toAutoClassifierInput` 返回空字符串是另一种机制。安全分类器收到空字符串时会跳过这个工具，不做风险评估。注释里说"security-relevant tools must override"，这是一个 opt-in（主动选择参与）机制：开发者需要显式覆盖这个方法，工具才会进入安全分类流程。默认跳过，不默认参与。

`assembleToolPool()` 内建工具排在 MCP 工具前（`tools.ts#L357–L381`）。这个顺序影响 prompt cache：内建工具列表末尾有 cache breakpoint，MCP 工具如果插到内建工具中间，breakpoint 就失效，所有下游请求都要重新计算 prompt cache，响应变慢、费用变高。

**技术债**：全集加载假设工具数量在可控规模内。当前规模下不是瓶颈，但工具数量一旦量级跳升到几百个，每次会话都加载全集再过滤，能力层会是第一个需要改的地方。

## 13.3 执行层的 13 条退出路径必须全部覆盖，任意一条缺失都是真实故障点

`while(true)`（`src/query.ts#L314`）是 Agent 和普通脚本的根本区别。普通脚本执行一次返回结果，Agent 需要循环直到任务真正完成。循环本身不难，难的是知道什么时候应该停。

循环的继续条件不是模型说"我完成了"，而是本轮是否产生了 `tool_use` 块（`query.ts#L563–L568`）：

```typescript
// We intentionally derive "need another model turn" from observed tool_use
// blocks instead of stop_reason. This keeps behavior stable across provider/
// SDK differences where stop_reason may be missing or delayed in stream events.
```

`stop_reason` 在流式路径下不可靠：消息在 `content_block_stop` 时落盘，`stop_reason` 在 `message_delta` 事件才到达。流式路径下落盘时 `stop_reason` 永远是 null，任何依赖 transcript 里 `stop_reason` 的恢复逻辑都会在流式路径下失效（`conversationRecovery.ts#L300–L306`）。用 `tool_use` 是否出现来判断继续条件，行为跨 provider 稳定。

`queryLoop()` 共有 13 条退出路径：

| 退出场景 | `reason` 值 | 类型 |
|---------|------------|------|
| 超出 blocking limit（无 auto-compact） | `'blocking_limit'` | 资源限制 |
| 图片尺寸错误 | `'image_error'` | 异常 |
| 模型调用抛出异常 | `'model_error'` | 异常 |
| 流式传输中被中断（Ctrl+C） | `'aborted_streaming'` | 用户中断 |
| prompt 过长且无法恢复 | `'prompt_too_long'` | 资源限制 |
| API 错误（限流/认证等） | `'completed'` | 异常（返回完成状态） |
| stop hook 阻止继续 | `'stop_hook_prevented'` | hook 干预 |
| token budget 耗尽/正常完成 | `'completed'` | 正常 |
| 工具执行中被中断 | `'aborted_tools'` | 用户中断 |
| hook 停止 | `'hook_stopped'` | hook 干预 |
| 超出 maxTurns | `'max_turns'` | 资源限制 |

API 错误（限流、认证失败等）返回 `'completed'` 而不是 `'error'`（inference：注释里没有直接说明原因，但 stop hook 触发死循环的机制可以解释这个选择）。如果 API 错误返回 `'error'`，stop hook 会在错误消息上触发处理逻辑，处理完成后可能继续重试，下一次调用碰到同样的限流，再触发 stop hook，形成死循环，频繁调用进一步加剧限流。把 API 错误归入 `'completed'`，stop hook 不会将其识别为需要干预的失败。

如果你的最小骨架只处理"正常完成"这一条路径，真实使用中有 12 种情况会让系统进入未定义行为。

### 13.3.1 先落盘再执行是恢复能力的前提，不是可选设计

`QueryEngine.ts#L443` 的注释：

```typescript
// Persist the user's message(s) to transcript BEFORE entering the query
// loop. ... Writing now makes the transcript resumable from the point the
// user message was accepted, even if no API response ever arrives.
if (persistSession && messagesFromUserInput.length > 0) {
  const transcriptPromise = recordTranscript(messages)
  if (isBareMode()) {
    void transcriptPromise  // --bare 模式：fire-and-forget
  } else {
    await transcriptPromise  // 正常模式：等待落盘完成
  }
}
```

`transcript`（持久化到磁盘的对话记录文件）在执行之前写入，使得进程被杀后可以从"用户消息已收到"这个状态恢复，而不是从"用户还没发消息"恢复。顺序是设计约束，不是实现细节。

`--bare` 模式（脚本调用场景，无需交互恢复）用 `void transcriptPromise`（fire-and-forget）。脚本调用不使用 `--resume`，而 `await` 在 SSD 上约 4ms、磁盘竞争时约 30ms，是整条启动路径上最大的可控延迟。`void` 不是漏掉了 `await`，是在脚本场景下有意接受"写入失败不影响执行"的权衡。

恢复时，`filterUnresolvedToolUses()` 过滤掉没有对应 `tool_result` 的 `tool_use` 块，防止同一条工具调用被执行两次。

## 13.4 可迁移原则

**三层对应三类失效，三层缺一层都会让你的 Agent 在整类场景下行为未定义。**

入口层缺失：所有请求的代价等同于最重的请求。轻操作不会失败，只是慢到用户以为系统在思考。

能力层权限逻辑分散：权限规则在新增工具时会被遗漏。系统不会报警，会安静地放行不该放行的工具。安全漏洞往往不是某处代码写错了，是某处本该有代码却没有写。

执行层退出路径不完整：少覆盖一条就是一个真实故障点。测试环境里 API 不限流、用户不 Ctrl+C、任务不超时，你看不到那 12 条路径有什么问题。生产环境里，它们都会出现。

这个原则的适用条件是：你在构建一个需要真实运行的 Agent，而不是用于演示的原型。演示原型可以只走快乐路径，生产 Agent 不能。

反例：如果你的 Agent 运行在受控环境里（任务类型单一、不会被中断、工具集固定不变），部分层可以简化，比如入口层退化为单一路径，能力层权限规则硬编码进工具列表。问题不是"能不能简化"，是"你是否知道简化掉了什么，以及简化带来了什么风险"。

原则是工具，不是教条。
