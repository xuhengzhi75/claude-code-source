# 第13章 最小可用 Agent 骨架

**本章核心主张**：一个能真正干活的 Agent 骨架需要三层，每层缺失会让系统在不同的维度失效。最小不是最简，而是刚好够用——去掉任何一层，系统就不再是 Agent。

---

一个能真正干活的 Agent，不需要从零发明，但也不是把 Claude Code 全量复制一遍。问题是：最少需要什么？

答案是三层，每层解决一个不同的问题：入口层解决"用户想做什么"，能力层解决"系统能做什么"，执行层解决"怎么把这件事做完"。这三层不是可以按需选择的，缺任何一层，Agent 就会在某类场景下失效。

## 13.1 入口层：零开销路径和懒加载

`src/entrypoints/cli.tsx` 是一个纯分流器。代码里的所有 import 都是动态的（`await import(...)`），在进入对应分支时才加载。这让快速路径的开销变成零：

| 触发条件 | 加载的模块 |
|---------|-----------|
| `--version` / `-v` | 零（`MACRO.VERSION` 在构建期内联为常量） |
| `--dump-system-prompt` | config + model + prompts |
| `ps/logs/attach/kill/--bg` | config + cli/bg |
| 完整主流程（兜底） | `main.js`（全量） |

`--version` 是唯一零模块加载路径，直接打印编译期内联的版本号后返回。其他快速路径只加载自己需要的模块，完整主流程是最重的路径。

这个设计的背后有一个工程决策值得注意：`feature()` 调用在 `if` 条件里内联，不提取为变量。这是为了让 Bun 构建工具做编译期死代码消除（DCE）——`DAEMON`、`BRIDGE_MODE` 等 feature flag 在外部构建版本中为 `false`，对应分支整块被裁掉，不出现在最终产物里。这不只是代码组织问题，是生产构件大小管理的策略。

**替代方案**：所有请求走同一条路径，在主程序里判断模式。这样入口层更简单，但代价是轻操作（如 `--version`）需要等待完整运行时加载。在 SSD 上约 4ms，在磁盘竞争时约 30ms——对 CLI 工具来说，30ms 的额外延迟是可感知的。

## 13.2 能力层：先全集后过滤，原因不只是方便

`src/tools.ts#L193` 的 `getAllBaseTools()` 和 `src/tools.ts#L265` 的 `filterToolsByDenyRules()` 的共同模式：先构建理论可用全集，再按条件过滤。

这个顺序的价值不只是代码组织，而是**权限规则的集中性**。如果反过来，先按条件判断再逐个添加，权限逻辑会散落在每个工具的注册代码里，新增工具时需要记得在每个地方加权限检查——这是一个容易遗漏的操作，遗漏会产生安全漏洞。

`src/Tool.ts#L763` 的 `TOOL_DEFAULTS` 和 `Tool.ts#L789` 的 `buildTool()` 保证了每个工具即使只实现最小接口，也会被安全默认值补齐：

```typescript
const TOOL_DEFAULTS = {
  isReadOnly: () => false,           // 默认假设有写操作（保守）
  isConcurrencyEnabled: () => false, // 默认不允许并发（保守）
  toAutoClassifierInput: () => '',   // 默认跳过安全分类（需显式覆盖）
}
```

两个保守默认值（`false`）防止了"忘了实现某项安全判断"导致的问题。`toAutoClassifierInput: () => ''` 是第三个默认值——返回空字符串意味着安全分类器会跳过这个工具，不做风险评估。注释里说"security-relevant tools must override"——这是一个需要主动选择参与（opt-in）的机制，而不是默认开启。

**技术债**：当前这个全集加载模式假设工具全集的加载成本可接受。如果工具数量增长到几百个，每次会话都加载全集再过滤，可能产生性能问题。当前规模下不是瓶颈，但工具数量一旦量级跳升，这里会是第一个需要改的地方。

## 13.3 执行层：循环终止的 13 条路径

`src/query.ts#L314` 的 `while(true)` 不是随意的——这是 Agent 和普通脚本的根本区别所在。普通脚本执行一次就结束，Agent 需要循环直到任务完成。

循环的继续条件不是模型说"我完成了"，而是本轮是否产生了 `tool_use`（L563–L568）：

```typescript
// We intentionally derive "need another model turn" from observed tool_use
// blocks instead of stop_reason. This keeps behavior stable across provider/
// SDK differences where stop_reason may be missing or delayed in stream events.
```

`stop_reason` 为什么不可靠？在流式传输路径下，消息在 `content_block_stop` 时落盘，而 `stop_reason` 在 `message_delta` 事件才到达。这意味着任何依赖 transcript 里 `stop_reason` 的恢复逻辑都是错的——在流式路径下，落盘时 `stop_reason` 永远是 null（`conversationRecovery.ts` L300–L306 的注释直接说明了这一点）。

`queryLoop()` 共有 **13 条退出路径**：

| 退出场景 | `reason` 值 | 类型 |
|---------|------------|------|
| 超出 blocking limit（无 auto-compact） | `'blocking_limit'` | 资源限制 |
| 图片尺寸错误 | `'image_error'` | 异常 |
| 模型调用抛出异常 | `'model_error'` | 异常 |
| 流式传输中被中断（Ctrl+C） | `'aborted_streaming'` | 用户中断 |
| prompt 过长且无法恢复 | `'prompt_too_long'` | 资源限制 |
| API 错误（限流/认证等） | **`'completed'`** | 异常（伪装成完成） |
| stop hook 阻止继续 | `'stop_hook_prevented'` | hook 干预 |
| token budget 耗尽/正常完成 | `'completed'` | 正常 |
| 工具执行中被中断 | `'aborted_tools'` | 用户中断 |
| hook 停止 | `'hook_stopped'` | hook 干预 |
| 超出 maxTurns | `'max_turns'` | 资源限制 |

最反直觉的一条：API 错误（限流、认证失败等）返回的是 `'completed'` 而不是 `'error'`。原因是：如果返回 `'error'`，stop hook 可能在错误消息上触发死循环（error → hook blocking → retry → error → …）。把 API 错误伪装成 `'completed'`，是为了让 stop hook 不把它当成需要处理的失败。

这 13 条退出路径，每一条对应一个真实场景，没有"以后再加"的。如果你构建的最小骨架只处理"正常完成"这一条路径，在真实使用中会有 12 种情况让你的系统进入未定义行为。

## 13.4 先落盘再执行：恢复的基础设施

`src/QueryEngine.ts#L443` 的注释写得很直接：

```typescript
// Persist the user's message(s) to transcript BEFORE entering the query
// loop. ... Writing now makes the transcript resumable from the point the
// user message was accepted, even if no API response ever arrives.
if (persistSession && messagesFromUserInput.length > 0) {
  const transcriptPromise = recordTranscript(messages)
  if (isBareMode()) {
    void transcriptPromise  // --bare 模式：fire-and-forget
  } else {
    await transcriptPromise  // 正常模式：等待落盘完成
  }
}
```

在执行之前先落盘，不是 nice-to-have，是恢复能力的基础设施。进程被杀后，下次启动时能从"用户消息已收到"这个状态恢复，而不是从"用户还没发消息"恢复。

`--bare` 模式用 `void transcriptPromise`（fire-and-forget）——注释说"Scripted calls don't --resume after kill-mid-request"，脚本调用不需要 `--resume`，而 `await` 在 SSD 上约 4ms、磁盘竞争时约 30ms，是整个启动路径中最大的可控延迟。这是一个有意识的性能/可靠性权衡，而不是遗漏了 `await`。

恢复时的去重机制：`filterUnresolvedToolUses()` 过滤掉没有对应 `tool_result` 的 `tool_use` 块，防止同一条工具调用被执行两次。

## 13.5 三层之间的关键边界

三层不是独立模块，有明确的数据流：入口层决定进不进 Agent 主流程，能力层决定这次执行可以用哪些工具，执行层用这些工具完成任务。

`QueryEngine` 在执行层和能力层之间做了一件重要的事（`QueryEngine.ts#L443`）：在进入循环之前，先把用户输入写入 transcript。这使得即使进程在执行中途被杀，已提交的用户输入不会丢失。

**最小骨架的验收标准**：给它一个需要多步工具调用才能完成的任务，它能自主循环执行到完成，中途不需要人工干预；进程被杀后，重新启动能从上次中断点继续，而不是从头开始。

如果做不到第一点，说明执行层的循环结构有问题，或者只处理了少数几条退出路径。如果做不到第二点，说明缺少落盘机制（先落盘再执行）或恢复逻辑（去重过滤、状态重建）。

## 13.6 心智模型验证题

你构建了一个 Agent，执行层是 `while(true)` 循环，退出条件是"本轮没有 `tool_use`"。现在 API 开始限流，每次调用返回 429 错误，但 SDK 把 429 封装成了一个 exception 而不是一条消息。系统会怎么处理？

答案：`while(true)` 里如果没有捕获 exception，429 会直接抛出，循环终止——这会产生 `'model_error'` 退出，但如果你没有 `try/catch`，进程会崩溃。如果有 `try/catch` 但只是打印日志后 `continue`，下一次循环还会碰到 429，形成死循环（频繁调用进一步加剧限流）。Claude Code 的处理是把 API 错误归到 `'completed'` 路径，让 stop hook 不把它当成需要重试的失败——这是 13 条退出路径中那条"伪装成 completed"的 API 错误路径的用意。

## 13.7 本章小结

最小可用 Agent 骨架需要三层：

**入口层**（[`cli.tsx#L33`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/entrypoints/cli.tsx#L33)）：先判断再加载，快路径零模块开销，feature flag 驱动编译期 DCE

**能力层**（[`tools.ts#L193`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/tools.ts#L193)）：先全集再过滤，权限规则集中不散落，保守默认值防遗漏

**执行层**（[`query.ts#L314`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/query.ts#L314)）：`while(true)` 循环，继续条件基于可观察事实（`tool_use`）而非模型自我报告（`stop_reason`），13 条退出路径每条都有明确出口

三层缺一不可。入口层决定系统的响应速度，能力层决定系统的安全边界，执行层决定系统能不能真正完成任务。最小不是最简，而是刚好够用。
