# 第13章 最小可用 Agent 骨架

一个能真正干活的 Agent，不需要从零发明，但也不是把 Claude Code 全量复制一遍。问题是：最少需要什么？

答案是三层，每层解决一个不同的问题：入口层解决"用户想做什么"，能力层解决"系统能做什么"，执行层解决"怎么把这件事做完"。这三层不是可以按需选择的，缺任何一层，Agent 就会在某类场景下失效。

## 13.1 入口层：先判断，再加载

`src/entrypoints/cli.tsx#L33` 的 `main()` 函数，第一件事不是启动主程序，而是判断当前请求属于哪种模式。`--version`（[`L41`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/entrypoints/cli.tsx#L41)）、`daemon`（[`L169`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/entrypoints/cli.tsx#L169)）、`--bg`（[`L189`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/entrypoints/cli.tsx#L189)）这些路径会直接处理完返回，不进入完整主流程。只有普通对话请求，才会走到最后的 `import('../main.js')`（[`L301`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/entrypoints/cli.tsx#L301)）。

这个设计的实际效果是：轻操作不需要把整套系统拉起来。对用户来说，常见命令响应更快；对系统来说，冷启动压力更小。

另一种做法是所有请求都走同一条路径，在主程序里判断模式。这样入口层更简单，但代价是轻操作（如 `--version`）需要等待完整运行时加载。对启动时间敏感的场景，这个代价是不可接受的。

## 13.2 能力层：先全集，再过滤

`src/tools.ts#L193` 的 `getAllBaseTools()` 和 `src/tools.ts#L265` 的 `filterToolsByDenyRules()` 的共同模式是：先构建理论可用全集，再按条件过滤。`assembleToolPool()`（[`tools.ts#L350`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/tools.ts#L350)）把内建工具和外部工具合并。

这个顺序很重要。如果反过来，先按条件判断再逐个添加，随着工具数量增加，逻辑会越来越难维护，权限边界也容易出现遗漏。

`src/Tool.ts#L763` 的 `TOOL_DEFAULTS` 和 `Tool.ts#L789` 的 `buildTool()` 保证了每个工具即使只实现最小接口，也会被安全默认值补齐。默认不可并发、默认非只读，这两个保守默认值防止了"忘了实现某项安全判断"导致的问题。

能力层的"先全集后过滤"假设工具全集的加载成本是可接受的。如果工具数量增长到几百个，每次会话都加载全集再过滤，可能产生性能问题。当前规模下这不是瓶颈，但工具数量一旦量级跳升，这里会是第一个需要改的地方。

## 13.3 执行层：循环，不是单次调用

`src/QueryEngine.ts#L213` 的 `submitMessage()` 进入 `src/query.ts#L243` 的 `queryLoop()`，后者是一个 `while(true)` 循环（[`query.ts#L314`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/query.ts#L314)）。每次迭代：调模型、收集工具调用、执行工具、判断是否继续。

继续的条件不是模型说"我完成了"，而是本轮是否产生了 `tool_use`（[`query.ts#L561`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/query.ts#L561) 的注释写得很直接：从观察到的 `tool_use` 块推导是否需要继续，而不是依赖 `stop_reason`，因为不同 provider/SDK 的 `stop_reason` 细节不总一致）。有工具调用就继续，没有就进入收尾。

执行层的循环结构是 Agent 和普通脚本的根本区别。普通脚本执行一次就结束，Agent 需要循环执行直到任务完成。循环的终止条件必须基于可观察的事实（工具调用是否出现），而不是模型的自我报告（`stop_reason`）。

## 13.4 三层之间的关系

这三层不是独立模块，而是有明确的数据流向：入口层决定进不进 Agent 主流程，能力层决定这次执行可以用哪些工具，执行层用这些工具完成任务。

`QueryEngine` 在执行层和能力层之间做了一件重要的事：在进入循环之前，先把用户输入写入 transcript（[`QueryEngine.ts#L443`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/QueryEngine.ts#L443)）。这意味着即使进程在执行中途被杀，已提交的用户输入不会丢失，下次恢复时有据可查。

这个细节在最小骨架里也值得保留：在开始执行之前，先把任务状态落盘。

## 13.5 最小骨架的验收标准

一个最小可用的 Agent 骨架，应该能通过以下检验：给它一个需要多步工具调用才能完成的任务，它能自主循环执行到完成，中途不需要人工干预。

如果做不到这一点，说明执行层的循环结构有问题。如果能做到，但换一个工具就需要大量改代码，说明能力层的接口设计有问题。如果启动很慢，说明入口层没有做快路径分流。

## 13.6 心智模型验证题

如果去掉执行层的循环结构，改成"调一次模型，返回结果"，会发生什么？

答案：模型在单次调用里可以返回工具调用请求，但没有循环结构，工具调用的结果无法反馈给模型，模型也无法根据工具结果继续执行。结果是 Agent 只能完成"不需要工具调用"的任务，或者"只需要一次工具调用"的任务。任何需要多步工具调用的任务（比如"读取文件，分析内容，写入结果"）都无法完成。

## 13.7 本章小结

最小可用 Agent 骨架需要三层：入口层（先判断再加载，[`cli.tsx#L33`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/entrypoints/cli.tsx#L33)）、能力层（先全集再过滤，[`tools.ts#L193`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/tools.ts#L193)）、执行层（循环而不是单次调用，[`query.ts#L314`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/query.ts#L314)）。
