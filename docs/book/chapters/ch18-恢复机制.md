# 第18章 恢复机制是玩具系统和真实系统的分水岭

急诊室的分诊不是"先来先处理"，而是按损伤类型分派到不同处置流程：出血止血、骨折固定、气道阻塞优先开气道。Claude Code 的恢复机制是同一个逻辑：不同类型的损坏走不同的修复路径，而不是一套通用"重新加载"。

进程在 API 请求中途被杀死，消息链的最后一条是用户消息，没有对应的 assistant 回复——API 认为这是非法结构。历史版本的 attachment 类型在新版本里已经改名，直接加载报错。上下文积累到几十万 token，超过模型处理上限，不能直接恢复。用户今天告诉 Agent 的偏好，明天开新会话时需要还在。

这四个问题，对应四个不同的修复层。

## 18.1 第一层：事实恢复

[`conversationRecovery.ts#L456`](https://github.com/xuhengzhi75/claude-code-source/blob/9e4d0c6d68748da4cc9623a89752ed2cf60af4ea/src/utils/conversationRecovery.ts#L456) 的 `loadConversationForResume()` 从持久化存储里加载历史会话，支持三种加载方式：最近会话、指定 sessionId、指定 jsonl 文件。

加载后，它沿着消息链找到主链（chain-walk），恢复消息列表、快照、元数据。这一层只做加载，不做任何修复。修复工作交给第二层。

chain-walk 假设消息链是线性的。如果消息链有分支（比如用户在同一个会话里发起了多个并行任务），chain-walk 可能无法正确处理所有分支——这是一个已知的边界条件，当前的设计假设会话是单主线推进的。

## 18.2 第二层：语义修复

[`conversationRecovery.ts#L164`](https://github.com/xuhengzhi75/claude-code-source/blob/9e4d0c6d68748da4cc9623a89752ed2cf60af4ea/src/utils/conversationRecovery.ts#L164) 的 `deserializeMessagesWithInterruptDetection()` 按顺序做五件事，每一步都依赖前一步的结果。

**第一步**（[`L169-L170`](https://github.com/xuhengzhi75/claude-code-source/blob/9e4d0c6d68748da4cc9623a89752ed2cf60af4ea/src/utils/conversationRecovery.ts#L169)）：`migrateLegacyAttachmentTypes()` 把历史版本的字段名映射到当前版本。这个函数的存在说明系统改过数据格式，且改动向后不兼容。文件系统存数据的代价：不能做 schema migration，只能在读取时处理所有历史格式。

**第二步**（[`L173-L184`](https://github.com/xuhengzhi75/claude-code-source/blob/9e4d0c6d68748da4cc9623a89752ed2cf60af4ea/src/utils/conversationRecovery.ts#L173)）：清理非法的 `permissionMode`。注释说明原因：这个字段是从磁盘读取的未验证 JSON，可能包含来自不同构建版本的模式值。

**第三步**（[`L186-L202`](https://github.com/xuhengzhi75/claude-code-source/blob/9e4d0c6d68748da4cc9623a89752ed2cf60af4ea/src/utils/conversationRecovery.ts#L186)）：过滤未配对的工具调用和孤立的 thinking 块。如果一个 `tool_use` 没有对应的 `tool_result`，或者一个 thinking 块后面没有 assistant 消息，这些消息被过滤掉。进程在工具执行过程中被杀死时，这种"半截的工具调用"是正常场景，不是异常。

**第四步**（[`L204-L224`](https://github.com/xuhengzhi75/claude-code-source/blob/9e4d0c6d68748da4cc9623a89752ed2cf60af4ea/src/utils/conversationRecovery.ts#L204)）：检测中断 turn。如果上次执行在 assistant 消息中途被中断，系统注入一条元消息：

```typescript
createUserMessage({
  content: 'Continue from where you left off.',
  isMeta: true,
})
```

`isTerminalToolResult`（[`conversationRecovery.ts`](https://github.com/xuhengzhi75/claude-code-source/blob/9e4d0c6d68748da4cc9623a89752ed2cf60af4ea/src/utils/conversationRecovery.ts) 信号25）处理了一个特殊情况：Brief 模式的 `tool_result` 结尾是合法的 turn 终态，不应该被检测为中断。中断检测必须区分"正常以工具结果结束的轮次"和"在工具结果中途被杀死的轮次"，否则会误注入 Continue 消息，让模型从一个实际上已完成的状态重新执行。

**第五步**（[`L226-L239`](https://github.com/xuhengzhi75/claude-code-source/blob/9e4d0c6d68748da4cc9623a89752ed2cf60af4ea/src/utils/conversationRecovery.ts#L226)）：补充 assistant sentinel。如果消息链的最后一条是用户消息，在后面插入一个合成的 assistant 消息，让消息链对 API 合法。

这五步的顺序是有意义的：先处理格式兼容性（旧版本字段），再处理合法性（非法字段值），再处理结构完整性（未配对消息），最后处理状态一致性（中断状态）。每一步的前提是前一步已经完成——在格式不兼容的数据上做合法性检查是无意义的。

还有一个不在这五步里的操作：`restoreSkillStateFromMessages()`（信号26）在反序列化之前从 attachment 回放技能状态。这个操作必须在反序列化前完成，否则下次 compact 会遗忘已激活的技能，导致压缩后技能状态丢失。

## 18.3 第三层：压缩后连续性

[`compact.ts#L326-L371`](https://github.com/xuhengzhi75/claude-code-source/blob/9e4d0c6d68748da4cc9623a89752ed2cf60af4ea/src/services/compact/compact.ts#L326) 的 `buildPostCompactMessages()` 在压缩后按固定协议重建最小上下文：

```
boundary → summary → kept → attachments → hooks
```

注释（[`L331`](https://github.com/xuhengzhi75/claude-code-source/blob/9e4d0c6d68748da4cc9623a89752ed2cf60af4ea/src/services/compact/compact.ts#L331)）明确写道：顺序即协议。这个顺序不是偏好，改变顺序会破坏依赖这个协议的 resume 逻辑。

压缩请求本身也可能触发 prompt-too-long 错误（信号29）——如果历史太长，连生成摘要的请求都装不进上下文。这时有截断重试逻辑：截掉部分历史再试。这是压缩机制的递归问题：用来压缩上下文的操作，本身也受上下文长度限制。

压缩完成后，系统重新注入 delta attachments（信号30），让模型在 post-compact 第一轮有完整的工具上下文。压缩把历史对话替换成摘要，但工具定义等上下文必须在压缩后重新注入，否则模型在新轮次里不知道自己有哪些工具可用。

[`annotateCompactBoundary()`（`L343-L352`）](https://github.com/xuhengzhi75/claude-code-source/blob/9e4d0c6d68748da4cc9623a89752ed2cf60af4ea/src/services/compact/compact.ts#L343) 在 boundary 消息上记录重连元数据，区分两种压缩模式：suffix-preserving（反应式压缩，保留最后一段）和 prefix-preserving（部分压缩，保留开头）。这个元数据让 resume 时能找到正确的接续点，而不是把摘要当成真实对话历史处理。

## 18.4 第四层：长期记忆

[`sessionMemory.ts#L460`](https://github.com/xuhengzhi75/claude-code-source/blob/9e4d0c6d68748da4cc9623a89752ed2cf60af4ea/src/services/SessionMemory/sessionMemory.ts#L460) 的 `createMemoryFileCanUseTool()` 揭示了记忆更新的权限设计：记忆更新子代理只被允许用 `FileEdit` 工具编辑指定的 memory 文件，任何其他工具调用都被拒绝。

记忆更新是在后台异步运行的（[`L318`，`runForkedAgent`](https://github.com/xuhengzhi75/claude-code-source/blob/9e4d0c6d68748da4cc9623a89752ed2cf60af4ea/src/services/SessionMemory/sessionMemory.ts#L318)），如果它能访问任意工具，就可能意外修改当前会话的状态。最小权限隔离防止后台操作污染主会话。

触发条件（[`L135-L166`](https://github.com/xuhengzhi75/claude-code-source/blob/9e4d0c6d68748da4cc9623a89752ed2cf60af4ea/src/services/SessionMemory/sessionMemory.ts#L135)）是双重阈值：token 增量必须超过 `minimumTokensBetweenUpdate`，同时工具调用次数也必须超过 `toolCallsBetweenUpdates`。注释（[`L165-L166`](https://github.com/xuhengzhi75/claude-code-source/blob/9e4d0c6d68748da4cc9623a89752ed2cf60af4ea/src/services/SessionMemory/sessionMemory.ts#L165)）强调：token 阈值是必要条件，即使工具调用次数满足，也必须等 token 阈值满足。这防止了在短对话里频繁触发记忆更新，避免把没什么实质内容的对话也持久化。

## 18.5 第五层：跨会话记忆整合

前四层解决的是单次会话的连续性。用户今天调试了一个复杂 bug，明天开新会话时 Agent 对这段历史一无所知——这是跨会话的遗忘问题。

`services/autoDream/autoDream.ts` 实现了跨会话的后台记忆整合。当会话积累到一定量后，系统 fork 出一个 subagent，在后台跑 `/dream` 命令，把历史会话压缩整理成结构化的 `MEMORY.md` 文件。

触发逻辑遵循"先便宜后昂贵"的顺序：先检查时间（距上次整理是否足够久），再检查会话数量（是否积累了足够的新内容），最后检查锁（是否已有进程在执行整理）。执行过程中加文件锁，整理失败则自动回滚。

整理结果是固定的 10 个模块：Session Title、Current State、Task Specification、Files and Functions、Workflow、Errors & Corrections、Codebase Documentation、Learnings、Key Results、Worklog。每个模块限制在约 2000 tokens，总量控制在 12000 tokens 以内。

和第四层（sessionMemory）的区别在于粒度：sessionMemory 是会话内的实时记忆更新，autoDream 是跨会话的周期性整合。前者保证当前会话的连续性，后者保证跨会话的知识积累。

## 18.6 主循环里的实时恢复分支

除了上面五层，`query.ts` 的主循环里还有几个内联的恢复分支，处理执行过程中的实时异常。

[`yieldMissingToolResultBlocks()`（`query.ts#L123`）](https://github.com/xuhengzhi75/claude-code-source/blob/9e4d0c6d68748da4cc9623a89752ed2cf60af4ea/src/query.ts#L123) 在工具调用结果缺失时补齐 `tool_result` 块。这个函数在三个地方被调用（[`L907`](https://github.com/xuhengzhi75/claude-code-source/blob/9e4d0c6d68748da4cc9623a89752ed2cf60af4ea/src/query.ts#L907)、[`L991`](https://github.com/xuhengzhi75/claude-code-source/blob/9e4d0c6d68748da4cc9623a89752ed2cf60af4ea/src/query.ts#L991)、[`L1032`](https://github.com/xuhengzhi75/claude-code-source/blob/9e4d0c6d68748da4cc9623a89752ed2cf60af4ea/src/query.ts#L1032)），分别对应：fallback 模型切换时、流式执行出错时、正常结束时。

三个调用点说明了一件事：消息链结构损坏不是偶发的边缘情况，而是在多种正常和异常路径下都可能发生。如果只在"看起来出错了"时才补齐，就会漏掉正常结束路径里的损坏情况。

## 18.7 边界与技术债

语义修复的五步流水线假设每种损坏类型是独立的，可以按顺序处理。如果两种损坏类型相互影响（比如旧版本的未配对工具调用格式和新版本的工具调用格式不同），流水线的当前顺序可能无法正确处理。这类情况需要在 `migrateLegacyAttachmentTypes()` 里专门处理，但历史上是否完整覆盖了所有版本的格式变化，需要对版本历史有完整了解才能评估。

autoDream 的整理是异步后台进行的，整理失败会回滚，但失败的原因不会主动通知用户。用户不知道跨会话记忆整合是否在正常工作，除非主动检查 `MEMORY.md` 的更新时间。这是后台操作的通用问题：无声地失败比明显地失败更难排查。

## 18.8 可迁移原则

**按损坏类型分层修复**：不同类型的恢复问题走不同的修复路径，而不是一套通用"重新加载"。适用条件：系统持久化数据需要跨版本兼容，且可能在执行中途被中断。反例：纯内存操作、不需要持久化的临时系统，直接重启比恢复更简单。

**后台操作必须最小权限隔离**：后台异步操作能访问的工具集必须严格限定，防止意外污染主流程。适用条件：任何后台异步子任务可能读写主流程状态的系统。反例：后台任务完全独立，和主流程没有共享资源。

**先便宜后昂贵的触发顺序**：触发条件的检查按成本从低到高排列，代价高的操作（比如 fork 子进程）只在代价低的条件都满足后才执行。适用条件：触发条件涉及多步检查且各步成本差异显著的系统。
