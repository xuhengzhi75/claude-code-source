# 第18章 恢复是玩具和真实系统的分水岭

直觉上，恢复应该很简单：把会话存到文件，需要时加载回来。但真实场景里，"存到文件"和"加载回来"之间有很多可能出错的地方。

进程在 API 请求中途被杀死，消息链的最后一条是用户消息，没有对应的 assistant 回复——这对 API 来说是不合法的结构。历史版本的 attachment 类型在新版本里已经改名，直接加载会报错。上下文积累到几十万 token，超过模型处理上限，不能直接恢复。用户今天告诉 Agent 的偏好，明天开新会话时需要还在。

这四个问题，分别对应四层恢复机制。

## 18.1 第一层：事实恢复（[`conversationRecovery.ts#L456`](https://github.com/xuhengzhi75/claude-code-source/blob/9e4d0c6d68748da4cc9623a89752ed2cf60af4ea/src/utils/conversationRecovery.ts#L456)）

`loadConversationForResume()` 从持久化存储里加载历史会话，支持三种加载方式：最近会话、指定 sessionId、指定 jsonl 文件。

加载后，它沿着消息链找到主链（chain-walk），恢复消息列表、快照、元数据。这一层解决的是"进程重启后能找回历史"的问题。

这一层本身不做任何修复，只是把原始数据加载进来。修复工作交给第二层。chain-walk 假设消息链是线性的。如果消息链有分支（比如用户在同一个会话里发起了多个并行任务），chain-walk 可能无法正确处理——这是一个已知的边界条件。

## 18.2 第二层：语义修复（[`conversationRecovery.ts#L164`](https://github.com/xuhengzhi75/claude-code-source/blob/9e4d0c6d68748da4cc9623a89752ed2cf60af4ea/src/utils/conversationRecovery.ts#L164)）

`deserializeMessagesWithInterruptDetection()` 是这条流水线里最复杂的一层。它按顺序做五件事，每一步都依赖前一步的结果。

首先（[`L169-L170`](https://github.com/xuhengzhi75/claude-code-source/blob/9e4d0c6d68748da4cc9623a89752ed2cf60af4ea/src/utils/conversationRecovery.ts#L169-L170)）迁移旧版本的 attachment 类型。`migrateLegacyAttachmentTypes()` 把历史版本的字段名映射到当前版本。这个函数的存在说明系统曾经改过数据格式，而且改动是向后不兼容的。

然后（[`L173-L184`](https://github.com/xuhengzhi75/claude-code-source/blob/9e4d0c6d68748da4cc9623a89752ed2cf60af4ea/src/utils/conversationRecovery.ts#L173-L184)）清理非法的 `permissionMode`。注释说明了原因：这个字段是从磁盘读取的未验证 JSON，可能包含来自不同构建版本的模式值。

接着（[`L186-L202`](https://github.com/xuhengzhi75/claude-code-source/blob/9e4d0c6d68748da4cc9623a89752ed2cf60af4ea/src/utils/conversationRecovery.ts#L186-L202)）过滤未配对的工具调用和孤立的 thinking 块。如果一个 `tool_use` 没有对应的 `tool_result`，或者一个 thinking 块后面没有 assistant 消息，这些消息会被过滤掉。

然后（[`L204-L224`](https://github.com/xuhengzhi75/claude-code-source/blob/9e4d0c6d68748da4cc9623a89752ed2cf60af4ea/src/utils/conversationRecovery.ts#L204-L224)）检测中断 turn。如果上次执行在 assistant 消息中途被中断，系统会注入一条元消息：

```typescript
createUserMessage({
  content: 'Continue from where you left off.',
  isMeta: true,
})
```

最后（[`L226-L239`](https://github.com/xuhengzhi75/claude-code-source/blob/9e4d0c6d68748da4cc9623a89752ed2cf60af4ea/src/utils/conversationRecovery.ts#L226-L239)）补充 assistant sentinel。如果消息链的最后一条是用户消息，系统会在后面插入一个合成的 assistant 消息，让消息链对 API 合法。注释解释了为什么要插入在特定位置：`removeInterruptedMessage` 的 `splice(idx, 2)` 需要找到正确的配对。

这五步的顺序揭示了一个通用模式：先处理格式兼容性（旧版本数据），再处理合法性（非法字段值），再处理结构完整性（未配对的消息），最后处理状态一致性（中断状态）。这个顺序适用于任何需要处理历史数据的系统。

## 18.3 第三层：压缩后连续性（[`compact.ts#L326-L371`](https://github.com/xuhengzhi75/claude-code-source/blob/9e4d0c6d68748da4cc9623a89752ed2cf60af4ea/src/services/compact/compact.ts#L326-L371)）

`buildPostCompactMessages()` 在压缩后按固定协议重建最小上下文，顺序是：

```
boundary → summary → kept → attachments → hooks
```

注释（[`L331`](https://github.com/xuhengzhi75/claude-code-source/blob/9e4d0c6d68748da4cc9623a89752ed2cf60af4ea/src/services/compact/compact.ts#L331)）说明了这个顺序是协议，不是偏好：

```
// 顺序即协议：boundary → summary → kept → attachments → hooks。
```

[`annotateCompactBoundary()`（`L343-L352`）](https://github.com/xuhengzhi75/claude-code-source/blob/9e4d0c6d68748da4cc9623a89752ed2cf60af4ea/src/services/compact/compact.ts#L343-L352) 在 boundary 消息上记录重连元数据，区分两种压缩模式：suffix-preserving（反应式压缩，保留最后一段）和 prefix-preserving（部分压缩，保留开头）。这个元数据让 resume 时能找到正确的接续点。

这一层解决的是"上下文太长，但不能丢失关键信息"的问题。压缩是有损的，但损失的是细节，保留的是结构和关键事实。

## 18.4 第四层：长期记忆（[`sessionMemory.ts#L460`](https://github.com/xuhengzhi75/claude-code-source/blob/9e4d0c6d68748da4cc9623a89752ed2cf60af4ea/src/services/SessionMemory/sessionMemory.ts#L460)）

`createMemoryFileCanUseTool()` 揭示了记忆更新的权限设计：

```typescript
return {
  behavior: 'deny' as const,
  message: `only ${FILE_EDIT_TOOL_NAME} on ${memoryPath} is allowed`,
}
```

记忆更新子代理只被允许做一件事：用 `FileEdit` 工具编辑指定的 memory 文件。任何其他工具调用都会被拒绝。

这个限制不是随意的。记忆更新是在后台异步运行的（[`L318`，`runForkedAgent`](https://github.com/xuhengzhi75/claude-code-source/blob/9e4d0c6d68748da4cc9623a89752ed2cf60af4ea/src/services/SessionMemory/sessionMemory.ts#L318)），如果它能访问任意工具，就可能意外修改当前会话的状态。最小权限隔离防止了这种副作用。

触发条件（[`L135-L166`](https://github.com/xuhengzhi75/claude-code-source/blob/9e4d0c6d68748da4cc9623a89752ed2cf60af4ea/src/services/SessionMemory/sessionMemory.ts#L135-L166)）是双重阈值：token 增量必须超过 `minimumTokensBetweenUpdate`，同时工具调用次数也必须超过 `toolCallsBetweenUpdates`。注释（[`L165-L166`](https://github.com/xuhengzhi75/claude-code-source/blob/9e4d0c6d68748da4cc9623a89752ed2cf60af4ea/src/services/SessionMemory/sessionMemory.ts#L165-L166)）强调：token 阈值是必要条件，即使工具调用次数满足，也必须等 token 阈值也满足才触发。这防止了在短对话里频繁触发记忆更新。

## 18.5 第五层：跨会话记忆整合（autoDream）

前四层解决的都是"单次会话"的连续性问题。但还有一个更长周期的问题：用户今天调试了一个复杂 bug，明天开新会话时，Agent 对这段历史一无所知。

`services/autoDream/autoDream.ts` 实现了跨会话的后台记忆整合。当会话积累到一定量后，系统会 fork 出一个 subagent，在后台跑 `/dream` 命令，把历史会话压缩整理成结构化的 `MEMORY.md` 文件。

触发逻辑遵循"先便宜后昂贵"的顺序：先检查时间（距上次整理是否足够久），再检查会话数量（是否积累了足够的新内容），最后检查锁（是否已有进程在执行整理）。执行过程中加文件锁，整理失败则自动回滚。

整理结果是固定的 10 个模块：Session Title、Current State、Task Specification、Files and Functions、Workflow、Errors & Corrections、Codebase Documentation、Learnings、Key Results、Worklog。每个模块限制在约 2000 tokens，总量控制在 12000 tokens 以内。

这个设计有两个值得注意的地方。第一，整理是异步后台进行的，不阻塞当前会话。第二，整理的触发不只在周期性检查时，也会在任务循环中动态发生：当累计上下文达到 10000 tokens 时首次触发，此后每增加 5000 tokens 或发生 3 次工具调用，就会再次触发。

和第四层（sessionMemory）的区别在于粒度：sessionMemory 是会话内的实时记忆更新，autoDream 是跨会话的周期性整合。前者保证当前会话的连续性，后者保证跨会话的知识积累。

## 18.6 主循环里的实时恢复分支

除了上面四层，`query.ts` 的主循环里还有几个内联的恢复分支，处理执行过程中的实时异常：

[`yieldMissingToolResultBlocks()`（`query.ts#L123`）](https://github.com/xuhengzhi75/claude-code-source/blob/9e4d0c6d68748da4cc9623a89752ed2cf60af4ea/src/query.ts#L123) 在工具调用结果缺失时补齐 `tool_result` 块。这个函数在三个地方被调用（[`L907`](https://github.com/xuhengzhi75/claude-code-source/blob/9e4d0c6d68748da4cc9623a89752ed2cf60af4ea/src/query.ts#L907)、[`L991`](https://github.com/xuhengzhi75/claude-code-source/blob/9e4d0c6d68748da4cc9623a89752ed2cf60af4ea/src/query.ts#L991)、[`L1032`](https://github.com/xuhengzhi75/claude-code-source/blob/9e4d0c6d68748da4cc9623a89752ed2cf60af4ea/src/query.ts#L1032)），分别对应：fallback 模型切换时、流式执行出错时、正常结束时。

这三个调用点说明：消息链结构损坏不是偶发的边缘情况，而是在多种正常和异常路径下都可能发生。

## 18.7 最小恢复实现

对业务 Agent 来说，不需要一次性实现 Claude Code 的全部恢复机制。按照这条流水线的逻辑，最小可用的恢复实现只需要两件事。

在开始执行任务时，把任务状态写入文件（任务 ID、描述、当前步骤）。执行完成后更新状态为"完成"。进程崩溃后，下次启动时读取这个文件，知道上次执行到哪里。这对应第一层（事实恢复）的最简版本。

在上下文接近上限时，触发一次摘要压缩：让模型把当前对话的关键信息总结成几段文字，用这个摘要替换历史对话，继续执行。这对应第三层（压缩后连续性）的最简版本。

第二层（语义修复）和第四层（长期记忆）可以在系统稳定后逐步添加。

## 18.8 心智模型验证题

如果去掉第二层（语义修复）的第五步（补充 assistant sentinel），会发生什么？

答案：当进程在用户消息发出后、API 响应到来前被杀死时，transcript 里的最后一条消息是用户消息，没有对应的 assistant 回复。恢复时，消息链的最后一条是用户消息，这对 Claude API 来说是不合法的结构（Claude API 要求消息链以 assistant 消息结尾，或者以 user 消息开始）。API 调用会失败，`--resume` 无法正常工作。

## 18.9 本章小结

恢复机制是玩具 Agent 和真实系统的分水岭。Claude Code 的恢复是五层流水线：事实恢复（加载历史）、语义修复（修复不合法结构）、压缩后连续性（保留关键信息）、长期记忆（会话内实时更新）、跨会话整合（autoDream 后台整理 MEMORY.md）。

每一层的存在都有具体的故障场景：语义修复的五个步骤对应五种不同的损坏类型，记忆更新的最小权限设计防止后台操作污染主会话，双重阈值防止频繁触发，autoDream 的先便宜后昂贵触发顺序防止无谓的整理开销。
