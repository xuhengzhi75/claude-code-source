# 第15章 提示词与任务指令

提示词决定模型在这个 Agent 里扮演什么角色，任务指令决定它在这次执行里要做什么。两者都重要，但作用层次不同，混淆它们会导致系统行为难以预测。

## 15.1 系统提示词的作用

系统提示词（system prompt）在每次对话开始时注入，告诉模型它是谁、能做什么、有哪些约束。Claude Code 的系统提示词由 `src/services/SystemPrompt/` 下的模块构建，包含工具描述、运行模式说明、权限边界、当前环境信息。

系统提示词的内容不是固定的，而是根据当前会话的配置动态生成。工具列表变了，系统提示词里的工具描述就变了。运行模式变了，约束说明就变了。这保证了模型对自己能力边界的认知，和系统实际允许的操作保持一致。

对业务 Agent 来说，系统提示词需要回答三个问题：这个 Agent 的职责是什么、它有哪些工具可以用、有哪些事情它不应该做。这三个问题如果在系统提示词里说清楚，模型在执行时的行为会更可预期。

## 15.2 任务指令和系统提示词的区别

系统提示词是角色设定，任务指令是具体任务。前者在会话开始时设定，后者在每次用户提交时给出。

一个常见的错误是把任务指令塞进系统提示词。比如"你是一个数据分析助手，你的任务是分析用户上传的 CSV 文件，找出异常数据，生成报告"。这里"你是一个数据分析助手"是角色设定，应该在系统提示词里；"分析用户上传的 CSV 文件，找出异常数据，生成报告"是具体任务，应该在用户消息里。

把任务指令放在系统提示词里的问题是：系统提示词在整个会话里是固定的，但任务会变。用户这次要分析 CSV，下次可能要分析 JSON，如果任务写死在系统提示词里，灵活性就没了。

## 15.3 指令的信息密度

模型处理提示词时，不是每个字都同等重要。信息密度低的提示词，模型会"读完但没记住"。

Claude Code 的系统提示词里，工具描述是信息密度最高的部分：每个工具的名称、用途、输入参数、使用场景，这些信息直接影响模型的工具选择。工具描述写得模糊，模型就会在不该用某个工具的时候用它，或者在该用的时候不用。

任务指令的信息密度同样重要。"帮我分析一下数据"和"读取 data.csv，找出 sales 列中超过均值两倍标准差的行，输出这些行的 id 和 sales 值"，模型执行的结果会完全不同。

对业务 Agent 来说，任务指令的质量比系统提示词的质量更容易影响单次执行结果。系统提示词决定上限，任务指令决定这次能不能达到上限。

## 15.4 命令系统的设计

`src/commands.ts` 管理的不只是工具，还有斜杠命令（slash commands）。斜杠命令是用户可以直接输入的快捷指令，比如 `/compact`、`/clear`、`/help`。

斜杠命令和工具的区别在于：工具是模型调用的，斜杠命令是用户直接触发的。斜杠命令通常用于控制 Agent 的运行状态（压缩上下文、清除历史、查看帮助），而不是执行业务任务。

对业务 Agent 来说，设计几个常用的斜杠命令可以显著改善使用体验：`/reset` 清除当前任务状态、`/status` 查看当前执行进度、`/help` 列出可用工具和命令。这些命令不需要经过模型，直接由系统处理，响应更快。

## 15.5 提示词的版本管理

提示词是代码，需要版本管理。这一点在业务 Agent 里经常被忽视，导致"改了提示词，行为变了，但不知道改了什么"。

Claude Code 的系统提示词通过代码构建，自然纳入了版本管理。对业务 Agent 来说，最简单的做法是把系统提示词存在一个独立文件里，和代码一起提交。每次修改提示词，都有对应的 commit 记录，出问题时可以回溯。

## 15.6 本章小结

系统提示词设定角色和约束，任务指令描述具体任务，两者不要混用。工具描述是系统提示词里信息密度最高的部分，直接影响模型的工具选择。任务指令的质量决定单次执行能不能达到系统设定的上限。
