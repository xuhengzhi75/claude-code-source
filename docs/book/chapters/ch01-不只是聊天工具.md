# 第1章 Claude Code 不只是聊天工具

## 1.1 一个让聊天工具崩溃的场景

你让 ChatGPT 帮你重构一个项目。它给出计划，你说开始，它逐文件输出修改建议。改到第 7 个文件时，网络断了。

重连之后，你问它"刚才改到哪了"。它说不知道。你只能从头来。

这不是 bug，这是设计假设：聊天工具认为每次对话独立，用户的需求可以在一次交互内被满足。问答场景下完全成立。任务执行场景下，它会在最不合时宜的时候崩溃。

Claude Code 的假设反过来：进程随时可能死掉，这是正常情况，不是异常。

## 1.2 "进程会死"这个假设在代码里的体现

打开 [`QueryEngine.ts#L443`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/QueryEngine.ts#L443)，进入执行循环之前有一行把用户消息写入 transcript 的代码。

注意顺序：先落盘，再执行。

大多数系统反过来——执行完成后保存历史。这个顺序决定了一件事：如果进程在 API 响应之前被杀死，用户的输入已经在磁盘上了，`--resume` 可以从这个点继续。"执行完成后保存"则意味着进程死掉时这一步的输入已经丢失。

这是整个恢复机制能不能工作的前提。

## 1.3 为什么用 JSONL，而不是数据库或单个 JSON 文件

Claude Code 用文件系统存 transcript——[`sessionStorage.ts#L204`](https://github.com/xuhengzhi75/claude-code-source/blob/main/src/utils/sessionStorage.ts#L204) 的 `getTranscriptPath()` 返回 `<sessionId>.jsonl`，普通文本文件。

JSONL（逐行 JSON）的关键属性是 append-only：每条消息独立追加到文件末尾，进程在任何时刻被杀，已写入的行都是完整可解析的。单个 JSON 文件要重写整个文件才能追加——进程被杀会留下损坏的 JSON；JSONL 的每次写入是原子的单行 append。

[`sessionStorage.ts#L1963`](https://github.com/xuhengzhi75/claude-code-source/blob/main/src/utils/sessionStorage.ts#L1963) 的注释直接说明了这个约束："The JSONL is append-only, so removed messages stay on disk"——"删除"消息不会真正删除磁盘内容，只在内存中过滤。这是设计约束，不是实现疏漏。

[`L225-L227`](https://github.com/xuhengzhi75/claude-code-source/blob/main/src/utils/sessionStorage.ts#L225) 有一处细节：`MAX_TRANSCRIPT_READ_BYTES = 50 * 1024 * 1024`（50MB），注释说"session JSONL can grow to multiple GB (inc-3930)"。超过 50MB 时读取路径只扫描最后 50MB，极长会话的早期历史在 `--resume` 时会被截断。这不是边界情况，是写进设计约束里的。

至于为什么不用数据库：并发写入、事务、复杂查询，这些优势在本地单机工具上用不上；文件的优势是零依赖，出问题直接 `cat` 就能看。如果 Claude Code 需要支持多机协作，文件系统会成为瓶颈。

## 1.4 `--resume` 返回 null 的精确路径

[`sessionStorage.ts#L139-L146`](https://github.com/xuhengzhi75/claude-code-source/blob/main/src/utils/sessionStorage.ts#L139) 定义了 `isTranscriptMessage`：只有 type 为 `user`/`assistant`/`attachment`/`system` 的条目才算消息。`queue-operation` 类型会被写入 JSONL，但读取时被静默跳过，不进入 `messages` Map。

`getLastSessionLog` 返回 null 的精确路径是 `messages.size === 0`（[`L3890`](https://github.com/xuhengzhi75/claude-code-source/blob/main/src/utils/sessionStorage.ts#L3890)）：

```
getLastSessionLog(sessionId)
  → loadSessionFile → loadTranscriptFile
    → 每行：if isTranscriptMessage(entry) → messages.set(uuid, entry)
    → queue-operation 不满足 → 被跳过
  → if messages.size === 0 → return null
```

"JSONL 里有内容但 `--resume` 失败"的场景：文件里只有任务队列操作记录，没有 user/assistant 消息，`messages.size === 0`，返回 null。报错只说"找不到会话"，不说"会话里没有对话消息"。

[`L3891-L3901`](https://github.com/xuhengzhi75/claude-code-source/blob/main/src/utils/sessionStorage.ts#L3891) 在 `--resume` 路径上有缓存预热——加载完 session 文件后，如果 `getSessionMessages` 缓存为空，就把 UUID 集合写入缓存，节省后续 `recordTranscript` 的重复读取。注释说节省了 170-227ms，是实测数据。

## 1.5 JSONL 格式演进需要向后兼容

[`sessionStorage.ts#L3621-L3626`](https://github.com/xuhengzhi75/claude-code-source/blob/main/src/utils/sessionStorage.ts#L3621) 的注释记录了一处格式演进：PR #24099 把 `progress` 消息从 `isTranscriptMessage` 里移除，但旧 JSONL 文件里仍然有这些条目。`loadTranscriptFile` 有专门的 `progressBridge` 逻辑处理遗留条目，防止 `buildConversationChain` 在遇到 progress 的 `parentUuid` 时截断链。

用文件系统存数据的代价：不能像数据库那样做 schema migration，只能在读取时处理所有历史格式。

## 1.6 继续条件：看事实，不看声明

聊天工具判断"任务完成"：模型返回 `stop_reason: end_turn`，完成。漏洞是模型可能在任务没完成时就返回 `end_turn`，因为它不知道自己还需要继续。

[`query.ts#L561`](https://github.com/xuhengzhi75/claude-code-source/blob/main/src/query.ts#L561) 的判断逻辑：不看 `stop_reason`，看有没有 `tool_use` 块。上一轮有工具调用就继续，没有才考虑停止。

模型可能声明错，但工具调用是客观发生的。

## 1.7 公式

```
Agent 实际能力 ≈ 模型能力 × 系统组织能力
```

乘法，不是加法。系统组织能力为零，模型再强也没用。系统组织能力包含四件事：知道当前在哪里（状态管理）、知道能做什么（工具契约）、知道中断后怎么继续（恢复机制）、知道多个任务如何协调（任务系统）。这四件事聊天工具都不需要，Claude Code 都需要。

验证：用户在 Claude Code 执行到第 5 步时强制关闭终端，重新打开后运行 `claude --resume`，从第几步继续？第 5 步——`QueryEngine.ts#L443` 在每步执行前都写入了 transcript。进阶：如果第 5 步只有 `queue-operation` 记录、没有 user/assistant 消息，`--resume` 会成功吗？

## 1.8 小结

Claude Code 和聊天工具的本质区别是对故障的态度：聊天工具假设进程会活着，Claude Code 假设进程会死。

这个假设在代码里体现为三个具体选择：先落盘再执行（[`QueryEngine.ts#L443`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/QueryEngine.ts#L443)），用 JSONL 的 append-only 原子写入（[`sessionStorage.ts#L1963`](https://github.com/xuhengzhi75/claude-code-source/blob/main/src/utils/sessionStorage.ts#L1963)），根据事实而非声明判断继续条件（[`query.ts#L561`](https://github.com/xuhengzhi75/claude-code-source/blob/main/src/query.ts#L561)）。
