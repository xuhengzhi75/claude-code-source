# 第1章 Claude Code 不只是聊天工具

## 1.1 一个让聊天工具崩溃的场景

你让 ChatGPT 帮你重构一个项目。它给出详细计划，你说"好，开始改"，它逐文件输出修改建议。改到第 7 个文件时，网络断了。

重连之后，你问它"刚才改到哪了"。

它说不知道。

你只能从头来。

这不是 ChatGPT 的 bug，这是聊天工具的设计假设：每次对话是独立的，用户的需求可以在一次交互内被满足。这个假设在问答场景下完全成立。在任务执行场景下，它会在最不合时宜的时候崩溃。

Claude Code 的设计假设是反过来的：进程随时可能死掉，这不是异常情况，而是正常情况。

一个假设的差异，导致了两套完全不同的架构。

## 1.2 "进程会死"这个假设在代码里的体现

打开 [`QueryEngine.ts#L443`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/QueryEngine.ts#L443)，在进入执行循环之前，有一行把用户消息写入 transcript 的代码。

注意顺序：先落盘，再执行。

大多数系统的做法是反过来的——执行完成后保存历史。这个顺序看起来无关紧要，但它决定了一件事：如果进程在 API 响应之前被杀死，用户的输入已经在磁盘上了，`--resume` 可以从这个点继续。

如果是"执行完成后保存"，进程死掉就意味着这一步的输入丢失，恢复只能从上一个完成的步骤开始。

这是整个恢复机制能不能工作的前提。

## 1.3 为什么不用数据库

Claude Code 用文件系统存 transcript——[`sessionStorage.ts#L202`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/utils/sessionStorage.ts#L202) 的 `getTranscriptPath()` 返回的是 `<sessionId>.jsonl`，一个普通的文本文件。

用数据库不是更专业吗？

数据库的优势是支持并发写入、事务、复杂查询。但 Claude Code 是本地单机工具，这些优势在这个场景下用不上。数据库的劣势是：增加依赖，出问题时需要专门工具才能查。

文件系统的优势是：零依赖，出问题直接 `cat` 就能看。这个优势在本地开发工具场景下比数据库的优势更值钱。

这是一个"在特定约束下做出的合理选择"，不是"最佳实践"。如果 Claude Code 需要支持多机协作，文件系统方案会成为瓶颈，需要换掉。

## 1.4 query loop 的继续条件

聊天工具判断"任务完成"的方式是：模型返回了 `stop_reason: end_turn`，就认为完成了。

这个判断在问答场景下没问题。在任务执行场景下有一个漏洞：模型可能在任务没完成的时候就返回 `end_turn`，因为它不知道自己还需要继续。

[`query.ts#L561`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/query.ts#L561) 的注释说明了 Claude Code 的判断逻辑：不看 `stop_reason`，看有没有 `tool_use` 块。如果上一轮有工具调用，就继续；如果没有，才考虑停止。

这个差异的含义是：Claude Code 根据已经发生的事实（有没有工具调用）来判断是否继续，模型的声明（stop_reason）只是参考。模型可能声明错，但工具调用是客观发生的。

## 1.5 用一个公式理解 Claude Code

```
Agent 实际能力 ≈ 模型能力 × 系统组织能力
```

乘法，不是加法。系统组织能力为零，模型再强也没用——系统无法把模型的单次输出串联成连续的任务执行。

系统组织能力包含四件事：知道当前在哪里（状态管理）、知道能做什么（工具契约）、知道中断后怎么继续（恢复机制）、知道多个任务如何协调（任务系统）。

这四件事，聊天工具都不需要，因为它的设计假设是每次交互独立完成。Claude Code 都需要，因为它的设计假设是任务会跨越多个时间点、多次中断、多个执行体。

验证题：用户在 Claude Code 执行到第 5 步时强制关闭终端，重新打开后运行 `claude --resume`，系统从第几步继续？答案是第 5 步——因为 `QueryEngine.ts#L443` 在每步执行前都写入了 transcript，第 5 步的输入已经在磁盘上了。

## 1.6 本章小结

Claude Code 和聊天工具的本质区别是对故障的态度：聊天工具假设进程会活着，Claude Code 假设进程会死。这个假设的差异，导致了持久化策略（先落盘再执行）、继续条件判断（看事实不看声明）、存储选择（文件系统而非数据库）的一系列不同。

设计一个系统，先问自己：它会在什么时候死？答案决定了架构。
