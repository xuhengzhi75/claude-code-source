# 第1章 Claude Code 不只是聊天工具

> 核心主张：Claude Code 的价值不在于模型更强，而在于它是一个把"连续执行"当成一等公民设计的任务系统——这个区别决定了它能完成聊天工具永远完不成的工作。

## 1.1 一个常见误解

很多人第一次看到 Claude Code，会觉得它和普通聊天工具差不多：输入一句话，得到一段回答。

这个观察没有错，但不完整。

Claude Code 可以把一项任务拆成多个步骤，并在同一次会话中持续推进。常见能力包括：理解需求并拆解步骤、调用工具（读写文件、执行命令、检索资料、操作浏览器等）、维护当前任务状态、在失败后继续执行、在权限范围内完成整段流程。

更准确的理解是：它是一个模型 + 工具 + 状态 + 编排的工作系统。

## 1.2 为什么会聊天不等于能完成工作

看一个简单例子。

你说："帮我整理这个项目并生成发布说明。"

普通聊天式 AI 往往可以给你模板，或者生成一段文案，但通常不会直接完成完整流程，例如：扫描本地仓库、识别真实改动、按规则生成 changelog、写入指定文件并执行校验。

Claude Code 这类系统的价值，在于它能把这些动作串成连续流程，并给出过程结果。

## 1.3 "连续执行"是一等公民，不是事后补丁

这是本书最重要的一个判断，也是最容易被忽视的一个。

普通聊天工具的设计中心是"单次回答质量"，连续执行是可选的附加功能。Claude Code 的设计中心是"任务完成"，单次回答只是完成任务的一个步骤。

这个区别在源码里有直接体现。`src/QueryEngine.ts` 的 `submitMessage()` 在进入执行循环之前，第一件事是把用户消息写入 transcript（[`QueryEngine.ts#L443-L462`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/QueryEngine.ts#L443)）。注释写得很直白：如果进程在 API 响应之前被杀，transcript 里已经有用户消息，`--resume` 就能从这个点继续。这不是事后加的容错，而是在设计时就把"中断后能继续"当成必须保证的属性。

一个把连续执行当成事后补丁的系统，不会在进入主循环之前就先落盘。

## 1.4 用系统视角理解 Claude Code

可以把 Claude Code 拆成几个部分：模型负责判断下一步，工具层负责执行外部动作（文件、命令、网络、浏览器等），上下文与状态保存任务约束、历史和中间结果，编排逻辑决定何时思考、何时调用工具、何时结束，恢复机制在中断或失败后继续推进。

用户感受到的好用，通常来自这些部分的协同，而不只是单次回答质量。

## 1.5 本书的拆解方式

本书不会把重点放在提示词清单，也不会做逐行源码讲解。我们主要回答四个问题：请求如何进入系统、命令和工具如何协作、运行时如何一步步推进任务、状态与恢复如何支撑长任务。

理解这四点后，Claude Code 的运行方式就会更清晰。

## 1.6 一个最小公式

```text
Agent 可用能力 ≈ 模型能力 × 系统组织能力
```

系统组织能力不足时（例如缺少工具管理、状态管理和恢复机制），模型再强也很难稳定完成长流程任务。系统组织清晰时，即使模型不是最强版本，也可能持续产出可用结果。

## 1.7 本章小结

Claude Code 是任务执行系统，关键能力是连续执行、工具调用、状态维护和恢复。它和聊天工具的本质区别不是功能多少，而是设计中心不同：聊天工具优化单次回答，Claude Code 优化任务完成。这个区别在源码的每一层都有体现，从入口的 fast-path 分流，到执行前的 transcript 落盘，再到 query loop 的继续条件判断。
