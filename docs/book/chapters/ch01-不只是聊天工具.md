# 第1章 Claude Code 不只是聊天工具

## 1.1 一个让聊天工具崩溃的场景

你让 ChatGPT 帮你重构一个项目。它给出详细计划，你说"好，开始改"，它逐文件输出修改建议。改到第 7 个文件时，网络断了。

重连之后，你问它"刚才改到哪了"。

它说不知道。

你只能从头来。

这不是 ChatGPT 的 bug，这是聊天工具的设计假设：每次对话是独立的，用户的需求可以在一次交互内被满足。这个假设在问答场景下完全成立。在任务执行场景下，它会在最不合时宜的时候崩溃。

Claude Code 的设计假设是反过来的：进程随时可能死掉，这不是异常情况，而是正常情况。

一个假设的差异，导致了两套完全不同的架构。

## 1.2 "进程会死"这个假设在代码里的体现

打开 [`QueryEngine.ts#L443`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/QueryEngine.ts#L443)，在进入执行循环之前，有一行把用户消息写入 transcript 的代码。

注意顺序：先落盘，再执行。

大多数系统的做法是反过来的——执行完成后保存历史。这个顺序看起来无关紧要，但它决定了一件事：如果进程在 API 响应之前被杀死，用户的输入已经在磁盘上了，`--resume` 可以从这个点继续。

如果是"执行完成后保存"，进程死掉就意味着这一步的输入丢失，恢复只能从上一个完成的步骤开始。

这是整个恢复机制能不能工作的前提。

## 1.3 为什么用 JSONL，而不是数据库或单个 JSON 文件

Claude Code 用文件系统存 transcript——[`sessionStorage.ts#L204`](https://github.com/xuhengzhi75/claude-code-source/blob/main/src/utils/sessionStorage.ts#L204) 的 `getTranscriptPath()` 返回 `<sessionId>.jsonl`，一个普通文本文件。

JSONL（逐行 JSON）不是随意的选择。它的关键属性是 append-only：每条消息独立追加到文件末尾，进程在任何时刻被杀，已写入的行都是完整可解析的。单个 JSON 文件要重写整个文件才能追加内容——进程被杀会留下损坏的 JSON；JSONL 的每次写入是原子的单行 append，不会损坏已有内容。

[`sessionStorage.ts#L1963`](https://github.com/xuhengzhi75/claude-code-source/blob/main/src/utils/sessionStorage.ts#L1963) 的注释直接说明了这个约束："The JSONL is append-only, so removed messages stay on disk"——"删除"消息不会真正删除磁盘上的内容，只是在内存中过滤。这是设计约束，不是实现疏漏。

数据库的优势是并发写入、事务、复杂查询，但 Claude Code 是本地单机工具，这些优势用不上。数据库的劣势是增加依赖，出问题需要专门工具才能查。文件系统的优势是零依赖，出问题直接 `cat` 就能看。这是"在特定约束下做出的合理选择"，不是"最佳实践"。如果 Claude Code 需要支持多机协作，文件系统方案会成为瓶颈。

[`sessionStorage.ts#L225-L227`](https://github.com/xuhengzhi75/claude-code-source/blob/main/src/utils/sessionStorage.ts#L225) 有一处细节：`MAX_TRANSCRIPT_READ_BYTES = 50 * 1024 * 1024`（50MB），注释说"session JSONL can grow to multiple GB (inc-3930)"。超过 50MB 时，读取路径只扫描最后 50MB，意味着极长会话的早期历史在 `--resume` 时会被截断。JSONL 文件可能大到需要专门的内存保护——这不是边界情况，是写进了设计约束里的。

## 1.4 `--resume` 返回 null 的精确路径

[`sessionStorage.ts#L139-L146`](https://github.com/xuhengzhi75/claude-code-source/blob/main/src/utils/sessionStorage.ts#L139) 定义了 `isTranscriptMessage`：只有 type 为 `user`/`assistant`/`attachment`/`system` 的条目才算消息。

`queue-operation` 类型的条目会被写入 JSONL 文件，但不满足这个检查，所以读取时被静默跳过，根本不进入 `messages` Map。`getLastSessionLog` 返回 null 的精确路径是 `messages.size === 0`（[`L3890`](https://github.com/xuhengzhi75/claude-code-source/blob/main/src/utils/sessionStorage.ts#L3890)）：

```
getLastSessionLog(sessionId)
  → loadSessionFile → loadTranscriptFile
    → 每行：if isTranscriptMessage(entry) → messages.set(uuid, entry)
    → queue-operation 不满足 → 被跳过
  → if messages.size === 0 → return null  ← 主触发点
  → findLatestMessage → if !lastMessage → return null  ← 次触发点
```

"JSONL 里有内容但 `--resume` 失败"的场景：文件里只有任务队列操作记录，没有任何 user/assistant 消息，`messages.size === 0`，返回 null。这个过滤逻辑对用户不可见——报错的只是"找不到会话"，不会说"会话里没有对话消息"。

另一个不显眼的设计：[`L3891-L3901`](https://github.com/xuhengzhi75/claude-code-source/blob/main/src/utils/sessionStorage.ts#L3891) 在 `--resume` 路径上有缓存预热——加载完 session 文件后，如果 `getSessionMessages` 缓存为空，就把 UUID 集合写入缓存，节省后续 `recordTranscript` 的重复读取。注释说节省了 170-227ms，是实测数据，不是估算。

## 1.5 JSONL 的格式演进需要向后兼容

[`sessionStorage.ts#L3621-L3626`](https://github.com/xuhengzhi75/claude-code-source/blob/main/src/utils/sessionStorage.ts#L3621) 的注释揭示了一处格式演进：PR #24099 把 `progress` 消息从 `isTranscriptMessage` 里移除，但旧的 JSONL 文件里仍然有 progress 条目。`loadTranscriptFile` 有专门的 `progressBridge` 逻辑来处理这些遗留条目，防止 `buildConversationChain` 在遇到 progress 的 `parentUuid` 时截断链。

这是"格式演进需要向后兼容"的典型案例。用文件系统存数据的代价是：不能像数据库那样做 schema migration，只能在读取时处理所有历史格式。

## 1.6 query loop 的继续条件

聊天工具判断"任务完成"的方式是：模型返回了 `stop_reason: end_turn`，就认为完成了。

这个判断在问答场景下没问题。在任务执行场景下有一个漏洞：模型可能在任务没完成的时候就返回 `end_turn`，因为它不知道自己还需要继续。

[`query.ts#L561`](https://github.com/xuhengzhi75/claude-code-source/blob/main/src/query.ts#L561) 的注释说明了 Claude Code 的判断逻辑：不看 `stop_reason`，看有没有 `tool_use` 块。如果上一轮有工具调用，就继续；如果没有，才考虑停止。

这个差异的含义是：Claude Code 根据已经发生的事实（有没有工具调用）来判断是否继续，模型的声明（stop_reason）只是参考。模型可能声明错，但工具调用是客观发生的。

## 1.7 用一个公式理解 Claude Code

```
Agent 实际能力 ≈ 模型能力 × 系统组织能力
```

乘法，不是加法。系统组织能力为零，模型再强也没用——系统无法把模型的单次输出串联成连续的任务执行。

系统组织能力包含四件事：知道当前在哪里（状态管理）、知道能做什么（工具契约）、知道中断后怎么继续（恢复机制）、知道多个任务如何协调（任务系统）。

这四件事，聊天工具都不需要，因为它的设计假设是每次交互独立完成。Claude Code 都需要，因为它的设计假设是任务会跨越多个时间点、多次中断、多个执行体。

验证题：用户在 Claude Code 执行到第 5 步时强制关闭终端，重新打开后运行 `claude --resume`，系统从第几步继续？答案是第 5 步——因为 `QueryEngine.ts#L443` 在每步执行前都写入了 transcript，第 5 步的输入已经在磁盘上了。进阶问题：如果第 5 步只有 `queue-operation` 记录、没有 user/assistant 消息，`--resume` 会成功还是返回 null？

## 1.8 本章小结

Claude Code 和聊天工具的本质区别是对故障的态度：聊天工具假设进程会活着，Claude Code 假设进程会死。

这个假设在代码里体现为三个具体选择：先落盘再执行（`QueryEngine.ts#L443`），用 JSONL 而非单个 JSON 文件（append-only 原子写入，`sessionStorage.ts#L1963`），根据事实而非声明判断继续条件（`query.ts#L561`）。

设计一个系统，先问自己：它会在什么时候死？答案决定了架构。
