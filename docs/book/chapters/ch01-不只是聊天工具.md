# 第1章 Claude Code 不只是聊天工具

> **核心主张：Claude Code 解决的第一性问题是"任务连续性"，而不是"回答质量"——这个设计中心的差异，导致它在架构上的每一层选择都和聊天工具相反。**

## 1.1 第一性问题：为什么聊天工具完不成真实工作

聊天工具的设计假设是：每次对话是独立的，用户的需求可以在一次交互内被满足。这个假设在问答场景下成立，在任务执行场景下不成立。

真实工作任务有三个特征，聊天工具无法处理：

**特征一：任务边界不确定。** "帮我整理这个项目"没有明确的终止条件。系统必须自己判断何时完成，而不是等用户说"好了"。

**特征二：执行过程不可预测。** 读文件可能失败，命令可能报错，中间结果可能需要重新规划。系统必须在执行中处理这些情况，而不是在执行前把所有情况都预见到。

**特征三：进度必须可恢复。** 用户可能在任务执行到一半时关闭终端。系统必须能从中断点继续，而不是从头开始。

这三个特征共同指向一个根本矛盾：**任务执行需要跨越多个时间点维持状态，但聊天工具的架构假设每次交互是无状态的。**

Claude Code 的架构选择，是把解决这个矛盾当成第一优先级。

## 1.2 "连续执行是一等公民"在源码里的体现

这不是一个抽象的设计理念，而是在源码的具体位置有直接体现。

`src/QueryEngine.ts#L443` 的注释写得很直白：在进入执行循环之前，先把用户消息写入 transcript。原因是：如果进程在 API 响应之前被杀，transcript 里已经有用户消息，`--resume` 就能从这个点继续。

这个设计选择的隐含假设是：**进程随时可能被杀死，这不是异常情况，而是正常情况。** 一个把连续执行当成事后补丁的系统，不会在进入主循环之前就先落盘——它会在执行完成后才写入，或者根本不写入。

对比：大多数聊天工具的实现是"执行完成后保存历史"。Claude Code 的实现是"执行开始前保存意图"。这个顺序的差异，决定了中断后能不能恢复。

## 1.3 替代方案：为什么不用数据库而用文件系统

Claude Code 用文件系统（jsonl 格式）存储 transcript，而不是数据库。这个选择有明确的隐含假设：**系统是单机运行的，不需要分布式协调。**

替代方案是用数据库（SQLite 或远程数据库）。数据库的优势是：支持并发写入、支持事务、支持复杂查询。劣势是：增加了依赖、增加了启动成本、出问题时需要专门工具才能查看。

文件系统方案的优势是：零依赖、可直接用文本编辑器查看、出问题时直接 `cat` 就能定位。这个优势在单机开发工具场景下比数据库的优势更重要。

**技术债：** 如果 Claude Code 需要支持多机协作或云端同步，文件系统方案会成为瓶颈。当前的设计假设单机场景，这个假设在企业部署场景下可能不成立。

## 1.4 可迁移的设计原则

从 Claude Code 的架构选择中，可以抽象出一条可迁移的原则：

**原则：系统的持久化策略应该反映它的故障模型，而不是它的正常路径。**

正常路径是：用户发出请求，系统执行，返回结果。故障路径是：进程在执行中途被杀死。如果持久化策略只考虑正常路径（执行完成后保存），故障路径就无法恢复。

适用条件：任务执行时间超过几秒，或者任务执行过程中有外部依赖（网络、文件系统、用户输入）。

反例：如果任务执行时间极短（毫秒级），且失败后重新执行的成本可以接受，那么"执行完成后保存"的策略就足够了。

## 1.5 心智模型：用一个公式理解 Claude Code

```text
Agent 可用能力 ≈ 模型能力 × 系统组织能力
```

这个公式的关键是乘法，不是加法。系统组织能力为零时，模型能力再强也没用——系统无法把模型的单次输出串联成连续的任务执行。

系统组织能力包含：状态管理（知道当前在哪里）、工具契约（知道能做什么）、恢复机制（知道中断后怎么继续）、任务系统（知道多个任务如何协调）。

**心智模型验证题：** 如果用户在 Claude Code 执行到第 5 步时强制关闭终端，然后重新打开并运行 `claude --resume`，系统能从第 5 步继续还是从第 1 步重新开始？答案是从第 5 步继续——因为 `QueryEngine.ts#L443` 在每步执行前都写入了 transcript。

## 1.6 本章小结

Claude Code 和聊天工具的本质区别不是功能多少，而是设计中心不同：聊天工具优化单次回答质量，Claude Code 优化任务连续性。这个区别在源码的每一层都有体现，从执行前的 transcript 落盘（[`QueryEngine.ts#L443`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/QueryEngine.ts#L443)），到 query loop 的继续条件判断（基于 `tool_use` 事实而非 `stop_reason` 声明），再到任务系统的持久化设计。

需要记住的核心概念只有一个：**任务连续性是一等公民，不是事后补丁。** 这一个概念，可以解释后面所有章节的架构选择。
