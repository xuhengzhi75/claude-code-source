# 第4章 入口与请求路由：系统怎么决定走哪条路径

这一章聚焦 Claude Code 的最前面一层。

当用户输入一条命令时，系统做的第一件事不是立刻进入完整主程序，也不是马上调用模型。它会先判断：这次请求到底属于哪一类路径，应该加载哪些模块，哪些东西根本没必要启动。

这一步看起来像小优化，实际上很重要。一个能长期运行的 Agent 系统，第一层就要学会分流。否则所有请求都走同一条重路径，冷启动、内存占用、模式切换和后续维护都会变差。

## 4.1 为什么入口层这么重要

很多人看 Agent，先看提示词和工具。
但从系统角度看，入口层决定了三件事：

- 这次请求属于什么模式
- 系统要不要加载完整运行时
- 后续执行链路从哪里接上

如果入口层做不好，后面的架构再漂亮，也会被一层混乱的启动逻辑拖住。

对 Claude Code 来说，入口层的目标很明确：
**在尽可能小的启动成本下，把不同请求分发到正确路径。**

## 4.2 `cli.tsx` 不只是入口文件，而是“参数路由层”

关键入口文件是：`src/entrypoints/cli.tsx`

这份文件最重要的设计不是“启动程序”，而是“先看参数，再决定是否继续深入”。

源码里能看到很明确的结构：

- 先读取 `process.argv`
- 先处理特别轻的 fast-path
- 只有在需要时才动态 `import(...)` 对应模块
- 都不命中时，最后才进入完整主程序

也就是说，`cli.tsx` 更像一个路由器，而不是一个把所有东西一次性拉起来的大入口。

这一点很值得学。因为 CLI 型 Agent 往往天然有很多运行模式：

- 普通命令行会话
- 后台任务
- daemon
- remote/bridge
- 特殊调试命令
- 只输出信息就退出的轻命令

如果这些路径都从同一个重入口直接灌进去，系统就会越来越难维护。

## 4.3 fast-path 的价值，不只是“更快”

在 `src/entrypoints/cli.tsx` 里，有些参数会走快速路径。
例如 `--version` 这类请求，不需要完整运行时，不需要工具装配，也不需要会话状态。

代码中的处理方式很直接：

- 先判断参数是否是 `--version` / `-v`
- 如果是，直接输出版本号后退出
- 不继续加载完整系统

这个设计表面上只是性能优化，实际上还有两个附加价值：

第一，它让轻命令的行为边界更清楚。
一个只看版本号的请求，不应该被复杂初始化拖慢，也不应该被其他模块副作用影响。

第二，它让启动过程更容易推理。
当你知道某类请求不会落入主干执行路径时，调试和维护都更简单。

所以 fast-path 的核心价值不是“省几毫秒”，而是：
**让不同复杂度的请求各走各的路径。**

## 4.4 动态导入说明系统在刻意控制启动成本

`cli.tsx` 里一个很明显的实现细节是，大量逻辑使用动态 `import(...)`。

含义很简单：

- 只有命中了某条路径，才加载那条路径需要的模块
- 没命中的能力，不参与当前这次启动

这和一开始就静态导入全部模块完全不同。

从架构角度，这相当于把“能力存在”与“本次真的要用”分开了。
系统可以支持很多模式，但每次只拿出当前要用的那一小块。

对普通读者来说，可以把它理解成：
一个成熟的 Agent 外壳，不会每次上电都把整座工厂全亮起来，而是先看今天要开哪条生产线。

## 4.5 入口层实际分流了哪些模式

从 `cli.tsx` 的参数判断可以看到，Claude Code 在入口层就区分了多类路径，例如：

- `--version` 这类纯信息输出
- `--dump-system-prompt` 这类调试/导出路径
- `remote-control` / `bridge` 相关路径
- `daemon` 路径
- `ps / logs / attach / kill / --bg` 这类后台会话管理路径
- template job 相关路径
- 其他普通路径，最后才进入主程序

这说明 Claude Code 的入口层不是“把请求交给 main 就算完”，而是在最前面就做了比较精细的模式拆分。

这个设计带来三个明显好处：

1. 每种模式都能有更清楚的启动边界。
2. 不同模式的依赖不会全部绑死在一起。
3. 后续增加新路径时，更容易局部扩展。

## 4.6 入口层还承担了部分环境预处理

`cli.tsx` 里不只做参数分流，也会在 very early 阶段处理少量环境相关动作。
例如：

- 调整 `COREPACK_ENABLE_AUTO_PIN`
- 某些远程环境下设置 `NODE_OPTIONS`
- 特定 feature flag 开启时注入一些实验/ablation 相关环境变量

这些代码说明一个事实：
入口层有时必须在“其他模块 import 之前”做少量设置，否则时机就晚了。

这类逻辑不适合滥用，但在运行时系统里很常见。
因为有些行为一旦错过启动窗口，后面再补就已经来不及。

所以从工程视角看，入口层除了路由，还承担了一部分“启动时序控制”。

## 4.7 入口层与主程序的边界

当请求不属于 fast-path，也不命中特殊模式时，`cli.tsx` 最后会进入 `main.tsx`。

这里的边界很清楚：

- `cli.tsx` 负责前置分流、最小启动、按需加载
- `main.tsx` 负责完整初始化、命令和工具装配、进入交互或非交互执行

这是一种很健康的职责切分。
如果入口文件既负责路由，又负责完整系统初始化，又负责会话执行，文件会很快变成维护噩梦。

Claude Code 这里的做法是先把入口压薄，只保留：

- 参数判断
- 路径分流
- 按需导入
- 极少数必须前置的环境准备

然后把真正复杂的启动工作交给 `main.tsx`。

## 4.8 对我们自己造 Agent 的启发

如果你以后要做一个简化版业务 Agent，这一章最值得抄的不是某个具体 flag，而是入口层的思路：

### 第一，不同请求不要共用一条重路径

至少应先区分：

- 轻量信息命令
- 正常对话/执行路径
- 后台任务或批处理路径
- 维护与调试路径

### 第二，按需加载比一次性全加载更健康

一开始项目小，大家容易把全部模块都 import 进来。
前期看不出问题，后期模式一多，启动就会越来越重。

### 第三，入口层要保持简单

入口层不是展示你会写多少逻辑的地方。
它的职责是让后面复杂系统有清楚的进入方式。

如果入口层已经很乱，通常说明系统边界也开始变乱了。

## 4.9 本章小结

这一章的核心结论可以压成一句话：

**Claude Code 的入口层不是简单开机按钮，而是一个负责模式分流、启动瘦身和前置时序控制的参数路由层。**

从源码上看，这个角色主要由 `src/entrypoints/cli.tsx` 承担。
它先把轻路径和特殊模式拦下来，再把普通路径送入 `main.tsx`。

理解这一层后，后面再看命令、工具、QueryEngine 和 query loop，就会更容易明白：为什么这个系统不是“一坨代码在跑”，而是一条被拆清楚的执行链。
