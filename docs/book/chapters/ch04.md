# 第4章 入口与请求路由：系统怎么决定走哪条路径

结论先说：`src/entrypoints/cli.tsx` 的核心作用不是“启动整个程序”，而是先在参数层做分流，只让当前请求进入需要的那条路径。

当用户输入一条命令时，Claude Code 不会立刻加载完整运行时。它会先判断：这次请求属于哪种模式，哪些模块真的需要加载，哪些能力这次根本不该启动。

这一步不是小优化，而是运行时边界的一部分。若所有请求都共用同一条重启动路径，冷启动、内存占用和模式维护成本都会上升。

## 4.1 `cli.tsx` 为什么更像参数路由层

从 `src/entrypoints/cli.tsx` 的控制流可以直接看到 4 个事实：

- 先读取 `process.argv`
- 先判断 fast-path
- 命中特殊路径后才动态 `import(...)` 对应模块
- 都不命中时，最后才进入完整主程序

所以这份文件最重要的职责不是“把系统启动起来”，而是“把不同请求送进不同入口”。

这也是为什么它更像参数路由层，而不是一个把所有逻辑都装进去的大入口文件。

## 4.2 fast-path 解决的不是体验小优化，而是路径隔离

以 `--version` 为例，代码会直接输出版本号并退出，不进入完整初始化。

这里至少隔离了 3 类成本：

- 不加载完整运行时
- 不装配命令和工具
- 不创建会话状态

这类分流让轻命令保持轻，也让后续控制流更容易推理。

对工程实现来说，fast-path 的价值是：
**把复杂度不同的请求拆开处理，而不是让所有请求都走主干。**

## 4.3 入口层实际分流了哪些模式

从 `cli.tsx` 里的分支可以直接看到，入口层至少处理了这些路径：

- `--version` 这类纯信息输出
- `--dump-system-prompt` 这类调试路径
- `remote-control / bridge` 路径
- `daemon` 路径
- `ps / logs / attach / kill / --bg` 这类后台会话管理路径
- `new / list / reply` 这类 template job 路径

这说明 Claude Code 在参数层就做了模式拆分，而不是把所有请求一股脑送进 `main.tsx`。

## 4.4 动态 `import(...)` 的事实和它背后的推断要分开说

从源码可以直接验证的事实是：`cli.tsx` 大量使用动态 `import(...)`，并且只在命中某条路径时才加载对应模块。

基于这个事实，可以做一个保守推断：这种按需加载有助于降低冷启动成本，也减少无关模块初始化。

这里要把两层说法分开：

- **事实**：存在按需动态导入
- **推断**：这种设计有助于启动性能和内存边界

这样写更严谨，也更方便后续做 evidence 对齐。

## 4.5 入口层还承担少量必须前置的环境准备

`cli.tsx` 不只做分流，也处理少量“必须早于其他模块 import”的准备工作，例如：

- 调整 `COREPACK_ENABLE_AUTO_PIN`
- 在特定远程环境中设置 `NODE_OPTIONS`
- 某些 feature flag 下预置实验相关环境变量

这些动作的重要性不在于数量，而在于时序。它们必须发生在更后面的模块加载之前，否则就会错过生效窗口。

所以入口层除了路由，还承担一部分启动时序控制。

## 4.6 它和 `main.tsx` 的边界是什么

当请求不命中 fast-path，也不属于特殊模式时，`cli.tsx` 才会把控制权交给 `main.tsx`。

这形成了清楚的职责切分：

- `cli.tsx` 负责参数分流、按需加载、少量前置环境准备
- `main.tsx` 负责完整初始化、命令装配、工具装配，以及进入交互或非交互执行

这条边界很重要。入口层越薄，后续系统边界越容易保持清楚。

## 4.7 对自己做 Agent 最值得学什么

这一章最值得抄的不是某个具体 flag，而是三个结构原则：

1. **不同请求不要共用一条重路径**
2. **能按需加载，就不要一开始全加载**
3. **入口层只做分流和前置时序，不承担完整业务执行**

这三条成立后，系统后面再长大，启动层也不容易先失控。

## 4.8 本章小结

`src/entrypoints/cli.tsx` 的核心作用，是在参数层先完成路径分流和按需加载，再把普通路径交给 `main.tsx`。

理解这一层后，后面看命令装配、`QueryEngine` 和 `query.ts` 就更容易，因为它们都不是在“所有请求都一样”的前提下运行，而是建立在入口层已经完成模式拆分的基础上。
