# 第14章 工具作为手脚

安全规则可以贴在总控室，也可以直接贴在每台机器上。前者依赖调度员记忆，后者规则跟着机器走，换人也不会丢。Claude Code 管理工具的方式选了后者——每个工具自带一整套安全声明，调用方只读声明。这一选择决定了整个工具层能随工具数量增长而不膨胀。

原则有边界：这套设计假设写工具的人诚实声明安全属性，系统无法检测有人把写操作声明成了只读。原则是工具，不是教条。

## 14.1 声明放在工具里，调用方复杂度不随工具数量增长

大多数系统的工具是薄壳——名字、参数、执行函数，安全检查散落在调用方。工具数量增加，调用方条件分支跟着增加；漏一个检查，系统就暴露一个缺口。

Claude Code 把安全语义翻转进工具定义本身。每个工具实现 `Tool` 接口时，同时随身携带七个声明（[`src/Tool.ts#L398-L474`](https://github.com/xuhengzhi75/claude-code-source/blob/9e4d0c6d68748da4cc9623a89752ed2cf60af4ea/src/Tool.ts#L398-L474)）：`call()` 执行逻辑、`inputSchema` 参数校验（Zod 约束）、`checkPermissions()` 调用前权限确认、`isReadOnly()` 是否只读（决定能否在 plan 模式运行）、`isDestructive()` 是否不可逆、`isConcurrencySafe()` 是否可以并发执行、`renderToolUse()` / `renderToolResult()` 终端渲染方式。调用方只读这些声明，不自行判断，工具换了声明跟着换，调用方逻辑不动。

未填的属性全部走默认值。`buildTool()`（[`Tool.ts#L789`](https://github.com/xuhengzhi75/claude-code-source/blob/9e4d0c6d68748da4cc9623a89752ed2cf60af4ea/src/Tool.ts#L789)）用 `{ ...TOOL_DEFAULTS, ...def }` 合并，`TOOL_DEFAULTS`（[`L763`](https://github.com/xuhengzhi75/claude-code-source/blob/9e4d0c6d68748da4cc9623a89752ed2cf60af4ea/src/Tool.ts#L763)）的注释直接说明设计意图：

```
// Defaults (fail-closed where it matters):
// - isConcurrencySafe → false (assume not safe)
// - isReadOnly → false (assume writes)
```

忘了声明 `isReadOnly: true`，工具在 plan 模式下不能用，只是比应有的更保守。但如果默认是 `true`，一个真正写文件的工具就会在 plan 模式下悄悄跑起来。两个遗漏的代价不对等，所以默认选限制。想要更宽松的权限，必须显式声明。

`toAutoClassifierInput` 默认返回空字符串，即"不需要安全分类"。这个字段是 auto 模式（YOLO 模式）（通过安全分类模型拦截高风险操作的非交互执行模式）的接口——系统把工具调用序列化成紧凑格式发给分类模型，判断要不要拦截。`BashTool` 提交 `input.command`，`FileWriteTool` 提交路径，读文件和搜索类工具用空字符串，因为它们不改状态（[`yoloClassifier.ts#L375`](https://github.com/xuhengzhi75/claude-code-source/blob/main/src/utils/permissions/yoloClassifier.ts#L375)）。

## 14.2 并发规则由工具声明驱动，不由调用方判断

后厨里切菜和洗碗可以同时做，但有人用燃气灶时，其他人不能换煤气罐——不管换煤气罐本身有多安全，都得等灶熄了再说。工具调度遵循同一逻辑。

`canExecuteTool`（[`StreamingToolExecutor.ts#L129`](https://github.com/xuhengzhi75/claude-code-source/blob/main/src/services/tools/StreamingToolExecutor.ts#L129)）的判断只有一条规则：

```typescript
return (
  executingTools.length === 0 ||
  (isConcurrencySafe && executingTools.every(t => t.isConcurrencySafe))
)
```

所有正在跑的工具都安全，且新工具也安全，才能并发。否则排队。模型同时发出 Grep、Glob、Bash（写命令）三个调用：Grep 先到立即跑，Glob 到时只有 Grep 在执行（安全）所以立即并发，Bash 到时不安全，等 Grep/Glob 跑完再开始。Bash 之后来的所有工具，哪怕只是读文件，也得等 Bash 跑完——怕的是读到 Bash 写了一半的中间状态。

`isConcurrencySafe` 的含义是"不产生竞态"，不是"只读"。`TaskCreateTool` 是写操作，但声明为 safe，因为内部用 `proper-lockfile` 保护，并发不产生竞态（[`TaskCreateTool.ts#L71`](https://github.com/xuhengzhi75/claude-code-source/blob/main/src/tools/TaskCreateTool/TaskCreateTool.ts#L71)）。`BashTool` 的 safe 值是动态的：`isReadOnly(input) ?? false`，`cat`/`ls` 是 safe，`rm`/`mv` 不是（[`BashTool.tsx#L434`](https://github.com/xuhengzhi75/claude-code-source/blob/main/src/tools/BashTool/BashTool.tsx#L434)）。`isConcurrencySafe` 抛异常或 input 解析失败时，降级为串行，不会让系统变得更宽松（[`toolOrchestration.ts#L100`](https://github.com/xuhengzhi75/claude-code-source/blob/main/src/services/tools/toolOrchestration.ts#L100)）。

Bash 出错有级联效果，其他工具出错没有。错误统一包装成 `tool_result`（`is_error: true`），格式 `<tool_use_error>错误信息</tool_use_error>`，模型自行决定要不要重试。但 Bash 有单独处理（[`StreamingToolExecutor.ts#L354`](https://github.com/xuhengzhi75/claude-code-source/blob/main/src/services/tools/StreamingToolExecutor.ts#L354)）：

```typescript
// Only Bash errors cancel siblings. Bash commands often have implicit
// dependency chains (e.g. mkdir fails → subsequent commands pointless).
if (tool.block.name === BASH_TOOL_NAME) {
  this.hasErrored = true
  this.siblingAbortController.abort('sibling_error')
}
```

`mkdir` 失败了，后续依赖这个目录的命令没意义继续跑；但读 `a.txt` 失败，不影响读 `b.txt`。两种不同的依赖结构，两种不同的取消策略。

两条执行路径，行为细节不同。流式路径（`StreamingToolExecutor`）在模型流式输出时就开始排队，不等所有工具块到齐，遇到不能并发的工具就 `break`（[`L129-L151`](https://github.com/xuhengzhi75/claude-code-source/blob/main/src/services/tools/StreamingToolExecutor.ts#L129)）。批处理路径（`toolOrchestration`）等所有工具到齐，用 `partitionToolCalls` 分批——连续安全工具一批并发，不安全工具单独串行，并发批次的 context 修改暂存后按顺序回放，避免竞态（[`L96-L123`](https://github.com/xuhengzhi75/claude-code-source/blob/main/src/services/tools/toolOrchestration.ts#L96)）。用哪条路由 `config.gates.streamingToolExecution` 控制。并发上限 `CLAUDE_CODE_MAX_TOOL_USE_CONCURRENCY`（默认 10）是资源与安全的双重阀门（[`toolOrchestration.ts#L165`](https://github.com/xuhengzhi75/claude-code-source/blob/main/src/services/tools/toolOrchestration.ts#L165)）。

## 14.3 工具池组装有顺序约束，内建工具必须排在 MCP 工具之前

工具池组装分三步（[`src/tools.ts`](https://github.com/xuhengzhi75/claude-code-source/blob/9e4d0c6d68748da4cc9623a89752ed2cf60af4ea/src/tools.ts)），每步的约束各有来源。

`getAllBaseTools()`（[`L193`](https://github.com/xuhengzhi75/claude-code-source/blob/9e4d0c6d68748da4cc9623a89752ed2cf60af4ea/src/tools.ts#L193)）产出所有内建工具的全集。`getTools(permissionContext)`（[`L274`](https://github.com/xuhengzhi75/claude-code-source/blob/9e4d0c6d68748da4cc9623a89752ed2cf60af4ea/src/tools.ts#L274)）按权限过滤，`filterToolsByDenyRules()` 把 deny 规则命中的工具从列表里删掉（[`L257`](https://github.com/xuhengzhi75/claude-code-source/docs/book/chapters/ch14-工具.md)）。被 deny 的工具直接消失——模型看不到，不是"能看到但调用失败"，省去模型反复尝试被拒工具的 token 开销。

`assembleToolPool(permissionContext, mcpTools)`（[`L350`](https://github.com/xuhengzhi75/claude-code-source/blob/9e4d0c6d68748da4cc9623a89752ed2cf60af4ea/src/tools.ts#L350)）合并内建工具和 MCP 工具，内建工具排前面。这个顺序有性能约束：内建工具列表末尾有 cache breakpoint（提示词缓存的分割标记），MCP 工具插到中间会让 breakpoint 失效，所有下游请求都得重新计算，响应变慢、费用变高（[`L359`](https://github.com/xuhengzhi75/claude-code-source/blob/9e4d0c6d68748da4cc9623a89752ed2cf60af4ea/src/tools.ts#L359)）。

MCP 工具通过 `fetchToolsForClient`（[`client.ts#L1743`](https://github.com/xuhengzhi75/claude-code-source/blob/main/src/services/mcp/client.ts#L1743)）从服务器拉取，包装成实现 `Tool` 接口的对象后，和内建工具走同一套调度逻辑。`isConcurrencySafe` 读 `tool.annotations?.readOnlyHint ?? false`（[`client.ts#L1795`](https://github.com/xuhengzhi75/claude-code-source/blob/main/src/services/mcp/client.ts#L1795)），由 MCP 服务器通过协议声明，不声明则默认 false。deny 规则支持 `mcp__server` 前缀匹配，一条规则封堵整个 MCP 服务器的所有工具（[`tools.ts#L256`](https://github.com/xuhengzhi75/claude-code-source/blob/9e4d0c6d68748da4cc9623a89752ed2cf60af4ea/src/tools.ts#L256)）。

## 可迁移原则

安全默认值应该选限制，不选许可。遗漏"声明限制"的后果是功能受限，遗漏"声明许可"的后果是安全缺口——两个遗漏的代价不对等，所以默认限制，让需要宽松的人主动声明。

这条原则的适用条件是：存在两种代价不对称的遗漏，且写工具的人足够可信。如果工具声明本身不可信（比如第三方插件），还需要独立验证层，光靠声明驱动是不够的。
