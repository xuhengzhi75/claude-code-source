# 第14章 工具作为手脚

如果安全语义放在调用方，会发生什么？每个调用工具的地方都需要检查：这个工具是只读的吗？可以并发执行吗？需要用户确认吗？随着工具数量增加，这些检查会散落在代码各处，遗漏一处就会产生安全漏洞。

Claude Code 的做法是把安全语义放在工具定义里：工具自己声明自己是否只读、是否可并发、是否需要确认。调用方只需要读取这些声明，不需要自己判断。

## 14.1 Tool 是厚接口，不是 RPC 壳

[`src/Tool.ts#L398-L474`](https://github.com/xuhengzhi75/claude-code-source/blob/9e4d0c6d68748da4cc9623a89752ed2cf60af4ea/src/Tool.ts#L398-L474) 定义了 `Tool` 接口的核心字段。一个工具同时携带：`inputSchema`（基于 Zod 的 JSON Schema，用于参数校验）、`call()`（实际执行逻辑）、`checkPermissions()`（权限检查，决定这次调用是否需要用户确认）、`isReadOnly()`（是否只读，决定能否在 plan 模式下运行）、`isDestructive()`（是否破坏性操作，注释说明"只在执行不可逆操作时设置"）、`isConcurrencySafe()`（是否可以并发执行）、`renderToolUse()` / `renderToolResult()`（在终端 UI 里如何渲染）。

这些字段不是装饰，而是系统在执行前用来做决策的依据。`isReadOnly()` 决定 plan 模式下能不能运行，`isConcurrencySafe()` 决定多个工具调用能不能同时执行，`checkPermissions()` 决定是否弹出确认对话框。

这个设计假设工具开发者会正确声明安全属性。如果工具开发者声明了错误的属性（比如把写操作声明为只读），系统无法检测到这个错误。安全语义的正确性依赖工具开发者的诚实性。

## 14.2 `buildTool()` 的默认值是 fail-closed

[`Tool.ts#L763-L775`](https://github.com/xuhengzhi75/claude-code-source/blob/9e4d0c6d68748da4cc9623a89752ed2cf60af4ea/src/Tool.ts#L763-L775) 定义了 `TOOL_DEFAULTS`：

```typescript
const TOOL_DEFAULTS = {
  isEnabled: () => true,
  isConcurrencySafe: (_input?: unknown) => false,
  isReadOnly: (_input?: unknown) => false,
  isDestructive: (_input?: unknown) => false,
  checkPermissions: (...) => Promise.resolve({ behavior: 'allow', updatedInput: input }),
  toAutoClassifierInput: (_input?: unknown) => '',
  userFacingName: (_input?: unknown) => '',
}
```

注释（[`L752-L758`](https://github.com/xuhengzhi75/claude-code-source/blob/9e4d0c6d68748da4cc9623a89752ed2cf60af4ea/src/Tool.ts#L752-L758)）明确说明了设计意图：

```
// Defaults (fail-closed where it matters):
// - isConcurrencySafe → false (assume not safe)
// - isReadOnly → false (assume writes)
// - isDestructive → false
```

[`buildTool()`（`L789`）](https://github.com/xuhengzhi75/claude-code-source/blob/9e4d0c6d68748da4cc9623a89752ed2cf60af4ea/src/Tool.ts#L789) 用 `{ ...TOOL_DEFAULTS, ...def }` 合并，工具定义里的字段覆盖默认值。

`toAutoClassifierInput` 默认返回 `''`，意思是"该工具对安全分类器没有意义"。这个字段是 auto 模式（YOLO 模式）安全分类器的输入编码接口——分类器把工具调用序列化为紧凑格式，发给专门的安全分类模型判断是否应该阻止。`BashTool` 返回 `input.command`，`FileWriteTool` 返回 `input.file_path`，`WebFetchTool` 返回 `input.url`；`GrepTool`、`GlobTool`、`FileReadTool` 等只读工具使用默认空字符串——它们不修改系统状态，不需要安全分类（[`src/utils/permissions/yoloClassifier.ts#L375-L416`](https://github.com/xuhengzhi75/claude-code-source/blob/main/src/utils/permissions/yoloClassifier.ts#L375)）。

这是 secure-by-default 的设计风格。工具作者如果想要更宽的权限（比如声明工具是只读的，允许在 plan 模式下运行），必须显式声明。如果忘了声明，系统会选择更保守的行为，而不是更宽松的行为。在任何需要安全约束的系统里，默认值应该是最保守的，而不是最宽松的——这把安全责任从"记得加限制"转移到"记得声明宽松"，减少了遗漏的可能性。

## 14.3 并发调度的反直觉约束

`isConcurrencySafe` 的精确调度条件出乎大多数人的预期。[`StreamingToolExecutor.ts#L129-L134`](https://github.com/xuhengzhi75/claude-code-source/blob/main/src/services/tools/StreamingToolExecutor.ts#L129) 的 `canExecuteTool`：

```typescript
private canExecuteTool(isConcurrencySafe: boolean): boolean {
  const executingTools = this.tools.filter(t => t.status === 'executing')
  return (
    executingTools.length === 0 ||
    (isConcurrencySafe && executingTools.every(t => t.isConcurrencySafe))
  )
}
```

并发的条件是"当前所有执行中的工具都是 safe，且新工具也是 safe"——两个条件必须同时满足。一旦有一个非 safe 工具在执行，所有后来的工具（包括 safe 的）都必须等待。

具体例子：模型同时发出 Grep + Glob + Bash 三个调用：
1. Grep 到达 → `canExecuteTool(true)` → 执行中工具为空 → 立即执行
2. Glob 到达 → `canExecuteTool(true)` → 执行中只有 Grep（safe）→ 立即并发
3. Bash 到达 → `canExecuteTool(false)` → 执行中有 Grep/Glob → 等待
4. Grep/Glob 完成 → `processQueue` 重新检查 → Bash 开始执行

非 safe 工具可能修改文件系统状态，safe 工具如果在此期间并发读取，可能读到中间态——这是约束如此严格的原因，代码里没有显式注释，但逻辑是显然的。

并发上限由环境变量 `CLAUDE_CODE_MAX_TOOL_USE_CONCURRENCY` 控制（默认 10），注释说是"资源与安全双重阀门"（[`toolOrchestration.ts#L165-L167`](https://github.com/xuhengzhi75/claude-code-source/blob/main/src/services/tools/toolOrchestration.ts#L165)）。

**`isConcurrencySafe` 的语义不是"只读"，而是"并发调用不产生竞态"。** `TaskCreateTool` 和 `TaskUpdateTool` 是写操作，但声明为 `safe = true`——因为它们内部有 `proper-lockfile` 保护，并发调用安全（[`TaskCreateTool.ts#L71`](https://github.com/xuhengzhi75/claude-code-source/blob/main/src/tools/TaskCreateTool/TaskCreateTool.ts#L71)）。`BashTool` 的 safe 值是动态的：`isReadOnly(input) ?? false`——只有只读命令才 safe，由 input 决定（[`BashTool.tsx#L434`](https://github.com/xuhengzhi75/claude-code-source/blob/main/src/tools/BashTool/BashTool.tsx#L434)）。

还有一个安全兜底（[`toolOrchestration.ts#L100-L101`](https://github.com/xuhengzhi75/claude-code-source/blob/main/src/services/tools/toolOrchestration.ts#L100) 注释）："只要 input 解析失败或 `isConcurrencySafe` 抛错，就降级为串行"——`isConcurrencySafe` 的失败路径是 fail-closed，不是 fail-open。

## 14.4 两套执行路径

并发调度有两套路径，行为不同。

**StreamingToolExecutor**（`config.gates.streamingToolExecution` 为 true 时使用）：工具在流式传输时就开始执行，不等待所有工具块到达。`canExecuteTool` 在每次 `addTool` 时检查，`processQueue` 遇到不能执行的非 safe 工具就 `break`，保证顺序（[`StreamingToolExecutor.ts#L129-L151`](https://github.com/xuhengzhi75/claude-code-source/blob/main/src/services/tools/StreamingToolExecutor.ts#L129)）。

**toolOrchestration**（非流式路径，`runTools`）：先用 `partitionToolCalls` 把所有工具分批——连续的 safe 工具合成一批并发执行，非 safe 工具单独一批串行执行。并发批次的 `contextModifier` 暂存后按顺序回放，避免并发写 context 竞态（[`toolOrchestration.ts#L96-L123`](https://github.com/xuhengzhi75/claude-code-source/blob/main/src/services/tools/toolOrchestration.ts#L96)）。

## 14.5 Bash 错误的级联取消

工具执行失败时，错误被包装为 `tool_result` 消息（`is_error: true`），内容格式为 `<tool_use_error>错误信息</tool_use_error>`。模型自行决定是否重试——系统不干预模型的决策。

但有一个级联取消机制（[`StreamingToolExecutor.ts#L354-L363`](https://github.com/xuhengzhi75/claude-code-source/blob/main/src/services/tools/StreamingToolExecutor.ts#L354)）：

```typescript
// Only Bash errors cancel siblings. Bash commands often have implicit
// dependency chains (e.g. mkdir fails → subsequent commands pointless).
// Read/WebFetch/etc are independent — one failure shouldn't nuke the rest.
if (tool.block.name === BASH_TOOL_NAME) {
  this.hasErrored = true
  this.siblingAbortController.abort('sibling_error')
}
```

只有 Bash 工具的错误会取消并发中的其他工具，其他工具（Read/WebFetch 等）的错误不影响兄弟工具。原因是 Bash 命令之间有隐式依赖链（`mkdir` 失败 → 后续依赖这个目录的命令没意义），而文件读取之间是独立的——一个文件读不到，不影响另一个文件的读取。

三种合成错误消息：`sibling_error`（"Cancelled: parallel tool call errored"）、`user_interrupted`（`REJECT_MESSAGE`，让 UI 显示正确文案）、`streaming_fallback`（"Streaming fallback - tool execution discarded"）。

## 14.6 工具池的三步组装

`src/tools.ts` 管理工具池，流程是三步。

先是 [`getAllBaseTools()`（`tools.ts#L193`）](https://github.com/xuhengzhi75/claude-code-source/blob/9e4d0c6d68748da4cc9623a89752ed2cf60af4ea/src/tools.ts#L193) 给出所有内建工具的理论全集。注释（[`L194-L196`](https://github.com/xuhengzhi75/claude-code-source/blob/9e4d0c6d68748da4cc9623a89752ed2cf60af4ea/src/tools.ts#L194-L196)）说明了设计意图：先全集、后过滤，便于统一审计工具可见性与 system prompt 缓存一致性。

然后是 [`getTools(permissionContext)`（`tools.ts#L274`）](https://github.com/xuhengzhi75/claude-code-source/blob/9e4d0c6d68748da4cc9623a89752ed2cf60af4ea/src/tools.ts#L274) 按当前权限上下文过滤。[`filterToolsByDenyRules()`（`tools.ts#L265`）](https://github.com/xuhengzhi75/claude-code-source/blob/9e4d0c6d68748da4cc9623a89752ed2cf60af4ea/src/tools.ts#L265) 根据 deny 规则裁剪——注释（[`L257-L264`](https://github.com/xuhengzhi75/claude-code-source/blob/9e4d0c6d68748da4cc9623a89752ed2cf60af4ea/src/tools.ts#L257-L264)）说明这个过滤在工具可见性层面生效，模型根本看不到被 deny 的工具，防止模型反复尝试调用被拒绝的工具浪费 token。

最后是 [`assembleToolPool(permissionContext, mcpTools)`（`tools.ts#L350`）](https://github.com/xuhengzhi75/claude-code-source/blob/9e4d0c6d68748da4cc9623a89752ed2cf60af4ea/src/tools.ts#L350) 把内建工具和 MCP 工具合并。

## 14.7 MCP 工具的注册路径

MCP 工具通过 `fetchToolsForClient`（[`client.ts#L1743`](https://github.com/xuhengzhi75/claude-code-source/blob/main/src/services/mcp/client.ts#L1743)）从 MCP 服务器动态拉取：调用协议的 `tools/list` 方法，对每个工具清理 Unicode，然后包装为实现 `Tool` 接口的对象。`inputSchema` 来自 MCP 服务器动态提供的 JSON Schema，不是静态定义的。

`isConcurrencySafe` 读取 `tool.annotations?.readOnlyHint ?? false`（[`client.ts#L1795-L1797`](https://github.com/xuhengzhi75/claude-code-source/blob/main/src/services/mcp/client.ts#L1795)）——由 MCP 服务器通过协议声明，不是硬编码 false。如果 MCP 服务器没有声明 `readOnlyHint`，默认 false（保守策略）。

`assembleToolPool` 把内建工具（按名排序）放前面，MCP 工具（按名排序）追加在后（[`tools.ts#L377-L380`](https://github.com/xuhengzhi75/claude-code-source/blob/9e4d0c6d68748da4cc9623a89752ed2cf60af4ea/src/tools.ts#L377)）。这个顺序服务于 cache breakpoint：服务器在内建工具列表末尾设置了 cache breakpoint，如果 MCP 工具插入到内建工具之间，breakpoint 失效，所有下游 cache key 都会失效，每次请求需要重新计算（[`tools.ts#L359-L364`](https://github.com/xuhengzhi75/claude-code-source/blob/9e4d0c6d68748da4cc9623a89752ed2cf60af4ea/src/tools.ts#L359)）。

deny 规则的 `mcp__server` 前缀可以一次性隐藏整个 MCP 服务器的所有工具——`getDenyRuleForTool` 支持前缀匹配，一条规则封堵一个服务器（[`tools.ts#L256-L275`](https://github.com/xuhengzhi75/claude-code-source/blob/9e4d0c6d68748da4cc9623a89752ed2cf60af4ea/src/tools.ts#L256)）。

## 14.8 心智模型验证题

如果一个工具忘了声明 `isReadOnly: true`，会发生什么？

`buildTool()` 用 `TOOL_DEFAULTS` 补齐，`isReadOnly` 默认 `false`（非只读），这个工具在 plan 模式下不能运行，即使它实际是只读的。工具能用，只是比应有的更保守——不会产生安全问题，但限制了使用场景。

进阶题：如果模型同时发出 TaskCreate + Bash + FileRead 三个调用，在 StreamingToolExecutor 路径下会怎么调度？提示：TaskCreate 是 safe（有锁保护），Bash 是 safe 或不 safe 取决于命令，FileRead 是 safe。如果 Bash 是只读命令（safe），三个工具可能全部并发；如果 Bash 是写命令（not safe），则要等 TaskCreate/FileRead 完成后 Bash 才能执行。

## 14.9 本章小结

工具系统的核心是把安全语义放在工具定义里，而不是放在调用方。

并发调度有一个反直觉约束：不是"safe 工具可以随时并发"，而是"必须当前所有执行中的工具都是 safe，新 safe 工具才能并发"（[`StreamingToolExecutor.ts#L129`](https://github.com/xuhengzhi75/claude-code-source/blob/main/src/services/tools/StreamingToolExecutor.ts#L129)）。`isConcurrencySafe` 的语义是"并发调用不产生竞态"，不是"只读"——TaskCreate/TaskUpdate 是写操作，但因为有文件锁保护而声明为 safe。

Bash 错误会取消兄弟工具，其他工具的错误不会——因为 Bash 命令之间有隐式依赖链，读操作之间是独立的（[`StreamingToolExecutor.ts#L354`](https://github.com/xuhengzhi75/claude-code-source/blob/main/src/services/tools/StreamingToolExecutor.ts#L354)）。

MCP 工具的 `isConcurrencySafe` 由服务器通过 `readOnlyHint` 声明，不是硬编码——但服务器不声明时默认 false（[`client.ts#L1795`](https://github.com/xuhengzhi75/claude-code-source/blob/main/src/services/mcp/client.ts#L1795)）。

安全语义放在工具里，不放在调用方。这个原则让系统在工具数量增长时，安全性不会随之退化。
