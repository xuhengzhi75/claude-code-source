# 第14章 工具作为手脚

假设你在管一个工厂，车间里有几十台机器。每台机器都有说明书，写着：这台机器会不会改动产品？它能不能和其他机器同时运行？操作前需不需要报告主管？

你可以把这些规则写在总控室，让调度员记住每台机器的情况。但调度员会换人，记忆会出错。更稳妥的做法是：把规则直接贴在机器上，谁来调度都能看到，不依赖任何人的记忆。

Claude Code 管理工具的方式，和这个一模一样。

## 14.1 工具随身携带自己的"说明书"

大多数系统把工具设计成薄壳：名字、参数、执行函数，仅此而已。安全检查、权限判断、并发控制，这些逻辑散落在调用方。工具越来越多，调用方越来越臃肿，漏掉一个检查就出问题。

Claude Code 反过来。每个工具随身携带一整套声明（[`src/Tool.ts#L398-L474`](https://github.com/xuhengzhi75/claude-code-source/blob/9e4d0c6d68748da4cc9623a89752ed2cf60af4ea/src/Tool.ts#L398-L474)）：

- `call()` — 实际做什么
- `inputSchema` — 接受什么参数（Zod 校验）
- `checkPermissions()` — 这次调用需要用户点确认吗
- `isReadOnly()` — 会不会改东西（决定能否在 plan 模式运行）
- `isDestructive()` — 是不是不可逆操作
- `isConcurrencySafe()` — 能不能和其他工具同时跑
- `renderToolUse()` / `renderToolResult()` — 在终端里怎么显示

调用方只需要读这些声明，不需要自己判断。工具换了，声明跟着换，调用方的逻辑不用动。

有一点代价：这套设计假设写工具的人会诚实地声明安全属性。如果有人把一个写操作声明成只读，系统检测不出来——这不是架构漏洞，而是信任边界的选择。

## 14.2 不声明就默认最保守

写一个新工具，懒得填安全属性，会怎样？

`buildTool()`（[`Tool.ts#L789`](https://github.com/xuhengzhi75/claude-code-source/blob/9e4d0c6d68748da4cc9623a89752ed2cf60af4ea/src/Tool.ts#L789)）用 `{ ...TOOL_DEFAULTS, ...def }` 合并，你没填的字段全部走默认值。[`TOOL_DEFAULTS`（`Tool.ts#L763-L775`）](https://github.com/xuhengzhi75/claude-code-source/blob/9e4d0c6d68748da4cc9623a89752ed2cf60af4ea/src/Tool.ts#L763-L775) 的注释写得很直接：

```
// Defaults (fail-closed where it matters):
// - isConcurrencySafe → false (assume not safe)
// - isReadOnly → false (assume writes)
// - isDestructive → false
```

翻译成人话：默认假设你的工具会写东西、不能并发、是有破坏性的。想要更宽松的权限，必须自己声明。

这个选择有反直觉的地方——你可能以为默认应该是"允许"，否则新工具不就什么都不能做了？但安全设计的逻辑正相反：遗漏一个"限制"的代价，比遗漏一个"放行"的代价大得多。如果你忘了声明 `isReadOnly: true`，最坏的结果是工具在 plan 模式下不能用，只是比应有的更保守。但如果你忘了声明 `isReadOnly: false`，而系统默认 true，那么一个真正会写文件的工具就在 plan 模式下跑起来了——这是真正的安全漏洞。

还有一个细节：`toAutoClassifierInput` 默认返回空字符串，意思是"这个工具不需要安全分类"。这个字段是 auto 模式（也叫 YOLO 模式）下的接口——系统把工具调用序列化成紧凑格式，发给专门的安全分类模型判断要不要拦截。`BashTool` 提交 `input.command`，`FileWriteTool` 提交 `input.file_path`，`WebFetchTool` 提交 `input.url`；读文件、搜索这类工具用默认空字符串，因为它们不改状态，不需要判断（[`yoloClassifier.ts#L375-L416`](https://github.com/xuhengzhi75/claude-code-source/blob/main/src/utils/permissions/yoloClassifier.ts#L375)）。

## 14.3 后厨调度：为什么安全工具也要排队等不安全工具

想象你在一家餐厅后厨。切菜可以和洗碗同时做，端盘子可以和备料同时做——这些工作互不干扰。但如果有人在用燃气灶，其他人就不能同时换煤气罐，不管换煤气罐本身多安全，都得等灶熄了再说。

Claude Code 的工具调度和这个逻辑一样。[`StreamingToolExecutor.ts#L129-L134`](https://github.com/xuhengzhi75/claude-code-source/blob/main/src/services/tools/StreamingToolExecutor.ts#L129) 的判断逻辑：

```typescript
private canExecuteTool(isConcurrencySafe: boolean): boolean {
  const executingTools = this.tools.filter(t => t.status === 'executing')
  return (
    executingTools.length === 0 ||
    (isConcurrencySafe && executingTools.every(t => t.isConcurrencySafe))
  )
}
```

翻译：**当前所有正在跑的工具都是安全的，且新来的工具也是安全的，才能并行。否则排队。**

举个例子：模型同时发出 Grep + Glob + Bash 三个调用。

1. Grep 到达 → 执行中工具为空 → 立即开跑
2. Glob 到达 → 执行中只有 Grep（安全）→ 立即并发
3. Bash 到达 → Bash 是写命令，不安全 → 等 Grep/Glob 跑完
4. Grep/Glob 完成 → 队列重新检查 → Bash 开始执行

为什么安全的 Grep 要等不安全的 Bash？因为方向反了——等的是 Bash 等前面的 Grep/Glob，不是 Grep/Glob 等 Bash。一旦 Bash 开始，后面来的所有工具——哪怕只是个读文件——都得排队等 Bash 跑完，怕的是读到 Bash 写了一半的中间状态。

并发上限是 `CLAUDE_CODE_MAX_TOOL_USE_CONCURRENCY`（默认 10），注释说是"资源与安全双重阀门"（[`toolOrchestration.ts#L165`](https://github.com/xuhengzhi75/claude-code-source/blob/main/src/services/tools/toolOrchestration.ts#L165)）。

最后一个容易搞错的点：**`isConcurrencySafe` 不等于"只读"。** `TaskCreateTool` 和 `TaskUpdateTool` 是写操作，但声明为 safe——因为它们内部用 `proper-lockfile` 保护，并发调用不会产生竞态（[`TaskCreateTool.ts#L71`](https://github.com/xuhengzhi75/claude-code-source/blob/main/src/tools/TaskCreateTool/TaskCreateTool.ts#L71)）。`BashTool` 更动态：它的 safe 值是 `isReadOnly(input) ?? false`，一条只读命令（`cat`、`ls`）是 safe，一条写命令（`rm`、`mv`）不是（[`BashTool.tsx#L434`](https://github.com/xuhengzhi75/claude-code-source/blob/main/src/tools/BashTool/BashTool.tsx#L434)）。同一个工具，命令不同，并发权限不同。

还有一个安全兜底（[`toolOrchestration.ts#L100-L101`](https://github.com/xuhengzhi75/claude-code-source/blob/main/src/services/tools/toolOrchestration.ts#L100)）：input 解析失败，或 `isConcurrencySafe` 本身抛异常，就降级为串行。错误路径不会让系统变得更宽松，只会更保守。

## 14.4 流式路径 vs 批处理路径

并发调度有两条路，行为不一样。

**流式路径**（`StreamingToolExecutor`）：工具在模型流式输出时就开始排队，不等所有工具块到齐。`canExecuteTool` 在每个工具到达时检查，队列里遇到不能并发的工具就 `break`，后面的全部等待（[`StreamingToolExecutor.ts#L129-L151`](https://github.com/xuhengzhi75/claude-code-source/blob/main/src/services/tools/StreamingToolExecutor.ts#L129)）。

**批处理路径**（`toolOrchestration`）：先等所有工具到齐，再用 `partitionToolCalls` 分批——连续的安全工具合为一批并发跑，不安全工具单独一批串行跑。并发批次的上下文修改暂存，等这批跑完再按顺序回放，避免并发写 context 出竞态（[`toolOrchestration.ts#L96-L123`](https://github.com/xuhengzhi75/claude-code-source/blob/main/src/services/tools/toolOrchestration.ts#L96)）。

两条路用哪条，由 `config.gates.streamingToolExecution` 控制。

## 14.5 Bash 出错，其他工具也跟着被取消

工具失败时，错误被包装成 `tool_result`（`is_error: true`），内容格式 `<tool_use_error>错误信息</tool_use_error>`。接下来怎么办，是模型的事——系统不干预，模型自己决定要不要重试。

但有一个特殊处理（[`StreamingToolExecutor.ts#L354-L363`](https://github.com/xuhengzhi75/claude-code-source/blob/main/src/services/tools/StreamingToolExecutor.ts#L354)）：

```typescript
// Only Bash errors cancel siblings. Bash commands often have implicit
// dependency chains (e.g. mkdir fails → subsequent commands pointless).
// Read/WebFetch/etc are independent — one failure shouldn't nuke the rest.
if (tool.block.name === BASH_TOOL_NAME) {
  this.hasErrored = true
  this.siblingAbortController.abort('sibling_error')
}
```

Bash 出错，会取消所有正在并发的其他工具。其他工具出错，不影响任何人。

为什么只有 Bash？注释给了答案：Bash 命令之间有隐式依赖链。`mkdir /tmp/foo` 失败了，后面所有依赖 `/tmp/foo` 的命令都没意义继续跑。但读文件和读文件之间是独立的，读 `a.txt` 失败，根本不影响读 `b.txt`。

三种取消消息：`sibling_error`（"Cancelled: parallel tool call errored"）、`user_interrupted`（让 UI 显示正确的中断文案）、`streaming_fallback`（"Streaming fallback - tool execution discarded"）。

## 14.6 工具池怎么组装起来的

系统启动时，工具池分三步组装（[`src/tools.ts`](https://github.com/xuhengzhi75/claude-code-source/blob/9e4d0c6d68748da4cc9623a89752ed2cf60af4ea/src/tools.ts)）：

先是 `getAllBaseTools()`（[`L193`](https://github.com/xuhengzhi75/claude-code-source/blob/9e4d0c6d68748da4cc9623a89752ed2cf60af4ea/src/tools.ts#L193)）列出所有内建工具。注释说这是先全集再过滤，便于统一审计工具可见性。

然后是 `getTools(permissionContext)`（[`L274`](https://github.com/xuhengzhi75/claude-code-source/blob/9e4d0c6d68748da4cc9623a89752ed2cf60af4ea/src/tools.ts#L274)）按当前权限过滤——`filterToolsByDenyRules()` 把 deny 规则覆盖的工具从列表里删掉。模型根本看不到被 deny 的工具，不是"能看到但调用失败"，而是直接不存在。好处是模型不会反复尝试调用一个总被拒绝的工具，浪费 token（[`L257-L264`](https://github.com/xuhengzhi75/claude-code-source/blob/9e4d0c6d68748da4cc9623a89752ed2cf60af4ea/src/tools.ts#L257)）。

最后是 `assembleToolPool(permissionContext, mcpTools)`（[`L350`](https://github.com/xuhengzhi75/claude-code-source/blob/9e4d0c6d68748da4cc9623a89752ed2cf60af4ea/src/tools.ts#L350)）把内建工具和 MCP 工具合并。顺序是内建工具排前面，MCP 工具追加在后（[`L377-L380`](https://github.com/xuhengzhi75/claude-code-source/blob/9e4d0c6d68748da4cc9623a89752ed2cf60af4ea/src/tools.ts#L377)）。为什么顺序重要？因为内建工具列表末尾有 cache breakpoint，MCP 工具如果插到内建工具中间，breakpoint 失效，每次请求都得重新计算——这会让响应变慢，费用变高（[`L359-L364`](https://github.com/xuhengzhi75/claude-code-source/blob/9e4d0c6d68748da4cc9623a89752ed2cf60af4ea/src/tools.ts#L359)）。

## 14.7 MCP 工具：动态接入，规则统一

除了内建工具，Claude Code 支持通过 MCP 协议接入外部工具。MCP 工具通过 `fetchToolsForClient`（[`client.ts#L1743`](https://github.com/xuhengzhi75/claude-code-source/blob/main/src/services/mcp/client.ts#L1743)）从服务器动态拉取：调用 `tools/list`，清理一下 Unicode，然后包装成实现 `Tool` 接口的对象。

包装后的 MCP 工具和内建工具走同一套调度逻辑，不区别对待。区别在于声明来源：内建工具的 `isConcurrencySafe` 是代码里写死的，MCP 工具的是读 `tool.annotations?.readOnlyHint ?? false`（[`client.ts#L1795`](https://github.com/xuhengzhi75/claude-code-source/blob/main/src/services/mcp/client.ts#L1795)）——由 MCP 服务器通过协议声明，服务器不声明就默认 false（保守策略）。

deny 规则支持 `mcp__server` 前缀匹配，一条规则可以封堵整个 MCP 服务器的所有工具，不用一个个列举（[`L256-L275`](https://github.com/xuhengzhi75/claude-code-source/blob/9e4d0c6d68748da4cc9623a89752ed2cf60af4ea/src/tools.ts#L256)）。

## 14.8 验证一下你的理解

如果一个工具忘了声明 `isReadOnly: true`，会发生什么？

系统用默认值 `false` 补齐，这个工具在 plan 模式下不能运行——即使它实际上什么都不会改。工具能用，只是比应有的更保守，不会有安全问题，只是限制了使用场景。

进阶一点：模型同时发出 TaskCreate + Bash（写命令）+ FileRead 三个调用，在流式路径下怎么调度？TaskCreate 是 safe（有锁），FileRead 是 safe，Bash 是写命令，not safe。

先到的两个（假设 TaskCreate 和 FileRead）并发跑。Bash 到达时，执行中有正在跑的工具，而 Bash 自己不是 safe——等待。TaskCreate 和 FileRead 完成后，Bash 开始执行。整个过程，FileRead 没有等 Bash，Bash 等了 TaskCreate/FileRead。

## 14.9 小结

这章的核心就一件事：**安全语义放在工具定义里，不放在调用方。**

这个决定带来了三个连锁效果：

默认值全部保守（fail-closed），写工具的人必须主动声明想要的宽松权限，不是被动声明限制。

并发调度的规则是"全员安全才能并发"，不是"我安全我就上"——一个不安全的工具会让整个队列暂停，直到它跑完。

`isConcurrencySafe` 的语义是"并发不产生竞态"，不是"只读"——TaskCreate 是写操作但可以并发，因为有锁；BashTool 根据命令内容动态判断，不是固定答案。

工具数量增加，调用方不需要知道每个工具的安全属性——那是工具自己的事。这是这套设计真正想解决的问题。
