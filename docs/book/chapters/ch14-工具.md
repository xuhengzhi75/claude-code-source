# 第14章 工具作为手脚

工厂里有几十台机器，安全规则可以贴在总控室，也可以直接贴在每台机器上。前者依赖调度员记忆，后者规则跟着机器走，换人也不会丢。Claude Code 管理工具的方式是后者。

## 14.1 每个工具自带一套规矩

大多数系统把工具设计成薄壳：名字、参数、执行函数，安全检查散落在调用方。工具越来越多，调用方越来越臃肿，漏掉一个检查就出问题。

Claude Code 反过来。每个工具随身携带一整套声明（[`src/Tool.ts#L398-L474`](https://github.com/xuhengzhi75/claude-code-source/blob/9e4d0c6d68748da4cc9623a89752ed2cf60af4ea/src/Tool.ts#L398-L474)）：

- `call()` — 实际做什么
- `inputSchema` — 接受什么参数（Zod 校验）
- `checkPermissions()` — 这次调用需要用户点确认吗
- `isReadOnly()` — 会不会改东西（决定能否在 plan 模式运行）
- `isDestructive()` — 是不是不可逆操作
- `isConcurrencySafe()` — 能不能和其他工具同时跑
- `renderToolUse()` / `renderToolResult()` — 在终端里怎么显示

调用方只需要读这些声明，不需要自己判断。工具换了，声明跟着换，调用方的逻辑不用动。代价是：这套设计假设写工具的人会诚实声明安全属性，系统无法检测有人把写操作声明成了只读。

## 14.2 不声明就默认最保守

写一个新工具，懒得填安全属性，会怎样？

`buildTool()`（[`Tool.ts#L789`](https://github.com/xuhengzhi75/claude-code-source/blob/9e4d0c6d68748da4cc9623a89752ed2cf60af4ea/src/Tool.ts#L789)）用 `{ ...TOOL_DEFAULTS, ...def }` 合并，没填的字段全部走默认值。[`TOOL_DEFAULTS`（`L763`）](https://github.com/xuhengzhi75/claude-code-source/blob/9e4d0c6d68748da4cc9623a89752ed2cf60af4ea/src/Tool.ts#L763) 的注释写得直接：

```
// Defaults (fail-closed where it matters):
// - isConcurrencySafe → false (assume not safe)
// - isReadOnly → false (assume writes)
```

想要更宽松的权限，必须显式声明。忘了声明 `isReadOnly: true`，工具在 plan 模式下不能用，只是比应有的更保守；但如果系统默认 true，一个真正会写文件的工具就在 plan 模式下跑起来了——两个遗漏的代价不对等，所以默认选保守。

`toAutoClassifierInput` 默认返回空字符串，意思是"不需要安全分类"。这个字段是 auto 模式（YOLO 模式）的接口——系统把工具调用序列化成紧凑格式，发给安全分类模型判断要不要拦截。`BashTool` 提交 `input.command`，`FileWriteTool` 提交路径，读文件和搜索类工具用空字符串，因为它们不改状态（[`yoloClassifier.ts#L375`](https://github.com/xuhengzhi75/claude-code-source/blob/main/src/utils/permissions/yoloClassifier.ts#L375)）。

## 14.3 安全工具也得等不安全工具跑完

后厨里切菜和洗碗可以同时做，但有人用燃气灶时，其他人不能换煤气罐——不管换煤气罐本身多安全，都得等灶熄了再说。

工具调度的逻辑一样。[`canExecuteTool`（`StreamingToolExecutor.ts#L129`）](https://github.com/xuhengzhi75/claude-code-source/blob/main/src/services/tools/StreamingToolExecutor.ts#L129) 的判断：

```typescript
return (
  executingTools.length === 0 ||
  (isConcurrencySafe && executingTools.every(t => t.isConcurrencySafe))
)
```

**所有正在跑的工具都安全，且新工具也安全，才能并行。否则排队。**

模型同时发出 Grep + Glob + Bash（写命令）三个调用：Grep 先到立即跑，Glob 到时执行中只有 Grep（安全）立即并发，Bash 到时不安全，等 Grep/Glob 跑完再开始。Bash 之后来的所有工具，哪怕只是读文件，也得等 Bash 跑完——怕的是读到 Bash 写了一半的中间状态。

几个容易搞错的细节：

**`isConcurrencySafe` 不等于只读。** `TaskCreateTool` 是写操作，但声明为 safe——因为内部有 `proper-lockfile` 保护，并发不产生竞态（[`TaskCreateTool.ts#L71`](https://github.com/xuhengzhi75/claude-code-source/blob/main/src/tools/TaskCreateTool/TaskCreateTool.ts#L71)）。`BashTool` 的 safe 值是动态的：`isReadOnly(input) ?? false`，`cat`/`ls` 是 safe，`rm`/`mv` 不是（[`BashTool.tsx#L434`](https://github.com/xuhengzhi75/claude-code-source/blob/main/src/tools/BashTool/BashTool.tsx#L434)）。

**出错时 fail-closed。** `isConcurrencySafe` 抛异常或 input 解析失败，降级为串行，不会让系统变得更宽松（[`toolOrchestration.ts#L100`](https://github.com/xuhengzhi75/claude-code-source/blob/main/src/services/tools/toolOrchestration.ts#L100)）。

并发上限 `CLAUDE_CODE_MAX_TOOL_USE_CONCURRENCY`（默认 10），是资源与安全的双重阀门（[`toolOrchestration.ts#L165`](https://github.com/xuhengzhi75/claude-code-source/blob/main/src/services/tools/toolOrchestration.ts#L165)）。

## 14.4 流式和批处理，执行方式不一样

两条执行路径，行为不同。

**流式路径**（`StreamingToolExecutor`）：工具在模型流式输出时就开始排队，不等所有工具块到齐，遇到不能并发的工具就 `break`（[`L129-L151`](https://github.com/xuhengzhi75/claude-code-source/blob/main/src/services/tools/StreamingToolExecutor.ts#L129)）。

**批处理路径**（`toolOrchestration`）：等所有工具到齐，用 `partitionToolCalls` 分批——连续安全工具一批并发，不安全工具单独串行。并发批次的 context 修改暂存后按顺序回放，避免竞态（[`L96-L123`](https://github.com/xuhengzhi75/claude-code-source/blob/main/src/services/tools/toolOrchestration.ts#L96)）。

用哪条路由 `config.gates.streamingToolExecution` 控制。

## 14.5 Bash 出错，兄弟工具跟着被取消

工具失败时，错误包装成 `tool_result`（`is_error: true`），内容格式 `<tool_use_error>错误信息</tool_use_error>`。模型自行决定要不要重试，系统不干预。

但 Bash 有特殊处理（[`StreamingToolExecutor.ts#L354`](https://github.com/xuhengzhi75/claude-code-source/blob/main/src/services/tools/StreamingToolExecutor.ts#L354)）：

```typescript
// Only Bash errors cancel siblings. Bash commands often have implicit
// dependency chains (e.g. mkdir fails → subsequent commands pointless).
if (tool.block.name === BASH_TOOL_NAME) {
  this.hasErrored = true
  this.siblingAbortController.abort('sibling_error')
}
```

Bash 出错，取消所有并发中的兄弟工具。其他工具出错，不影响任何人。`mkdir` 失败了，后面依赖这个目录的命令没意义继续跑；但读 `a.txt` 失败，不影响读 `b.txt`——这是两种不同的依赖结构。

## 14.6 工具池按固定顺序拼装，顺序不能乱

三步（[`src/tools.ts`](https://github.com/xuhengzhi75/claude-code-source/blob/9e4d0c6d68748da4cc9623a89752ed2cf60af4ea/src/tools.ts)）：

`getAllBaseTools()`（[`L193`](https://github.com/xuhengzhi75/claude-code-source/blob/9e4d0c6d68748da4cc9623a89752ed2cf60af4ea/src/tools.ts#L193)）给出所有内建工具的全集。

`getTools(permissionContext)`（[`L274`](https://github.com/xuhengzhi75/claude-code-source/blob/9e4d0c6d68748da4cc9623a89752ed2cf60af4ea/src/tools.ts#L274)）按权限过滤。`filterToolsByDenyRules()` 把 deny 规则覆盖的工具从列表里删掉——模型直接看不到，不是"能看到但调用失败"，省去模型反复尝试被拒工具的 token 浪费（[`L257`](https://github.com/xuhengzhi75/claude-code-source/blob/9e4d0c6d68748da4cc9623a89752ed2cf60af4ea/src/tools.ts#L257)）。

`assembleToolPool(permissionContext, mcpTools)`（[`L350`](https://github.com/xuhengzhi75/claude-code-source/blob/9e4d0c6d68748da4cc9623a89752ed2cf60af4ea/src/tools.ts#L350)）合并内建工具和 MCP 工具，内建工具排前面。这个顺序不是随意的：内建工具列表末尾有 cache breakpoint，MCP 工具插到中间会让 breakpoint 失效，所有下游请求都得重新计算，响应变慢、费用变高（[`L359`](https://github.com/xuhengzhi75/claude-code-source/blob/9e4d0c6d68748da4cc9623a89752ed2cf60af4ea/src/tools.ts#L359)）。

## 14.7 MCP 工具插进来，和内建工具一视同仁

MCP 工具通过 `fetchToolsForClient`（[`client.ts#L1743`](https://github.com/xuhengzhi75/claude-code-source/blob/main/src/services/mcp/client.ts#L1743)）从服务器拉取，包装成实现 `Tool` 接口的对象后，和内建工具走同一套调度逻辑。区别在于安全声明来源：`isConcurrencySafe` 读 `tool.annotations?.readOnlyHint ?? false`（[`client.ts#L1795`](https://github.com/xuhengzhi75/claude-code-source/blob/main/src/services/mcp/client.ts#L1795)），由服务器通过协议声明，不声明则默认 false。

deny 规则支持 `mcp__server` 前缀匹配，一条规则封堵整个 MCP 服务器的所有工具（[`tools.ts#L256`](https://github.com/xuhengzhi75/claude-code-source/blob/9e4d0c6d68748da4cc9623a89752ed2cf60af4ea/src/tools.ts#L256)）。

## 14.8 小结

安全语义放在工具定义里，不放在调用方——这是这章所有细节的来源。

默认 fail-closed，声明宽松比声明限制更难被遗漏。并发规则是"全员安全才并发"，`isConcurrencySafe` 是"不产生竞态"而不是"只读"。Bash 出错级联取消兄弟工具，其他工具出错只影响自己。工具数量增加，调用方的逻辑不变。
