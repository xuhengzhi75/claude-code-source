# 第9章 护城河不只是模型

**核心主张（可被反驳）：** Claude Code 真正难以复制的不是某项功能，而是多个系统层协同积累的边界处理能力——每一层都是真实踩坑后加上去的，复制代码复制不了踩坑历史。

## 9.1 第一性问题：为什么模型能力不等于产品护城河

同一个底层模型，不同系统的实际表现差距可以很大。原因不是模型，是系统。

模型能力可以通过换版本快速提升，系统能力——连续性工程、并发一致性、可运维性——需要在真实场景中持续打磨。这两类能力的积累速度不同，这就是护城河的来源。

竞争对手可以拿到 Claude Code 的源码（这次泄露事件已经发生），但拿到源码不等于拿到护城河。护城河在代码背后。

## 9.2 连续性工程：每层防腐处理背后都有一段历史

`src/utils/conversationRecovery.ts#L77` 的 `migrateLegacyAttachmentTypes()` 把旧版本的 `new_file` 类型迁移成当前的 `file` 类型，把 `new_directory` 迁移成 `directory`（[`L88`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/utils/conversationRecovery.ts#L88)、[`L99`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/utils/conversationRecovery.ts#L99)）。

这个函数的存在说明系统在演进过程中改变过数据格式，同时需要保持对旧数据的兼容。竞争对手复制了这段代码，但他们的系统没有旧数据，这段代码对他们没有实际意义。他们需要在自己的系统演进中积累自己的迁移代码。

`conversationRecovery.ts#L164` 的 `deserializeMessagesWithInterruptDetection()` 更能说明问题。它在恢复时做了多层处理：迁移 legacy attachment 类型（[`L170`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/utils/conversationRecovery.ts#L170)）、检测中断 turn（[`L272`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/utils/conversationRecovery.ts#L272)）、在必要时补 assistant sentinel（[`L226`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/utils/conversationRecovery.ts#L226)）。每一层都对应一类真实出现过的边界情况，没有踩过坑的人不会想到要加这一层。

## 9.3 类型安全遥测：把规范变成编译器约束

`src/services/analytics/index.ts` 里有两个看起来奇怪的类型：

```typescript
type AnalyticsMetadata_I_VERIFIED_THIS_IS_NOT_CODE_OR_FILEPATHS = never
type AnalyticsMetadata_I_VERIFIED_THIS_IS_PII_TAGGED = never
```

这是用类型系统强制执行安全规则。开发者在提交遥测代码时，必须显式声明"我已确认这个字符串不包含代码路径或未打标的 PII"，否则 TypeScript 编译报错。

文档规范靠人记，编译器约束靠机器检查。这个模式把"不能做什么"从规范层推进到了构建层，安全检查不依赖代码审查者的记忆力。

## 9.4 并发一致性：并发 bug 在低压场景下不出现

`src/utils/tasks.ts#L505` 的 `getTaskListLockPath()` 返回任务列表的锁文件路径。`tasks.ts#L621` 的 `claimTaskWithBusyCheck()` 在这把锁内完成两步：检查 agent 是否空闲，以及认领任务（[`tasks.ts#L532`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/utils/tasks.ts#L532) 的注释明确写了：使用 task-list-level lock 防止 TOCTOU 竞态）。

并发 bug 的特点是在低压场景下不出现，在高压场景下才暴露。这类 bug 不通过代码审查发现，只有在真实的多 Agent 并发场景下才会暴露。修复一个往往引入另一个，边界情况需要长期实战积累才能补全。

## 9.5 可运维性：出问题时能直接看文件

任务以 JSON 文件形式存在（[`tasks.ts#L232`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/utils/tasks.ts#L232) 的 `getTaskPath()` 返回 `<taskId>.json`），会话历史以 JSONL 格式落盘（[`sessionStorage.ts#L202`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/utils/sessionStorage.ts#L202) 的 `getTranscriptPath()` 返回 `<sessionId>.jsonl`）。出问题时，不需要特殊工具，直接读文件就能定位。

对企业级落地来说，这一点比很多功能更重要。纯黑盒系统即使功能更强，也会在企业采购环节遇到阻力，因为运维团队看不进去、出事了排查不了。

## 9.6 Kill Switch：远程控制而不发新版本

Claude Code 内置了多层 Kill Switch，通过 GrowthBook 服务端配置下发（[`growthbook.ts#L622`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/services/analytics/growthbook.ts#L622) 的 `initializeGrowthBook()`），客户端按固定间隔轮询（[`growthbook.ts#L1087`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/services/analytics/growthbook.ts#L1087) 的 `setupPeriodicGrowthBookRefresh()`：外部用户 6 小时、内部用户 20 分钟）。消费侧通过 [`growthbook.ts#L734`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/services/analytics/growthbook.ts#L734) 的 `getFeatureValue_CACHED_MAY_BE_STALE()` 读取，函数名里的 `CACHED_MAY_BE_STALE` 直接说明了这个时间窗口的存在。

这和 feature flag 的目的不同。Feature flag 是产品迭代工具，Kill Switch 是安全控制工具。当模型版本出现意外行为时，Anthropic 可以在几分钟内通过 Kill Switch 限制受影响的功能，而竞争对手需要发布新版本、等用户更新。这个响应速度差距在安全事件里至关重要。

## 9.7 反蒸馏机制：主动干扰模型知识提取

[`claude.ts#L301`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/services/api/claude.ts#L301) 里有一项通过 feature flag 控制的机制：在 API 请求体中加入 `anti_distillation: ['fake_tools']`（[`L312`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/services/api/claude.ts#L312)）。触发条件是三重检查：入口必须是 CLI（`CLAUDE_CODE_ENTRYPOINT === 'cli'`）、必须是一方客户端（`shouldIncludeFirstPartyOnlyBetas()`）、且 GrowthBook feature flag `tengu_anti_distill_fake_tool_injection` 为 true。这会指示 API 在响应中注入虚假工具定义，污染外部观察者的训练数据。

[`streamlinedTransform.ts#L130`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/utils/streamlinedTransform.ts#L130) 的 `createStreamlinedTransformer()` 是第二层：去除 thinking content，把工具调用简化为类别统计，而不是完整的调用细节。让外部观察者即使捕获到输出，也无法还原完整的推理链路和工具使用模式。

两层机制分工不同：`fake_tools` 是主动污染（注入错误数据），`streamlinedTransform` 是信息隐藏（删除关键细节）。两者从数据质量和数据完整性两个维度同时降低竞争对手蒸馏的效果。

这个机制的边界条件是：假设竞争对手主要通过抓取 Claude Code 输出来获取训练数据。如果对手有其他数据来源（直接调用 API、购买数据集），这两层机制的效果会大打折扣。

## 9.8 Session Memory：随使用时间增长的迁移成本

`src/services/SessionMemory/` 在后台持续运行，用受限子 agent 提取对话信息，写入 session memory 文件（[`sessionMemory.ts#L460`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/services/SessionMemory/sessionMemory.ts#L460) 的 `createMemoryFileCanUseTool()` 限制只能访问 memoryPath）。

这类护城河的特殊性在于时间维度：用户用得越久，积累的上下文越多，迁移成本越高。竞争对手可以做出功能相似的产品，但让已积累大量 session memory 的用户放弃，需要提供压倒性的优势，不只是"差不多"。

设计 AI 产品时值得问：一年后用户积累的数据和习惯在哪里？如果答案是"什么都没有"，产品就没有时间护城河。

## 9.9 excluded-strings.txt：构建门禁而非人工审查

`excluded-strings.txt` 列出了绝对不能出现在外部构建产物里的字符串：内部模型代号、API Key 前缀、敏感信息。构建系统在打包后对输出做 grep，命中就直接失败。

这是构建阶段的自动化防线，而不是代码审查阶段的人工检查。把"不能出现在生产环境"的约束从人工检查变成自动化门禁，密钥、内部域名、测试账号都可以加进来。

这次 Claude Code 源码泄露事件说明了这道防线的边界：它覆盖构建产物里的字符串，但没有覆盖 source map 文件。source map 包含完整的 TypeScript 源码映射，发布时不应该打包进去，这次事故的根因是发布流程漏掉了"剔除 source map"这一步，不是 excluded-strings 机制本身的失效。

## 9.10 边界与技术债

这里描述的护城河几乎都依赖单机本地文件系统。规模化到分布式后，文件锁不可靠、JSONL append 的原子性假设不成立、任务目录扫描成为瓶颈。这些边界条件是设计层面的已知约束，不是实现层面的疏漏。

反蒸馏机制假设外部观察者通过抓取输出来学习，这个假设随着数据获取渠道多元化会逐步失效。

## 9.11 可迁移原则

把安全约束从文档变成编译器检查（类型安全遥测），比依赖人工记忆更可靠。适用条件是：约束可以被表达为类型系统能检查的形式。反例是：约束依赖运行时上下文（比如"这个字段在 staging 环境可以为空，在 production 环境不能为空"），类型系统无法静态检查，只能靠运行时校验。

## 9.12 本章不覆盖项

本章不覆盖具体的恢复机制实现细节（见第10章），也不覆盖任务系统的并发原语（见第8章）。这里分析护城河的来源和可复制性，而不是每个机制的实现方式。
