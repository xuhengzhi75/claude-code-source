# 第9章 护城河不只是模型

**核心主张（可被反驳）：** Claude Code 真正难以复制的不是某项功能，而是多层系统能力在真实边界情况中的历史沉积——源码可以复制，踩坑历史不能。

---

假设你继承了一家餐厅，老板把所有菜谱都给了你。菜谱准确，食材一样，但你做出来的味道就是差一点。差的不是菜谱，是老板在每道菜上踩过的几百个坑。护城河是坑，不是菜谱。

Claude Code 源码已经泄露。这件事本身说明了一个问题：如果护城河在源码里，护城河就已经不存在了。但竞争者拿到源码后，能直接超越吗？

本章分析护城河的三个来源，以及每个来源的边界条件。

---

## 9.1 边界处理能力来自踩坑，不来自设计

系统演进过程中，每次数据格式变更都会留下迁移代码；每次并发 bug 修复都会留下锁机制。这些代码的价值不在逻辑本身，而在于它们对应的那段历史。

`src/utils/conversationRecovery.ts#L77` 的 `migrateLegacyAttachmentTypes()` 把旧版本的 `new_file` 类型迁移成 `file`，把 `new_directory` 迁移成 `directory`（[`L88`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/utils/conversationRecovery.ts#L88)、[`L99`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/utils/conversationRecovery.ts#L99)）。复制这段代码的竞争对手，系统里没有旧格式数据，这段代码对他们没有实际意义。他们需要在自己的系统演进中积累自己的迁移代码。

`conversationRecovery.ts#L164` 的 `deserializeMessagesWithInterruptDetection()` 更直接。它在恢复时做三层处理：迁移 legacy attachment 类型（[`L170`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/utils/conversationRecovery.ts#L170)）、检测中断 turn（`turn` 指一轮对话：从用户发消息到 AI 回复完毕算一个 turn）（[`L272`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/utils/conversationRecovery.ts#L272)）、在必要时补 assistant sentinel（[`L226`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/utils/conversationRecovery.ts#L226)）。每一层对应一类真实出现过的边界情况，没有踩过这个坑的人不会想到要加这一层。

并发 bug 的积累路径更难复制。`src/utils/tasks.ts#L505` 的 `getTaskListLockPath()` 返回任务列表锁文件路径，`tasks.ts#L621` 的 `claimTaskWithBusyCheck()` 在锁内完成检查和认领两步（[`tasks.ts#L532`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/utils/tasks.ts#L532) 的注释明确写了：使用 task-list-level lock 防止 TOCTOU 竞态）。TOCTOU（检查与使用之间的时间窗口竞态）在低压场景下不出现，在多 Agent 并发的高压场景才暴露。这类 bug 不通过代码审查发现，只有真实实战才能暴露，修复一个往往引入另一个。

## 9.2 把规范变成编译器约束，才能脱离人工记忆

文档规范靠人记，编译器约束靠机器检查。两种方式在规模扩大时的可靠性差距是指数级的。

`src/services/analytics/index.ts` 里有两个类型：

```typescript
type AnalyticsMetadata_I_VERIFIED_THIS_IS_NOT_CODE_OR_FILEPATHS = never
type AnalyticsMetadata_I_VERIFIED_THIS_IS_PII_TAGGED = never
```

这是把安全规范变成编译器约束的具体做法。`never`（TypeScript 中的空类型，赋值给 `never` 类型的变量会导致编译报错）被用作强制声明机制——开发者提交遥测代码时，必须显式断言"我已确认这个字符串不包含代码路径或未打标的 PII"，否则 TypeScript 编译失败。安全检查不依赖代码审查者的记忆力，而是由构建系统自动执行。

`excluded-strings.txt` 是同一模式的另一个实例。它列出了绝对不能出现在外部构建产物里的字符串：内部模型代号、API Key 前缀、敏感信息。构建系统在打包后对输出做 grep，命中就直接失败。把"不能出现在生产环境"的约束从 review 阶段的人工检查推进到了构建阶段的自动门禁，密钥、内部域名、测试账号都可以加进去。

这次 Claude Code 源码泄露事件同时标定了这道防线的边界：它覆盖构建产物里的字符串，但没有覆盖 source map 文件。source map（调试辅助文件，包含 TypeScript 源码到 JavaScript 的完整映射，不应出现在生产包里）包含完整的源码映射，这次事故的根因是发布流程漏掉了"剔除 source map"这一步，不是 excluded-strings 机制本身的失效。

## 9.3 运营控制权差距在安全事件里被放大

功能差距在日常使用中可以接受，响应速度差距在安全事件里是决定性的。这个差距来源于是否有独立于版本发布的运营控制层。

Claude Code 内置多层 Kill Switch，通过 GrowthBook 服务端配置下发（[`growthbook.ts#L622`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/services/analytics/growthbook.ts#L622) 的 `initializeGrowthBook()`）。客户端按固定间隔轮询：外部用户 6 小时、内部用户 20 分钟（[`growthbook.ts#L1087`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/services/analytics/growthbook.ts#L1087) 的 `setupPeriodicGrowthBookRefresh()`）。消费侧通过 [`growthbook.ts#L734`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/services/analytics/growthbook.ts#L734) 的 `getFeatureValue_CACHED_MAY_BE_STALE()` 读取，函数名里的 `CACHED_MAY_BE_STALE` 明确说明了这个时间窗口的存在。当模型版本出现意外行为时，Anthropic 可以在几分钟内通过 Kill Switch 限制受影响功能；竞争对手没有这一层，需要发布新版本、等用户更新。

反蒸馏机制针对的是另一类控制权：训练数据的质量控制。[`claude.ts#L301`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/services/api/claude.ts#L301) 里有一项通过 feature flag 控制的机制：在 API 请求体中加入 `anti_distillation: ['fake_tools']`（[`L312`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/services/api/claude.ts#L312)）。触发条件是三重检查：入口必须是 CLI（`CLAUDE_CODE_ENTRYPOINT === 'cli'`）、必须是一方客户端（`shouldIncludeFirstPartyOnlyBetas()`）、且 GrowthBook feature flag `tengu_anti_distill_fake_tool_injection` 为 true。这会指示 API 在响应中注入虚假工具定义，污染外部观察者的训练数据。

[`streamlinedTransform.ts#L130`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/utils/streamlinedTransform.ts#L130) 的 `createStreamlinedTransformer()` 是第二层：去除 thinking content，把工具调用简化为类别统计，而不是完整的调用细节。让外部观察者即使捕获到输出，也无法还原完整的推理链路和工具使用模式。两层机制分工清晰：`fake_tools` 是主动污染，注入错误数据；`streamlinedTransform` 是信息隐藏，删除关键细节。两者从数据质量和数据完整性两个维度同时降低竞争对手蒸馏的效果。

可运维性也是控制权的一部分。任务以 JSON 文件形式存在（[`tasks.ts#L232`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/utils/tasks.ts#L232) 的 `getTaskPath()` 返回 `<taskId>.json`），会话历史以 JSONL 格式落盘（[`sessionStorage.ts#L202`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/utils/sessionStorage.ts#L202) 的 `getTranscriptPath()` 返回 `<sessionId>.jsonl`）。出问题时，不需要特殊工具，直接读文件定位。纯黑盒系统即使功能更强，在企业采购环节也会遇到阻力，因为运维团队出事了排查不了。

## 9.4 边界条件与可迁移原则

这里描述的护城河几乎都依赖单机本地文件系统。分布式场景下，文件锁不可靠，JSONL append 的原子性假设不成立，任务目录扫描成为瓶颈。这些是设计层面的已知约束，不是实现层面的疏漏。

反蒸馏机制的有效性建立在一个假设上：竞争对手主要通过抓取 Claude Code 输出来获取训练数据。如果对手有其他数据来源（直接调用 API、购买数据集），这两层机制的效果会下降。（inference）

Session Memory 的护城河性质与上面三个不同，它依赖时间维度。`src/services/SessionMemory/` 在后台持续运行，用受限子 agent 提取对话信息，写入 session memory 文件（[`sessionMemory.ts#L460`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/services/SessionMemory/sessionMemory.ts#L460) 的 `createMemoryFileCanUseTool()` 限制只能访问 memoryPath）。用户用得越久，积累的上下文越多，迁移成本越高。竞争对手可以做出功能相似的产品，但让已积累大量 session memory 的用户迁移，需要提供压倒性的优势。设计 AI 产品时值得问：一年后用户积累的数据和习惯在哪里？如果答案是"什么都没有"，产品就没有时间护城河。

**可迁移原则：** 把约束推进到构建层比留在 review 层更可靠。适用条件是约束可以被表达为类型系统或构建脚本能检查的形式。反例是约束依赖运行时上下文（比如"这个字段在 staging 环境可以为空，在 production 环境不能为空"），类型系统无法静态检查，只能靠运行时校验。原则是工具，不是教条。
