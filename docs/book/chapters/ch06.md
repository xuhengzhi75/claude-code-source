# 第6章 QueryEngine：为什么它是会话编排核心

结论先说：`src/QueryEngine.ts` 解决的不是再包一层循环，而是把一次请求组织成可持续、可记录、可恢复的会话执行过程。

如果只看单轮执行，很多人会以为 Agent 只是调模型 → 调工具 → 返回结果。但一旦流程跨多轮，问题就变成：消息历史放在哪、transcript 何时写入、usage 怎么累计、错误和结果怎么稳定输出给 SDK。`QueryEngine` 处理的正是这些会话级问题。

## 6.1 它和 `query.ts` 的职责边界

Claude Code 把两层问题分开了：

- `src/QueryEngine.ts`：会话级编排、持久化、计量、对外事件整理
- `src/query.ts`：单轮状态机推进、继续判断、恢复分支、工具执行循环

这条边界很关键。若两者混在一起，后续加 transcript、compact、SDK 兼容、恢复逻辑时，控制流会迅速失控。

## 6.2 `submitMessage()` 为什么是关键入口

`submitMessage()` 接住的不是一段 prompt，而是一整次 turn 的执行组织工作。它至少做 6 类事情：接住用户输入、准备工具和命令和模型和上下文、把输入并入会话消息状态、在进入主循环前写 transcript、调用 `query(...)` 进入单轮执行循环、把内部消息整理成外部可消费的事件和结果。

所以它不是简单转发，而是会话级入口。

## 6.3 从源码可以直接看到哪些职责

在 `src/QueryEngine.ts` 中，有几类职责是可以直接对到代码结构的。

### 会话消息状态

类内部维护 `mutableMessages`。这说明消息历史是会话级资产，而不是某个局部函数里的临时变量。

### 持久化与恢复友好性

源码中能看到 `recordTranscript` 和 `flushSessionStorage`。这说明系统不会等到最后才统一记账，而是把落盘动作贴近执行路径，以降低中断后的恢复成本。

### 对外事件归并

文件里还能看到 `buildSystemInitMessage`、`permission_denials`、`usage` 等输出归并点。`QueryEngine` 不只接内核，还负责把内部消息流整理成 SDK 可消费的事件流。

## 6.4 为什么这层会直接影响稳定性

如果会话编排层没做好，典型故障会很直接：transcript 断裂、usage 漏计、工具结果挂错回合、外部调用方拿到不一致事件流、中断后不知道从哪里恢复。

这些问题并不属于模型聪不聪明，而属于会话是不是被稳定组织起来。`QueryEngine` 的价值就在这里。

## 6.5 对自己做 Agent 最值得学什么

如果你要做一个缩小版业务 Agent，这一章最值得带走的不是把 `QueryEngine` 原样复刻，而是先守住两条边界：把会话编排和单轮执行拆开；让消息、transcript、usage 这些会话资产有稳定归宿。

只要这两条成立，系统后面再加恢复、任务和 SDK 适配时，复杂度才有地方收口。

## 6.6 本章小结

`QueryEngine` 的核心价值，不是它替代了 `query.ts`，而是它把单轮执行循环包进了一个更完整的会话容器里。

它往上承接入口和能力装配后的运行条件，往下驱动 `query.ts`，同时对外整理 transcript、usage 和事件结果。这也是它为什么更像会话编排核心，而不是一个普通中间层。
