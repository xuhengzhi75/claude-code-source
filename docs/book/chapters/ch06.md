# 第6章 QueryEngine：为什么它是会话编排核心

前两章讲了入口分流和能力装配。
接下来要进入 Claude Code 真正开始“组织一次工作”的位置。

如果把整个系统想成一台工作机器，那么 `QueryEngine` 更像总控台。
它不负责每一个底层细节，但它负责把这次会话组织起来，让请求能进入执行循环，又能把结果、状态和记录收回来。

这一章的目标很简单：讲清楚为什么 `src/QueryEngine.ts` 是 Claude Code 的会话编排核心。

## 6.1 为什么需要一层“会话编排”

很多初学者做 Agent，会把流程写成一个大循环：

- 收到输入
- 调模型
- 如果有工具就执行
- 再继续问模型
- 最后输出结果

这种写法在最小 demo 阶段能跑，但很快就会出现问题：

- 消息历史放在哪里
- 每轮执行前后什么时候持久化
- 成本和 usage 怎么统计
- 中断后如何恢复
- SDK 或 UI 侧怎么拿到统一事件流
- 工具权限、系统上下文、用户上下文怎么一起带进去

这时你就会发现，真正难的不是“循环跑不跑”，而是：
**一次会话怎样被稳定地组织起来。**

`QueryEngine` 就是在解决这个问题。

## 6.2 `QueryEngine` 和 `query.ts` 的职责边界

Claude Code 的一个很值得学的设计，是它没有把所有逻辑都塞进一个文件。

从源码关系看：

- `src/QueryEngine.ts`：偏会话级编排
- `src/query.ts`：偏单轮状态机推进

这个边界很重要。

你可以把它理解成：

- `QueryEngine` 管整场会话
- `query.ts` 管当前这轮怎么继续

两者配合，但不是同一层问题。

如果没有这层拆分，系统会很快把“会话生命周期”和“单轮执行细节”搅在一起，之后想加恢复、压缩、SDK 兼容、事件输出时就会越来越痛苦。

## 6.3 `submitMessage()` 是进入主执行链的关键入口

`QueryEngine` 最重要的方法之一是 `submitMessage()`。

从会话编排角度看，它至少负责几类事情：

- 接住新的用户输入
- 准备这次执行所需的上下文
- 把用户输入并入当前消息列表
- 在进入主循环前做必要持久化
- 调用 `query(...)` 把执行权交给单轮状态机
- 持续消费循环产出的消息和事件
- 在结束时更新 usage、cost、状态与输出结果

这说明 `submitMessage()` 不是“转发一下请求”而已。
它像一个会话调度入口，把一次请求纳入同一条长期会话轨道。

## 6.4 为什么说它是“会话级”核心

一个很实在的判断标准是：
如果拿掉 `QueryEngine`，只剩 `query.ts`，系统能不能优雅地跑完整个产品级会话？

答案通常是不行。

因为 `query.ts` 更像执行内核，它关注的是当前轮次如何继续推进。
但产品级会话还需要另外一层处理：

- 历史消息如何保留
- 初始消息如何注入
- transcript 何时写入
- SDK 事件如何映射
- 总 usage 和 cost 如何累计
- 文件状态、权限拒绝、memory、plugin 等状态如何跨轮保留

这些事情都不是单轮循环本身该扛的。

所以 `QueryEngine` 的价值在于，它把“执行循环”包进了一个更完整的会话容器里。

## 6.5 从源码能看到哪些编排职责

从 `src/QueryEngine.ts` 的结构看，这一层至少在承担以下职责：

### 第一，管理会话消息状态

类内部维护 `mutableMessages`，说明消息历史不是局部变量，而是会话级状态。

这意味着：
- 一次提交不是孤立的
- 后续轮次会延续前面形成的消息链
- 系统可以在同一会话里持续推进任务

### 第二，管理执行环境与工具上下文

配置对象里带有：

- `tools`
- `commands`
- `mcpClients`
- `canUseTool`
- `thinkingConfig`
- `fallbackModel`
- `maxTurns`
- `taskBudget`

这说明 QueryEngine 接住的不是一段 prompt，而是一整套本次会话的运行条件。

### 第三，负责持久化与恢复友好性

源码中能看到 `recordTranscript`、`flushSessionStorage` 等能力接入。

这类动作非常关键。
因为一个真实工作的 Agent，不应该等所有事情做完后才记账。越接近执行现场地落盘，系统在中断后的恢复能力越强。

### 第四，负责把底层事件整理成对外输出

QueryEngine 会把 assistant、user、progress、attachment 等内部流转消息整理成 SDK 可消费的事件。

这意味着它还是“内部执行内核”和“外部消费接口”之间的桥梁。

## 6.6 为什么这一层会直接影响稳定性

很多人以为稳定性主要由模型决定。
其实在 Agent 里，稳定性很大一部分来自会话组织能力。

如果编排层混乱，常见问题会非常多：

- 消息顺序错乱
- 工具结果挂不到正确回合
- usage 统计不完整
- transcript 断裂
- 中断后恢复位置不清楚
- 外部调用方拿到不一致的事件流

`QueryEngine` 的存在，就是把这些跨轮、跨模块、跨接口的问题集中收口。

所以它不是“又一层包装”，而是稳定性基础设施。

## 6.7 对自己做业务 Agent 的启发

如果你以后做一个简化版业务 Agent，这一章最应该带走的不是把 QueryEngine 一比一照抄，而是学它的分层观念。

### 至少把“会话编排”和“单轮执行”拆开

哪怕你的系统还很小，也建议区分：

- 一层负责管理整个会话的状态、历史、上下文、落盘
- 一层负责当前轮次如何继续或结束

### 尽量让消息状态成为会话资产

不要把所有消息都散落在临时变量里。
只要任务会跨多轮推进，消息历史就应该被稳定管理。

### 让持久化尽量靠近执行路径

不要等最终成功才保存。
中间状态越早落下，恢复能力越强。

## 6.8 本章小结

这一章的核心结论是：

**`QueryEngine` 之所以关键，不是因为它做了最复杂的状态机，而是因为它负责把一次请求组织成可持续的会话执行过程。**

它承上启下：

- 往上承接入口和能力装配后的运行条件
- 往下驱动 `query.ts` 的单轮执行循环
- 往外输出稳定的事件、记录和结果

理解这一层之后，下一章再看 `query.ts`，就更容易明白：真正的循环引擎是怎么决定继续、调用工具、恢复和收尾的。
