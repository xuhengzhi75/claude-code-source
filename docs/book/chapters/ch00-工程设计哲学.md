2026 年 3 月 31 日，Anthropic 在发布 Claude Code v2.1.88 时，将一个 59.8 MB 的 JavaScript source map 文件意外打包进了 npm 包，完整源码随之泄露。代码库约 1900 个文件、512,000 行 TypeScript，涵盖核心 LLM 引擎、工具权限系统、多 Agent 协调机制，以及 44 个尚未发布的功能开关。

这份源码有一个值得注意的细节：贯穿全库的注释，写法上更像是在对代码库里工作的 AI Agent 说话，而不是对人类读者。

这篇文章想回答一个问题：Claude Code 的工程设计，背后有没有一条贯穿始终的逻辑？有，只有一句话：**用确定性的工程框架，驯化不确定的模型行为。** 下面从三个层次展开：Agent 怎么思考、信息怎么管理、系统怎么约束。

---

## 第一部分：Agent 的三种推理模式

从学术论文到生产代码，Claude Code 的 Agent 推理机制经历了三次演进，每一次都在解决上一次留下的问题。

### ReAct：推理与行动交错

要理解 Claude Code，必须先理解 ReAct。

ReAct 来自普林斯顿大学和 Google 在 2022 年联合发表的论文《[ReAct: Synergizing Reasoning and Acting in Language Models](https://arxiv.org/abs/2210.03629)》（ICLR 2023）。它是 Reasoning（推理）和 Acting（行动）两个词的合并，但它不是一个概念，而是一套强制机制：不允许模型直接给出答案，而是强迫它走三步——**Thought（思考）→ Action（行动）→ Observation（观察）**，循环往复，直到任务完成。

这篇论文的第一作者是姚顺雨（Shunyu Yao），发表时他还是普林斯顿大学的博士在读生。有一个容易被误解的冷知识：这篇论文不是他的博士毕业论文，他真正的博士论文是 2025 年完成的《[Language Agents: From Next-Token Prediction to Digital Automation](https://arxiv.org/abs/2503.03905)》，ReAct 只是其中的一个章节。但 ReAct 的影响力已经远远超出了一篇会议论文该有的体量——今天每一次 Claude Code 的任务执行，背后都在跑这套机制。

在 ReAct 之前，有两条路都走不通。纯推理（Chain of Thought）让模型只靠参数知识推导，遇到训练截止日期之后的信息就产生幻觉；纯行动让模型直接执行操作，没有内部规划，方向一偏就一路错到底。论文在 HotpotQA 和 WebShop 等基准测试上证明了这一点：ReAct 的洞察是**只有交错才能纠错**——推理为行动提供方向，行动的结果随时修正推理。

[Google Research 的官方博客](https://blog.research.google/2022/11/react-synergizing-reasoning-and-acting.html)从另一个角度解读了 ReAct，提出了一个关键词：**白盒化**。因为每一个 Thought 步骤都留在上下文里，开发者可以一步步回溯模型的决策过程，也可以在任意节点插入修正。这种可解释性，是让 AI 系统真正可信赖的工程基础。

在 Claude Code 的源码里，这套机制落地成了 `query.ts` 文件里的 `queryLoop` 函数，核心是一个 `while (true)` 大循环：每轮先调用 `callModel` 流式输出推理内容，如果输出包含工具调用块就进入 Action 阶段，执行 `runTools`，工具结果追加到 `messages` 数组后推动下一次循环。

这里有个容易猜错的细节：**只读工具才并发，有副作用的写操作是串行执行的**。这个分组逻辑在接口文档里没有说明，只有打开源码才能看到。这个循环支撑着超过 40 个工具，按职责分五层：执行层（Bash Tool、REPL Tool）、文件层（File Read/Edit/Grep）、信息层（Web Search/Fetch）、编排层（Agent Tool）、扩展层（MCP 协议）。每一个工具的输出，都是 ReAct 循环下一轮的 Observation。

### Plan-and-Execute：规划与执行分离

ReAct 解决了"怎么行动"的问题，但没有解决"怎么完成长任务"的问题。

让 ReAct 去"重构整个后端、跑测试、提 PR"，它会做着做着忘了最初要干嘛，在某个细节里越陷越深，最后全局目标丢了。这不是 ReAct 的设计失误，而是一个根本矛盾：**规划和执行是两种不同的认知任务**。规划需要全局视角，需要在不确定条件下做延迟判断；执行需要聚焦当前步骤，需要快速行动。让同一个循环同时承担这两件事，本身就是矛盾的。

[LangChain 的官方博客](https://blog.langchain.com/plan-and-execute-agents/)把 Plan-and-Execute 落地成了三步流程：Planner 先列出多步任务清单，Executor 逐步执行每一步，执行结束后系统触发 Re-planning，Planner 再次判断——完成就结束，没完就生成新清单再跑一轮。Planner 和 Executor 是两个独立组件，结构层面的分离。

Claude Code 的做法出乎意料：**不拆组件，用同一个 `while(true)` 循环，靠权限状态切换和计划文件，把 Plan、Execute、Re-plan 三件事都跑在里面。**

具体来说：模型判断任务复杂后调用 `EnterPlanMode` 工具，把权限状态切到 `plan`，系统立刻拒绝所有文件写操作——这是强制约束，不是靠模型自觉。被限制住之后，模型只能做一件事：探索代码库、设计方案、把任务清单写进计划文件。规划完成后，模型调用 `ExitPlanMode` 提交计划，等用户审批，确认后权限恢复，模型照着计划文件开始改代码。

Re-plan 不是执行结束后才触发的独立步骤，而是嵌在执行里的能力：计划文件是磁盘上的普通文件，执行阶段模型有写权限，随时可以改，改了下一步就按新内容走。LangChain 用架构分离，Claude Code 用状态切换，两种方式背后的想法是一样的——**先规划再执行，随时能重新评估，是让 Agent 真正走完长任务的核心设计。**

### Reflexion：失败后的自我纠错

很多 Agent 不是不会做，而是会反复犯同一种错。第一次失败还能接受，第二次还是错，第三次还在同一个坑里打转。问题不是能力，而是没有"失败后的学习机制"。

Reflexion 要解决的正是这个问题。它来自 2023 年 NeurIPS 的论文《[Reflexion: Language Agents with Verbal Reinforcement Learning](https://arxiv.org/abs/2303.11366)》（[NeurIPS 2023 收录](https://proceedings.neurips.cc/paper_files/paper/2023/hash/1b44b878bb782e6954cd888628510e90-Abstract-Conference.html)），把 Agent 拆成四个组件：Actor 负责执行，Evaluator 给结果打分，失败后触发 Self-Reflection 用自然语言写下"哪里做错了、下次怎么改"，最后存入 Memory 供下一轮读取。论文的核心洞察是：**用自然语言文本替代传统强化学习的标量信号**，更丰富、更可解释、更容易跨轮次传递。

Anthropic 在 2024 年 12 月发布的《[Building Effective Agents](https://www.anthropic.com/research/building-effective-agents)》里把这套思路翻译成了工程师能直接落地的流程：Draft 生成初稿，Review 审查挑错，Refine 带反馈重写。三个环节和论文角色一一对应。

在 Claude Code 的源码里，Reflexion 没有做成单独模块，而是散在循环里，分三层落地：

**第一层：工具错误注入。** 每次工具调用失败，错误以 `is_error: true` 的格式直接注入对话上下文，模型下一轮就能读到"刚才哪一步报错了"，据此调整行动。这是最小粒度的 Evaluator 闭环。

**第二层：熔断机制。** `denialTracking.ts` 里写了硬限制：`maxConsecutive = 3`，连续 3 次操作被拒绝，Agent 直接中止，强制人工介入。这是工程层的最大重试次数上限，防止 Agent 在同一个错误里无限打转。

**第三层：语言记忆。** 当上下文窗口接近上限，Claude Code 触发 `autoCompact`，对整段对话做结构化总结。总结 prompt 里明确写了第 4 条：**Errors and fixes**——把本轮遇到了哪些错误、怎么修的，用自然语言记下来。下一轮开始时，这段失败历史作为摘要注入新上下文。这正是 Verbal Memory 的工程实现。

值得注意的是，这三个模式共用同一个 `while(true)` 循环骨架。在 ReAct 里它推进行动，在 Plan-and-Execute 里它统筹全局，在 Reflexion 里它记住失败、吸收教训。**只要职责一变，整个模式就会长出完全不一样的能力。同一段代码，三种工程哲学。**

## 第二部分：信息是有限资源

Agent 怎么思考，解决的是推理问题。但在推理发生之前，还有一个更基础的问题：**上下文窗口里应该有什么信息、粒度多粗、何时展开？**

这是信息编排的问题。Claude Code 在这个层面有两套设计，分别对应"当前任务需要什么工具知识"和"跨会话需要记住什么"。

### 渐进式披露：Skills 的按需加载

Anthropic 在 2025 年 9 月发布的《[Effective context engineering for AI agents](https://www.anthropic.com/engineering/effective-context-engineering-for-ai-agents)》里提出了一个关键概念：**context rot**——上下文窗口里的 token 越多，模型提取信息的能力越弱。信息不是消失了，而是被稀释到模型注意不到了。解法是 Agent 只维护轻量级摘要，按需动态加载，而不是预先填满上下文。

这套思路在 Claude Code 的 Skills 机制里有最典型的实现。Skills 可以有几十上百个，每个展开后最多 5000 tokens。如果全部预加载，上下文会爆；如果完全不加载，模型就不知道自己有什么能力。Skills 用渐进式披露解了这个矛盾，整个运作是一套完整的四步闭环：

**第一步**：会话开始时只注入 YAML frontmatter，约 100 tokens。模型知道有哪些技能，但看不到正文内容。

**第二步**：模型匹配到任务后调用 SkillTool，完整正文才被加载，全文上限 5000 tokens，不调用就永远不进上下文。

**第三步**：执行完成后正文被剥离，只留约 200 token 的结果摘要，用完即收，上下文零污染。

**第四步**：`auto compact` 触发后只重新注入用过的技能。源码注释里明确写道：重注入完整列表收益微乎其微。

100 tokens 进来，5000 tokens 展开，用完压成 200 tokens 的摘要，压缩后只取回用过的。**渐进式披露不会让模型更聪明，它只是让有限的注意力用在更重要的地方。**

### 记忆系统：只存不可派生的认知

如果说渐进式披露解决的是"当前会话里信息怎么管"，那记忆系统解决的是"跨会话之间什么值得保留"。

整个记忆系统的设计哲学只有一句话：**只存不可派生的认知。**

源码注释第一行就写得很清楚：记忆只能存无法从当前项目状态派生出来的上下文。代码模式、架构决策、git 历史、文件结构，这些随时可以查到的一律不能存，即使用户主动要求记下来也不行。记忆不是全量归档，是只存工具查不到的那部分认知。

在这个约束下，源码定义了四种记忆类型：`user`（这个用户是谁，比如"Go 老手初次接触 React 前端"，存下来下次解释就用后端类比）、`feedback`（用户给的纠偏，改错要记，被确认的做法也要记——只记错误会让模型越来越保守）、`project`（当前项目进行中的事，注意相对日期要转成绝对日期，"周四"要存成具体日期，否则时间一过记忆就失效了）、`reference`（外部系统指针，Grafana 看板在哪、Linear 项目叫什么）。

记忆系统分三层运作：

**存得准**：每条记忆写成独立的 markdown 文件，文件头带 `name`、`description`、`type` 三个字段。`description` 特别关键，不是给人看的，是给后续召回时做相关性判断用的，描述越精准召回越准。所有文件的指针汇聚在记忆索引 `MEMORY.md` 里，严格控制在 200 行 25KB 以内——这个上限不是随便定的，因为这份索引会在每次会话启动时注入进系统提示，超出就截断，索引失效。

**取得准**：取分三个时机。启动时静态加载，把记忆索引前 200 行塞进系统提示；会话中动态召回，每次对话转折点触发相关记忆查找，先并发扫所有记忆文件的前 30 行只读 frontmatter，再把这份列表丢给 Sonnet 做旁路调用——这个 Sonnet 调用只有 256 个 token 的预算，就是读 description 做匹配，最多选 5 个相关文件注入当前上下文；后台自动提取，触发时机非常精确，是主模型产出最终回复且没有工具调用的那一刻。还有一个反直觉的设计：已经用过的文件在 Sonnet 调用之前就被过滤掉了，5 个槽位全部留给新文件，不浪费。

**维护不过期**：这是最容易被忽视的一层。没有维护机制，记忆系统就是一个正在腐烂的时间胶囊。Claude Code 用三个机制对抗记忆腐化：`autoDream`（源码里叫 `DreamTask`，后台子代理周期运行，触发要过三道门——时间超过 24 小时、会话累计 5 个、分布式锁防止多进程同时跑，四个阶段依次完成定向、采集、整合、剪枝）、团队记忆隔离（`private` 目录只属于当前用户，`team` 子目录跨贡献者共享，团队记忆里绝对不能存 API key 或凭证）、新鲜度警告（超过一天的记忆会自动附上提示："这条记忆是若干天前的，用之前先验证当前代码"）。

**不是让模型"记住一切"，是让模型"记对该记的"。**

## 第三部分：Harness——把一切串起来

前两部分讲的是 Claude Code 内部的设计细节。这一部分要退一步，看清楚这些设计背后共同的工程直觉，以及它在更大尺度上是怎么运作的。

### 外部 Harness：Anthropic 的长任务工程实践

Anthropic 在 2025 年 11 月发布的《[Effective harnesses for long-running agents](https://www.anthropic.com/engineering/effective-harnesses-for-long-running-agents)》里，把长任务 Agent 的核心问题讲得很清楚。

单体 Agent 在长任务里容易出两个问题：一是"上下文焦虑"（Context Anxiety），任务一长，模型明明没到极限，却开始提前敷衍收尾；二是"过度自信"，模型总高估自己的完成度，看着像做完了，实际上跑起来全是 Bug。这两个问题的根源是一样的：**Agent 不是在一个永远在线的大脑里工作，而是在离散的上下文窗口里接力**，每次进入新窗口，它都面临三个致命伤——上下文分段、记忆不连续、状态不干净。

解法不是赌一个更强的单体模型，而是把"做事"和"验收"拆开，搭一个三层架构：Planner 先把一句很短的需求扩写成完整的详细文档，Generator 负责分阶段实现代码，Evaluator 负责用自动化方式检查结果（通过 Playwright 点页面、看状态，再决定这一轮算不算过关）。

Anthropic 拿复古游戏开发做了个极具反差的对比：**单 Agent 基线系统只花了 20 分钟、9 美元，但核心玩法是坏的；全套 Harness 架构跑了约 6 小时、烧了 200 美元，把最初的一句话需求扩展成了 16 个特性和 10 个冲刺周期，最终交付了更完整、可验证的结果。** 这组数据不是在证明 Harness 更便宜，而是在证明它换来了一个真正可以被测试、被追踪、被修正的工程过程。

更有意思的是，Anthropic 没有把这套架构当成永久答案。随着模型能力变强，他们开始主动"拆除"这些结构——先是移除了原本必需的 Sprint 契约，又把 Evaluator 从每轮检查改成了运行末尾的单次检查。**Harness 的每一层，都是在弥补模型当前还做不到的事。一旦模型变强，昨天还必须的脚手架，今天就可能变成负担。**

### 内部 Harness：Claude Code 自己就是一套 Harness

外部 Harness 是工程师在 Claude Code 外面搭的脚手架。但打开源码会发现，Claude Code 里面早就自己搭好了一套，而且它就管自己叫 Harness。这套内部 Harness 从四个维度约束模型行为。

**第一层：信任门控——先确认身份，再交出工具**

源码注释里反复出现一个词："延后"。MCP 资源预取、插件加载、语言服务初始化，全都延后到信任验证通过之后才启动。这不是偶然的代码风格，而是一个刻意的设计原则：所有能力扩展——插件、技能、MCP 接入、会话钩子——都在信任边界之后。启动流程本身就是第一道 Harness：先确认你是谁、你能做什么，然后才把工具交出去。

**第二层：工具约束——三层过滤，从宽到严**

Anthropic 在 2025 年 10 月的[沙箱博客](https://www.anthropic.com/engineering/claude-code-sandboxing)里提到，内部测试中沙箱机制减少了 84% 的权限提示。2026 年 3 月的 [auto mode 博客](https://www.anthropic.com/engineering/claude-code-auto-mode)进一步解释了背后的问题：用户接受了 93% 的权限提示，长期下来产生"审批疲劳"，反而让安全形同虚设。

源码里的工具约束正是这个问题的工程解法，分三层递进：第一层是内置安全白名单，文件读取、代码搜索、计划模式切换等不改变状态的操作直接放行，不进入后续判断；第二层是项目内文件操作，在当前仓库目录内的写操作直接允许，可通过版本控制追溯；第三层是 transcript 分类器，Shell 命令、外部网络请求、项目目录外的文件操作才会到这里，由模型分两阶段判断——先用单 token 快速过滤，只有被标记的操作才进入链式推理，避免每次都付出完整推理的代价。

在 Bash 工具上，这套约束还叠加了九个独立的专属校验模块，仅安全检查一个模块就定义了 23 种规则，覆盖路径校验、只读校验、破坏性命令警告等场景。

**第三层：运行边界——三道防线对抗失控**

Anthropic 在 [harness-design 博客](https://www.anthropic.com/engineering/harness-design-long-running-apps)（2026 年 3 月）里区分了两种应对上下文增长的策略：**压缩（compaction）**保留对话连续性，把早期内容原地摘要，同一个 Agent 继续跑；**重置（context reset）**给 Agent 一个干净的起点，代价是需要结构化的交接文件。两者解决的问题不同——压缩解决的是上下文爆满，重置解决的是"上下文焦虑"（模型在接近窗口上限时提前收尾）。

源码里的三道防线对应这两种策略加一个兜底：硬熔断（最大轮次上限和 token 预算上限，触发任何一道引擎就停止，每轮结束记录停止原因）、软压缩（`autoCompact` 在接近上下文窗口上限时自动压缩对话历史，让长任务继续推进）、断点恢复（每次会话被完整序列化到磁盘，包含会话 ID、全部消息记录和 token 统计，熔断之后下一轮可以从中断点接上）。

**第四层：分工 Agent——不是一个模型扛所有**

harness-design 博客里描述的三 Agent 架构（Planner 扩写需求 → Generator 分阶段实现 → Evaluator 用 Playwright 点击验收）和源码里内置 Agent 的分工直接对应：代码探索 Agent 负责理解代码库、方案规划 Agent 负责制定执行计划、实现验证 Agent 负责检查结果正确性。

这些内置 Agent 大部分有功能开关门控，可以按实验需求动态开关——代码探索和方案规划受 A/B 实验控制，实现验证 Agent 还在实验阶段，需要 feature flag 才会加载。更关键的是，Agent 工具支持动态分裂（Fork），子 Agent 可以在运行时从父 Agent 继承完整上下文派生出来，也可以从磁盘恢复之前中断的状态继续执行。

---

把这四个维度合在一起，会发现一件有意思的事：**工程师在 Claude Code 外面搭 Harness，用循环、用规划、用验收；而 Claude Code 里面早就自己搭好了一套。外面是 Harness，打开里面还是 Harness，同样的工程直觉在每一层都在重演。**

当我们用 Claude Code 做事的时候，我们其实是在用一个 Harness 驾驭另一个 Harness。Claude Code 里面那一套是 Model Harness，约束裸模型；外面工程师搭的那一套是 Agent Harness，约束已经被 Harness 过的 Agent。两个尺度，同一个工程直觉。

---

## 结语：工程化才是 AI 能力的真正边界

读完这批源码，我最深的感受是：**很多看起来像"模型不行"的失败，实际上是 Harness 没搭好。**

没有交接资产，没有增量节奏，没有统一验收标准，没有记忆维护机制，没有渐进式的信息编排——这些工程层面的缺失，会让一个本来够用的模型表现得像一个不够用的模型。

反过来，一套好的工程框架，能让一个不稳定的模型，在确定的边界内稳定地交付。这不是在降低对模型的要求，而是在正确地使用模型。

Claude Code 的源码告诉我们：**决定 AI 系统上限的，不是模型性能，而是工程化框架。** 想让 Agent 从"会回答"升级到"会干活"，核心就在于这套 Harness 能否把不确定的模型，驯化成一个可交付的工程系统。

这是 Anthropic 用 51 万行代码写下的答案。
