# 第5章 能力装配：命令、工具、技能和插件怎么拼成可用系统

结论先说：Claude Code 不是把模型直接接到一堆能力上，而是先经过一层装配与过滤，再决定当前会话真正能看到什么、能调用什么。

这层的意义，不是工具更多，而是能力边界更清楚。在真实系统里，模型上限往往不只由模型本身决定，还由三件事决定：当前能看到哪些动作、哪些动作受权限约束、不同来源的能力如何被统一组织。

## 5.1 什么叫能力装配

这里说的能力装配，不是列一个工具清单，而是做 3 件事：准备理论上可用的能力全集，按当前环境、权限和模式过滤，把留下的能力整理成统一接口。

这个过程决定了当前会话的工作边界。同一个模型接在不同装配层后面，表现可能差很多，因为模型能操作的世界并不相同。

## 5.2 命令和工具都不是看到什么就全给什么

从 `src/commands.ts` 可以看到，命令来自多个来源：内建命令、skill 命令、插件命令、workflow 命令，以及动态发现的命令。合并之后还会继续做可用性和启用状态过滤。

从 `src/tools.ts` 可以看到，工具也不是直接全量暴露。系统会先得到理论工具全集，再根据权限上下文与运行模式过滤出当前会话真正可用的集合。

这两层的共同点是：系统先整理，再暴露。

## 5.3 工具层为什么不只是函数集合

`src/Tool.ts` 说明工具并不只是可调用函数。工具还携带一整套运行时契约，例如：输入输出结构、权限检查、只读或破坏性语义、并发安全声明、UI 展示和摘要方式、与 MCP 工具的映射。

这意味着工具层本身就是运行时治理层，而不是外挂仓库。

这个设计的价值很直接：新工具更容易稳定接入，权限边界更容易统一处理，工具结果更容易回到主循环，不同来源的能力更容易用同一套方式管理。

## 5.4 为什么模型前面要先放一层治理

如果把模型直接连到脚本、接口和插件上，前期开发会看起来更快，但很快会遇到几个问题：哪些能力该默认开放，哪些能力必须先做权限判断，多来源能力重名时怎么办，某些模式下哪些能力根本不该出现。

Claude Code 在这一层的做法是先治理，再开放。这会增加一点前置设计成本，但能换来更清楚的边界和更稳定的行为。

## 5.5 命令、工具、技能、插件分别扮演什么角色

可以把这几类能力来源简化理解成：命令偏用户可见的控制入口，工具偏模型在执行中调用的动作接口，技能偏可复用的方法包装和经验单元，插件偏外部扩展能力来源。

它们不是同一个层次，但都会影响当前会话能做什么。能力装配层的作用，就是把这些来源整理成一个可运行的能力池。

## 5.6 对自己做 Agent 最值得抄的三件事

先定义能力边界，再扩能力数量；给能力统一接口，而不是各写各的调用方式；把理论能力和当前可见能力分开。

这三件事越早做，系统越不容易在后期失控。

## 5.7 本章小结

Claude Code 的能力装配层做的不是堆功能，而是先整理命令、工具、技能和插件，再把当前会话真正可用的那部分暴露给模型。

这样做的结果是，模型面对的不是原始能力堆，而是一套经过过滤、统一和边界治理的动作池。后面再看 `QueryEngine`，就更容易理解：执行核心为什么能建立在更稳定的能力边界之上。
