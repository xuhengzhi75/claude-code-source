# 第5章 能力装配：工具池是怎么拼出来的

## 5.1 四类能力来源，一个工具池

Claude Code 的能力来自四个地方：内建工具（`BashTool`、`FileReadTool` 等）、MCP 工具（外部服务器提供）、Skill（`.claude/skills/` 目录下的 SKILL.md）、命令（`/help`、`/clear` 等斜杠命令）。

这四类东西经常被混淆，但它们的受众完全不同：

命令是给**用户**用的，用户在终端里输入 `/clear` 清空对话，模型不知道这件事发生了。工具是给**模型**用的，模型在推理时决定调用 `read_file`，用户看到的是结果。Skill 是给**模型**用的提示词包装，把常见的工具调用序列封装成更高层的能力，本质上是一段注入到 system prompt 的指令。MCP 工具和内建工具一样，都是模型可以调用的工具，只是来源不同。

最终暴露给模型的只有工具（内建 + MCP）和 Skill，命令对模型不可见。

## 5.2 内建工具必须排在 MCP 工具之前

[`tools.ts#L348`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/tools.ts#L348) 的 `assembleToolPool()` 注释解释了一个不明显的约束：

```typescript
// 组合策略是"内建优先、MCP补充"：
// 同名冲突时优先保留内建定义，确保核心行为与权限语义稳定。
//
// Sort each partition for prompt-cache stability, keeping built-ins as a
// contiguous prefix. The server's claude_code_system_cache_policy places a
// global cache breakpoint after the last prefix-matched built-in tool;
// a flat sort would interleave MCP tools into built-ins and invalidate
// all downstream cache keys whenever an MCP tool sorts between existing
// built-ins.
return uniqBy(
  [...builtInTools].sort(byName).concat(allowedMcpTools.sort(byName)),
  'name',
)
```

内建工具和 MCP 工具**分别排序后拼接**，而不是混在一起排序。原因是：服务端在内建工具列表末尾放了一个 prompt cache 断点。如果 MCP 工具插入到内建工具之间（比如一个叫 `bash_helper` 的 MCP 工具按字母顺序排在 `BashTool` 和 `FileEditTool` 之间），断点位置就变了，整个 system prompt 的缓存就失效了。

缓存失效意味着每次请求都要重新传输和处理完整的 system prompt，成本直接上升。

这是一个"看起来是排序规则，实际上是缓存工程"的设计。如果有人"优化"这段代码，把两个列表合并后统一排序，功能完全正常，但每次有新 MCP 工具时缓存都会失效。

## 5.3 同名冲突：内建工具静默胜出

`assembleToolPool()` 用 `uniqBy(..., 'name')` 去重，保留第一个出现的工具。由于内建工具在前，同名时内建工具胜出，MCP 工具被静默丢弃。

这防止了一类攻击：恶意或有 bug 的 MCP 工具通过重名覆盖内建工具的行为。比如一个 MCP 工具命名为 `bash`，如果没有内建优先规则，它会替换掉内建的 `BashTool`，模型调用 `bash` 时实际执行的是 MCP 工具的逻辑。

静默丢弃的代价是：MCP 工具作者不会收到任何提示，工具就是不出现在工具池里。如果 MCP 工具作者不知道这个规则，可能会困惑为什么自己的工具没有生效。

## 5.4 deny rules：在模型看到工具之前就过滤

[`tools.ts#L265`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/tools.ts#L265) 的 `filterToolsByDenyRules()` 注释说明了一个关键点：

```typescript
// Uses the same matcher as the runtime permission check (step 1a), so MCP
// server-prefix rules like `mcp__server` strip all tools from that server
// before the model sees them — not just at call time.
```

deny rules 在工具池组装阶段就生效，不是在模型调用工具时才检查。这意味着被 deny 的工具根本不会出现在发给模型的工具列表里，模型不知道这些工具存在。

如果 deny rules 只在调用时检查，模型会看到工具、尝试调用、然后被拒绝。这会浪费 token（模型生成了一个无效的工具调用），也会让模型困惑（为什么这个工具不能用）。提前过滤让模型的工具视图和实际可用能力保持一致。

## 5.5 SIMPLE 模式：工具集的极简版本

[`tools.ts#L276`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/tools.ts#L276) 有一个 `CLAUDE_CODE_SIMPLE` 环境变量控制的简化模式：

```typescript
if (isEnvTruthy(process.env.CLAUDE_CODE_SIMPLE)) {
  const simpleTools: Tool[] = [BashTool, FileReadTool, FileEditTool]
  return filterToolsByDenyRules(simpleTools, permissionContext)
}
```

简化模式只保留三个工具：Bash、文件读、文件编辑。这是给嵌入场景用的——当 Claude Code 被集成到其他系统里，不需要完整工具集时，可以用这个模式减少 system prompt 的体积，降低 token 消耗。

## 5.6 验证题

如果你安装了一个 MCP 工具，它的名字恰好和内建的 `read_file` 工具相同，会发生什么？

答案：`assembleToolPool()` 在去重时保留内建工具，MCP 工具被静默丢弃。模型看到的是内建的 `read_file`，你的 MCP 工具不会出现在工具列表里，也不会有任何错误提示。

## 5.7 本章小结

工具池的组装有三个关键约束：内建工具必须排在 MCP 工具之前（[`tools.ts#L368`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/tools.ts#L368)，为了 prompt cache 稳定性）、同名时内建工具静默胜出（防止 MCP 覆盖内建行为）、deny rules 在组装阶段就过滤（让模型的工具视图和实际可用能力一致）。

需要记住的核心概念：**工具排序不只是展示顺序，还是 prompt cache 的工程约束——顺序错了不报错，只是缓存失效。**
