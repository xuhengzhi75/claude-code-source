# 第5章 能力装配：工具池是怎么拼出来的

酒店后台有四套系统：客房服务、门禁卡、前台终端、WiFi密码。每个系统对应不同的使用者，客人、员工、系统管理员各拿各的。把这四套东西混在一张清单里发给所有人，不仅混乱，还不安全。

Claude Code 同样有四类能力来源，划分对象不同，职责不同。

## 5.1 四类来源，划分对象不同

Claude Code 的能力来自：内建工具（`BashTool`、`FileReadTool` 等）、MCP 工具（外部服务器提供）、Skill（`SKILL.md` 提示词包装）、命令（`/help`、`/clear` 等斜杠命令）。

命令是给用户用的，用户在终端输入 `/clear` 清空对话，模型不知道这件事发生了。工具是给模型用的，模型在推理时决定调用 `read_file`，用户看到的是结果。Skill 是注入 system prompt 的提示词段落，把常见的工具调用序列包装成更高层指令。MCP 工具和内建工具一样，都是模型可以调用的，只是来源不同。

最终暴露给模型的是工具（内建 + MCP）和 Skill。命令对模型不可见。

核心主张：**能力装配层的价值在于，它在模型看到工具之前就决定了边界，而不是在调用时逐条检查。**

## 5.2 为什么内建工具必须排在 MCP 工具之前

[`tools.ts#L348`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/tools.ts#L348) 的 `assembleToolPool()` 注释说明了一个不明显的约束：

```typescript
// Sort each partition for prompt-cache stability, keeping built-ins as a
// contiguous prefix. The server's claude_code_system_cache_policy places a
// global cache breakpoint after the last prefix-matched built-in tool;
// a flat sort would interleave MCP tools into built-ins and invalidate
// all downstream cache keys whenever an MCP tool sorts between existing
// built-ins.
return uniqBy(
  [...builtInTools].sort(byName).concat(allowedMcpTools.sort(byName)),
  'name',
)
```

两个列表分别排序后拼接，而不是合并后统一排序。原因是服务端在内建工具列表末尾放了一个 prompt cache 断点。一个叫 `bash_helper` 的 MCP 工具按字母顺序会排在 `BashTool` 和 `FileEditTool` 之间，断点位置随之移动，整个 system prompt 的缓存失效，下一次请求必须重新传输完整上下文。

这个排序规则表面上是整洁，实质上是缓存工程。把两个列表合并统一排序，功能完全正常，但每次添加新 MCP 工具都会导致缓存失效，响应延迟和 token 成本双双上升。这类隐式约束通常不会写在 README 里，只有代码注释里留着。

## 5.3 同名冲突：内建工具静默胜出

`assembleToolPool()` 用 `uniqBy(..., 'name')` 去重，保留第一个出现的。内建工具在前，同名时 MCP 工具被静默丢弃。

这防止了一类覆盖攻击：恶意或有 bug 的 MCP 工具通过重名替换内建工具的行为。一个 MCP 工具命名为 `bash`，如果没有内建优先规则，模型调用 `bash` 时实际执行的会是 MCP 工具的逻辑。

代价是：MCP 工具作者不会收到任何提示，工具直接不出现。如果开发者不知道这条规则，他只能困惑为什么自己的工具无效。这是一个在安全性和可调试性之间做出的选择，系统选择了安全。

## 5.4 deny rules：在模型看到工具之前过滤

[`tools.ts#L265`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/tools.ts#L265) 的 `filterToolsByDenyRules()` 注释说：

```typescript
// Uses the same matcher as the runtime permission check (step 1a), so MCP
// server-prefix rules like `mcp__server` strip all tools from that server
// before the model sees them — not just at call time.
```

deny rules 在工具池组装阶段生效，被 deny 的工具不出现在发给模型的工具列表里。如果只在调用时检查，模型会看到工具、尝试调用、然后被拒绝——浪费 token，也让模型困惑。提前过滤让模型的工具视图和实际可用能力保持一致。

一条 `mcp__server` 前缀规则可以封堵整个 MCP 服务器的所有工具，不需要逐条列举。

## 5.5 `CLAUDE_CODE_SIMPLE`：工具集的极简变体

[`tools.ts#L276`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/tools.ts#L276)：

```typescript
if (isEnvTruthy(process.env.CLAUDE_CODE_SIMPLE)) {
  const simpleTools: Tool[] = [BashTool, FileReadTool, FileEditTool]
  return filterToolsByDenyRules(simpleTools, permissionContext)
}
```

简化模式只保留三个工具：Bash、文件读、文件编辑。这是嵌入场景的选项，当 Claude Code 被集成到其他系统时，减少 system prompt 体积，降低 token 消耗。deny rules 依然生效，嵌入方可以在三个工具上继续叠加限制。

## 5.6 边界与技术债

当前设计的隐含假设：**工具数量在可管理范围内，全集加载不成问题。** `getAllBaseTools()` 必须与 Statsig 动态配置同步（[`tools.ts#L191`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/tools.ts#L191)），否则 system prompt 缓存会跨用户失效。这个同步关系目前没有编译时检查，靠注释约定。

同名冲突静默处理是另一个已知问题：开发者没有可靠的方式在运行时知道自己的 MCP 工具是否被覆盖，只能翻工具列表手动核对。

## 5.7 可迁移原则

权限规则前置到能力注册阶段，而不是调用阶段。适用条件：权限检查成本高（token 浪费、用户困惑），且权限规则相对静态，不需要在调用时动态计算。反例：权限规则依赖运行时上下文（比如当前文件的所有者、执行时的网络状态），无法在注册阶段确定，只能在调用时检查。

## 5.8 本章不覆盖项

工具内部的权限检查机制（`checkPermissions()`）和并发控制（`isConcurrencySafe()`）在第14章展开。命令系统的动态过滤逻辑（`meetsAvailabilityRequirement()`）在第15章。Skill 的注入时机和 system prompt 的缓存分区机制在第15章。
