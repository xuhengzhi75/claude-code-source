# 第5章 能力装配：工具池的边界在哪里决定

餐厅后厨备菜，刀具、调料、炉灶分别锁在对应的储物柜里，厨师进厨房之前就只能拿到他那把锁对应的东西。不是到了炒锅面前再问"你能用这把刀吗"，而是进门的时候就已经定了。

Claude Code 的工具池装配逻辑是同一个思路：在模型看到工具之前，系统就已经决定了它能看到哪些工具。本章的核心主张是——**工具池的三个组装规则（分区排序、内建优先、前置过滤）都是这个约束的具体落地，不是独立的工程决策。**

能力来自四个来源：内建工具（`BashTool`、`FileReadTool` 等系统内置的工具类）、MCP 工具（外部服务器提供、通过 MCP 协议接入的工具）、Skill（注入 system prompt 的提示词段落，把常见工具调用序列包装成高层指令）、命令（`/help`、`/clear` 等斜杠指令）。命令是给用户用的，模型不知道 `/clear` 发生了。最终进入模型视野的只有工具（内建 + MCP）和 Skill。

## 5.1 排序分区是缓存工程，整洁只是副产品

[`tools.ts#L348`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/tools.ts#L348) 的 `assembleToolPool()` 把内建工具和 MCP 工具分别排序，再拼接，而不是合并成一个列表统一排序：

```typescript
// Sort each partition for prompt-cache stability, keeping built-ins as a
// contiguous prefix. The server's claude_code_system_cache_policy places a
// global cache breakpoint after the last prefix-matched built-in tool;
// a flat sort would interleave MCP tools into built-ins and invalidate
// all downstream cache keys whenever an MCP tool sorts between existing
// built-ins.
return uniqBy(
  [...builtInTools].sort(byName).concat(allowedMcpTools.sort(byName)),
  'name',
)
```

服务端在内建工具列表末尾放了一个 prompt cache（提示词缓存）断点。断点后面的内容变化不会让断点前面的缓存失效。问题在于：如果把两个列表合并统一排序，一个叫 `bash_helper` 的 MCP 工具按字母顺序会插入到 `BashTool` 和 `FileEditTool` 之间，断点的位置随之移动，整个 system prompt 的缓存键重新计算，下一次请求必须重传完整上下文。

分区排序让内建工具列表保持稳定。MCP 工具无论怎么增减，都只影响断点之后的部分，之前的缓存照常命中。把两个列表合并排序，功能完全正确，但每次安装新 MCP 工具都会让缓存失效，响应延迟和 token 成本双双上升。这个约束只存在于代码注释里，没有编译时检查。

## 5.2 内建工具静默优先是安全约束，调试可见性是代价

`assembleToolPool()` 用 `uniqBy(..., 'name')` 去重，保留第一个出现的。内建工具在列表前部，同名时 MCP 工具被静默丢弃。

静默丢弃封堵了覆盖攻击（tool shadowing attack）：MCP 工具如果能通过同名替换内建工具，一个命名为 `bash` 的 MCP 工具就能在模型调用 `bash` 时注入恶意逻辑，而模型对此一无所知。内建优先规则让这条路在工具池组装阶段就被堵死，不需要在调用时再做检查。

代价是调试体验。MCP 工具作者不会收到任何错误或警告，工具直接从可用列表里消失。开发者只能手动翻工具列表核对。这个取舍目前没有改善计划，属于已知缺口（inference）。系统在安全性和调试可见性之间选择了前者。

## 5.3 deny rules 前置让模型视图和实际能力一致

[`tools.ts#L265`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/tools.ts#L265) 的 `filterToolsByDenyRules()` 注释说明了这一点：

```typescript
// Uses the same matcher as the runtime permission check (step 1a), so MCP
// server-prefix rules like `mcp__server` strip all tools from that server
// before the model sees them — not just at call time.
```

deny rules（拒绝规则）在工具池组装阶段生效，被拒绝的工具不出现在发送给模型的工具列表里。如果改为调用时检查，模型会看到工具、尝试调用、然后被拒绝——浪费一次推理 token，且模型下一轮可能再次尝试，因为它的工具视图里这个工具始终存在。

一条 `mcp__server` 前缀规则可以封堵整个 MCP 服务器的所有工具，不需要逐条列举。前置过滤让模型看到的工具集和它实际能调用的工具集保持一致，消除了"看得到用不了"的认知错位。

## 5.4 边界与可迁移原则

**极简变体。** [`tools.ts#L276`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/tools.ts#L276) 的 `CLAUDE_CODE_SIMPLE` 模式把工具集缩减为三个：

```typescript
if (isEnvTruthy(process.env.CLAUDE_CODE_SIMPLE)) {
  const simpleTools: Tool[] = [BashTool, FileReadTool, FileEditTool]
  return filterToolsByDenyRules(simpleTools, permissionContext)
}
```

嵌入场景（Claude Code 作为子进程被其他系统集成时）用这个模式减少 system prompt 体积。deny rules 在三个工具上照样生效，嵌入方可以继续叠加限制。

**隐含假设与技术债。** 当前设计假设工具数量在可管理范围内，全集加载没有性能压力。`getAllBaseTools()` 必须与 Statsig 动态配置同步（[`tools.ts#L191`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/tools.ts#L191)），否则 system prompt 缓存会跨用户失效，这个依赖关系目前靠注释约定，没有编译时保障。同名冲突的静默处理是另一个已知问题，开发者缺乏可靠的运行时诊断手段。

**可迁移原则。** 权限检查前置到能力注册阶段而不是调用阶段，适用条件有两个：权限检查本身成本高（token、延迟、模型困惑任意一项），且权限规则在注册时就能确定。反例是权限规则依赖运行时上下文——比如当前操作的文件所有者、网络状态、执行时的用户身份——这类规则在注册阶段无法确定，只能在调用时检查。原则是工具，不是教条。

## 5.5 本章不覆盖项

工具内部的权限检查（`checkPermissions()`）和并发控制（`isConcurrencySafe()`）在第14章展开。命令系统的动态过滤逻辑（`meetsAvailabilityRequirement()`）在第15章。Skill 的注入时机和 system prompt 的缓存分区机制在第15章。
