# 第5章 能力装配：命令、工具、技能和插件怎么拼成可用系统

> 核心主张：Claude Code 的能力装配层说明"先全集后过滤"不是性能优化，而是权限治理策略——把权限规则集中在过滤层，而不是散落在每个工具实现里，这样新增工具时不需要记得在每个地方加权限检查。

Claude Code 不是把模型直接接到一堆能力上，而是先经过一层装配与过滤，再决定当前会话真正能看到什么、能调用什么。

这层的意义，不是工具更多，而是能力边界更清楚。在真实系统里，模型上限往往不只由模型本身决定，还由三件事决定：当前能看到哪些动作、哪些动作受权限约束、不同来源的能力如何被统一组织。

## 5.1 什么叫能力装配

这里说的能力装配，不是列一个工具清单，而是做 3 件事：准备理论上可用的能力全集，按当前环境、权限和模式过滤，把留下的能力整理成统一接口。

这个过程决定了当前会话的工作边界。同一个模型接在不同装配层后面，表现可能差很多，因为模型能操作的世界并不相同。

## 5.2 命令和工具都不是看到什么就全给什么

从 `src/commands.ts` 可以看到，命令来自多个来源：内建命令、skill 命令、插件命令、workflow 命令，以及动态发现的命令。合并之后还会继续做可用性和启用状态过滤。

从 `src/tools.ts` 可以看到，工具也不是直接全量暴露。`getAllBaseTools()`（[`tools.ts#L193`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/tools.ts#L193)）先给出理论工具全集，`filterToolsByDenyRules()`（[`tools.ts#L265`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/tools.ts#L265)）再按权限上下文与运行模式过滤出当前会话真正可用的集合。

这两层的共同点是：系统先整理，再暴露。

## 5.3 工具层为什么不只是函数集合

`src/Tool.ts#L763` 定义了 `TOOL_DEFAULTS`，其中有两个保守默认值：`isReadOnly: false`（默认非只读）和默认不可并发。这意味着新工具如果忘了声明自己的安全属性，系统会按最严格的方式处理，而不是按最宽松的方式。

这是一个反直觉的设计选择：通常我们会认为"默认允许"比"默认拒绝"更方便开发。但 `TOOL_DEFAULTS` 的选择是"默认拒绝"——新工具必须主动声明自己是只读的、可并发的，才能获得对应的宽松处理。这把安全边界当成默认值，而不是需要主动开启的选项。

`buildTool()`（[`Tool.ts#L789`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/Tool.ts#L789)）保证了每个工具即使只实现最小接口，也会被安全默认值补齐。

这意味着工具层本身就是运行时治理层，而不是外挂仓库。

## 5.4 为什么模型前面要先放一层治理

如果把模型直接连到脚本、接口和插件上，前期开发会看起来更快，但很快会遇到几个问题：哪些能力该默认开放，哪些能力必须先做权限判断，多来源能力重名时怎么办，某些模式下哪些能力根本不该出现。

Claude Code 在这一层的做法是先治理，再开放。`assembleToolPool()`（[`tools.ts#L350`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/tools.ts#L350)）在合并内建工具和 MCP 工具时，重名时保持内建优先——这个规则防止了外部 MCP 工具通过重名覆盖内建工具的行为。这会增加一点前置设计成本，但能换来更清楚的边界和更稳定的行为。

## 5.5 命令、工具、技能、插件分别扮演什么角色

可以把这几类能力来源简化理解成：命令偏用户可见的控制入口，工具偏模型在执行中调用的动作接口，技能偏可复用的方法包装和经验单元，插件偏外部扩展能力来源。

它们不是同一个层次，但都会影响当前会话能做什么。能力装配层的作用，就是把这些来源整理成一个可运行的能力池。

## 5.6 对自己做 Agent 最值得抄的三件事

先定义能力边界，再扩能力数量；给能力统一接口，而不是各写各的调用方式；把理论能力和当前可见能力分开。

这三件事越早做，系统越不容易在后期失控。

## 5.7 本章小结

Claude Code 的能力装配层做的不是堆功能，而是先整理命令、工具、技能和插件，再把当前会话真正可用的那部分暴露给模型。

"先全集后过滤"的顺序不是随意的：权限规则集中在 `filterToolsByDenyRules()` 一处，新增工具时不需要在每个工具实现里记得加权限检查。这样做的结果是，模型面对的不是原始能力堆，而是一套经过过滤、统一和边界治理的动作池。后面再看 `QueryEngine`，就更容易理解：执行核心为什么能建立在更稳定的能力边界之上。
