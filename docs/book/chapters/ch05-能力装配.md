# 第5章 能力装配：命令、工具、技能和插件怎么拼成可用系统

> **核心主张：能力装配层解决的第一性问题是"权限治理的集中化"——把权限规则集中在过滤层，而不是散落在每个工具实现里，这样新增工具时不需要在每个地方记得加权限检查。**

## 5.1 第一性问题：为什么权限不能分散在工具里

假设每个工具自己负责权限检查，会发生什么？

新增工具时，开发者需要记得在工具实现里加权限检查。如果忘了，工具就会在不该可用的场景下出现。随着工具数量增加，权限规则散落在几十个文件里，修改一条规则需要找到所有相关工具并逐一修改。

Claude Code 的做法是把权限检查集中在 `filterToolsByDenyRules()`（[`tools.ts#L265`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/tools.ts#L265)）。新增工具时，工具实现不需要关心权限，权限规则在过滤层统一处理。

**这个设计的代价：** 过滤层需要了解所有工具的语义，才能正确判断哪些工具在哪些场景下应该被过滤。随着工具数量增加，过滤层的复杂度也会增加。

## 5.2 "先全集后过滤"的顺序不是随意的

`getAllBaseTools()`（[`tools.ts#L193`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/tools.ts#L193)）先给出理论工具全集，`filterToolsByDenyRules()` 再按权限上下文与运行模式过滤出当前会话真正可用的集合。

为什么不反过来——先按模式决定加载哪些工具，再合并？

反向做法的问题是：权限规则需要在多个地方维护（每个模式的加载逻辑里都要有），而且"理论上存在哪些工具"这个信息会变得不透明。"先全集后过滤"让理论能力和实际能力分开：`getAllBaseTools()` 回答"系统能做什么"，`filterToolsByDenyRules()` 回答"当前会话能做什么"。

**隐含假设：** 这个设计假设工具全集的加载成本是可接受的。如果工具数量增长到几百个，每次会话都加载全集再过滤，可能产生性能问题。

## 5.3 保守默认值：安全边界作为默认值

`src/Tool.ts#L763` 定义了 `TOOL_DEFAULTS`，其中有两个保守默认值：`isReadOnly: false`（默认非只读）和默认不可并发。

这是一个反直觉的设计选择。通常我们认为"默认允许"比"默认拒绝"更方便开发——新工具不需要声明任何属性就能工作。但 `TOOL_DEFAULTS` 的选择是"默认拒绝"：新工具必须主动声明自己是只读的、可并发的，才能获得对应的宽松处理。

**隐含假设：** 这个设计假设工具开发者会主动声明安全属性。如果开发者不了解这个约定，工具会以最严格的方式运行，但不会有任何错误提示——工具能用，只是比应有的更保守。

`buildTool()`（[`Tool.ts#L789`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/Tool.ts#L789)）保证了每个工具即使只实现最小接口，也会被安全默认值补齐。

## 5.4 内建工具优先：防止外部工具覆盖内建行为

`assembleToolPool()`（[`tools.ts#L350`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/tools.ts#L350)）在合并内建工具和 MCP 工具时，重名时保持内建优先。

**这条规则防止了什么？** 外部 MCP 工具可以通过重名覆盖内建工具的行为。比如，一个恶意或有 bug 的 MCP 工具命名为 `read_file`，如果没有内建优先规则，它会替换掉内建的文件读取工具。

**替代方案：** 另一种做法是命名空间隔离——内建工具用 `builtin.read_file`，MCP 工具用 `mcp.read_file`，完全避免重名冲突。命名空间隔离的优势是更透明（用户能看到工具来源），劣势是模型需要学习命名空间约定，提示词也需要相应调整。

内建优先的选择是在安全性和简洁性之间的权衡：模型不需要了解命名空间，但外部工具无法扩展内建工具的行为。

## 5.5 命令、工具、技能、插件的职责边界

这四类能力来源的职责是不同的：

命令（commands）是用户可见的控制入口，比如 `/help`、`/clear`。它们的受众是用户，不是模型。工具（tools）是模型在执行中调用的动作接口，比如 `read_file`、`bash`。它们的受众是模型，不是用户。技能（skills）是可复用的方法包装，把常见的工具调用序列封装成更高层的能力。插件（MCP）是外部扩展能力来源，通过标准协议接入。

**技术债：** 命令和工具的边界在某些场景下是模糊的。斜杠命令（如 `/force-snip`）可以修改消息历史，这是工具层的能力，但它通过命令接口暴露。这种混用在当前规模下可以接受，但随着命令数量增加，可能需要更清晰的边界。

## 5.6 可迁移的设计原则

**原则：把权限规则集中在过滤层，而不是分散在能力实现里。**

适用条件：系统有多个能力来源，权限规则需要跨能力统一执行。

反例：如果每个能力的权限规则完全不同，集中过滤层会变得非常复杂，可能不如分散管理。

**原则：用保守默认值保护安全边界。**

新增能力时，默认按最严格的方式处理，而不是最宽松的方式。这把安全责任从"记得加限制"转移到"记得声明宽松"，减少了遗漏的可能性。

## 5.7 心智模型验证题

如果一个 MCP 工具和内建工具同名，会发生什么？

答案：`assembleToolPool()` 在合并时保持内建优先，MCP 工具会被丢弃（或重命名）。模型看到的是内建工具，而不是 MCP 工具。这个行为是静默的——没有错误提示，MCP 工具只是不出现在工具池里。

## 5.8 本章小结

能力装配层的核心价值是权限治理的集中化：`filterToolsByDenyRules()` 是权限规则的唯一执行点，新增工具不需要在工具实现里加权限检查。"先全集后过滤"让理论能力和实际能力分开，保守默认值把安全边界作为默认值，内建优先防止外部工具覆盖内建行为。

这三个设计共同保证了：能力数量增加时，权限治理的复杂度不会线性增长。
