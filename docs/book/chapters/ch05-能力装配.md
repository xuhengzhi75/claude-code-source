# 第5章 能力装配：工具池在模型看到它之前就已定型

厨师进厨房的时候，储物柜已经按岗位锁好了。不是站到炒锅面前再问"这把刀能不能用"，而是进门的那一刻就已经定了。

Claude Code 的工具池装配逻辑是同样的时序选择：在工具列表发送给模型之前，系统就已经完成了全部裁剪。分区排序、内建优先、deny rules 前置，三个规则都是这个时序约束的具体落地，不是独立的工程决策。理解了时序，三个规则就变得可预测；不理解时序，三个规则看起来像三个不相关的设计偏好。

工具来自四个来源：内建工具（`BashTool`、`FileReadTool` 等系统内置的工具类）、MCP 工具（通过 MCP 协议接入的外部服务器工具）、Skill（注入 system prompt 的提示词段落，把常见工具调用序列包装成高层指令）、命令（`/help`、`/clear` 等斜杠指令）。命令是给用户用的，模型不感知 `/clear` 发生了什么。最终进入模型视野的是工具（内建 + MCP）和 Skill。

## 5.1 分区排序保护缓存，合并排序让每次安装 MCP 工具都付出代价

[`tools.ts#L348`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/tools.ts#L348) 的 `assembleToolPool()` 把内建工具和 MCP 工具分别排序后再拼接，而不是合并成一个列表统一排序：

```typescript
// Sort each partition for prompt-cache stability, keeping built-ins as a
// contiguous prefix. The server's claude_code_system_cache_policy places a
// global cache breakpoint after the last prefix-matched built-in tool;
// a flat sort would interleave MCP tools into built-ins and invalidate
// all downstream cache keys whenever an MCP tool sorts between existing
// built-ins.
return uniqBy(
  [...builtInTools].sort(byName).concat(allowedMcpTools.sort(byName)),
  'name',
)
```

服务端在内建工具列表末尾放了一个 prompt cache（提示词缓存）断点（cache breakpoint）。断点之前的内容不变，缓存就能命中；断点之后的变化不影响断点之前的缓存键。合并排序会把一个叫 `bash_helper` 的 MCP 工具按字母顺序插入 `BashTool` 和 `FileEditTool` 之间，断点位置随之移动，整个 system prompt 的缓存键重新计算，下一次请求必须重传完整上下文。

分区排序让内建工具列表保持稳定。MCP 工具无论怎么增减，只影响断点之后的部分，之前的缓存照常命中。两种方式功能上完全等价，分区排序只是额外保护了缓存命中率，代价是每次安装新 MCP 工具时响应延迟和 token 成本都会上升的风险被消除了。这个约束只活在代码注释里，没有编译时检查——注释被删掉，规则就消失了。

## 5.2 内建优先用静默丢弃封堵覆盖攻击，调试可见性是已知代价

`assembleToolPool()` 用 `uniqBy(..., 'name')` 去重，保留列表中第一个出现的同名工具。内建工具排在列表前部，同名时 MCP 工具被静默丢弃。

这个机制封堵了工具覆盖攻击（tool shadowing attack）：如果 MCP 工具能通过同名替换内建工具，一个命名为 `bash` 的 MCP 工具就能在模型调用 `bash` 时注入恶意逻辑，而模型对此没有任何感知能力。内建优先规则让这条路在工具池组装阶段就被堵死，不需要等到调用时再做检查。

调试体验为此付出了代价。MCP 工具作者不会收到任何错误或警告，工具直接从可用列表里消失。开发者只能手动翻工具列表核对是否有同名冲突。这个可见性缺口目前没有改善计划，属于已知技术债（inference）。

## 5.3 deny rules 前置消除"看得到用不了"的认知错位

[`tools.ts#L265`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/tools.ts#L265) 的 `filterToolsByDenyRules()` 在工具池组装阶段生效：

```typescript
// Uses the same matcher as the runtime permission check (step 1a), so MCP
// server-prefix rules like `mcp__server` strip all tools from that server
// before the model sees them — not just at call time.
```

deny rules（拒绝规则）在工具列表发送给模型之前过滤完毕。被拒绝的工具不出现在模型收到的工具列表里。如果改为调用时检查，模型会看到工具、尝试调用、收到拒绝响应，然后可能在下一轮继续尝试，因为它的工具视图里这个工具始终存在，它没有理由认为这次调用会和上次有什么不同。

一条 `mcp__server` 前缀规则可以封堵整个 MCP 服务器的所有工具，不需要逐条列举。前置过滤让模型看到的工具集和它实际能调用的工具集保持一致，两个视图重合，"看得到用不了"的推理浪费就不存在了。

## 5.4 边界条件改变，三个规则的假设就可能失效

[`tools.ts#L276`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/tools.ts#L276) 的 `CLAUDE_CODE_SIMPLE` 模式把工具集缩减为三个，用于嵌入场景（Claude Code 作为子进程被外部系统集成时）：

```typescript
if (isEnvTruthy(process.env.CLAUDE_CODE_SIMPLE)) {
  const simpleTools: Tool[] = [BashTool, FileReadTool, FileEditTool]
  return filterToolsByDenyRules(simpleTools, permissionContext)
}
```

deny rules 在三个工具上照样生效，嵌入方可以继续叠加限制。这个模式同时减少了 system prompt 体积，降低了嵌入场景的 token 开销。

当前三个规则都隐含着工具数量在可管理范围内这个假设——全集加载没有性能压力，因为工具列表不长。`getAllBaseTools()` 必须与 Statsig 动态配置同步（[`tools.ts#L191`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/tools.ts#L191)），否则 system prompt 缓存会跨用户失效；这个依赖关系目前靠注释约定，没有编译时保障。同名冲突的静默处理是另一个已知缺口，开发者缺乏可靠的运行时诊断手段。

权限检查前置到能力注册阶段而不是调用阶段，适用条件有两个：权限检查本身成本高（token、延迟、模型困惑任意一项），且权限规则在注册时就能确定。反例是权限规则依赖运行时上下文——比如当前操作的文件所有者、网络状态、执行时的用户身份。这类规则在注册阶段无法确定，只能在调用时检查。原则是工具，不是教条。

工具内部的权限检查（`checkPermissions()`）和并发控制（`isConcurrencySafe()`）在第14章展开。命令系统的动态过滤逻辑（`meetsAvailabilityRequirement()`）和 Skill 的注入时机、system prompt 的缓存分区机制在第15章。
