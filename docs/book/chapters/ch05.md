# 第5章 能力装配：命令、工具、技能和插件怎么拼成可用系统

前一章讲的是入口分流。
这一章开始进入 Claude Code 的第二层：系统不是直接把模型放出来工作，而是先决定本次会话到底拥有哪些能力。

这是很多 Agent 产品最容易被忽略的一层。
表面上看，模型会不会做事，似乎取决于模型本身。实际上，在真实系统里，更常决定上限的是：

- 系统给了它哪些命令和工具
- 这些能力是怎么暴露出去的
- 哪些能力默认可见，哪些需要过滤
- 权限边界在哪里

Claude Code 在这层做得比较系统，所以它给人的感觉不是“会几个功能”，而是“像一台可以持续工作的机器”。

## 5.1 什么叫能力装配

这里说的“能力装配”，不是简单列一个工具清单。
它指的是：

1. 系统先准备理论上可用的能力全集
2. 再根据当前环境、权限、模式和可用性，把本次真正能用的那一部分挑出来
3. 最后把这些能力组织成模型可以调用、系统可以管理的统一接口

这个过程决定了会话的工作边界。

同一个模型，如果接在两套不同装配系统后面，表现会非常不一样。
因为模型看到的不是“世界本身”，而是系统允许它操作的那一层世界。

## 5.2 命令不是写死的，而是动态生成的

和很多人直觉不同，Claude Code 的命令并不是启动时写死的一份固定菜单。

从 `src/commands.ts` 可以看出，命令来源会被合并：

- 内建命令
- skill 目录命令
- 插件命令
- workflow 命令
- 动态发现的 skill 命令

合并后还要继续做筛选，例如：

- 当前账号或环境是否满足可用条件
- 某个命令是否被显式禁用
- 重名命令如何处理

这意味着命令系统不是“仓库里有什么就全给模型看”，而是：
**每次会话都重新计算当前可见命令集。**

这很重要，因为真实运行环境是会变化的。
登录状态可能变化，插件可能变化，某些能力可能被 feature gate 裁掉，某些命令也可能在当前上下文根本不该暴露。

## 5.3 工具系统走的是“先全集，再过滤”

工具层的设计也很类似。

从 `src/tools.ts` 的结构看，可以概括成两步：

1. 先得到理论上的基础工具全集
2. 再根据权限与运行模式过滤出本次真正可用的集合

这里最值得注意的不是工具多，而是它的过滤发生在“暴露给模型之前”。

也就是说，Claude Code 不只是事后拦截危险动作，而是在一开始就尽量不把不该用的工具交给当前会话。

这和“先全开放，再靠模型自己克制”完全不是一个思路。

## 5.4 工具不只是函数集合，而是统一契约

在源码里，工具并不是简单的调用函数。
`src/Tool.ts` 这一层承担的是统一协议。

从架构上看，一个工具除了执行逻辑，还往往会携带这些信息：

- 输入输出约定
- 权限检查
- 是否只读
- 是否具破坏性
- 是否可并发
- UI 如何展示
- 摘要如何生成
- 与 MCP 工具如何映射

这说明 Claude Code 的工具层不是“外挂功能仓库”，而是运行时的一部分。

统一契约的意义非常现实：

- 系统可以更稳定地接新工具
- 权限控制更容易落地
- UI 和日志能统一展示结果
- 工具结果能稳定回到主循环里

这也是为什么真正成熟的 Agent，不是工具越多越厉害，而是工具协议越统一越稳。

## 5.5 为什么要在模型前面放一层能力治理

如果把模型直接连到一堆脚本和接口上，看起来开发会更快。
但真实系统很快就会遇到问题：

- 哪些能力该默认开放
- 哪些操作必须先过权限
- 多来源工具重名怎么办
- 插件和内建能力冲突怎么办
- 某些模式下哪些能力根本不该出现

Claude Code 在这一层的处理方式很像企业里的权限系统和服务注册中心：
先治理，再暴露。

这会让前期复杂一点，但后期收益很大。
因为模型不是直接面对原始能力堆，而是面对一套已经被整理过的、边界较清楚的动作系统。

## 5.6 命令、工具、技能、插件之间是什么关系

如果从普通读者视角理解，可以把这几类东西想成不同层次的能力来源：

- **命令（commands）**：偏用户可见的操作入口和控制指令
- **工具（tools）**：偏模型在执行过程中可调用的动作接口
- **技能（skills）**：偏可复用的任务经验、命令扩展或工作方法包装
- **插件（plugins）**：偏外部扩展能力来源

它们不是一回事，但最后都会影响“当前会话能做什么”。

所以系统需要一层装配逻辑，把这些不同来源的能力拼成一个当前可运行的能力池。

这就是为什么前面说，Claude Code 强的不只是工具本身，而是它对能力来源做了系统化组织。

## 5.7 能力装配层和入口层的关系

前一章讲过，入口层决定请求走哪条路径。
这一章讲的能力装配层，则是在进入主程序后回答另一个问题：

**既然已经进入这次会话，那么这次会话到底拥有哪些手和脚。**

两层配合起来，系统才完整：

- 入口层解决“走哪条路”
- 装配层解决“带什么能力上路”

如果没有入口分流，所有模式都会落到同样重的初始化。
如果没有能力装配，所有会话又会看到同样粗糙的一套能力。

成熟系统需要两者同时成立。

## 5.8 对自己做业务 Agent 的启发

如果你以后要做一个简化版 Agent，这一层非常值得先学，而不是最后补。

一个更健康的顺序通常是：

### 先定义能力边界，再加能力数量

不要一开始就拼命接工具。
先想清楚：

- 这个 Agent 在当前场景下需要哪些动作
- 哪些动作默认开放
- 哪些动作要受权限控制
- 哪些动作只读，哪些动作会改环境

### 让能力有统一接口

哪怕你只有 3 个工具，也应该让它们至少有统一的：

- 名字
- 输入
- 输出
- 错误处理
- 权限标签

### 把“可见能力”和“理论能力”分开

仓库里可以有很多工具，但每次会话只开放一部分。
这个边界越早建立，系统越不容易失控。

## 5.9 本章小结

这一章的核心结论是：

**Claude Code 在模型前面放了一层能力装配系统，用来动态组织命令、工具、技能和插件，并在暴露前先完成过滤与治理。**

这样做的结果是，模型拿到的不是一堆杂乱能力，而是一套当前上下文下可用、可控、可追踪的能力池。

理解这一层后，下一章再看 `QueryEngine`，就会更容易明白：
为什么执行核心不是孤立运转的，而是建立在一套已经装配好的能力边界之上。
