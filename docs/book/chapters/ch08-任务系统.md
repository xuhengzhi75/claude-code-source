# 第8章 任务系统：状态从内存提升到文件系统

想象一个场景：你让 Claude Code 帮你重构一个大型项目，它已经跑了 20 分钟，改了 30 个文件，还有 10 个文件没改完。这时候你的电脑死机了。

重启之后，Claude Code 对刚才发生的一切一无所知。20 分钟的工作，全部消失。

这不是假设。这是对话循环的天然局限：状态活在进程内存里，进程退出，状态消失。对话循环解决的是"这一轮怎么继续"，但它没有回答"进程重启后任务在哪里"。

任务系统解决的就是这个问题。

## 8.1 任务对象化：状态从内存提升到文件系统

[`tasks.ts#L76`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/utils/tasks.ts#L76) 定义了 `TaskSchema`：

```typescript
export const TaskSchema = lazySchema(() =>
  z.object({
    id: z.string(),
    subject: z.string(),
    description: z.string(),
    owner: z.string().optional(), // agent ID
    status: TaskStatusSchema(),   // 'pending' | 'in_progress' | 'completed'
    blocks: z.array(z.string()),
    blockedBy: z.array(z.string()),
    metadata: z.record(z.string(), z.unknown()).optional(),
  }),
)
```

任务状态不是内存里的变量，而是写入文件系统的 JSON 对象。进程重启后，任务状态仍然存在。这是任务系统和普通变量管理的根本区别。

这个设计有一个隐含假设：文件系统是可靠的持久化介质。在单机环境下，这个假设成立。在分布式环境下（多台机器共享任务），文件系统不再是可靠的共享存储，需要换成数据库或分布式存储。文件系统的选择是在简单性和功能性之间的权衡：对单机场景足够，对分布式场景不够。

## 8.2 高水位文件：防止 ID 复用

[`tasks.ts#L92`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/utils/tasks.ts#L92) 的 `HIGH_WATER_MARK_FILE`（`.highwatermark`）记录已分配过的最大任务 ID。

为什么不直接用"当前最大任务 ID + 1"？

考虑这个场景：任务 1、2、3 存在，任务 3 完成后被删除，当前最大 ID 变成 2，新任务被分配 ID 3。但历史日志里的 ID 3 指向的是旧任务，新任务和旧任务的日志会混淆，状态追踪错乱。

高水位文件保证任务 ID 单调递增，即使旧任务被删除，新任务也不会复用旧 ID。

这个文件是一个单点写入。如果多个进程同时分配任务 ID，可能产生竞态。当前实现依赖文件锁来保证原子性，但文件锁在某些文件系统（NFS）上不可靠——在 NFS 挂载的环境里跑多 Agent，这里是第一个会出问题的地方。

## 8.3 原子认领：防止 TOCTOU 竞态

[`tasks.ts#L621`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/utils/tasks.ts#L621) 的 `claimTaskWithBusyCheck()` 注释里写得很清楚：

```typescript
// 关键一致性点：把"检查 agent 是否繁忙"与"抢占任务"放进同一把 list-level 锁。
// 这是典型的 check-then-act 场景，分离会导致 TOCTOU 竞争
// （两个 agent 同时看到空闲并同时认领）。
```

TOCTOU 是 time-of-check to time-of-use 的缩写。具体场景是：Agent A 检查任务 T 是否空闲，发现空闲。Agent B 也检查任务 T，发现空闲。Agent A 认领任务 T。Agent B 也认领任务 T。结果是两个 agent 同时执行同一个任务。

把检查和认领放在同一把锁内，保证了这两步是原子的：一旦 Agent A 开始检查，Agent B 必须等待，直到 Agent A 完成认领或放弃。

锁的参数也有讲究（[`tasks.ts#L94-L108`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/utils/tasks.ts#L94-L108)）：

```typescript
// Budget sized for ~10+ concurrent swarm agents: each critical section does
// readdir + N×readFile + writeFile (~50-100ms on slow disks), so the last
// caller in a 10-way race needs ~900ms. retries=30 gives ~2.6s total wait.
const LOCK_OPTIONS = {
  retries: { retries: 30, minTimeout: 5, maxTimeout: 100 },
}
```

`retries: 30` 是为 10 个并发 agent 的 swarm 场景设计的。每个临界区大约需要 50-100ms，10 个 agent 同时竞争时，最后一个等待者最多需要 900ms，30 次重试给出约 2.6 秒的总等待时间。这个数字不是随意选的，是根据实际场景估算的。

## 8.4 任务系统和主循环的接缝

任务系统不是完全绕开主循环独立存在的。`src/query.ts#L1584` 里有 `task-notification` 相关处理，`query.ts#L1694` 里有 `BG_SESSIONS` 下的 task summary 生成。

主循环在执行过程中，可以感知任务系统的状态变化，也可以向任务系统报告执行进度。这些接缝是隐式的——需要读代码才能找到所有接缝点。如果未来需要修改任务系统的接口，需要找到所有接缝点并逐一修改。没有显式接口声明，接口变更的成本就会落在下一个读代码的人身上。

## 8.4.1 Fork Sub-Agent：Prompt Cache 共享设计

普通的 Sub-Agent 从零开始，主 Agent 需要给它一份完整的背景介绍才能开工。Fork Sub-Agent 直接继承主 Agent 的完整对话历史和系统提示，减少上下文传递的成本。

Fork 机制还有一个更深的设计意图：让多个克隆实例共享 Prompt Cache。来自 `tools/AgentTool/forkSubagent.ts` 的注释：

> "For prompt cache sharing, all fork children must produce byte-identical API request prefixes."

所有 Fork 实例的前缀消息被刻意设计成完全一致，只有最后那一段指令不同。这样 API 服务器可以缓存前面的大段内容，只需计算最后那一点差异。并行克隆 10 个实例，API 调用成本远低于 10 倍，因为大部分内容被缓存了。

每个 Fork 实例收到的指令格式也有严格约束：

```typescript
export function buildChildMessage(directive: string): string {
  return `<fork-boilerplate>
STOP. READ THIS FIRST.

You are a forked worker process. You are NOT the main agent.

RULES:
1. Do NOT spawn sub-agents; execute directly.  // 禁止再克隆，防止无限嵌套
2. Do NOT converse, ask questions              // 不许废话，直接干活
3. USE your tools directly: Bash, Read, Write // 用工具行动，别聊天
4. Your response MUST begin with "Scope:"    // 必须先声明自己的工作范围

Output format:
  Scope:         // 我负责什么
  Result:        // 我做完了什么
  Key files:     // 涉及哪些文件
  Files changed: // 修改了哪些文件（含 git commit hash）
  Issues:        // 发现了什么问题
</fork-boilerplate>

${directive}
`
}
```

这个格式约束有两个作用：一是保证前缀字节完全一致（缓存命中），二是强制子实例先声明工作范围再行动，避免越界操作。

## 8.4.2 DreamTask：后台记忆整理

有一种特殊的后台任务叫 DreamTask，职责是趁 Claude Code 空闲时在后台整理记忆（Memory Consolidation）。它不介入正常执行流程，等空闲时将工作日志整理成结构化的记忆文件，供下次使用。

从 `tasks/DreamTask/DreamTask.ts` 可以看到它的内部阶段：orient（定向）→ gather（收集）→ consolidate（整合）→ prune（裁剪）。对外只暴露"开始"和"更新中"两种状态，内部四阶段对调用方透明。

这个设计的价值在于：记忆整理是计算密集型操作，放在主流程里会阻塞用户交互，放在后台则完全无感。代价是整理结果有延迟，不是实时可用的。

## 8.4.3 Coordinator 模式：理解合成而非转发

协调器模式（Coordinator Mode）通过环境变量 `CLAUDE_CODE_COORDINATOR_MODE=1` 激活。在这个模式下，主 Agent 完全转变角色，只做三件事：给 Worker 下达任务、接收 Worker 的汇报（以 `<task-notification>` XML 格式）、综合所有信息决定下一步。

源码注释中写着：

> "Parallelism is your superpower. Workers are async. Launch independent workers concurrently whenever possible."

协调器模式最容易犯的错误是"懒惰的委托"——把研究 Agent 的汇报原封不动转发给执行 Agent：

```
// 反模式（懒惰的委托——错的）
AgentTool({ prompt: "Based on your findings, fix the auth bug" })

// 正确做法（经过理解合成的指令——对的）
AgentTool({ prompt: "Fix the null pointer in src/auth/validate.ts:42.
  The user field on Session is undefined when sessions expire but the token
  remains cached. Add a null check before user.id access — if null, return 401." })
```

协调器必须真正看懂研究 Agent 的汇报，然后写出一份经过消化的新指令，包含具体文件路径、行号、修改内容。任务不能只是转发，需要经过理解再重新表达。这个原则不只适用于 AI 系统，任何多层级的任务分派都面临同样的问题。

## 8.5 KAIROS：动态权限的探索方向

当前任务系统的权限模型是静态的——任务在创建时确定权限范围，执行过程中不变。但 KAIROS（Claude Code 内部的自主代理研究项目）探索了一个更动态的方向：把终端的焦点状态作为权限信号。

具体来说，当用户的终端窗口处于前台（用户正在看着屏幕）时，Agent 可以执行更高风险的操作；当终端处于后台（用户不在场）时，Agent 应该降低操作风险等级，或者暂停等待用户确认。

这个设计的直觉是：用户在场时，可以立即发现和纠正错误；用户不在场时，错误可能在被发现之前已经造成不可逆的影响。

现有的 `TaskSchema` 没有动态权限字段，权限在任务创建时固定。如果要支持 KAIROS 式的动态权限，需要在任务状态里加入"当前有效权限"字段，并在每次执行操作前重新评估。

这个方向有一个隐含假设：终端焦点状态等价于"用户在监督"。但用户可能开着终端去做别的事，或者终端在前台但用户没有在看。这个假设在大多数情况下成立，但不是严格的安全保证。

## 8.6 最小任务系统的必要条件

如果要做一个缩小版业务 Agent，最小任务系统至少需要五件事：独立任务 ID（保证任务可以被唯一标识）、明确状态（pending / running / done / failed）、固定结果落点（任务输出有固定的存储位置）、认领机制（防止重复执行）、失败后的重试或人工接管入口。

这五件事不一定复杂，但缺任何一项，长任务都会很快失序。

## 8.7 心智模型验证题

如果去掉高水位文件，只用当前最大任务 ID + 1 作为新任务 ID，会发生什么？

答案：当任务被删除后，当前最大任务 ID 可能减小。比如，任务 1、2、3 存在，删除任务 3，当前最大 ID 变成 2，新任务被分配 ID 3。但历史日志里的 ID 3 指向的是旧任务，新任务和旧任务的日志会混淆。高水位文件保证 ID 只增不减，避免了这个问题。

## 8.8 本章小结

任务系统的核心价值是把任务状态从进程内存提升到文件系统，让长任务在进程重启后能够恢复，让并发 agent 不重复认领。任务状态写入文件系统（[`tasks.ts#L76`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/utils/tasks.ts#L76)），高水位文件防止 ID 复用（[`tasks.ts#L92`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/utils/tasks.ts#L92)），原子认领防止并发竞态（[`tasks.ts#L621`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/utils/tasks.ts#L621)）——这三个设计共同保证了任务系统在真实场景下的可靠性。

多 Agent 场景下还有三个值得关注的设计：Fork Sub-Agent 通过保证前缀字节完全一致来共享 Prompt Cache，让并行克隆的 API 成本远低于线性增长；DreamTask 把记忆整理放到后台空闲时执行，四阶段（orient→gather→consolidate→prune）对外透明；Coordinator 模式要求协调者对研究结果做"理解合成"再下达指令，而不是原封不动转发——这个原则适用于任何多层级的任务分派系统。
