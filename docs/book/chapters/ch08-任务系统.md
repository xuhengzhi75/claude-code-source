# 第8章 任务系统与长任务：为什么真正能干活的 Agent 不能只有对话循环

> **核心主张：任务系统解决的第一性问题是"状态持久性"——把任务状态从进程内存提升到文件系统，让进程重启后任务不丢失，让多个 agent 并发时不重复认领。这个提升不是功能增强，而是让长任务在真实环境里能够可靠运行的前提。**

## 8.1 第一性问题：对话循环为什么不足以承载长任务

单次 query loop 解决的是"这一轮怎么继续"，但长任务还会引入五类问题：

第一，会话结束后任务是否仍然存在。对话循环的状态在进程退出时消失，如果任务需要跨会话持续，对话循环无法支持。

第二，执行是否要转到后台。用户关闭终端后，任务应该继续运行，而不是随进程退出。

第三，任务状态是否独立保存。如果任务状态只在内存里，进程崩溃后状态丢失，任务需要从头开始。

第四，中间结果是否有固定落点。长任务的中间结果需要持久化，否则崩溃后无法恢复到中间状态。

第五，并发执行时如何避免重复认领。多个 agent 并发时，同一个任务可能被多个 agent 同时认领，导致重复执行。

这五类问题都需要任务状态独立于进程内存存在。

## 8.2 任务对象化：状态从内存提升到文件系统

`src/utils/tasks.ts#L69` 定义了 `TASK_STATUSES`：`['pending', 'in_progress', 'completed']`。任务状态不是内存里的变量，而是写入文件系统的 JSON 对象（[`tasks.ts#L76`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/utils/tasks.ts#L76) 的 `TaskSchema`）。

这意味着进程重启后，任务状态不会丢失。这是任务系统和普通变量管理的根本区别：普通变量在进程退出时消失，文件系统里的任务状态在进程退出后仍然存在。

**隐含假设：** 这个设计假设文件系统是可靠的持久化介质。在单机环境下，这个假设成立。在分布式环境下（多台机器共享任务），文件系统不再是可靠的共享存储，需要换成数据库或分布式存储。

**替代方案：** 另一种做法是用数据库（SQLite、PostgreSQL）存储任务状态。数据库的优势是支持事务、查询更灵活、支持分布式。劣势是引入了额外的依赖，部署更复杂。文件系统的选择是在简单性和功能性之间的权衡：对单机场景足够，对分布式场景不够。

## 8.3 高水位文件：防止 ID 复用

`src/utils/tasks.ts#L92` 的 `HIGH_WATER_MARK_FILE`（`.highwatermark`）记录已分配过的最大任务 ID。

**为什么需要这个文件？** 如果任务 ID 在删除后被重新使用，新任务可能和历史记录混淆。比如，任务 ID 5 完成后被删除，新任务被分配 ID 5，但历史日志里的 ID 5 指向的是旧任务。这会导致状态追踪错乱。

高水位文件保证任务 ID 单调递增，即使旧任务被删除，新任务也不会复用旧 ID。

**技术债：** 高水位文件是一个单点写入，如果多个进程同时分配任务 ID，可能产生竞态。当前实现依赖文件锁来保证原子性，但文件锁在某些文件系统（NFS）上不可靠。

## 8.4 原子认领：防止 TOCTOU 竞态

`src/utils/tasks.ts#L621` 的 `claimTaskWithBusyCheck()` 把检查 agent 是否空闲和认领任务这两步放在同一把 list-level lock 内完成。

注释里明确写了：这是为了防止 TOCTOU（time-of-check to time-of-use）竞态——如果检查和认领之间有间隙，两个 agent 可能同时通过检查，然后同时认领同一个任务。

**TOCTOU 竞态的具体场景：** Agent A 检查任务 T 是否空闲，发现空闲。Agent B 也检查任务 T，发现空闲。Agent A 认领任务 T。Agent B 也认领任务 T。结果是两个 agent 同时执行同一个任务。

把检查和认领放在同一把锁内，保证了这两步是原子的：一旦 Agent A 开始检查，Agent B 必须等待，直到 Agent A 完成认领或放弃。

**可迁移原则：** 任何"检查-然后-操作"的模式都有 TOCTOU 竞态风险。解决方案是把检查和操作放在同一个原子操作里，而不是分两步执行。

## 8.5 任务系统和主循环的接缝

任务系统并不只存在于 `tasks.ts` 里。`src/query.ts#L1584` 里有 `task-notification` 相关处理，`query.ts#L1694` 里有 `BG_SESSIONS` 下的 task summary 生成。

这说明任务系统不是完全绕开运行时主循环独立存在，而是和主循环在若干关键位置接起来。主循环在执行过程中，可以感知任务系统的状态变化，也可以向任务系统报告执行进度。

**技术债：** 任务系统和主循环的接缝是隐式的——需要读代码才能找到所有接缝点。如果未来需要修改任务系统的接口，需要找到所有接缝点并逐一修改。

## 8.6 终端焦点状态：权限信号的演进方向

当前任务系统的权限模型是静态的——任务在创建时确定权限范围，执行过程中不变。但 KAIROS（Claude Code 内部的自主代理研究项目）探索了一个更动态的方向：把终端的焦点状态作为权限信号。

具体来说，当用户的终端窗口处于前台（用户正在看着屏幕）时，Agent 可以执行更高风险的操作；当终端处于后台（用户不在场）时，Agent 应该降低操作风险等级，或者暂停等待用户确认。这个设计的直觉是：用户在场时，可以立即发现和纠正错误；用户不在场时，错误可能在被发现之前已经造成不可逆的影响。

这个方向对当前任务系统的影响是：任务的权限边界不再是静态配置，而是随运行时上下文动态调整。这需要任务系统能够感知外部状态（终端焦点），并在执行过程中动态修改权限约束。

**当前任务系统的局限：** 现有的 `TaskSchema`（[`tasks.ts#L76`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/utils/tasks.ts#L76)）没有动态权限字段，权限在任务创建时固定。如果要支持 KAIROS 式的动态权限，需要在任务状态里加入"当前有效权限"字段，并在每次执行操作前重新评估。

**隐含假设：** 终端焦点状态作为权限信号，假设"用户在场"等价于"用户在监督"。但用户可能开着终端去做别的事，或者终端在前台但用户没有在看。这个假设在大多数情况下成立，但不是严格的安全保证。

**可迁移原则：** 权限不一定是静态的。在需要人机协作的场景里，把运行时上下文（用户是否在场、任务风险等级、当前系统状态）纳入权限决策，可以在自动化效率和安全控制之间找到更好的平衡点。

## 8.7 最小任务系统的必要条件

如果要做一个缩小版业务 Agent，最小任务系统至少需要：独立任务 ID（保证任务可以被唯一标识）、明确状态（pending / running / done / failed）、固定结果落点（任务输出有固定的存储位置）、认领机制（防止重复执行）、失败后的重试或人工接管入口。

这五件事不一定复杂，但缺任何一项，长任务都会很快失序。

## 8.8 心智模型验证题

如果去掉高水位文件，只用当前最大任务 ID + 1 作为新任务 ID，会发生什么？

答案：当任务被删除后，当前最大任务 ID 可能减小。比如，任务 1、2、3 存在，删除任务 3，当前最大 ID 变成 2，新任务被分配 ID 3。但历史日志里的 ID 3 指向的是旧任务，新任务和旧任务的日志会混淆。高水位文件保证 ID 只增不减，避免了这个问题。

## 8.9 本章小结

任务系统的核心价值是把任务状态从进程内存提升到文件系统，让长任务在进程重启后能够恢复，让并发 agent 不重复认领。任务状态写入文件系统（[`tasks.ts#L76`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/utils/tasks.ts#L76)），高水位文件防止 ID 复用（[`tasks.ts#L92`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/utils/tasks.ts#L92)），原子认领防止并发竞态（[`tasks.ts#L621`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/utils/tasks.ts#L621)）——这三个设计共同保证了任务系统在真实场景下的可靠性。
