# 第8章 任务系统与长任务：为什么真正能干活的 Agent 不能只有对话循环

> 核心主张：任务系统的价值不是让界面上多了任务概念，而是把"任务状态"从对话循环的局部变量提升成独立的持久化对象——这个提升让进程重启后任务不丢失，让多个 agent 并发时不重复认领。

只靠当前对话里的 query loop，Claude Code 可以完成多轮推进，但还不足以承载后台任务、长时任务和并发任务。要让系统真正长期干活，任务必须被提升成独立运行时对象。

这里的关键不是有没有任务列表，而是任务是否有自己的状态、归属、输出落点和恢复路径。只要任务持续时间拉长，这些问题就会从可选项变成必选项。

## 8.1 为什么对话循环不足以承载长任务

单次 query loop 解决的是这一轮怎么继续，但长任务还会引入另外 5 类问题：会话结束后任务是否仍然存在、执行是否要转到后台、任务状态是否独立保存、中间结果是否有固定落点、并发执行时如何避免重复认领。

继续多问几轮不是长任务结构，任务对象化才是。

## 8.2 从源码能看到什么叫任务对象化

`src/utils/tasks.ts#L69` 定义了 `TASK_STATUSES`：`['pending', 'in_progress', 'completed']`。任务状态不是内存里的变量，而是写入文件系统的 JSON 对象（[`tasks.ts#L76`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/utils/tasks.ts#L76) 的 `TaskSchema`）。

这意味着进程重启后，任务状态不会丢失。这是任务系统和普通变量管理的根本区别：普通变量在进程退出时消失，文件系统里的任务状态在进程退出后仍然存在。

`src/utils/tasks.ts#L92` 的 `HIGH_WATER_MARK_FILE`（`.highwatermark`）记录已分配过的最大任务 ID。这防止了任务 ID 在删除后被重新使用——如果 ID 被复用，新任务可能和历史记录混淆，导致状态追踪错乱。

## 8.3 任务系统和主循环其实有接缝

任务系统并不只存在于 `tasks.ts` 里。`src/query.ts#L1584` 里有 `task-notification` 相关处理，`query.ts#L1694` 里有 `BG_SESSIONS` 下的 task summary 生成。

这说明任务系统不是完全绕开运行时主循环独立存在，而是和主循环在若干关键位置接起来。主循环在执行过程中，可以感知任务系统的状态变化，也可以向任务系统报告执行进度。

## 8.4 任务系统真正解决的是什么

它真正解决的不是列出任务，而是一致性问题：同一任务会不会被重复执行，任务认领后会不会被别人误碰，失败后状态会不会漂移，中间输出能不能追踪，最终结果归属是否清楚。

`src/utils/tasks.ts#L621` 的 `claimTaskWithBusyCheck()` 把检查 agent 是否空闲和认领任务这两步放在同一把 list-level lock 内完成。注释里明确写了：这是为了防止 TOCTOU（time-of-check to time-of-use）竞态——如果检查和认领之间有间隙，两个 agent 可能同时通过检查，然后同时认领同一个任务。

这些问题一旦处理不好，系统就算能跑很多轮，也很难稳定承担真实工作。

## 8.5 对自己做 Agent 最低限度要做到什么

如果你要做一个缩小版业务 Agent，最小任务系统至少要有：独立任务 ID、明确状态（例如 pending / running / done / failed）、固定结果落点、认领机制、失败后的重试或人工接管入口。

这 5 件事不一定复杂，但缺任何一项，长任务都会很快失序。

## 8.6 本章小结

任务系统之所以重要，不是因为它让界面上多了任务概念，而是因为它把长时执行、后台推进、状态治理和恢复路径从对话附属物提升成了独立运行时结构。任务状态写入文件系统（[`tasks.ts#L76`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/utils/tasks.ts#L76)），高水位文件防止 ID 复用（[`tasks.ts#L92`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/utils/tasks.ts#L92)），原子认领防止并发竞态（[`tasks.ts#L621`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/utils/tasks.ts#L621)）——这三个设计共同保证了任务系统在真实场景下的可靠性。

到这里为止，Claude Code 的主干已经清楚了：入口分流、能力装配、会话编排、运行时循环、任务系统。后面再看恢复、护城河和缩小版实现，就会更容易知道哪些能力是可删的，哪些是不能删的。
