# 第11章 为什么很多 Agent 看起来像但干不了真活

**本章核心主张**：演示 Agent 和能真正干活的 Agent 之间的距离，不是功能多少的距离，是对"出错"这件事的处理密度。前者假设一切顺利，后者要求每个出错路径都有明确的出口。

---

演示场景和真实场景的差距，不是功能多少，而是"边界情况处理密度"。演示任务通常边界清晰、输入规范、不需要中断恢复、失败了可以重来。真实工作场景则相反：任务边界模糊、输入包含历史遗留的奇怪写法、进行到一半可能被打断。

很多 Agent 的设计隐含地假设了"任务会正常完成"。这个假设在演示场景下成立，在真实场景下不成立。

## 11.1 三种中止来源，为什么不能用 instanceof

`src/utils/errors.ts` 的 `isAbortError()` 函数只有几行，但隐藏了一段演进历史：

```typescript
// We can't use instanceof for the SDK class because minified builds will
// mangle class names, breaking instanceof checks.
export function isAbortError(error: unknown): boolean {
  return (
    error instanceof AbortError ||
    error instanceof DOMException ||
    hasExactErrorMessage(error, 'Request was aborted.')
  )
}
```

三种来源不是一开始就分开的，是随着依赖栈演进才逐渐分出来的：最初可能只有自定义 `AbortError`；引入 `AbortController` / `AbortSignal`（Web 标准）后，产生了 `DOMException`；Anthropic SDK 自己也有 `APIUserAbortError`，但 minified 构建会把 class name 混淆，导致 `instanceof` 失效，最终只能用字符串匹配 `'Request was aborted.'`。

这个演进路径说明：不同类型的错误对应不同的处理策略，这个判断不是设计决策，是现实约束。用户主动中止不需要重试，网络超时需要重试，把它们混在一起处理是设计问题，而不是实现问题。

**隐含假设**：这个设计假设"来源不同的中止，语义相同"——用户点 Ctrl+C、AbortController.abort()、SDK 内部超时，都意味着"不要继续这个请求"。如果你的系统需要区分"用户主动中止"和"系统强制中止"的后续行为（比如前者需要清理草稿，后者不需要），这里需要重新分层。

## 11.2 一个 flag 背后烧掉的数千次 API 调用

`src/query.ts` 的 `State` 对象里有一个字段：`hasAttemptedReactiveCompact: boolean`。它存在的原因，代码注释在 L1306-L1310 写得很直接：

> "Resetting to false here caused an infinite loop: compact → still too long → error → stop hook blocking → compact → … burning thousands of API calls."

这不是随意加的 flag，是修复了一个真实的无限循环 bug。

这个 flag 在整个 `queryLoop()` 里出现了 7 次，每次出现都防着一个具体的失效场景：

| 行号 | 操作 | 防护的场景 |
|------|------|-----------|
| L282 | 初始化为 `false` | 每次新 query 重置状态 |
| L1094–L1095 | 注释说明 | 防止 post-compact turn 再次 media-error 导致螺旋 |
| L1135 | 传入 `tryReactiveCompact()` | 告知是否已尝试过压缩 |
| L1171 | 设为 `true` | compact 成功后标记，防止再次触发 |
| **L1311** | **保持原值** | stop hook blocking 后继续，**不重置** |
| L1347 | 重置为 `false` | token budget continuation，新的预算周期 |
| L1736 | 重置为 `false` | 正常进入下一 turn |

L1311 是最反直觉的。stop hook blocking 之后 `continue`，`hasAttemptedReactiveCompact` 保持原值而不重置——注释解释了原因：如果 compact 已经跑过但上下文仍然过长，重置这个 flag 会让 compact 再次触发，然后 stop hook 再次阻断，形成死循环，"烧掉数千次 API 调用"。

**这是被动管理和主动管理之间的真实代价差距**：被动等 API 报 `context window exceeded` 再处理，当前轮次的工作已经失败，需要从上一个检查点恢复。主动压缩需要知道当前 token 用量和压缩触发阈值，维护更多状态——但主动管理的失败路径是可控的，被动管理的失败路径会连锁。

## 11.3 State 对象的膨胀轨迹

`src/query.ts` L206–L219 的 `State` 类型定义是这样的：

```typescript
type State = {
  messages: Message[]
  toolUseContext: ToolUseContext
  autoCompactTracking: AutoCompactTrackingState | undefined
  maxOutputTokensRecoveryCount: number
  hasAttemptedReactiveCompact: boolean    // 补丁：防无限循环
  maxOutputTokensOverride: number | undefined
  pendingToolUseSummary: Promise<...> | undefined
  stopHookActive: boolean | undefined     // 补丁：stop hook 功能加入时追加
  turnCount: number
  transition: Continue | undefined        // 补丁：调试辅助，测试断言用
}
```

L203–L205 的注释写了："Keep this shape focused on runtime continuity concerns only"——这个"Keep focused"的告诫本身就在说：这个对象已经有膨胀的趋势，需要刻意约束。

至少三个字段是明显的后期补丁：`hasAttemptedReactiveCompact`（修复无限循环）、`stopHookActive`（stop hook 功能加入时追加）、`transition`（注释说"Lets tests assert recovery paths fired without inspecting message contents"，是调试辅助）。

集中状态管理的价值在于：任何时刻都能知道系统处于哪个状态，以及从这个状态能做什么操作。但这个价值有一个前提：`State` 对象需要保持聚焦。随着功能迭代，所有"循环间需要保持的变量"都往 `State` 里放，是压力最小的路径——也是让集中管理失去意义的路径。

**技术债**：当前 `State` 有 10 个字段，3 个是后期补丁。如果不定期清理，`State` 会变成一个"什么都往里放"的垃圾桶。注释里的告诫是清醒的，但告诫不等于约束。

## 11.4 中断类型区分：一种比另一种更难处理

`src/utils/conversationRecovery.ts` L139–L227 定义了 `TurnInterruptionState`，内部实际是三态：

- `none`：turn 正常完成（最后消息是 assistant）
- `interrupted_prompt`：用户发了消息但 assistant 还没响应（最后消息是普通 user 消息）
- `interrupted_turn`：turn 进行中被打断（最后消息是 tool_result 或 attachment）

对外只暴露两态：`interrupted_turn` 会被自动转换为 `interrupted_prompt`——在最后一条消息后追加一条 `"Continue from where you left off."` 的 meta 消息，然后按 `interrupted_prompt` 处理。

这个"自动转换"的设计意图是**语义修复而不是 transcript 回放**：先从磁盘恢复事实（发生了什么），再把尾部重写成 API 合法的、可续跑的对话形状（模型能从哪里继续）。

有一个边界情况值得注意（L313–L322）：Brief mode（`#20467`）下，completed turn 合法地以 tool_result 结尾，因为 brief mode 会丢弃 trailing assistant text block。如果不特殊处理，每次 brief mode 会话恢复都会被误判为 `interrupted_turn`，注入一条幽灵 "Continue from where you left off."。这个特殊处理是后来补上的，在代码里是一个 issue 编号标注，是典型的"实际踩坑后修复"。

## 11.5 权限边界：双重白名单而不是宽松过滤

`src/services/SessionMemory/sessionMemory.ts` L466–L490 的 `createMemoryFileCanUseTool()` 实现了细粒度权限：

```typescript
export function createMemoryFileCanUseTool(memoryPath: string): CanUseToolFn {
  return async (tool: Tool, input: unknown) => {
    if (
      tool.name === FILE_EDIT_TOOL_NAME &&
      typeof input === 'object' && input !== null &&
      'file_path' in input
    ) {
      const filePath = input.file_path
      if (typeof filePath === 'string' && filePath === memoryPath) {  // 严格相等
        return { behavior: 'allow' as const, updatedInput: input }
      }
    }
    return { behavior: 'deny' as const, message: `only ${FILE_EDIT_TOOL_NAME} on ${memoryPath} is allowed` }
  }
}
```

两重限制：工具名必须是 `FILE_EDIT_TOOL_NAME`（即 `str_replace_based_edit_tool`），且 `file_path` 必须**严格等于**（`===`）传入的 `memoryPath`。

注意路径检查用的是严格相等而不是前缀匹配——这意味着 `../` 路径遍历攻击天然无效。不是专门防的，是严格相等的副产品。

**可迁移原则**：副作用操作应在独立的最小权限上下文中执行。适用条件：副作用操作的影响范围可以被明确界定。反例：如果副作用操作需要读取主上下文的大量状态，隔离的成本可能超过收益。

## 11.6 心智模型验证题

一个 Agent 系统，在每次压缩上下文后，会把 `hasAttemptedReactiveCompact` 重置为 `false`，以便下次还能触发压缩。这个设计有什么问题？

答案：如果压缩后上下文仍然过长（比如 tool result 本身很大），下次 turn 会再次触发压缩，压缩后仍然过长，再次触发，形成死循环。Claude Code 的修复方式是：压缩后标记为 `true`，在 stop hook blocking 路径下**不重置**这个标记，即使 turn 继续执行。只有在 token budget continuation（新的预算周期）和正常进入下一 turn 时，才重置。这个细节区分了"同一个上下文需要再次压缩"（不允许）和"新的 turn 可以压缩"（允许）。

## 11.7 本章小结

很多 Agent 在演示里好用，在真实工作中暴露问题，核心原因是三个：

**错误来源不区分**：`isAbortError()` 的三种来源统一判断（`errors.ts`），不是追求优雅，是 minified 构建破坏 `instanceof` 的现实约束推着走的结果。

**失败路径没有闭环**：`hasAttemptedReactiveCompact` 这一个 flag 背后是一个烧掉数千次 API 调用的真实 bug（`query.ts#L1306`）。每个防护机制的存在都有来历，没有来历的地方就是没有被真实场景测试过。

**状态管理在功能迭代里悄悄失控**：`State` 对象从 7 个字段长到 10 个，3 个是补丁，注释里有告诫但没有约束（`query.ts#L203`）。

演示和真实之间的距离，不是功能的距离，是对"出错"这件事的态度距离。
