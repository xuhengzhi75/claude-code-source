# 第11章 为什么很多 Agent 看起来像但干不了真活

**核心主张**：演示 Agent 和能干活的 Agent 之间的差距，是对出错路径的处理密度。功能数量决定演示效果；错误处理密度决定系统能走多远。

---

厨房里摆一台煤气灶，能点火、能调温、能关闭，演示给人看完全没问题。真正能用来开餐馆的灶，还需要回答：气管堵了怎么处理、打火失败了是什么原因、有人在火上时断气了怎么恢复。两台灶的功能清单一样，但第二台多了一整套故障应对逻辑。

三步工具调用的任务，第二步超时，演示 Agent 抛异常停掉；能干活的 Agent 需要区分三种来源：用户主动中止、网络断开、还是上下文撑不住了。来源不同，后续处理不同。这个区分需要代码，代码需要维护，维护过程中会出新 bug，修 bug 又加新状态。这条链条的长度，就是演示和生产之间的距离。

## 11.1 中止信号的三种来源无法用类型系统区分

`src/utils/errors.ts` 的 `isAbortError()` 只有几行，但这几行字的背后是一段演进历史：

```typescript
// We can't use instanceof for the SDK class because minified builds will
// mangle class names, breaking instanceof checks.
export function isAbortError(error: unknown): boolean {
  return (
    error instanceof AbortError ||
    error instanceof DOMException ||
    hasExactErrorMessage(error, 'Request was aborted.')
  )
}
```

三条判断路径对应三个来源。最初可能只有自定义 `AbortError`；引入 `AbortController` / `AbortSignal`（Web 标准的取消信号机制）之后产生了 `DOMException`；Anthropic SDK 自己也会抛中止错误，但 minified 构建（生产构建时压缩混淆代码）会把类名改掉，`instanceof` 判断因此失效，最后只能靠字符串 `'Request was aborted.'` 兜底。

字符串匹配是降级路径，不是偏好。`instanceof` 是类型安全的；字符串匹配在 SDK 改错误消息时静默失败。这里的技术债是：一旦 SDK 改了那条消息，第三路径失效，且没有编译期报错。

这个函数把用户主动中止（Ctrl+C）和 SDK 内部超时统一判断为"不需要继续这个请求"。如果系统需要区分这两种中止的后续行为，比如用户主动中止需要清理草稿而系统超时不需要，当前实现需要重新分层。

## 11.2 一个布尔 flag 保护的是整条压缩循环

`src/query.ts` 的 `State` 对象里有一个字段 `hasAttemptedReactiveCompact: boolean`（reactive compact，响应式上下文压缩：当上下文超出窗口时主动触发压缩）。代码注释在 L1306–L1310 写得直接：

> "Resetting to false here caused an infinite loop: compact → still too long → error → stop hook blocking → compact → … burning thousands of API calls."

这条注释记录的是真实发生过的事，不是假设场景。

这个 flag 在 `queryLoop()` 里出现 7 次，每次都在防一个具体的失效场景：

| 行号 | 操作 | 防护的失效 |
|------|------|-----------|
| L282 | 初始化为 `false` | 每次新 query 重置状态 |
| L1094–L1095 | 注释 | 防止 post-compact turn 再次 media-error 导致螺旋 |
| L1135 | 传入 `tryReactiveCompact()` | 告知是否已尝试压缩 |
| L1171 | 设为 `true` | compact 成功后标记，防止再次触发 |
| L1311 | 保持原值 | stop hook blocking 后继续，不重置 |
| L1347 | 重置为 `false` | token budget continuation，新预算周期 |
| L1736 | 重置为 `false` | 正常进入下一 turn |

L1311 最反直觉。stop hook（stop hook，会话停止钩子：在 AI 准备结束 turn 时触发的扩展点，可阻断或修改停止行为）阻断之后 `continue`，`hasAttemptedReactiveCompact` 保持原值而不重置。原因是：如果 compact 已经跑过但上下文仍然过长，重置这个 flag 会让 compact 再次触发，stop hook 再次阻断，进入死循环。"数千次 API 调用"对应真实的资金消耗，不是修辞。

被动等 API 报 `context window exceeded` 再处理，当前轮次的工作已经失败，需要从检查点恢复。主动压缩绕开了这个失败，但引入了需要维护的状态，而状态维护本身有出错机会。两种策略都有代价，区别在于主动管理的失败路径是可控的，被动管理的失败路径是连锁的。

## 11.3 State 对象的膨胀轨迹是错误处理历史的可见记录

`src/query.ts` L203–L205 的注释写了："Keep this shape focused on runtime continuity concerns only"（保持这个对象只关注运行时连续性）。这条告诫说明对象有膨胀趋势，需要刻意约束。告诫本身不等于约束。

L206–L219 的 `State` 当前有 10 个字段：

```typescript
type State = {
  messages: Message[]
  toolUseContext: ToolUseContext
  autoCompactTracking: AutoCompactTrackingState | undefined
  maxOutputTokensRecoveryCount: number
  hasAttemptedReactiveCompact: boolean    // 补丁：防无限循环
  maxOutputTokensOverride: number | undefined
  pendingToolUseSummary: Promise<...> | undefined
  stopHookActive: boolean | undefined     // 补丁：stop hook 功能加入时追加
  turnCount: number
  transition: Continue | undefined        // 补丁：调试辅助，测试断言用
}
```

至少三个字段是后期补丁：`hasAttemptedReactiveCompact` 修复了真实的无限循环；`stopHookActive` 在 stop hook 功能加入时追加；`transition` 的注释说"Lets tests assert recovery paths fired without inspecting message contents"（让测试断言恢复路径已触发），是调试辅助字段进入了生产对象。

集中状态管理（把循环间需要保持的变量放在一个对象里）的价值在于：任何时刻都能知道系统处于哪个状态，以及从这个状态能做什么操作。这个价值有一个前提：`State` 对象保持聚焦。所有"循环间需要保持的变量"都往 `State` 里放，是压力最小的路径。当调试辅助字段进入生产状态对象，状态对象已经不只是运行时连续性的载体。10 个字段、3 个补丁，这条演进轨迹记录了错误处理加固的历史。

## 11.4 会话中断恢复先修事实再修语义

`src/utils/conversationRecovery.ts` L139–L227 定义了 `TurnInterruptionState`，内部有三种状态：

- `none`：turn（一轮对话：从用户发消息到 AI 回复完毕算一个 turn）正常完成，最后消息是 assistant
- `interrupted_prompt`：用户发了消息但 assistant 还没响应，最后消息是普通 user 消息
- `interrupted_turn`：turn 进行中被打断，最后消息是 tool_result 或 attachment

对外只暴露两态：`interrupted_turn` 被自动转换为 `interrupted_prompt`，在最后一条消息后追加一条 `"Continue from where you left off."` 的 meta 消息，然后按 `interrupted_prompt` 处理。

这个设计分两步走：先从磁盘恢复事实（发生了什么），再把尾部重写成 API 合法的、可续跑的对话形状（模型能从哪里继续）。事实恢复和语义修复分离，每一步的职责是明确的。

边界情况在 L313–L322：Brief mode（`#20467`，Brief mode：一种会话模式，AI 回复完毕后会丢弃尾部的 assistant text block）下，已完成的 turn 合法地以 tool_result 结尾，因为 trailing assistant text 被丢掉了。如果不特殊处理，每次 Brief mode 会话恢复都会被判定为 `interrupted_turn`，注入一条幽灵 `"Continue from where you left off."`。这个特殊处理是踩坑后补上的，不是前期设计时预见到的。

## 11.5 权限隔离：双重白名单让副作用操作的影响范围可以被精确界定

`src/services/SessionMemory/sessionMemory.ts` L466–L490 的 `createMemoryFileCanUseTool()` 实现了细粒度权限控制：

```typescript
export function createMemoryFileCanUseTool(memoryPath: string): CanUseToolFn {
  return async (tool: Tool, input: unknown) => {
    if (
      tool.name === FILE_EDIT_TOOL_NAME &&
      typeof input === 'object' && input !== null &&
      'file_path' in input
    ) {
      const filePath = input.file_path
      if (typeof filePath === 'string' && filePath === memoryPath) {
        return { behavior: 'allow' as const, updatedInput: input }
      }
    }
    return { behavior: 'deny' as const, message: `only ${FILE_EDIT_TOOL_NAME} on ${memoryPath} is allowed` }
  }
}
```

两重限制：工具名必须是 `FILE_EDIT_TOOL_NAME`（即 `str_replace_based_edit_tool`），且 `file_path` 必须严格等于传入的 `memoryPath`。路径检查用严格相等而不是前缀匹配，`../` 路径遍历天然无效，不是专门防的，是严格相等的副产品。

**可迁移原则**：副作用操作应在独立的最小权限上下文中执行。适用条件：副作用操作的影响范围可以被明确界定，且这个界定不依赖主上下文的大量状态。反例：如果副作用操作需要读取主上下文的大量共享状态，隔离的接口成本可能超过它带来的安全收益。原则是工具，不是教条。

## 心智模型验证题

一个 Agent 系统，在每次压缩上下文后，会把 `hasAttemptedReactiveCompact` 重置为 `false`，以便下次还能触发压缩。这个设计有什么问题？

压缩后上下文仍然过长（比如 tool result 本身很大），下次 turn 会再次触发压缩，压缩后仍然过长，再次触发，形成死循环。Claude Code 的修复方式是：压缩后标记为 `true`，在 stop hook blocking 路径下不重置，即使 turn 继续执行。只有在 token budget continuation（新的预算周期）和正常进入下一 turn 时才重置。这个细节区分了两种语义：同一个上下文需要再次压缩（不允许）和新的 turn 可以压缩（允许）。
