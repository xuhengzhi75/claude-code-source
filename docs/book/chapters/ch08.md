# 第8章 任务系统与长任务：为什么真正能干活的 Agent 不能只有对话循环

结论先说：只靠当前对话里的 query loop，Claude Code 可以完成多轮推进，但还不足以承载后台任务、长时任务和并发任务。要让系统真正长期干活，任务必须被提升成独立运行时对象。

这里的关键不是“有没有任务列表”，而是任务是否有自己的状态、归属、输出落点和恢复路径。只要任务持续时间拉长，这些问题就会从可选项变成必选项。

## 8.1 为什么对话循环不足以承载长任务

单次 query loop 解决的是“这一轮怎么继续”，但长任务还会引入另外 5 类问题：

- 会话结束后任务是否仍然存在
- 执行是否要转到后台
- 任务状态是否独立保存
- 中间结果是否有固定落点
- 并发执行时如何避免重复认领

这说明“继续多问几轮”不是长任务结构，任务对象化才是。

## 8.2 从源码能看到什么叫任务对象化

从仓库结构可以直接看到任务相关模块：

- `src/Task.ts`
- `src/tasks.ts`
- `src/tasks/*`

这些文件说明任务在 Claude Code 里不是展示层小组件，而是独立建模的一层。围绕它们，已有材料也反复出现状态生命周期、认领、输出、终态和恢复这些问题。

## 8.3 任务系统和主循环其实有接缝

任务系统并不只存在于 `Task` 模块里。`src/query.ts` 中也能看到任务接缝，例如：

- `task-notification` 相关处理
- `BG_SESSIONS` 下的 task summary 生成
- 主循环里对任务相关消息和摘要的消费

这说明任务系统不是完全绕开运行时主循环独立存在，而是和主循环在若干关键位置接起来。

## 8.4 任务系统真正解决的是什么

它真正解决的不是“列出任务”，而是一致性问题：

- 同一任务会不会被重复执行
- 任务认领后会不会被别人误碰
- 失败后状态会不会漂移
- 中间输出能不能追踪
- 最终结果归属是否清楚

这些问题一旦处理不好，系统就算能跑很多轮，也很难稳定承担真实工作。

## 8.5 对自己做 Agent 最低限度要做到什么

如果你要做一个缩小版业务 Agent，最小任务系统至少要有：

1. **独立任务 ID**
2. **明确状态**，例如 pending / running / done / failed
3. **固定结果落点**
4. **认领机制**
5. **失败后的重试或人工接管入口**

这 5 件事不一定复杂，但缺任何一项，长任务都会很快失序。

## 8.6 本章小结

任务系统之所以重要，不是因为它让界面上多了任务概念，而是因为它把长时执行、后台推进、状态治理和恢复路径从“对话附属物”提升成了独立运行时结构。

到这里为止，Claude Code 的主干已经清楚了：入口分流、能力装配、会话编排、运行时循环、任务系统。后面再看恢复、护城河和缩小版实现，就会更容易知道哪些能力是可删的，哪些是不能删的。
