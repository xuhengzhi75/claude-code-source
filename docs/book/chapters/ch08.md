# 第8章 任务系统与长任务：为什么真正能干活的 Agent 不能只有对话循环

前一章讲的是 query loop，也就是一轮一轮的运行时引擎。
但如果系统只靠当前会话里的循环硬扛，很多真实工作还是会很快碰到上限。

原因很简单：真实工作不只是“继续多问几轮”，还会涉及更长时间尺度的推进。

例如：

- 任务需要后台持续运行
- 多个任务需要排队或并行
- 某个任务中断后要知道从哪继续
- 执行结果要被稳定写回
- 多个执行体不能互相抢同一件事

这些问题，已经超出单次 query loop 能优雅处理的范围。
所以 Claude Code 还需要任务系统这一层。

## 8.1 为什么长任务必须有单独结构

很多人一开始做 Agent，会把长任务理解成“让模型继续聊更久”。
这通常不够。

因为一旦任务进入真实环境，你就会遇到几个问题：

- 当前会话可能会结束
- 执行可能要转到后台
- 任务状态需要独立保存
- 结果要能被之后的流程接手
- 并发执行时要避免重复认领

这时就会发现，单一对话循环主要解决的是“当前轮怎么推进”，不是“长期任务如何存在”。

任务系统存在的意义，就是把任务从一次对话里提升成可管理对象。

## 8.2 从源码看，任务不是 UI 小组件，而是运行时对象

从仓库结构可以看到：

- `src/Task.ts`
- `src/tasks.ts`
- `src/tasks/*`

这说明任务在 Claude Code 里不是前端展示层的概念，而是独立建模的一层。

前面已有架构笔记提到，这套任务系统会涉及：

- 任务定义
- 状态生命周期
- 注册与认领
- 输出写入
- 终态管理
- 并发一致性控制

也就是说，任务不是“便于用户看一眼的列表”，而是执行流程的一部分。

这是很多玩具 Agent 和成熟 Agent 的差别。
前者把任务当成说明文字，后者把任务当成运行时实体。

## 8.3 为什么任务系统和恢复机制天然绑在一起

只要任务会持续存在，恢复问题就一定会出现。

你至少要回答这些问题：

- 任务做到哪一步了
- 上一次是谁在处理
- 中间产物写到哪里了
- 如果进程断了，下一次怎么接上
- 如果任务已完成，谁来标记终态

因此任务系统和恢复机制通常不能分开看。

在 Claude Code 里，围绕任务还有 `task-recovery-map`、`recovery-and-continuity` 这类说明材料，正说明它不是把任务当一层表面功能，而是把它看作长期连续性的组成部分。

## 8.4 任务系统真正解决的不是“列任务”，而是“一致性”

对普通读者来说，任务系统最容易被误解成 todo list。
其实真正难的地方不是列任务，而是保证任务不会乱。

真实系统里最难的是这些一致性问题：

- 同一任务会不会被重复执行
- 任务被认领后，其他执行体是否还能误碰
- 失败后状态是否正确更新
- 中间输出是否可追踪
- 最终结果是否有清楚归属

如果这些问题不解决，任务系统就只是一层装饰。

Claude Code 之所以值得学，不是因为它也有任务列表，而是因为它把任务和执行、恢复、状态管理放在了一起考虑。

## 8.5 为什么这层会直接影响“能不能干真活”

当任务持续时间变长后，系统的差异会迅速放大。

一个没有任务系统的 Agent，可能在短流程里看起来很聪明。
但一旦进入：

- 长时间运行
- 多步交付
- 后台执行
- 中断恢复
- 多任务并行

它就容易开始失序。

所以从商业价值角度看，任务系统往往是“能演示”和“能工作”的分界线之一。

这也是为什么 Claude Code 的参考价值不只是 prompt 和工具，而在于它把长期执行结构也一起做进了系统。

## 8.6 对自己做业务 Agent 的启发

如果你现在就想做一个缩小版业务 Agent，不需要一开始把任务系统做得很复杂。
但至少要先有一个最小版本。

最低限度建议包含：

1. **任务有独立 ID**
2. **任务有明确状态**，例如 pending / running / done / failed
3. **任务结果有固定落点**
4. **执行者有认领机制**
5. **失败后可重试或可人工接管**

只要这 5 件事成立，你的系统就已经跨过了“只有对话，没有任务”的门槛。

## 8.7 本章小结

这一章的核心结论是：

**真正能长期干活的 Agent，不能只靠当前对话循环，还必须把任务提升成独立、可追踪、可恢复、可并发治理的运行时对象。**

Claude Code 的任务系统价值就在这里。
它不是锦上添花的管理界面，而是长任务执行结构的一部分。

到这里为止，我们已经把 Claude Code 的主干链路走通了：

- 入口分流
- 能力装配
- 会话编排
- 运行时循环
- 任务系统

后面再往下看，就可以开始讨论它为什么形成技术壁垒，以及哪些部分值得我们照着做，哪些暂时不该太早复制。
