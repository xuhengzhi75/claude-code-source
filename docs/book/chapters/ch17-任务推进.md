# 第17章 任务系统解决三类并发失效，而不只是管理状态

调度室黑板上只有一块，所有司机认领任务都在同一块黑板上操作。如果两个司机同时看到"送货到 5 号仓"还没人领，都举手认领，就要出问题。不是黑板不够用，是认领动作没有原子化。

Claude Code 的任务系统处理的是同类问题：在多个 Agent 并发运行时，任务可能被重复认领、可能因 Agent 宕机永久卡死、可能因通知失败导致状态回滚。`claude ps` 实时显示执行进度，只是这套基础设施露出水面的最小接口。

任务系统三个设计决策，每个决策对应一类并发失效。理解这三个决策，就理解了多 Agent 系统的存活底线。

## 17.1 任务认领必须原子化，否则会被重复认领

check-then-act 是并发 Bug 的经典来源。检查"这个任务有没有人认领"和"把我的 ID 写进去"是两步，两步之间存在时间窗口。两个 Agent 同时通过检查，同时认领同一个任务，各自以为任务归自己所有，重复执行。

[`tasks.ts#L621`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/utils/tasks.ts#L621) 的 `claimTaskWithBusyCheck()` 用任务列表级锁（锁整个任务池而不是单个任务文件）把检查和认领原子化：

```typescript
async function claimTaskWithBusyCheck(...) {
  const lockPath = await ensureTaskListLockFile(taskListId)
  release = await lockfile.lock(lockPath, LOCK_OPTIONS)
  // 在同一把锁内：读所有任务、检查 agent 是否繁忙、检查任务是否可认领、认领任务
}
```

这里用的是任务列表级锁，而不是任务级锁。普通的 `updateTask()` 修改单个任务时只锁那一个任务文件，其他任务并发修改不受影响。但认领任务涉及 busy check，需要读取该 Agent 的所有当前任务——这个读取必须和后续写入在同一把锁内，所以要锁整个列表。

`ClaimTaskResult` 的失败理由有五种：`task_not_found`、`already_claimed`、`already_resolved`、`blocked`、`agent_busy`。`agent_busy` 返回 `busyWithTasks` 字段，列出该 Agent 正在处理的任务 ID，让调度方可以判断是等待这个 Agent 还是分配给其他空闲 Agent。

`updateTaskUnsafe` 是内部无锁变体，供已持锁调用方使用（[`tasks.ts#L355-L371`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/utils/tasks.ts#L355)）。`proper-lockfile` 不可重入，持锁后再次尝试加锁会死锁。已经持有列表级锁的调用方需要调用 `updateTaskUnsafe` 而不是 `updateTask`，绕过内部加锁逻辑。数据库事务同一连接可以嵌套，文件锁不行。

协作空间的主键由 [`tasks.ts#L199`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/utils/tasks.ts#L199) 的 `getTaskListId()` 确定，有五级优先级：

```typescript
export function getTaskListId(): string {
  if (process.env.CLAUDE_CODE_TASK_LIST_ID) {
    return process.env.CLAUDE_CODE_TASK_LIST_ID  // 1. 显式环境变量
  }
  const teammateCtx = getTeammateContext()
  if (teammateCtx) {
    return teammateCtx.teamName  // 2. 进程内 teammate 上下文
  }
  return getTeamName() || leaderTeamName || getSessionId()
  // 3. CLAUDE_CODE_TEAM_NAME 环境变量
  // 4. leader 注册的名字
  // 5. session ID
}
```

tmux/iTerm2 模式下，外部 teammate 通过 `CLAUDE_CODE_TEAM_NAME` 知道自己属于哪个 team；进程内 teammate 通过 `teammateCtx.teamName` 知道；leader 调 `setLeaderTeamName()` 注册。三种启动方式都收敛到同一个 taskListId，写入 `~/.claude/tasks/<taskListId>/` 下同一目录。

`setLeaderTeamName()` 的副作用值得注意：leader 创建 team 时不只是改变量，还触发 `notifyTasksUpdated()` 事件——taskListId 变化时 UI 需要重新加载任务列表。

## 17.2 两个数据结构决策防止状态腐化

状态腐化不会触发报错，只会让任务静默卡死。两个字段层面的设计决策专门对抗这种无声失效。

### 双向阻塞图：用空间换时间，删除时级联清理

`blocks` 和 `blockedBy` 是双向阻塞图。[`tasks.ts#L458`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/utils/tasks.ts#L458) 的 `blockTask()` 同时更新两端：A blocks B 时，A 的 `blocks` 加入 B 的 ID，B 的 `blockedBy` 加入 A 的 ID。只维护单向时查询某任务的前置依赖需要全表扫描；双向维护让查询是 O(1)，代价是写入时维护一致性。

`deleteTask()` 删除任务时级联清理两端，把被删任务从所有其他任务的 `blocks` 和 `blockedBy` 里移除。不做级联清理的后果：某任务的 `blockedBy` 里存着一个已删除任务的 ID，系统认为它仍在被阻塞，永远不会被认领，没有任何报错，只是静默卡死。

### 高水位标记防 ID 重用，操作顺序是关键

任务状态存在文件系统里，每个任务是 `<id>.json`。删除任务时文件被删，ID 空出来。如果下一个任务复用了这个 ID，历史记录里会出现混淆。

[`tasks.ts#L92`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/utils/tasks.ts#L92) 的高水位标记（`.highwatermark` 文件）记录曾经分配过的最大 ID。`createTask()` 在分配新 ID 时取"当前文件中最大 ID"和"高水位标记"的较大值，保证新 ID 总是递增。

`deleteTask()` 先把被删任务的 ID 写入高水位标记，再删文件。这个顺序是关键：如果先删文件，然后写高水位标记时进程崩溃，下一次 `createTask()` 可能分配到被删任务的 ID。先写高水位标记，即使删除失败，ID 也不会被重用，宁可 ID 空着不用，不能 ID 重用。

`resetTaskList()` 在 swarm 开始新一轮协作前清空任务池，删除所有任务文件，但先把当前最高 ID 写入高水位标记，保证新一轮 ID 从上一轮结束的地方继续递增。

这是零依赖方案：不用数据库，不用分布式 ID 生成器，一个文本文件解决问题。多机部署时，文件系统方案失效，需要换成数据库序列或分布式 ID 生成器。

### activeForm 字段将 UI 展示协议耦合进 schema（inference）

[`tasks.ts#L76`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/utils/tasks.ts#L76) 的 `TaskSchema` 定义了任务字段：

```typescript
z.object({
  id: z.string(),
  subject: z.string(),
  description: z.string(),
  activeForm: z.string().optional(), // "Running tests" 而不是 "run test suite"
  owner: z.string().optional(),
  status: TaskStatusSchema(),         // 'pending' | 'in_progress' | 'completed'
  blocks: z.array(z.string()),
  blockedBy: z.array(z.string()),
  metadata: z.record(z.string(), z.unknown()).optional(),
})
```

`activeForm` 是进行时动作短语，专门为 UI spinner 设计。当状态是 `in_progress` 时，UI 直接显示这个字段作为进度文案。这个字段存在意味着任务 schema 同时承担了 UI 展示协议：任务创建者提供文案，消费者直接用。推断这是有意的耦合设计，消除了"UI 自己解读任务状态生成文案"的模糊地带。

## 17.3 Agent 宕机后任务必须归还，通知失败不能阻断写入

宕机是多 Agent 系统的常见场景，不是边缘情况。一个 Agent 认领了任务，进程被杀，任务卡在 `in_progress`，没有人完成。整个 swarm 陷入死锁：其他 Agent 看到任务有 owner，不去认领，但 owner 已经死了。

[`tasks.ts#L818`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/utils/tasks.ts#L818) 的 `unassignTeammateTasks()` 处理这种情况：

```typescript
export async function unassignTeammateTasks(
  teamName: string,
  teammateId: string,
  teammateName: string,
  reason: 'terminated' | 'shutdown',
): Promise<UnassignTasksResult>
```

当 teammate 被终止或主动关闭时，调用方触发这个函数。它找出该 Agent 持有的所有未完成任务，把 `owner` 清空，状态重置为 `pending`，生成通知注入给其他 Agent：

```
<teammateName> was terminated. 2 task(s) were unassigned: #3 "Write unit tests", #7 "Update docs".
Use TaskList to check availability and TaskUpdate with owner to reassign them to idle teammates.
```

`reason` 参数区分 `terminated` 和 `shutdown`，生成的通知措辞不同，方便人工排障判断是异常退出还是正常退出。任务归还把悬挂的 `in_progress` 变成可恢复的 `pending`，是 swarm 的存活能力，不是可选优化。

通知机制有两处：进程内用 `notifyTasksUpdated()`，跨 Agent 用 `enqueuePendingNotification()`。

[`task/framework.ts`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/utils/task/framework.ts) 的 `notifyTasksUpdated()` 用 try/catch 包住 `emit()`：

```typescript
export function notifyTasksUpdated(): void {
  try {
    tasksUpdated.emit()
  } catch {
    // Ignore listener errors — task mutations must not fail due to notification issues
  }
}
```

任务持久化是核心操作，UI 刷新是副效应。通知失败导致任务写入回滚，会让系统在 UI 有 bug 时整体不可用。

跨 Agent 的异步通知用 XML 格式（inference: XML 比 JSON 在训练数据里有更多先例，模型解析更稳定）：

```xml
<task-notification>
  <task-id>3</task-id>
  <task-type>background_agent</task-type>
  <status>completed</status>
  <summary>Task "Write unit tests" completed successfully</summary>
</task-notification>
```

这条消息被注入下一次迭代的 tool results，接收方 Agent 在下一轮循环里看到状态变化。

主循环和任务系统有两处接缝。第一处（[`query.ts#L1577`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/query.ts#L1577)）：进程内多个 Agent 共享命令队列，靠 `agentId` 字段区分"这条通知是给我的"，主线程处理 `agentId === undefined`，subagent 处理 `agentId === currentAgentId`。第二处（[`query.ts#L1685`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/query.ts#L1685)）：`BG_SESSIONS` 模式下，主循环每次迭代结束时，只有主线程（`!toolUseContext.agentId`）调用 `taskSummaryModule.maybeGenerateTaskSummary()`，避免多个 Agent 并发写入摘要冲突。

可观测性有两条路径。文件系统路径：每个任务是 `<id>.json`，输出写入 `<id>.output`，[`diskOutput.ts`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/utils/task/diskOutput.ts) 的 `getTaskOutputDelta()` 用字节偏移读增量，不加载整个文件，适合人工排障时直接 `cat`。SDK 事件流路径：[`sdkProgress.ts`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/utils/task/sdkProgress.ts) 的 `emitTaskProgress()` 推送结构化进度，包含已用 token 数、工具调用次数、执行时长，供 CI 系统和监控平台程序化消费。两者并存，互不依赖，服务对象不同。

## 17.4 边界与技术债

文件系统任务队列是单机方案。`~/.claude/tasks/<taskListId>/` 下的文件在同一台机器上的 Agent 间共享，跨机器共享需要网络文件系统或换用数据库。NFS 在高频写入时性能显著下降。

`LOCK_OPTIONS` 里的 `retries` 配置决定并发 Agent 等待上限。注释里说每个 critical section 约需 50-100ms，10 个 Agent 排队最后一个约需 900ms，`retries=30` 给了约 2.6s 总等待时间。并发 Agent 数量超过这个假设时，等不到锁的 Agent 返回 `task_not_found`，表现为任务被漏认领、Agent 空转。

## 17.5 可迁移原则

任务认领必须原子化。check-then-act 在同一把锁内完成，不能分步执行。适用条件：多消费者竞争有限资源的任何系统。反例：单消费者场景，不存在竞态，加锁只增加开销。原则是工具，不是教条。

Agent 宕机后的资源必须有归还机制。持有任务的 Agent 死掉，任务状态必须自动归还为可认领，否则系统会积累越来越多的悬挂状态。适用条件：任何有任务所有权的多 Agent 系统。反例：任务是幂等的、重复执行无害，可以用超时机制而不是显式归还。原则是工具，不是教条。

通知失败不应阻断主操作。持久化是核心，通知是副效应，不能调换优先级。适用条件：所有带异步通知的状态变更。反例：通知本身是事务的一部分（比如消息队列投递失败需要回滚），这时主操作依赖通知成功，不能隔离。原则是工具，不是教条。

## 17.6 本章不覆盖项

任务系统和恢复机制的交互（任务状态在进程重启后如何恢复）在第18章覆盖。多 Agent 协调器（leader/worker 模式、邮箱系统）是独立的 `coordinator/` 实现，本章只覆盖基础任务调度层。
