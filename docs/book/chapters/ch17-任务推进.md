# 第17章 任务推进与运行时结构

`claude ps` 命令能实时显示后台任务的执行进度。这不是魔法，而是任务系统和主循环之间有明确的接缝：主循环在执行过程中定期向任务系统报告进度，任务系统把这些进度写入文件，`claude ps` 读取文件展示。

没有这个接缝，任务就是黑盒。有了接缝，长任务的执行状态才对外可见。

## 17.1 任务的生命周期

[`src/utils/tasks.ts#L76`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/utils/tasks.ts#L76) 的 `TaskSchema` 定义了任务的数据结构：任务有 ID、类型、状态、归属的任务列表。[`tasks.ts#L287`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/utils/tasks.ts#L287) 的 `createTask()` 和 [`tasks.ts#L373`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/utils/tasks.ts#L373) 的 `updateTask()` 管理任务的持久化和状态更新。

任务状态存在文件系统里，不在内存里。这意味着进程重启后，任务状态不会丢失。这是任务系统和普通变量管理的根本区别。

[`src/utils/tasks.ts#L621`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/utils/tasks.ts#L621) 的 `claimTaskWithBusyCheck()` 是认领任务的关键函数。它在同一把锁内完成两个操作：检查当前 agent 是否空闲，以及认领任务。这两步必须原子完成，否则两个 agent 可能同时通过检查，然后同时认领同一个任务。

[`tasks.ts#L92`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/utils/tasks.ts#L92) 的高水位文件（`.highwatermark`）记录已分配过的最大任务 ID。这防止了任务 ID 在删除后被重新使用，避免新任务和历史记录混淆。

## 17.2 主循环的运行时结构

[`src/query.ts#L243`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/query.ts#L243) 的 `queryLoop()` 是任务执行的运行时引擎。它的结构是一个 `while(true)` 循环（[`query.ts#L314`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/query.ts#L314)），每次迭代包含：预处理上下文、调模型、执行工具调用、判断是否继续。

循环的继续条件不是模型说"我还没完成"，而是本轮是否产生了工具调用（[`query.ts#L561`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/query.ts#L561)）。有工具调用就继续，没有就进入收尾。这个判断比依赖模型的文字输出更稳定。

循环内部维护一个 `State` 对象（[`query.ts#L282`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/query.ts#L282)），记录跨迭代的可变状态：当前消息列表、工具调用上下文、输出 token 恢复计数、是否已尝试压缩。这个状态对象让恢复路径可以"整体替换状态再继续"，而不是在局部变量里打补丁。

## 17.3 任务系统和主循环的接缝

任务系统和主循环不是完全独立的。`src/query.ts#L1584` 里有 `task-notification` 相关处理：subagent 只处理发给自己的 task-notification，不处理其他 agent 的。`query.ts#L1694` 在 `BG_SESSIONS` 模式下，主循环会定期生成 task summary，让 `claude ps` 命令能看到后台任务的执行进度。

这些接缝的作用是：主循环在执行过程中，可以感知任务系统的状态变化，也可以向任务系统报告执行进度。这让长任务的执行状态对外可见，不是黑盒。

任务系统和主循环的接缝是隐式的——需要读代码才能找到所有接缝点。如果未来需要修改任务系统的接口，需要找到所有接缝点并逐一修改。一个更显式的做法是引入事件系统，让任务系统和主循环通过事件通信，而不是直接调用。这个改造没有做，代价是接口变更时的搜索成本。

## 17.4 多 Agent 协作的基础

[`src/utils/tasks.ts#L199`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/utils/tasks.ts#L199) 的 `getTaskListId()` 决定任务写到哪个列表，优先级是：显式环境变量 `CLAUDE_CODE_TASK_LIST_ID` > teammate 上下文 > team 名 > 当前 session。

注释里解释了这个设计的意图：`taskListId` 是"协作空间主键"，任何执行实体（主会话、同进程 teammate、tmux teammate）最终都要收敛到同一 ID，这样任务文件目录才能成为跨进程共享的单一真实来源。

这个设计保证了主会话和 teammate 会话能汇聚到同一个任务池，不会各自维护独立的任务列表。在多 Agent 协作系统里，需要一个"协作空间主键"来保证所有 Agent 操作的是同一个任务池。这个主键的确定逻辑需要处理多种情况（显式指定、从上下文推断、默认值），并且有明确的优先级。

## 17.5 运行时结构的可观测性

任务状态存在文件里，可以直接查看。这是 Claude Code 任务系统的一个设计选择：用可读文件而不是数据库，降低排障成本。[`tasks.ts#L232`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/utils/tasks.ts#L232) 的 `getTaskPath()` 返回 `<taskId>.json`，每个任务是一个独立的 JSON 文件。

出现问题时，可以直接看任务文件，知道每个任务的当前状态、认领时间、执行结果。不需要查数据库，不需要看日志，文件本身就是状态的真相。

## 17.6 心智模型验证题

如果去掉 `query.ts#L1694` 的 task summary 生成，会发生什么？

答案：`claude ps` 命令无法显示后台任务的执行进度。用户只能看到任务的状态（pending/in_progress/completed），但看不到任务当前执行到哪一步、正在做什么。对于长时间运行的后台任务，用户无法判断任务是在正常执行还是已经卡住。

## 17.7 本章小结

任务系统把任务作为独立运行时对象管理（[`tasks.ts#L76`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/utils/tasks.ts#L76)），状态持久化在文件系统里，支持中断后恢复。主循环用 `while(true)` 结构驱动执行（[`query.ts#L314`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/query.ts#L314)），继续条件基于工具调用事实（[`query.ts#L561`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/query.ts#L561)）而不是模型文字输出。任务系统和主循环之间有明确的接缝（[`query.ts#L1694`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/query.ts#L1694)），让长任务的执行状态对外可见。
