# 第17章 任务系统是多 Agent 协作的基础设施，而不是状态存储

一个仓库里只有一个货架清单，所有叉车工看同一份，认领货物时更新同一份。这不是文件管理，是协调机制。谁在搬什么、哪些货物可以认领、搬完后怎么通知其他人——这些问题的答案，共同决定了仓库能不能高效运转。

Claude Code 的任务系统处理的是同类问题：在多个 Agent 并发运行时，如何保证任务不被重复认领、不被遗忘、不因某个 Agent 宕机而永远卡死。`claude ps` 能实时显示执行进度，只是这套基础设施对外暴露出来的最小接口。

## 17.1 任务的数据结构：activeForm 字段为什么存在

[`tasks.ts#L76`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/utils/tasks.ts#L76) 的 `TaskSchema` 定义了任务字段，其中 `activeForm` 值得单独说明：

```typescript
z.object({
  id: z.string(),
  subject: z.string(),
  description: z.string(),
  activeForm: z.string().optional(), // "Running tests" 而不是 "run test suite"
  owner: z.string().optional(),
  status: TaskStatusSchema(),         // 'pending' | 'in_progress' | 'completed'
  blocks: z.array(z.string()),
  blockedBy: z.array(z.string()),
  metadata: z.record(z.string(), z.unknown()).optional(),
})
```

`activeForm` 是进行时动作短语，专门为 UI spinner 设计——当状态是 `in_progress` 时，UI 直接显示这个字段作为进度文案，不需要自己生成文案。这个字段的存在说明任务 schema 不只是数据存储规范，还包含了 UI 展示协议。数据层和展示层的这种耦合是有意的：任务创建者提供文案，消费者直接用，不存在"UI 自己解读任务状态生成文案"的模糊地带。

## 17.2 双向阻塞图：为什么要维护两端

`blocks` 和 `blockedBy` 是双向阻塞图。[`tasks.ts#L458`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/utils/tasks.ts#L458) 的 `blockTask()` 同时更新两端：A blocks B 时，A 的 `blocks` 加入 B 的 ID，B 的 `blockedBy` 加入 A 的 ID。

只维护单向（比如只存"A blocks B"而不存"B blocked by A"）在读取时需要全表扫描找到所有阻塞 B 的任务。双向维护用空间换时间——查询某任务的前置依赖是 O(1)，代价是写入时需要维护一致性。

`deleteTask()` 删除任务时级联清理：把该任务从所有其他任务的 `blocks` 和 `blockedBy` 里移除，防止悬挂引用。在多 Agent 并发创建/删除任务时，如果不做级联清理，图会逐渐腐化——某个任务的 `blockedBy` 里存着一个已删除任务的 ID，系统认为它还在被阻塞，永远不会被认领。这个问题不会触发报错，只会让任务静默卡死。

## 17.3 高水位标记：为什么要防止 ID 重用

任务状态存在文件系统里，每个任务是 `<id>.json`。删除任务时，文件被删除，ID 空出来。如果下一个任务复用了这个 ID，历史记录里会出现混淆：同一个 ID 对应过两个不同任务。

[`tasks.ts#L92`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/utils/tasks.ts#L92) 的高水位标记（`.highwatermark` 文件）解决这个问题。它记录曾经分配过的最大 ID。`createTask()` 在分配新 ID 时取"当前文件中最大 ID"和"高水位标记"的较大值，保证新 ID 总是递增。

`deleteTask()` 在删除文件之前，先把被删任务的 ID 写入高水位标记（如果它比当前标记更大）。这个顺序是关键：如果先删文件，然后写高水位标记时进程崩溃，高水位标记没有更新，下一次 `createTask()` 可能分配到被删任务的 ID，导致 ID 重用。先写高水位标记，即使后续删除失败，ID 也不会被重用——这是"宁可 ID 不用，不能 ID 重用"的权衡。

`resetTaskList()` 在多 Agent swarm 开始新一轮协作前清空任务池，删除所有任务文件，但先把当前最高 ID 写入高水位标记，保证新一轮的 ID 从上一轮结束的地方继续递增。

这是一个零依赖的方案：不用数据库，不用分布式 ID 生成器，一个文本文件解决问题。每次分配 ID 都需要读高水位标记文件，对于任务创建频率来说这个开销可以忽略。多机部署时，文件系统方案失效，需要换成数据库序列或分布式 ID 生成器。

## 17.4 两种锁粒度：为什么认领任务需要更粗的锁

任务系统有两种锁：任务级锁（锁单个任务文件）和任务列表级锁（锁整个任务池）。

`updateTask()` 在修改一个任务时只锁这一个任务的文件，其他任务的并发修改不受影响。

`claimTask()` 需要更粗的锁。认领任务包含两步：检查 Agent 是否空闲（busy check），以及认领任务（write owner）。这是典型的 check-then-act 场景——如果两步之间有时间窗口，两个 Agent 可能同时通过检查，同时认领同一个任务。

[`tasks.ts#L621`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/utils/tasks.ts#L621) 的 `claimTaskWithBusyCheck()` 用任务列表级锁把检查和认领原子化：

```typescript
async function claimTaskWithBusyCheck(...) {
  const lockPath = await ensureTaskListLockFile(taskListId)
  release = await lockfile.lock(lockPath, LOCK_OPTIONS)
  // 在同一把锁内：读所有任务、检查 agent 是否繁忙、检查任务是否可认领、认领任务
}
```

`ClaimTaskResult` 的失败理由有五种：`task_not_found`、`already_claimed`、`already_resolved`、`blocked`、`agent_busy`。`agent_busy` 这种情况返回 `busyWithTasks` 字段，列出该 Agent 正在处理的任务 ID，让调度方可以判断：等待这个 Agent，还是把新任务分配给其他空闲 Agent。

`updateTaskUnsafe` 是内部无锁变体，供已持锁调用方使用（[`tasks.ts#L355-L371`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/utils/tasks.ts#L355)）。`proper-lockfile` 不可重入——如果持锁后再次尝试加锁，会死锁。已持锁的调用方需要调用 `updateTaskUnsafe` 而不是 `updateTask`，绕过内部加锁逻辑。这是文件锁方案的固有约束，在数据库事务里，同一个连接可以嵌套事务；文件锁不行。

锁实现用的是文件锁，不是数据库事务。文件锁的优势是零依赖、可以直接检查锁文件的存在；缺点是只在单机上有效。Agent 分布在多台机器上时，文件锁失效。

## 17.5 协作空间主键：五种启动方式收敛到同一任务池

多 Agent 系统里，所有 Agent 必须操作同一个任务池，不能各自维护独立列表。任务池用目录区分：`~/.claude/tasks/<taskListId>/`。

[`tasks.ts#L199`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/utils/tasks.ts#L199) 的 `getTaskListId()` 有五级优先级：

```typescript
export function getTaskListId(): string {
  if (process.env.CLAUDE_CODE_TASK_LIST_ID) {
    return process.env.CLAUDE_CODE_TASK_LIST_ID  // 1. 显式环境变量
  }
  const teammateCtx = getTeammateContext()
  if (teammateCtx) {
    return teammateCtx.teamName  // 2. 进程内 teammate 上下文
  }
  return getTeamName() || leaderTeamName || getSessionId()
  // 3. CLAUDE_CODE_TEAM_NAME 环境变量
  // 4. leader 注册的名字
  // 5. session ID
}
```

tmux/iTerm2 模式下，外部 teammate 进程通过 `CLAUDE_CODE_TEAM_NAME` 环境变量知道自己属于哪个 team；进程内 teammate 通过 `teammateCtx.teamName` 知道；leader 自己调 `setLeaderTeamName()` 注册。三种启动方式最终都收敛到同一个 taskListId，写入同一个目录。

`setLeaderTeamName()` 有一个副作用：

```typescript
export function setLeaderTeamName(teamName: string): void {
  if (leaderTeamName === teamName) return
  leaderTeamName = teamName
  notifyTasksUpdated()  // 通知订阅者：任务池变了
}
```

leader 创建 team 时不只是改了一个变量，还触发了"tasks updated"事件。UI 订阅了这个事件——当 taskListId 变化时，UI 需要重新加载任务列表，因为它们现在看的是一个不同的目录。

## 17.6 Agent 宕机后的任务归还

Agent 宕机是多 Agent 系统的常见场景。一个 Agent 认领了任务，然后进程被杀，任务卡在 `in_progress` 状态，没有人完成它。

[`tasks.ts#L818`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/utils/tasks.ts#L818) 的 `unassignTeammateTasks()` 处理这种情况：

```typescript
export async function unassignTeammateTasks(
  teamName: string,
  teammateId: string,
  teammateName: string,
  reason: 'terminated' | 'shutdown',
): Promise<UnassignTasksResult>
```

当一个 teammate 被终止或主动关闭时，调用方触发这个函数。它找出该 Agent 持有的所有未完成任务，把 `owner` 清空，状态重置为 `pending`，并生成一条通知消息注入给其他 Agent：

```
<teammateName> was terminated. 2 task(s) were unassigned: #3 "Write unit tests", #7 "Update docs".
Use TaskList to check availability and TaskUpdate with owner to reassign them to idle teammates.
```

`reason` 参数区分了 `terminated` 和 `shutdown`，生成的通知措辞不同，方便人工排障时判断是异常退出还是正常退出。

不做任务归还的后果：宕机 Agent 的任务永远卡在 `in_progress`，整个 swarm 陷入死锁——其他 Agent 看到任务有 owner，不去认领，但 owner 已经死了。任务归还把悬挂状态变成了可恢复的 `pending` 状态，这是多 Agent 系统的生存能力，不是锦上添花。

## 17.7 主循环与任务系统的接缝

主循环 `queryLoop()` 和任务系统之间有两处接缝。

第一处（[`query.ts#L1577`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/query.ts#L1577)）：subagent 只处理发给自己的 task-notification，不处理其他 Agent 的。进程内多个 Agent 共享同一个命令队列，靠 `agentId` 字段区分"这条通知是给我的"。主线程处理 `agentId === undefined` 的命令，subagent 处理 `agentId === currentAgentId` 的命令。

第二处（[`query.ts#L1685`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/query.ts#L1685)）：在 `BG_SESSIONS` 模式下，主循环每次迭代结束时，调用 `taskSummaryModule.maybeGenerateTaskSummary()`，生成后台任务摘要写入文件，供 `claude ps` 读取。这个调用只发生在非 subagent 的主线程上（`!toolUseContext.agentId`），避免多个 Agent 都在生成摘要造成写入冲突。

这两处接缝是隐式的——没有统一的事件总线或 TaskSystem 接口，而是在循环内直接调用。引入事件系统可以让接缝更显式，但会增加事件顺序、重复触发、事件丢失的处理复杂度。Claude Code 选择了简单直接的耦合，代价是修改接缝逻辑时需要找到所有调用点逐一修改。

## 17.8 通知系统：任务写入不因通知失败而失败

[`task/framework.ts`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/utils/task/framework.ts) 的 `notifyTasksUpdated()` 用 try/catch 包住 `emit()`：

```typescript
export function notifyTasksUpdated(): void {
  try {
    tasksUpdated.emit()
  } catch {
    // Ignore listener errors — task mutations must not fail due to notification issues
  }
}
```

注释写得很清楚：任务写入操作不能因为通知失败而失败。任务持久化是核心操作，UI 刷新是副效应。调换优先级——通知失败导致任务写入回滚——会让系统在 UI 有 bug 时整体不可用。

跨 Agent 的异步通知走 `enqueuePendingNotification()`，消息格式是 XML：

```xml
<task-notification>
  <task-id>3</task-id>
  <task-type>background_agent</task-type>
  <status>completed</status>
  <summary>Task "Write unit tests" completed successfully</summary>
</task-notification>
```

XML 格式是有意的选择——模型对 XML 标签的解析在训练数据里有足够的先例，比 JSON 更稳定。这条消息被注入下一次迭代的 tool results，接收方 Agent 在下一轮循环里看到状态变化。

## 17.9 可观测性的两种实现

任务系统有两种可观测性机制，服务不同的消费方。

文件系统可见性：每个任务是 `<id>.json`，任务输出写入 `<id>.output`。出了问题可以直接 `cat` 任务文件，不需要查数据库或日志。[`diskOutput.ts`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/utils/task/diskOutput.ts) 的 `getTaskOutputDelta()` 用字节偏移读增量输出，每次只读新增的部分，不加载整个文件。对于持续写入的长任务，增量读取是必要的——每次轮询全量加载会随任务时长线性增长。

SDK 事件流：[`sdkProgress.ts`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/utils/task/sdkProgress.ts) 的 `emitTaskProgress()` 向 SDK 事件队列推送结构化进度，包括已用 token 数、工具调用次数、执行时长。这个接口是为程序化消费设计的——CI 系统、监控平台可以订阅 SDK 事件流，实时获取 Agent 的进度。

两种机制的服务对象不同：文件系统可见性服务人工排障，SDK 事件流服务程序化集成。两者并存，互不依赖。

## 17.10 边界与技术债

文件系统任务队列是单机方案。`~/.claude/tasks/<taskListId>/` 下的文件在同一台机器上的 Agent 间共享，跨机器共享需要网络文件系统或换用数据库，NFS 在高频写入时性能显著下降。

`LOCK_OPTIONS` 里的 `retries` 配置决定了并发 Agent 的等待上限。注释里说每个 critical section 需要约 50-100ms，10 个 Agent 排队最后一个需要约 900ms 等待，`retries=30` 给了约 2.6s 的总等待时间。并发 Agent 数量超过这个假设时，等不到锁的 Agent 会返回 `task_not_found`，表现为任务被漏认领、Agent 空转。

## 17.11 可迁移原则

**任务认领必须原子化**：check-then-act 在同一把锁内完成，不能分步执行。适用条件：多消费者竞争有限资源的任何系统。反例：单消费者场景，不存在竞态，加锁只增加开销。

**Agent 宕机后的资源必须有归还机制**：持有任务的 Agent 死掉，任务状态必须自动归还为可认领，否则系统会积累越来越多的悬挂状态。适用条件：任何有任务所有权的多 Agent 系统。反例：任务是幂等的、重复执行无害，可以用超时机制而不是显式归还。

**通知失败不应阻断主操作**：持久化是核心，通知是副效应，不能调换优先级。适用条件：所有带异步通知的状态变更。反例：通知本身是事务的一部分（比如消息队列投递失败需要回滚），这时调换优先级是正确的。

任务系统和恢复机制的交互（任务状态在进程重启后如何恢复）在第18章覆盖。多 Agent 协调器（leader/worker 模式、邮箱系统）是独立的 `coordinator/` 实现，本章只覆盖基础任务调度层。
