# 第2章 Claude Code 为什么强

## 2.1 大多数 Agent 系统的死法

你用过那种"演示很惊艳、真用就崩"的 Agent 工具吗？

演示时，任务顺利完成，工具调用一气呵成。真正用起来，任务跑到一半进程挂了，重启后从头来；两个 Agent 同时工作，同一个任务被执行了两遍；上下文积累太多，API 报错，整个会话废掉。

这三类问题有一个共同根源：**系统把"出错"当成了不该发生的事，而不是必然会发生的事。** 一旦出错，系统不知道该怎么办，只能停下来。

Claude Code 的设计假设是反过来的。

## 2.2 状态集中：任何时刻都知道自己在哪

[`query.ts#L275`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/query.ts#L275) 的 `State` 对象把所有跨迭代的可变状态集中在一处：

```typescript
let state: State = {
  messages: params.messages,
  toolUseContext: params.toolUseContext,
  hasAttemptedReactiveCompact: false,
  turnCount: 1,
  // ...
}
```

这不是代码整洁的问题。关键在于：**恢复路径可以原子替换这个对象。** 如果状态散落在十几个局部变量里，恢复时需要逐一重建，漏掉任何一个都会导致状态不一致。集中在一个对象里，恢复就是把这个对象换掉。

`hasAttemptedReactiveCompact` 这个 flag 值得单独说一下。它的作用是防止压缩无限循环：压缩 → 仍然过长 → 报错 → 再次压缩 → 仍然过长 → 报错……如果没有这个 flag，系统会在这个循环里烧掉数千次 API 调用。这个 flag 不是随意加的，是修复了一个真实发生过的无限循环 bug 之后补上的。

## 2.3 工具默认值：忘了声明就按最严格的来

[`Tool.ts#L761`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/Tool.ts#L761) 的注释写得很直白：

```typescript
// 默认值采用"安全兜底"思路：未知工具先按不可并发/非只读处理，
// 由具体工具显式声明更宽松能力，避免遗漏实现时误放权。
const TOOL_DEFAULTS = {
  isConcurrencySafe: (_input?: unknown) => false,  // 默认不可并发
  isReadOnly: (_input?: unknown) => false,          // 默认非只读（即假设会写）
  // ...
}
```

这是一个反直觉的选择。通常的做法是"默认允许，需要时限制"——开发更方便，新工具接入不需要声明太多东西。

Claude Code 反过来：**默认拒绝，需要时开放。** 新工具如果忘了声明 `isReadOnly: true`，系统会把它当成写操作处理，触发更严格的权限检查。最坏情况是性能下降，不会导致数据被意外修改。

如果默认允许，工具作者忘了声明，系统会按最宽松的方式处理。最坏情况是一个只读工具被当成写操作来并发控制，或者一个写操作工具在不该并发的时候并发执行，导致数据损坏。

两种默认值的区别：一个把"忘了声明"的代价压在性能上，一个把代价压在数据安全上。

## 2.4 原子认领：TOCTOU 竞争是怎么发生的

[`tasks.ts#L621`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/utils/tasks.ts#L621) 的注释直接点名了这个问题：

```typescript
// 关键一致性点：把"检查 agent 是否繁忙"与"抢占任务"放进同一把 list-level 锁。
// 这是典型的 check-then-act 场景，分离会导致 TOCTOU 竞争
// （两个 agent 同时看到空闲并同时认领）。
async function claimTaskWithBusyCheck(...) {
  const lockPath = await ensureTaskListLockFile(taskListId)
  release = await lockfile.lock(lockPath, LOCK_OPTIONS)
  // 在锁内：检查 + 认领，两步原子完成
}
```

TOCTOU（time-of-check to time-of-use）是并发 bug 的经典形态：Agent A 检查任务未被认领，Agent B 也检查任务未被认领，然后 A 认领，B 也认领，同一个任务被执行两遍。

这类 bug 在低并发时几乎不出现，在高并发时才暴露，而且复现困难、修复成本高。`claimTaskWithBusyCheck` 的设计是把检查和认领放在同一把锁内，从根本上消除了这个时间窗口。

注意这里用的是文件锁（`.lock` 文件），而不是数据库事务。文件锁的优势是零依赖、可以直接看到锁文件的存在；劣势是只能在单机上工作。如果 Agent 分布在多台机器上，文件锁失效，需要换成 Redis 或数据库的分布式锁。

## 2.5 中断检测：三种死法对应三种救法

进程被杀有三种情况，对应三种不同的恢复策略。[`conversationRecovery.ts#L272`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/utils/conversationRecovery.ts#L272) 的 `detectTurnInterruption()` 把这三种情况区分开来：

第一种：用户消息发出去了，API 还没响应，进程就死了。这时 transcript 里有用户消息，没有 assistant 回复。恢复时需要重新发这条消息。

第二种：API 开始响应了，模型回复到一半，进程死了。这时 transcript 里有一条不完整的 assistant 消息。恢复时需要把这条不完整的消息标记为中断，补一个 sentinel，让 API 能接受这个不完整的历史。

第三种：正常结束。不需要特殊处理。

如果不区分这三种情况，统一报错让用户重试，用户会丢失已经完成的工作。区分之后，每种情况都有对应的恢复路径，用户重新打开后能从中断点继续。

## 2.6 三件事的共同逻辑

状态集中、保守默认值、原子认领，这三个设计看起来是独立的，但背后是同一个逻辑：

**把"出错"当成正常情况来设计。**

进程随时可能死，恢复路径必须能原子替换状态——状态集中由此而来。工具作者随时可能忘了声明，系统必须在遗漏时选择安全的一侧——保守默认值由此而来。并发冲突随时可能发生，检查和认领之间不能有时间窗口——原子认领由此而来。

这三个设计的共同代价是：接入成本更高（工具需要显式声明属性）、性能有一定损耗（文件锁的开销）、代码更复杂（状态集中管理需要纪律）。但这些代价在真实使用中是值得的，因为它们换来的是系统在出错时能继续工作，而不是停下来。

## 2.7 验证题

如果你在 Claude Code 里写了一个新工具，忘了声明 `isReadOnly: true`，会发生什么？

答案：系统会把它当成写操作处理，触发更严格的权限检查，可能需要用户手动确认。工具能用，但比应该的更慢、更麻烦。不会发生数据损坏。这就是保守默认值的代价和收益。

## 2.8 本章小结

Claude Code 的强来自一个贯穿所有设计的假设：出错是正常情况，不是边界情况。这个假设体现在状态集中（[`query.ts#L275`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/query.ts#L275)）、保守默认值（[`Tool.ts#L763`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/Tool.ts#L763)）、原子认领（[`tasks.ts#L621`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/utils/tasks.ts#L621)）三个地方。
