# 第2章 Claude Code 为什么强

一个系统能在任意时刻崩溃，该如何设计？不是说要把每种错误都 catch 住，而是说：在崩溃真的发生时，系统的状态能否支撑恢复？Claude Code 用三个具体决策回答了这个问题。

## 2.1 状态集中，恢复就是整体替换

恢复的代价取决于状态散落的程度。[`query.ts#L275`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/query.ts#L275) 把所有跨迭代的可变状态收敛到单一的 `State` 对象：

```typescript
let state: State = {
  messages: params.messages,
  toolUseContext: params.toolUseContext,
  hasAttemptedReactiveCompact: false,
  turnCount: 1,
  // ...
}
```

状态散落在十几个局部变量里时，崩溃后恢复需要逐一重建，漏掉任何一个都会导致状态不一致。集中在单一对象后，恢复就是把这个对象整体替换，没有中间状态，也没有遗漏的可能。

`hasAttemptedReactiveCompact` 这个字段值得单独说明。它的作用是阻断一个确实发生过的无限循环：上下文过长触发压缩，压缩后仍然超长，再次触发压缩，如此往复，最终烧掉数千次 API 调用才被人工干预。这个 flag 不是防御性设计，是事故之后打上的补丁。它的存在说明状态集中这个决策是在真实故障中被校验过的。

## 2.2 漏配工具只影响性能，不会出错

[`Tool.ts#L761`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/Tool.ts#L761) 的注释写明了设计意图，叫 "fail-closed"：

```typescript
// Defaults (fail-closed where it matters):
// - isConcurrencySafe → false (assume not safe)
// - isReadOnly → false (assume writes)
const TOOL_DEFAULTS = {
  isConcurrencySafe: (_input?: unknown) => false,
  isReadOnly: (_input?: unknown) => false,
}
```

fail-closed 的意思是：遗漏声明时，系统走最严格的路径，而不是最宽松的路径。

两种方向的区别在于"遗漏"的后果落在哪里。工具作者忘记声明 `isReadOnly: true`，系统把这个工具当写操作处理，触发更严格的权限检查，用户可能多一次确认弹窗。这是性能代价，不是正确性问题。相反，如果默认宽松，工具作者忘记声明 `isConcurrencySafe: false`，一个有竞态问题的写工具在不该并发时并发执行，导致数据损坏，且低并发下不复现、高并发下才暴露，排查成本极高。

这是刻意的权衡：把遗漏的代价定向压在可见的性能降级上，而不是不可见的数据损坏上。

## 2.3 检查和认领合一，竞态在结构上不存在

TOCTOU（time-of-check to time-of-use，检查时到使用时之间的状态可能已改变）是并发 bug 的经典形态。两个 Agent 同时检查到某任务未被认领，于是各自认领，同一个任务执行两遍。这类 bug 在低并发时几乎不出现，高并发时才暴露，难以复现，修复成本高。

[`tasks.ts#L621`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/utils/tasks.ts#L621) 的注释直接点名了要解决的问题：

```typescript
// 关键一致性点：把"检查 agent 是否繁忙"与"抢占任务"放进同一把 list-level 锁。
// 这是典型的 check-then-act 场景，分离会导致 TOCTOU 竞争
// （两个 agent 同时看到空闲并同时认领）。
async function claimTaskWithBusyCheck(...) {
  const lockPath = await ensureTaskListLockFile(taskListId)
  release = await lockfile.lock(lockPath, LOCK_OPTIONS)
  // 在锁内：检查 + 认领，两步原子完成
}
```

把检查和认领放入同一把锁，时间窗口在结构上不存在，不依赖执行时序的侥幸。

这里用文件锁，不用数据库事务。好处是零依赖，出问题直接查文件系统状态。代价是方案只在单机上有效：多机协作时文件锁失效，需要换成分布式锁。这个边界是明确的设计约束，不是遗漏。

## 2.4 进程崩溃分三种，恢复方式各不同

进程终止有三种情形，恢复路径不同。[`conversationRecovery.ts#L272`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/utils/conversationRecovery.ts#L272) 的 `detectTurnInterruption()` 把这三种情形区分开来处理。

第一种：用户消息已写入 transcript，API 还没响应，进程终止。恢复时重发这条消息。

第二种：API 已开始响应，模型回复写到一半，进程终止。transcript 里存在一条不完整的 assistant 消息。恢复时把这条消息标记为中断，补入一个 sentinel 条目，让后续 API 调用能接受这段不完整的历史记录。sentinel 是一种占位标记，用于标识历史中存在中断点，防止 API 因格式不合法而拒绝请求。

第三种：正常结束，不需要处理。

统一报错让用户从头重试会丢失已完成的工作。区分三种情形后，`--resume` 能从中断点继续，已完成的工具调用不重复执行。

## 2.5 这套设计，出了单机就不管用

状态集中假设进程是单线程的。如果同一个 `State` 被多个协程并发修改，集中本身不提供线程安全保证，需要额外的锁机制。

文件锁的原子认领假设任务存储在本地文件系统。分布式部署时这个假设失效，方案需要整体替换。

保守默认值假设工具作者会在需要时显式声明宽松权限。如果团队内所有工具都没人填声明，所有操作都按最严格路径执行，性能退化为最坏情况。

三个设计都以"单机、单用户、非极高并发"为前提。在这个范围内，每个决策都有明确的收益。超出这个范围，评估需要从头来。

## 2.6 把代价引到看得见的地方

三个决策背后有一个共同模式：把"遗漏或异常"的代价，定向引导到可观察、可测量的路径上，而不是让它沉默地破坏正确性。

这个模式适用于任何需要在部分故障下保持正确的系统：默认拒绝而非默认允许的权限系统、先记录再执行的事务日志、先写检查点再继续的批处理任务。反例是当故障本身不可分割时，或者中间状态没有意义时，强行引入这套模式只会增加复杂度，得不到对应的收益。

原则是工具，不是教条。
