# 第2章 Claude Code 为什么强

> **核心主张：Claude Code 的强来自"系统可观测性"——在任何时刻，系统都能精确知道自己处于什么状态，这个属性让长流程任务在中断后能继续，而不是从头来。这不是功能堆砌，而是一个贯穿所有设计决策的约束。**

## 2.1 第一性问题：为什么"功能多"不等于"能干活"

很多 Agent 系统功能看起来很多，但在真实工作中很快暴露问题。根本原因不是功能不够，而是系统在任何时刻都不知道自己处于什么状态。

"系统不知道自己的状态"会导致三类具体问题：

**问题一：中断后无法恢复。** 进程被杀后，系统不知道上次执行到哪里，只能从头开始。

**问题二：并发时状态错乱。** 多个执行体同时工作时，系统不知道哪个任务已经被认领，导致重复执行。

**问题三：上下文超限时无法降级。** 上下文积累到超过模型处理上限时，系统不知道哪些内容可以压缩，只能报错停止。

Claude Code 在这三个方向上都有专门的设计，而不是把它们当成边界情况处理。

## 2.2 可观测性的三个层次

**层次一：会话状态可观测。**

`src/query.ts#L282` 的 `State` 对象把所有跨迭代的可变状态集中在一处：`messages`、`toolUseContext`、`hasAttemptedReactiveCompact`、`turnCount` 等。这不是代码整洁的问题，而是可观测性的问题：任何时刻都能通过这一个对象知道系统处于什么状态。

隐含假设：**状态集中管理的前提是单线程执行。** 如果多个线程同时修改这个对象，集中管理反而会成为竞态条件的来源。Claude Code 的 query loop 是单线程的，这个假设成立。

**层次二：任务状态可观测。**

`src/utils/tasks.ts#L76` 的 `TaskSchema` 把任务状态写入文件系统，而不是内存。这意味着即使进程重启，任务状态也不会丢失。`tasks.ts#L92` 的 `.highwatermark` 文件记录已分配过的最大任务 ID，防止 ID 复用导致新任务和历史记录混淆。

**层次三：中断状态可观测。**

`src/utils/conversationRecovery.ts#L272` 的 `detectTurnInterruption()` 在加载历史消息时，区分三种情况：`interrupted_turn`（模型回复到一半被中断）、`interrupted_prompt`（用户消息发出后 API 还没响应就被中断）、正常结束。这三种情况对应不同的恢复策略，不是统一报错让用户重试。

## 2.3 工具契约：安全边界的默认值选择

`src/Tool.ts#L763` 的 `TOOL_DEFAULTS` 有两个保守默认值：`isReadOnly: false`（默认非只读）、默认不可并发。

这是一个反直觉的设计选择。通常的做法是"默认允许，需要时限制"，因为这样开发更方便。Claude Code 的选择是"默认拒绝，需要时开放"——新工具必须主动声明自己是只读的、可并发的，才能获得对应的宽松处理。

**为什么选择保守默认值？** 因为工具的安全属性是工具作者最清楚的，而不是调用方。如果默认允许，工具作者忘了声明安全属性，系统会按最宽松的方式处理，可能导致意外的破坏性操作。如果默认拒绝，工具作者忘了声明，系统会按最严格的方式处理，最坏情况是性能下降，不会导致数据损坏。

**技术债：** 保守默认值会导致新工具接入时需要显式声明更多属性，增加了接入成本。在工具数量少时，这个成本可以接受；在工具数量很多时，可能需要更细粒度的默认值分类。

## 2.4 并发一致性：原子操作的必要性

`src/utils/tasks.ts#L621` 的 `claimTaskWithBusyCheck()` 把检查 agent 是否空闲和认领任务这两步放在同一把锁内完成。

**为什么必须原子？** 如果检查和认领之间有间隙，两个 agent 可能同时通过检查（都看到任务未被认领），然后同时认领同一个任务，导致重复执行。这类 TOCTOU（time-of-check to time-of-use）竞态在低并发时不出现，在高并发时才暴露，修复成本很高。

**替代方案：** 用数据库的事务来保证原子性，而不是文件锁。数据库事务的优势是支持更复杂的原子操作，劣势是增加了依赖。文件锁方案的优势是零依赖、可观察（`.lock` 文件可以直接看到），劣势是在极高并发下性能会退化。

**边界条件：** 文件锁方案假设所有 agent 都在同一台机器上运行。如果 agent 分布在多台机器上，文件锁无法跨机器工作，需要换成分布式锁（如 Redis）。

## 2.5 可迁移的设计原则

从 Claude Code 的架构中，可以抽象出两条可迁移的原则：

**原则一：状态集中，不要分散。** 把所有可变状态集中在一个对象里，而不是散落在多个局部变量中。这让系统在任何时刻都能通过一个对象知道自己的状态，也让恢复路径可以原子替换状态。

适用条件：单线程执行，或者有明确的状态所有权。反例：多线程并发修改同一个状态对象，会引入竞态条件。

**原则二：安全属性用保守默认值，而不是宽松默认值。** 新组件如果忘了声明安全属性，系统应该按最严格的方式处理，而不是按最宽松的方式。

适用条件：安全属性的声明成本低于安全事故的修复成本。反例：如果安全属性的声明非常繁琐，开发者会倾向于绕过声明，保守默认值反而会被忽视。

## 2.6 心智模型验证题

读完这章，你应该能回答：如果两个 Claude Code agent 同时看到同一个 `pending` 状态的任务，会发生什么？

答案：`claimTaskWithBusyCheck()` 用文件锁保证了原子性，只有一个 agent 能成功认领，另一个会收到"任务已被认领"的结果。这不是靠运气，而是靠锁的互斥性保证的。

## 2.7 本章小结

Claude Code 的强来自三个层次的可观测性：会话状态（[`query.ts#L282`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/query.ts#L282)）、任务状态（[`tasks.ts#L76`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/utils/tasks.ts#L76)）、中断状态（[`conversationRecovery.ts#L272`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/utils/conversationRecovery.ts#L272)）。这三个层次共同保证了系统在任何时刻都知道自己处于什么状态，这是长流程任务能稳定执行的根本原因。

需要记住的核心概念：**可观测性是稳定性的前提，不是稳定性的结果。**
