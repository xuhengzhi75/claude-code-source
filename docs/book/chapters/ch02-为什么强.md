# 第2章 Claude Code 为什么强

> 核心主张：Claude Code 的强不是因为工具多，而是因为它把"系统在任何时刻都知道自己处于什么状态"当成设计约束——这个约束让长流程任务在中断后能继续，而不是从头来。

## 先看动作，再看结构

从行为看，Claude Code 会持续执行一组动作：接收任务、读取环境信息、决定下一步、调用工具、将结果并回当前流程、在未完成时继续推进。

这些动作单看都不新鲜，难点在于把它们稳定串联。系统需要清楚：当前进度在哪里、为什么继续、何时结束、失败后如何衔接、哪些动作可做、哪些动作受限。这是运行时架构问题，不只是提示词问题。

从仓库结构看，这套能力通常可以分成几层：入口层接收请求并区分运行模式，能力装配层整理命令、工具、插件与可见能力，会话编排层组织会话启动、持续和记录，单轮执行层处理一轮中的状态变化和工具调用，任务/历史/恢复/压缩/记忆支撑长任务连续运行。

## 强在连续工作能力

普通聊天系统更适合一问一答，问答结束后流程通常也结束。Claude Code 处理的是持续任务，需要管理过程而不是只生成文本。

这意味着系统要具备：记录当前任务状态、判断是否需要继续下一轮、保存中间结果以便恢复、在上下文变长时做压缩同时保留关键事实。

所谓强，常体现在较长流程中的稳定性：出错更少，中断后更容易继续，步骤边界更清晰。

## 强在统一的工具契约

在 Agent 系统里，工具不只是外挂功能，更像统一管理的动作接口。系统需要知道：每个工具能做什么、不能做什么，默认风险与权限边界，是否允许并发，调用结果如何回到主流程。

`src/Tool.ts#L763` 定义了 `TOOL_DEFAULTS`，其中两个保守默认值值得注意：默认不可并发（`isReadOnly: false`）、默认非只读。这意味着新工具如果忘了声明自己的安全属性，系统会按最严格的方式处理，而不是按最宽松的方式。这是把安全边界当成默认值，而不是当成需要主动开启的选项。

统一契约的价值很实际：行为更一致，权限边界更清楚，新工具接入成本更低。

## 强在状态与恢复

真实任务经常中断，上下文也会持续增长。系统是否可靠，往往看它中断后能否继续。

`src/utils/conversationRecovery.ts#L164` 的 `deserializeMessagesWithInterruptDetection()` 在加载历史消息时，会检测对话是否在中途被打断。检测逻辑分三种情况：`interrupted_turn`（模型回复到一半被中断）、`interrupted_prompt`（用户消息发出后 API 还没响应就被中断）、正常结束。这三种情况对应不同的恢复策略，不是统一报错让用户重试。

恢复至少包含两层：事实恢复（找回已执行步骤、产出结果、当前进度）和语义恢复（把这些事实整理成可继续执行的上下文）。系统关注的重点是工作连续性，不只是保存聊天记录。

## 强在任务系统与并发一致性

很多产品有任务列表，但执行阶段会遇到更复杂的问题：任务由谁认领、是否会重复执行、失败后如何更新状态、多个执行体并行时如何避免互相覆盖。

`src/utils/tasks.ts#L621` 的 `claimTaskWithBusyCheck()` 在同一把锁内完成两个操作：检查当前 agent 是否空闲，以及认领任务。这两步必须原子完成，否则两个 agent 可能同时通过检查，然后同时认领同一个任务。这个细节说明任务系统不是简单的状态标记，而是考虑了并发场景下的一致性。

## 强在系统化

很多 Agent 产品功能看起来很多，但功能数量不等于成熟度。Claude Code 的参考价值在于，它呈现了一套较完整的运行结构：入口、能力装配、会话编排、单轮执行、任务系统、历史、恢复、压缩、权限。这些部分协同后，系统才更稳定。

## 对读者最有用的结论

如果你想从 Claude Code 学习，可优先关注以下顺序：状态组织、工具接入、历史记录、恢复机制、任务追踪。

这些能力会直接影响 Agent 是否能进入真实工作场景。骨架稳定之后，再扩展工具数量和自动化复杂度，风险更低。
