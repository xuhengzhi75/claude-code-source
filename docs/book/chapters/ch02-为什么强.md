# 第2章 故障后可恢复，是系统约束，不是开发者自律

厨房里的标准做法是：食材用完放回原位，刀洗完挂回刀架。道理不是整洁，是下一个人来的时候能接着用，不用先花时间找东西。软件里的对应问题是：进程崩溃之后，谁来保证下一次启动能从中断点接续，而不是从头来一遍？

"约好了大家都仔细"不是答案。答案是把恢复能力编码进系统结构：崩溃发生时，状态有地方找；遗漏声明时，代价落在性能上不是正确性上；两个执行者同时抢任务时，结构上只有一个人能拿到。

这套逻辑的适用条件是"故障局部可见"。进程崩了能重启，消息丢了有记录，这些都算局部可见。如果故障本身不可分割——比如网络脑裂，两边都认为自己是主节点——中间状态没有意义，强行引入这套模式只会增加复杂度。原则是工具，不是教条。

---

## 2.1 状态集中让恢复只有一步

恢复的代价取决于状态散落的程度。[`query.ts#L275`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/query.ts#L275) 把所有跨迭代的可变状态收敛到单一的 `State` 对象：

```typescript
let state: State = {
  messages: params.messages,
  toolUseContext: params.toolUseContext,
  hasAttemptedReactiveCompact: false,
  turnCount: 1,
  // ...
}
```

状态散落在十几个局部变量里时，崩溃后恢复需要逐一重建每一个，漏掉任何一个就导致状态不一致。收敛到单一对象后，恢复就是整体替换这个对象，没有中间状态，也没有遗漏的可能。

`hasAttemptedReactiveCompact` 这个字段承载了一段真实事故的教训：它阻断的是"上下文过长触发压缩、压缩后仍然超长、再次触发压缩"这个无限循环。这个 flag 是事故之后打上的补丁，不是防御性设计。它的存在说明状态集中策略经过了真实故障的校验。

这个决策以"进程单线程"为前提。同一个 `State` 对象被多个协程并发修改时，集中本身不提供线程安全保证。

## 2.2 保守默认值把遗漏代价压在性能上

遗漏的代价落在哪里，由默认方向决定。[`Tool.ts#L761`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/Tool.ts#L761) 的注释称之为 "fail-closed"——遗漏声明时走最严格路径，而非最宽松路径：

```typescript
// Defaults (fail-closed where it matters):
// - isConcurrencySafe → false (assume not safe)
// - isReadOnly → false (assume writes)
const TOOL_DEFAULTS = {
  isConcurrencySafe: (_input?: unknown) => false,
  isReadOnly: (_input?: unknown) => false,
}
```

工具作者忘记声明 `isReadOnly: true`，系统把这个工具当写操作处理，触发更严格的权限检查，用户多一次确认弹窗。这是性能代价，正确性不受影响。

宽松默认的后果在另一个维度上：工具作者忘记声明 `isConcurrencySafe: false`，一个有竞态问题的写工具在不该并发时并发执行，导致数据损坏。这类 bug 在低并发下几乎不复现，高并发下才暴露，排查成本极高。

两个方向的漏填后果本质不对称：多一次弹窗，和数据损坏且难以复现。保守默认值是刻意的权衡，把遗漏的代价定向压在可见的性能降级上。

## 2.3 检查与认领同锁，竞态在结构上不存在

TOCTOU（time-of-check to time-of-use，检查时到使用时之间状态可能已改变）是并发 bug 的经典形态。两个 Agent 同时检查到某任务未被认领，各自认领，同一个任务执行两遍。这类 bug 在低并发时几乎不出现，高并发时才暴露，难以复现。

[`tasks.ts#L621`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/utils/tasks.ts#L621) 的注释直接点名了要解决的问题：

```typescript
// 关键一致性点：把"检查 agent 是否繁忙"与"抢占任务"放进同一把 list-level 锁。
// 这是典型的 check-then-act 场景，分离会导致 TOCTOU 竞争
// （两个 agent 同时看到空闲并同时认领）。
async function claimTaskWithBusyCheck(...) {
  const lockPath = await ensureTaskListLockFile(taskListId)
  release = await lockfile.lock(lockPath, LOCK_OPTIONS)
  // 在锁内：检查 + 认领，两步原子完成
}
```

把检查和认领放入同一把锁，时间窗口在结构上不存在，不依赖执行时序的侥幸。

这里用文件锁，不用数据库事务。好处是零依赖，出问题直接查文件系统状态。代价是方案只在单机上有效：多机协作时文件锁失效，需要换成分布式锁。这个边界是明确的设计约束。

## 2.4 中断恢复按终止情形分类，避免重复执行

统一报错让用户从头重试会丢失已完成的工作。进程终止有三种情形，恢复路径不同，需要区分处理。

[`conversationRecovery.ts#L272`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/utils/conversationRecovery.ts#L272) 的 `detectTurnInterruption()` 把三种情形分开。

情形一：用户消息已写入 transcript（持久化到磁盘的对话记录文件），API 还没响应，进程终止。恢复时重发这条消息。

情形二：API 已开始响应，模型回复写到一半，进程终止。transcript 里存在一条不完整的 assistant 消息。恢复时把这条消息标记为中断，补入一个 sentinel 条目。sentinel 是用于标识历史中存在中断点的占位标记，它让后续 API 调用能接受这段不完整的历史，而不会因格式不合法拒绝请求。

情形三：正常结束，不需要处理。

区分三种情形后，`--resume` 能从中断点继续，已完成的工具调用不重复执行。

## 2.5 四个决策共享同一个边界

四个决策都以"单机、单用户、非极高并发"为前提，超出这个范围时评估需要从头来。

状态集中假设进程是单线程的。文件锁的原子认领假设任务存储在本地文件系统，分布式部署时这个假设失效。保守默认值假设工具作者会在需要时显式声明宽松权限，如果团队内所有工具都没人填声明，所有操作都按最严格路径执行，性能退化为最坏情况。

这套模式在满足两个条件时可以直接迁移：故障局部可见，中间状态有意义。满足这两条的场景都能用——默认拒绝的权限系统、先记录再执行的事务日志、先写检查点再继续的批处理任务。两个条件缺一，强行引入只会增加复杂度。
