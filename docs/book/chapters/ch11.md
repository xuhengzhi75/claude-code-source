# 第11章 为什么很多 Agent 看起来像但干不了真活

这一章不是批评某个具体产品，而是分析一类现象：很多 Agent 在演示时表现不错，但在真实工作场景中很快暴露出问题。这些问题有规律可循。

## 11.1 演示场景和真实场景的差距

演示场景通常有几个特点：任务边界清晰、输入格式规范、不需要中断恢复、不需要处理并发、失败了可以重来。

真实工作场景则相反：任务边界模糊（用户说"帮我整理一下这个项目"，没有明确终止条件）、输入格式不规范（代码里有历史遗留的奇怪写法）、进行到一半可能被打断、多个任务可能同时进行、失败了需要从中间状态恢复。

这两类场景的差距，决定了一个 Agent 在演示里好用，不代表在真实工作里好用。

## 11.2 工具调用的错误处理

Claude Code 的工具调用有明确的错误处理层级。`src/utils/errors.ts` 里定义了 `AbortError`、`InterruptedError`、`ContextWindowExceededError` 等不同类型，每种类型对应不同的处理策略。

`AbortError` 表示用户主动中止，不需要重试，直接清理状态。`InterruptedError` 表示被外部信号中断，可能需要恢复。`ContextWindowExceededError` 表示上下文超限，需要触发压缩流程。

很多 Agent 实现里，工具调用失败只有一种处理方式：报错、停止、让用户重试。这在演示里看起来没问题，因为演示任务通常不会触发这些边界情况。但在真实使用中，这些情况会频繁出现。

## 11.3 状态管理的一致性

`src/services/SessionManager/sessionManager.ts` 维护了一个完整的会话状态机。状态转换有明确的路径：`idle → running → paused → idle`，或者 `running → error → idle`。

状态机的价值在于：任何时刻都能知道系统处于哪个状态，以及从这个状态能做什么操作。如果没有状态机，系统的行为取决于各个组件的局部状态，这些局部状态可能不一致，导致难以预测的行为。

很多 Agent 实现里，状态是隐式的：通过变量是否为 null、数组是否为空、某个 flag 是否为 true 来判断。这在简单场景下够用，但在并发、中断、恢复等复杂场景下，隐式状态很容易出现不一致。

## 11.4 上下文窗口的主动管理

`src/services/QueryEngine/queryEngine.ts` 在每轮对话前检查 token 使用量，如果接近上限，主动触发压缩。这是主动管理，不是被动等待 API 报错。

被动等待的问题是：当 API 报 context window exceeded 时，当前轮次的工作已经失败，需要从上一个检查点恢复。如果没有检查点机制，就需要从头开始。

主动管理需要知道当前 token 使用量，以及压缩的触发阈值。这两个信息都需要从 API 响应里提取，并在系统内维护。

## 11.5 工具权限的边界

`src/services/PermissionManager/permissionManager.ts` 管理工具的执行权限。不同工具有不同的权限级别，有些操作需要用户确认，有些操作在特定模式下才允许。

权限管理的价值不只是安全，还有可预测性：用户知道 Agent 能做什么、不能做什么，不会出现 Agent 意外执行了用户没有预期的操作。

很多 Agent 实现里，工具权限是全有或全无的：要么所有工具都可以用，要么需要用户逐一确认。前者不安全，后者体验差。细粒度的权限管理需要对工具的风险等级有明确的分类，以及对用户意图的准确判断。

## 11.6 长任务的进度可见性

`src/services/TaskManager/taskManager.ts` 维护任务的状态和进度。任务有明确的生命周期：创建、认领、执行、完成或失败。每个状态变化都有记录。

进度可见性对用户体验很重要：用户知道任务在进行中，知道大概到哪一步了，知道如果出问题在哪里查看。如果任务是黑盒，用户只能等待，不知道是在正常执行还是已经卡住。

实现进度可见性需要在任务执行的关键节点写入状态，并提供查询接口。这不是核心功能，但对真实使用体验影响很大。

## 11.7 这些差距的本质

上面列的 5 类问题，本质上都是同一件事：在真实场景中，系统需要处理的情况比演示场景复杂得多，而这些复杂情况的处理需要专门的设计。

这些设计不是一次性完成的，而是在真实使用中逐步发现、逐步补齐的。这就是为什么一个经过真实场景打磨的系统，和一个刚完成基础功能的系统，在演示里看起来差不多，但在真实使用中差距很大。

## 11.8 本章小结

很多 Agent 在演示里好用，在真实工作中暴露问题，原因在于：错误处理不完整、状态管理不一致、上下文窗口被动管理、工具权限粗粒度、长任务进度不可见。

这些问题的共同特点是：在演示场景下不会触发，在真实使用中频繁出现。下一章从学习者的角度看：面对 Claude Code 的架构，该学什么，先别急着抄什么。
