# 第7篇：源码都泄露了，为什么还是难以复制

2024 年，Claude Code 的源码意外泄露了。

任何人都可以下载，看到完整的实现细节。

那么，竞争对手拿到源码之后，能直接做出一个一样好用的产品吗？

答案是：很难。

---

## 菜谱可以复制，踩坑历史不能

想象一家开了 20 年的老餐厅，老板把所有菜谱都给了你。

菜谱准确，食材一样，但你做出来的味道就是差一点。

差的不是菜谱，是老板在每道菜上踩过的几百个坑。

Claude Code 的源码就是这份菜谱。代码可以复制，但代码背后的故障历史不能。

## 一个函数，一段历史

源码里有一个函数叫 `migrateLegacyAttachmentTypes()`，它的作用是把旧版本的数据格式转换成新格式。

竞争对手可以把这个函数复制过去。但这个函数存在的原因是：Claude Code 在某次版本升级时改变了数据格式，需要处理老用户的历史数据。

竞争对手的系统没有这段升级历史，这个函数对他们没有意义。他们需要在自己的系统演进中，踩自己的坑，积累自己的迁移代码。

## 25 万次 API 调用换来的一行代码

源码里有一段注释，记录了一次真实的事故：

> 1279 个会话里，压缩连续失败 50 次以上，最极端的一个会话失败了 3272 次，全球每天因此浪费约 25 万次 API 调用。

解决方案只有一行代码：把最大连续失败次数限制为 3。

这行代码很简单，任何人都能写出来。但没有经历过这次事故，没有看到那 25 万次浪费的数据，你不会知道这行代码是必要的，也不会知道为什么是 3 而不是 10 或者无限。

**代码是结论，注释是推导过程，事故是前提。** 复制代码，拿不走前提。

## 把安全规范变成编译器约束

Claude Code 有一个有意思的设计：把安全规范写进类型系统里。

比如，当开发者要提交包含用户数据的遥测代码时，必须在代码里显式声明"我已确认这个数据不包含敏感信息"。如果没有这个声明，代码编译不通过。

这不是靠人工审查来保证的，是靠编译器来强制执行的。

普通团队的做法是：写一份文档，告诉开发者"提交遥测数据时要注意隐私"。但文档靠人记，人会忘，会犯错。

把规范变成编译器约束，靠机器检查，不依赖任何人的记忆力。

这种能力不是一天建成的，是在多次安全事故之后，逐渐把教训固化进工程流程里的结果。

## 用户用得越久，迁移成本越高

Claude Code 有一个"记忆"功能：在后台持续整理对话中的重要信息，保存到文件里。

用户用得越久，积累的上下文越多，Claude Code 对这个用户的了解越深，帮助越精准。

竞争对手可以做出功能相似的产品，但让已经积累了大量历史的用户迁移，需要提供压倒性的优势。

这是一种时间护城河：不是靠技术壁垒，而是靠时间积累。

## 快速响应的能力

还有一个差距，在日常使用中不明显，但在出问题的时候很关键。

Claude Code 内置了一套"紧急开关"机制。如果某个功能出现了意外行为，Anthropic 可以在几分钟内通过服务端配置，限制或关闭这个功能，不需要发布新版本，不需要等用户更新。

竞争对手如果没有这一层，出了问题只能发布新版本，等用户更新，这个过程可能需要几天甚至几周。

**功能差距在日常使用中可以接受，响应速度差距在安全事件里是决定性的。**
