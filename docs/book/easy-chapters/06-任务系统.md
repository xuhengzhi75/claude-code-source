# 第6篇：长任务为什么不会半途而废

你让 Claude Code 帮你重构 30 个文件。

做到第 15 个的时候，你的电脑崩溃了。

重启之后，它会从头开始，还是从第 16 个继续？

答案是：从第 16 个继续。

这不是魔法，是一套专门设计的任务系统。

---

## 任务存在文件里，不是内存里

普通程序的状态存在内存里。进程一死，状态全没。

Claude Code 的任务系统把任务状态存在文件里。每个任务是一个 JSON 文件，放在你电脑上的一个固定目录里。

进程死了，文件还在。重启之后，系统读取文件，知道哪些任务完成了，哪些还没完，从没完的地方继续。

这个设计的好处不只是崩溃恢复。你还可以直接打开这些文件看看任务进展，不需要任何特殊工具，`cat` 一下就行。

## 多个 AI 同时工作时，怎么防止抢任务

Claude Code 支持多个 AI 实例同时工作，分头处理不同的任务。

这里有一个经典的并发问题：如果两个 AI 同时看到"任务 5 还没人做"，都去认领，会发生什么？

答案是：两个都以为任务归自己，各自去做，同一个任务被执行了两遍。

Claude Code 的解决方案是：**把"检查任务是否可认领"和"认领任务"这两步，放在同一把锁里原子完成。**

就像超市收银台，一次只能一个人结账。你在结账的时候，后面的人必须等。等你结完，下一个人才能开始。

这样，两个 AI 不可能同时认领同一个任务。先到的那个认领成功，后到的看到任务已经有人了，去找别的任务。

## 任务 ID 永远不重复

任务系统有一个小细节：任务 ID 是单调递增的，永远不会重复使用。

为什么这很重要？

假设任务 3 完成了，被删除了。如果下一个新任务也被分配了 ID 3，历史记录里就会出现混乱：同一个 ID 对应两个不同的任务，排查问题时完全对不上。

Claude Code 用一个"高水位标记"文件来解决这个问题：记录曾经分配过的最大 ID。新任务的 ID 永远比这个数字大，不管中间删除了多少任务。

## 一个 AI 崩溃了，它的任务怎么办

多个 AI 协作时，如果其中一个崩溃了，它正在做的任务会怎样？

如果没有处理机制，这些任务会永远卡在"进行中"的状态，其他 AI 看到有人在做，不去接手，但实际上没有人在做。整个系统陷入死锁。

Claude Code 的处理方式是：当检测到一个 AI 实例退出时，把它持有的所有未完成任务重置为"待认领"状态，并通知其他 AI 实例："某某 AI 退出了，它的任务 3 和任务 7 现在可以认领了。"

其他 AI 收到通知，去认领这些任务，继续完成。

## 通知失败了，任务状态不会回滚

任务系统里有一个设计原则：**通知失败，不影响任务状态的保存。**

当一个任务状态发生变化时，系统会通知相关的 AI 实例。但如果通知失败了（比如网络问题），任务状态已经写入文件了，不会因为通知失败而回滚。

为什么这样设计？因为任务状态的持久化是核心，通知只是"顺便告诉一声"。如果通知失败就回滚任务状态，那么任何一个 UI 的 bug 都可能导致整个任务系统不可用。

**核心操作不能被副效应绑架。**
