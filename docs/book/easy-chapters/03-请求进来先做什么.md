# 第3篇：一个请求进来，它先做什么

你在终端输入一条命令，按下回车。

在 Claude Code 真正开始"思考"之前，它已经做了很多事。这些事大多数人不会注意到，但它们决定了你等待的时间，也决定了系统的稳定性。

---

## 先判断你要做什么，再决定加载什么

Claude Code 启动的第一件事，不是初始化 AI，不是读取配置，而是看你输入的是什么。

如果你输入的是 `claude --version`，它会立刻打印版本号，然后退出。整个过程不读任何文件，不连接任何服务，几乎是瞬间完成的。

如果你输入的是 `claude ps`（查看后台任务），它也会走一条快速通道，直接处理完返回，不会启动 AI 系统。

只有当你真的要让它帮你做事——比如"帮我重构这段代码"——它才会走完整的启动流程。

这个设计的逻辑很简单：**你要的东西越轻，它加载的东西越少。** 不需要的东西，一概不加载。

## 为什么这样设计

想象一家餐厅。你走进去，只是问"今天有没有位置"，服务员不会先把所有菜单都拿出来，再问你要什么。他只需要看一眼预约本，告诉你有没有位置，你就可以走了。

Claude Code 的入口层做的是同样的事：先判断你要什么，再决定准备什么。

这不只是为了速度，也是为了稳定性。加载的东西越少，出错的可能性越小。一个只需要打印版本号的操作，不应该因为 AI 服务连接失败而报错。

## 在你等待的那几秒里

当你真的发出一个需要 AI 处理的请求时，Claude Code 会进入完整的启动流程。

在这个过程中，有一件事值得一提：它会提前触发一些"预热"操作。

比如，在 macOS 上，读取系统钥匙串（存储你的 API 密钥的地方）需要一点时间。Claude Code 会在加载其他模块的同时，提前发起这个读取请求，让两件事并行进行。等其他模块加载完，钥匙串也读好了，不需要额外等待。

这种"同时做多件事"的设计，让启动时间缩短了大约 65 毫秒。听起来不多，但如果你每天用几十次，积累起来就很可观了。

## 有一个细节很有意思

Claude Code 的代码里有一些"开关"，用来控制某些功能是否启用。

这些开关有一个特殊的要求：它们必须直接写在判断语句里，不能提取成变量。

原因是：这些开关是在编译时（代码打包的时候）就确定的，不是运行时才判断的。如果某个功能被关闭，编译器会直接把这段代码从最终产物里删掉，就好像它从来不存在一样。

但如果你把开关提取成一个变量，编译器就看不出来这个变量在运行时会是什么值，没办法做这个优化。被关闭的功能代码就会留在产物里，不仅增加体积，还会把未发布的功能暴露出来。

这是一个很小的细节，但它说明了一件事：**代码里的很多"奇怪写法"，背后都有具体的工程原因，不是随意为之。**

## 最终到达哪里

经过这一系列判断和准备，你的请求最终会到达一个叫做"查询循环"的地方。

这是 Claude Code 真正开始工作的地方——它在这里和 AI 模型交互，调用工具，处理结果，一轮一轮地推进，直到任务完成。

这个循环的细节，我们在第2篇已经聊过一部分（怎么判断任务完成），后面还会继续深入。
