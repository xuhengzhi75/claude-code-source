# 第5篇：它怎么记住你说过的话

你和 Claude Code 聊了很长时间，做了很多事。

中途你的电脑崩溃了，重启之后，它还记得你们做到哪里了吗？

答案是：大部分情况下，记得。

这背后有一套专门的设计。

---

## 状态和上下文，是两件不同的事

先区分两个概念，因为它们经常被混淆。

**上下文**是 AI 模型在当前这一轮能看到的信息：你说了什么、它回答了什么、工具返回了什么结果。这些信息存在内存里，进程结束就消失了。

**状态**是跨越多轮、甚至跨越多次会话都需要保留的信息：任务进行到哪一步了、有哪些重要的发现需要记住。这些信息存在文件里，进程重启后还在。

这个区别很重要。如果把所有东西都放在内存里，一旦崩溃，什么都没了。如果把所有东西都写到文件里，又太慢、太重。

Claude Code 的做法是：**该放内存的放内存，该落盘的落盘，两者分开管理。**

## 模型每轮能看到什么

每次 AI 模型开始处理你的请求，它能看到的信息分三层：

第一层是"背景信息"：当前的 Git 仓库状态、今天的日期、你的配置文件内容。这些信息在会话开始时就注入了，整个过程不变。

第二层是"对话历史"：你说过的每一句话、它回答的每一段内容、工具调用的每一个结果。这一层增长最快，对话越长，这里的内容越多。

第三层是"记忆文件"：把重要信息存到文件里，需要时再读取。这是跨会话保留信息的机制，不依赖对话历史的连续性。

## 崩溃之后为什么还能恢复

Claude Code 有一个设计原则：**先记录，再执行。**

当你发出一个请求，在 Claude Code 真正开始处理之前，它会先把你的请求写入磁盘上的对话记录文件。

这样，如果处理过程中进程崩溃了，你的请求已经在磁盘上了。重启之后，系统知道上次做到哪里，可以从那个点继续。

如果反过来——先执行，执行完再保存——那么崩溃发生在执行过程中时，这一步的输入就丢失了，恢复时不知道从哪里继续。

## 记忆更新是在后台悄悄做的

Claude Code 有一个"记忆更新"机制：在对话进行到一定程度时，它会在后台把重要信息整理并保存到记忆文件里。

这个过程是异步的，不会打断你正在做的事。

有意思的是，负责更新记忆的"子进程"权限被严格限制了：它只能编辑指定的记忆文件，不能做任何其他操作。

为什么要这么限制？因为如果记忆更新进程能访问任意工具，它就可能意外触发文件写入、命令执行，干扰主流程正在做的事。把"记忆更新"和"执行任务"严格分开，两件事互不干扰。

## 一个防止无限循环的小机制

上下文窗口有大小限制。如果对话太长，超出了限制，Claude Code 会尝试压缩历史记录，把不重要的内容删掉，腾出空间。

但有一种情况很危险：压缩之后还是太长，再次触发压缩，再次太长，再次压缩……无限循环，一直在烧 API 费用。

为了防止这种情况，Claude Code 记录了一个标志：是否已经尝试过压缩。如果压缩过一次还是不够，就不再尝试，而是让错误暴露出来，让你知道出了问题。

**宁可明显失败，也不陷入无声的无限循环。** 这是一个很实用的设计原则。
