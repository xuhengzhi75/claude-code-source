# 第2篇：它是怎么知道"任务还没完"的

你有没有注意到，Claude Code 在帮你做事的时候，不会做到一半就停下来说"好了"。它会一直做，直到真的做完为止。

这件事听起来理所当然，但背后有一个很有意思的设计选择。

---

## 普通 AI 工具怎么判断"完成了"

大多数 AI 工具的逻辑很简单：模型说"我做完了"，就停。

这在回答问题的时候没问题。你问"北京今天天气怎么样"，模型回答完，对话结束，完全合理。

但在执行任务的时候，这个逻辑有一个漏洞：**模型可能在任务没完成的时候，就以为自己完成了。**

比如你让它帮你检查 20 个文件，它检查了 5 个，觉得"差不多了"，就说"完成了"。这不是模型在撒谎，是它真的不确定自己还需不需要继续。

## Claude Code 的判断方式

Claude Code 不听模型说什么，它看模型做了什么。

具体来说，它看的是：**这一轮，模型有没有调用工具？**

如果有——比如读了一个文件、执行了一条命令、搜索了一段代码——那就说明任务还在进行中，继续下一轮。

如果没有，模型只是输出了一段文字，没有做任何操作，那才考虑停止。

这个区别很关键：工具调用是客观发生的事实，不是模型的主观判断。模型可能说错，但它要么调用了工具，要么没有，这是可以直接观察到的。

## 为什么这样更可靠

想象一个场景：你让 Claude Code 帮你重构一批文件，它正在逐个处理。

处理到第 8 个文件时，模型输出了一段总结，语气听起来像是"好了，我做完了"。

如果按"模型说完就停"的逻辑，任务就在这里结束了，剩下 12 个文件没有处理。

但 Claude Code 不看这段总结，它看：这一轮有没有工具调用？有——模型刚才读了第 8 个文件。那就继续，处理第 9 个。

直到某一轮，模型没有调用任何工具，只是输出了最终的总结，Claude Code 才判断任务完成。

## 这背后还有一个故事

这个设计不是一开始就有的。

早期版本的 Claude Code 曾经出现过另一个问题：任务卡住了，但系统不停，一直在循环。最极端的情况，一个会话里连续失败了 3000 多次，每次都在重试，每次都失败，白白浪费了大量资源。

后来的修复方案是：每种"出了问题还要继续"的情况，都设一个上限。上下文太长了，最多压缩一次；输出被截断了，最多恢复三次。超过上限，就让错误暴露出来，而不是无限重试。

这些上限的数字不是拍脑袋定的，是从真实的失败数据里算出来的。

## 这对你意味着什么

Claude Code 在执行任务时，你不需要担心它"做到一半就停"。它的停止条件是客观的，不依赖它自己的判断。

但反过来，如果你发现它一直在转、迟迟不停，也不要慌——它可能真的还在处理，或者遇到了需要你介入的问题。

判断一个 AI 工具是否适合做长任务，可以问：它的"完成"判断是基于客观事实，还是基于模型的自我声明？
