# 任务系统与恢复系统关系图（非技术导读）

> 这份文档回答一个常见问题：
> “任务在跑的过程中如果中断，系统怎么知道该从哪里接着来？”

## 一句话先懂

任务系统负责“谁在做什么、做到哪一步”；恢复系统负责“中断后如何把上下文和结构补齐”。两者配合，才能真正连续工作。

---

## 关系总图（纯文本）

```text
                    任务维度（Task）

        [创建任务] -> [任务JSON落盘] -> [认领/更新状态]
             |             |               |
             |             |               +-- 锁保护（.lock）
             |             +-- 高水位（.highwatermark，防ID回绕）
             |
             +-----------------------------------------+
                                                       |
                                                       v
                    会话维度（Conversation）

 [用户消息/工具结果] -> [transcript/history 持久化] -> [可resume加载]
                               |                         |
                               |                         +-- 反序列化修复结构
                               |                             （补齐中断痕迹、清理非法片段）
                               |
                               +-> [compact压缩后重建关键上下文]
                                     - boundary/summary
                                     - 附件重放（技能、计划、异步任务等）
                                                       |
                                                       v
                                          [继续下一轮执行]
```

## 两套系统各自做什么

## A. 任务系统（`src/utils/tasks.ts` 等）

它更像“任务白板 + 取号机 + 认领机制”：

- 任务以 JSON 文件形式存在（可读、可查、可恢复）。
- 通过 `.lock` 锁避免多人并发时改乱同一条任务。
- 通过 `.highwatermark` 保证任务编号不会因为重置而回绕复用。
- `getTaskListId()` 用于决定“这条任务属于哪个任务池”（团队/会话维度）。

**结果**：即使某个进程掉线，任务状态仍在磁盘上，不会“凭空消失”。

## B. 恢复系统（`conversationRecovery.ts` / `compact.ts` 等）

它更像“会话黑匣子 + 自动续飞”：

- `loadConversationForResume`：把可恢复会话加载回来。
- `deserializeMessagesWithInterruptDetection`：修复结构问题，比如未配对片段、异常中断痕迹。
- `restoreSkillStateFromMessages`：从历史消息里恢复技能相关状态。
- compact 后会按协议重建最小关键上下文（边界、摘要、附件）。

**结果**：恢复后不是“只剩摘要”，而是还能带着关键上下文继续工作。

## 任务—恢复联动的四种典型场景

### 场景 1：执行中断（例如进程被关闭）
- 任务侧：任务状态仍在 JSON 文件中。
- 会话侧：历史消息与 transcript 可被 resume 读取。
- 恢复侧：反序列化阶段会清理/补齐异常结构。

### 场景 2：上下文太长
- 会话侧：触发 compact，压缩旧内容。
- 恢复侧：保留 boundary/summary，并重放关键附件。
- 任务侧：任务本身不丢，因为状态在独立任务文件里。

### 场景 3：多人并发协作
- 任务侧：锁机制保证认领与状态更新的原子性。
- 会话侧：各自会话可独立恢复。
- 综合结果：减少“同一任务被多人重复接单”的风险。

### 场景 4：跨会话继续
- 任务侧：按 taskListId 找到同一任务池。
- 会话侧：按 sessionId 或历史链恢复对话。
- 综合结果：业务任务与对话记忆都能接续，而不是只恢复其中一半。

## 给非技术读者的判断标准（验收视角）

如果下面 3 条都成立，这套“任务-恢复联动”就算健康：

1. 中断后能找到“做过什么”（有任务文件和会话记录）。
2. 恢复后能继续推进“下一步”，而不是只能重看历史。
3. 并发下不会频繁出现状态互相覆盖或重复认领。
