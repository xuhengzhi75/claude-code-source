# 我们该从 Claude Code 学什么（以及先别急着抄什么）

> 目标读者：
> - 几乎不懂技术、但想理解“为什么它看起来很稳”的普通读者
> - 正在做业务 Agent 的产品/工程同学，想用最省钱省力的方式借鉴

---

## 先给结论（30 秒版）

如果你只记住一句话：

**先抄“工程秩序”，再抄“高级技巧”。**

这里的“工程秩序”指三件事：
1. 入口分流（快路径）
2. 统一能力池（命令/工具先全集、后过滤）
3. 可恢复主循环（失败后能继续，不是直接崩）

这三件事在源码里都很清楚，且 ROI（投入产出比）最高。

---

## 一、普通读者应该学什么

把 Claude Code 想成一家“高压值班中心”：

- **前台分流快**：简单请求走快通道，不拖累全系统。
- **中台规则清**：不是谁都能调用所有能力，要按身份和权限过滤。
- **后台能续命**：中途出错、超长、被打断，都有补救流程。

这就是它“看起来聪明”背后更关键的原因：

**它不是只靠模型回答得好，而是靠“出问题也不容易死”。**

---

## 二、做业务 Agent 的人：高性价比借鉴顺序（建议按这个来）

## 第 1 步（最先做）：入口分流，先把系统跑快

### 你该借鉴
- 给最常见、最轻量的操作做 fast-path
- 让重模块按需加载（dynamic import），不要开机全加载

### 源码依据
- `src/entrypoints/cli.tsx`
  - `main()` 里先判断 `--version`、`daemon`、`remote-control`、`--bg` 等路径
  - 常规路径最后才 `import('../main.js')`

### 为什么值
这是典型“小改动，大体感提升”：
- 冷启动更快
- 普通用户第一印象更好
- 还没动 AI 主逻辑就能见效

---

## 第 2 步：把命令/工具做成“先全集、后过滤”

### 你该借鉴
- 先维护一个理论可用全集
- 再按账号状态、权限、运行模式做过滤
- 保持一个稳定的组装入口，别散落在各处 if/else

### 源码依据
- `src/commands.ts`
  - `loadAllCommands(...)` + `getCommands(...)`
  - `meetsAvailabilityRequirement(...)`
  - `memoize` 做缓存，但可用性判断每次重算
- `src/tools.ts`
  - `getAllBaseTools()`
  - `filterToolsByDenyRules(...)`
  - `assembleToolPool(...)`（内建 + MCP 合并）

### 为什么值
这一步能明显降低“业务增长后系统变乱”的风险：
- 加新工具/新命令不容易破坏老行为
- 权限与风险边界更容易审计

---

## 第 3 步：定义统一工具契约（默认值要保守）

### 你该借鉴
- 给工具统一接口：输入、执行、权限、只读性、并发性
- 给默认值“偏保守”（比如默认不可并发、默认非只读）

### 源码依据
- `src/Tool.ts`
  - `TOOL_DEFAULTS`
  - `buildTool(...)`
  - `isReadOnly`、`canRunConcurrently` 等约束位

### 为什么值
避免“某个新工具少写了一段安全逻辑”导致线上事故。

---

## 第 4 步：把“可恢复循环”搭起来（这是长期稳定性的分水岭）

### 你该借鉴
- 明确主循环状态（State）
- 明确继续条件（是否有 tool_use 等）
- 明确失败恢复分支（超长、输出截断、fallback、补齐结果）

### 源码依据
- `src/QueryEngine.ts`
  - `for await (const message of query(...))`
  - `recordTranscript(...)`、`flushSessionStorage()`
  - `discoveredSkillNames.clear()`（轮级清理）
- `src/query.ts`
  - `type State` + `while (true)`
  - `needsFollowUp` 决定继续或收尾
  - `MAX_OUTPUT_TOKENS_RECOVERY_LIMIT`
  - `FallbackTriggeredError`
  - `yieldMissingToolResultBlocks(...)`
  - `tryReactiveCompact(...)`

### 为什么值
这一步决定你能不能从“demo 可用”变成“线上可托管”。

---

## 第 5 步（再往后）：任务协作与并发一致性

### 你该借鉴
- 用“文件 + 锁”先实现可见、可恢复、可审计的任务系统
- 对认领任务等竞争动作做原子化（check-then-act 在同一锁内）

### 源码依据
- `src/utils/tasks.ts`
  - `.highwatermark`（ID 不回绕）
  - `.lock`（列表级锁）
  - `claimTaskWithBusyCheck(...)`
  - `updateTask(...)`、`resetTaskList(...)`
  - `getTaskListId()`（协作归属）

### 为什么值
多 Agent 场景里，这比“再加一个 fancy 模型”更能降低事故率。

---

## 三、什么“不该急着抄”

下面这些不是不能学，而是**不适合一上来就重投入**：

1. **复杂恢复分支全量复刻**
   - 例如 prompt-too-long、max_output_tokens、fallback、compact 多机制并行。
   - 先做一条最小可恢复链路，再逐步补。

2. **全家桶式多运行形态**
   - daemon / bridge / bg sessions / SDK 全开会让系统复杂度陡增。
   - 业务早期先保一条主路径跑稳。

3. **过早做“超级插件生态”**
   - 没有稳定契约前，开放太多扩展点会导致长期维护债务。

一句话：

**先把“不会死”做好，再把“很强大”做出来。**

---

## 四、给非技术团队的落地版本（90 天）

- **第 1-2 周**：入口快路径 + 核心日志落盘
- **第 3-4 周**：命令/工具统一组装 + 权限过滤
- **第 5-8 周**：主循环状态机 + 基础恢复（至少 2 种常见错误）
- **第 9-12 周**：任务认领锁 + 可观测指标（恢复成功率、冲突率）

这套顺序的好处是：
- 每一阶段都能对业务产生可见收益
- 不需要先投入巨量底层研发
- 出问题时可定位、可回滚

---

## 五、验收标准（判断你是否“学对了”）

如果你借鉴后满足以下四条，方向基本正确：

1. 常见命令明显更快（冷启动和轻操作）
2. 权限错误从“莫名其妙”变成“可解释、可复现”
3. 会话中断后能恢复，不需要人工拼聊天记录
4. 多 Agent 不再频繁抢同一任务

---

## 六、最后一句

Claude Code 最值得学的，不是“某个神秘技巧”，而是它对现实问题的工程排序：

**先稳、再快、再强。**

对业务 Agent 团队来说，这正是最高性价比的方法论。
