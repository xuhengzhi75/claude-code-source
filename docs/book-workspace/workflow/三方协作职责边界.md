## ⚠️ 安全与隐私规则（强制，不可绕过）

**本仓库为 GitHub 公开仓库，任何推送内容对互联网公开可见。**

小许在执行任何 push 操作前，必须确认以下内容未被包含在提交中：

### 绝对禁止推送的内容

| 类别 | 具体示例 |
|---|---|
| API Key / Token | GitHub token、OpenClaw 密钥、SSO token、任何 Bearer token |
| 内部系统账号信息 | misId、员工 UID、内网账号密码 |
| 美团内部 URL | sankuai.com 内网链接、内部系统地址 |
| 公司内部文件 | ONES 工作项内容、学城文档正文、内部 PRD、内部邮件 |
| 个人隐私信息 | 手机号、身份证号、家庭住址等 |
| 沙箱/设备配置 | sandboxId、device-auth.json 内容、openclaw.json 内容 |

### 执行规则

1. **每次 push 前自查**：检查本次 diff，确认无上述内容
2. **遇到边界情况先停下**：不确定某段内容是否敏感时，先问 Gordon，不推
3. **已推送敏感内容的处理**：立即通知 Gordon，不要尝试自行处理（git 历史需要 force push 清除，需 xuhengzhi75 操作）
4. **内部链接替换原则**：如章节中需引用内部资料作为背景，改用公开描述或匿名化后再写入

### 协作内容的来源审查

小许在执行写作/润色任务时，如果 Gordon 提供的原始材料中包含内部信息：
- 提取技术观点和方法论，丢弃内部上下文
- 不将原始材料的文字直接写入仓库文件
- 如有疑问，先确认哪些内容可以公开

## 边界裁定规则

当职责边界不清时，按以下优先级处理：

1. **技术准确性问题** → 以 xuhengzhi75 的源码判断为准
2. **内容方向问题** → Gordon 决策
3. **格式/工程问题** → 小许自行处理，事后通知
4. **有争议的章节改动** → 不改，先在此文档补充边界，再推进
