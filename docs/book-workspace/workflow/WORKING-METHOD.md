# WORKING METHOD

## 目标
把这个仓库作为一个长期的源码拆解、书稿沉淀与协作写作工程来推进，而不是零散写文档。

## 一、源码阅读方法
默认按以下顺序阅读：

1. 先立总判断：这套系统真正强在哪
2. 再看控制权链路：谁把控制权交给谁
3. 再看能力边界：这次会话到底开放了什么能力
4. 再看主循环：何时继续、何时结束、何时恢复
5. 最后看边界 case：哪里最容易断、为什么要这么补

### 源码阅读铁律
- 先按控制权转移读，不按目录树平铺读
- 每个关键结论都要区分 `verified` / `inference`
- 先找不可删组件，再谈功能细节
- 优先写系统矛盾与设计取舍，少写功能说明书
- 每章至少要产出一个非显然结论

## 二、写作原则
- 结论先行，不写“本章将……”式预告
- 信息密度优先，不靠同义重复撑篇幅
- 每段至少一个新信息单元 + 一个可复核锚点
- 推断必须明确标注，不能写成已证实事实
- 少写模块罗列，多写系统级解释

## 三、分析侧 → 写作侧交接规则
分析侧交付至少包含：
1. 一句话结论
2. 本章最该讲的 3~5 个点
3. 不要重复讲的点
4. 关键源码锚点（文件 + 函数/分支/对象）
5. 哪些判断只能写成 inference
6. 章节边界（本章讲到哪，不讲到哪）

## 四、协作规则
- 源码分析侧不直接抢写正文
- 写作侧不重复深挖同一批源码
- 开工前先 pull
- 完成一点就 push
- 优先轮动，不攒大包

## 五、验收方法
一个源码解读要通过，至少满足：
1. 证据纪律达标
2. 有系统级解释
3. 至少两条非显然结论
4. 信息密度合格
5. 能说明设计取舍与边界 case

## 六、项目推进原则
- 先把一轮主干做完并验收，再并行开后续章节
- 不在已达标章节上无限打磨空转
- 定期落盘记忆，定期压缩上下文
- 关键规划、规则、分工必须写入仓库文档
