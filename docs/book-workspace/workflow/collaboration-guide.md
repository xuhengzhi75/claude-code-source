# 协作规范指南（collaboration guide）

更新时间：2026-04-01
适用范围：`docs/book/` 写作与源码分析协作

---

## 1. 目标与原则

本指南用于统一“源码分析侧 + 写作侧”协作方式，避免重复劳动与冲突改写。

核心目标：
- **分工清晰**：谁产出分析材料，谁落章节正文。
- **轮动推进**：分析和写作交替进行，小步快跑。
- **接口明确**：交接时信息完整，写作侧可直接开写。
- **远端优先**：`origin/main` 始终是最新协作基线。

---

## 2. 分工边界（谁做什么）

### 2.1 源码分析侧（Analysis）
负责：
- 阅读指定源码，抽取模块职责、关键流程、边界条件、恢复逻辑、设计取舍。
- 产出高价值注释与 `architecture-notes/*.md`。
- 给写作侧提供可直接使用的“交接卡”（结论 + 锚点 + 边界）。

不负责：
- 不直接写章节正文（`docs/book/chapters/*.md`）。
- 不与写作侧并行改同一章节正文。

### 2.2 写作侧（Writing）
负责：
- 基于 analysis notes、源码注释、requirements 写章节正文。
- 统一叙述风格与信息密度，提升可读性。
- 维护 `references/references/chapter-evidence-map.md` 对证据链进行映射。

不负责：
- 不重复深挖已明确由分析侧覆盖的同一批源码。
- 不绕开分析结论直接拍板高风险技术判断。

---

## 3. 轮动机制（默认节奏）

采用固定轮动：
1. 分析侧完成一个主题分析并 push。
2. 写作侧 pull 后据此写对应章节并 push。
3. 分析侧再 pull，进入下一个主题。
4. 周期重复（分析 → 写作 → 分析 → 写作）。

执行要求：
- 每次只推进一个清晰主题，不攒“大包”。
- 每完成一小段可独立审阅的成果就提交并推送。

---

## 4. 交接接口（handoff contract）

分析侧向写作侧交付时，至少包含以下 6 项：

1. **一句话结论**（标注 `verified` / `inference`）
2. **本章最该讲的 3~5 点**（机制、边界、恢复/治理、取舍）
3. **不要重复讲的点**（避免同义重复和章节重叠）
4. **关键源码锚点**（`路径::函数/分支/对象`）
5. **章节边界**（本章覆盖与不覆盖范围）
6. **写法约束**（哪些只能保守推断）

建议直接使用模板文件：
- `docs/book-workspace/workflow/analysis-handoff-template.md`

---

## 5. Pull/Push 纪律（强制）

### 5.1 开始前
- 必须先执行：`git pull --rebase origin main`
- 若存在冲突，先解决冲突并确认基线后再开工。

### 5.2 进行中
- 单主题小步提交，提交信息建议：
  - `docs(book): <简短说明>`
  - `docs(code): <简短说明>`

### 5.3 完成后
- 必须立即执行：`git push origin main`
- 禁止在本地长时间积压未推送成果。

纪律底线：
- **远端优先于本地**；
- **先 pull 再改，改完就 push**。

---

## 6. 何时可以开写（写作侧准入条件）

满足以下条件即可开写：
1. 对应主题已在 `planning/planning/completed-coverage-and-writing-queue.md` 标记为可写（优先级 A/B/C）。
2. 已收到分析侧交接卡（至少包含结论、锚点、边界）。
3. 证据来源可追溯到 `architecture-notes` 或源码锚点。
4. 章节依赖的前置内容已可引用（无需补新分析）。

建议优先顺序：
- 先写优先级 A（证据最扎实），再写 B（明确 inference），最后推进 C（新章节）。

---

## 7. 何时不要抢活（禁止并行抢占）

以下场景**不要开写/不要抢改**：
1. 分析侧正在处理同一主题，尚未交付交接卡。
2. 章节结论依赖新的源码判断，但 analysis notes 尚未覆盖。
3. 同一文件已有对方进行中的改写（未 push/未合并）。
4. 只为“赶进度”而重复做已完成分析或重写既有结论。
5. 需要跨章节重构边界但未先对齐协作规则。

处理方式：
- 先在协作文档补充边界与接口，再进入正文改写；
- 不能确认责任归属时，默认“分析先行、写作后置”。

---

## 8. 冲突预防与升级处理

冲突预防：
- 同主题只允许一侧主导；另一侧仅消费其交付。
- 任何人启动新任务前必须 pull 最新远端。
- 发现边界不清时，先更新协作文档，不直接改正文。

升级处理（出现争议时）：
1. 以源码锚点和 evidence map 为准；
2. 无证据结论降级为 inference；
3. 暂缓定稿，回到分析侧补证据后再写。
