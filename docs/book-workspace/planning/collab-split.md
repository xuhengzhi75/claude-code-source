## 协作模式（重要，请先读这里）

**固定分工，不重复作业：**

1. **我（源码分析侧）**
   - 负责深入阅读指定源码文件
   - 负责在源码里补高价值注释，或补 `architecture-notes/` 技术笔记
   - 负责给出：模块职责、关键函数、边界条件、设计意图、源码锚点、写作注意事项
   - **不直接写章节正文**

2. **对方（写作侧）**
   - 负责根据源码注释、`architecture-notes/` 笔记和 `planning/planning/requirements.md` 写章节正文
   - 负责统一风格、控制信息密度、处理读者可读性
   - 负责维护 `references/references/chapter-evidence-map.md`
   - **不重复做源码深挖，不与源码分析侧抢同一份工作**

3. **轮动机制**
   - 我先分析一个主题并落注释/笔记 → 立即 push
   - 对方拉取后据此写对应章节 → 立即 push
   - 我再拉取远端，分析下一主题 → 立即 push
   - 双方按“分析 → 写作 → 分析 → 写作”轮动，不等大包攒完再一起交

这个模式的目标只有一个：
**避免两边重复作业，同时保持远端持续有最新进展。**

---

## 协作铁律（必须遵守）

1. **每完成一点工作，立即 push 远端**
   - 不攒一大堆再推
   - 远端始终作为最新协作基线

2. **开始新任务前先 pull 远端**
   - 任何一方开始写之前，先同步 `origin/main`
   - 避免在旧基线上重复劳动或产生冲突

3. **源码分析侧不写正文，写作侧不深挖同一批源码**
   - 防止两边都在写同一件事

4. **章节和技术笔记分开维护**
   - 源码分析成果优先落在：源码注释、`architecture-notes/`、必要的协作文档说明
   - 正文章节优先由写作侧维护

5. **有不确定边界时，优先追加协作文档，不抢写文件**
   - 宁可先写清分工和接口，也不要两边直接改同一块内容

## 写作接口约定（我给对方的提示模板）

当我交付一个新主题给写作侧时，默认应附带以下信息：

1. **一句话模块结论**
2. **本章最该讲的 3~5 个点**
3. **不要重复讲的内容**
4. **哪些判断只能写成 inference**
5. **关键源码锚点（文件 + 函数/分支/对象）**
6. **建议章节边界：这章讲到哪，不要越到下一章**

这能让写作侧直接进入写作，不需要再重新猜模块边界。

## 冲突预防规则

1. 我不直接改对方正在写的章节正文。
2. 对方不重复重做我已经给出的源码分析与注释。
3. 若同一主题需要双方都接触，优先由我先产出分析材料，对方后写。
4. 每次开始前先 `git pull`，每完成一小步就 `git push`。
5. 提交信息格式：`docs(book): <简短说明>` 或 `docs(code): <简短说明>`。

---

## 写作风格提醒（给写作侧）

1. 每章开头先给结论，不写“本章将……”式预告。
2. 控制信息密度，不能靠同义重复撑字数。
3. 源码引用写成 `` `src/xxx.ts` ``，尽量带函数/分支名。
4. 推断性判断要明确写成保守推断，不写成已证实事实。
5. 不重复分析侧已经写清的技术细节，把重点放在读者能读懂的表达与结构收束上。
