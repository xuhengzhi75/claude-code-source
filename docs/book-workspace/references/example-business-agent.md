# Example Business Agent v0

## 目标
做一个“运营分析 Agent”最小版本，用来处理日常业务分析问题，而不是追求通用万能 Agent。

## 为什么选这个方向
- 输入和输出边界相对清楚
- 可以从读取数据、整理问题、生成分析结论开始
- 容易嵌入真实工作流
- 很适合作为 Claude-Code-like 架构的第一种业务落地样本

## 最小模块
1. `entrypoint`
   - 区分交互式提问、批量任务、定时任务
2. `capability assembly`
   - 暴露有限工具：读文件、读表格、执行 SQL、生成报告
3. `query engine`
   - 管理消息历史、上下文、会话状态
4. `query loop`
   - 执行“理解问题 → 取数 → 分析 → 生成结论 → 必要时继续”
5. `task system`
   - 跟踪长时间分析任务、报表生成、失败重试
6. `recovery`
   - 记录中间结果和任务状态，支持中断后续跑

## 最小工具集合
- `read_data`
- `run_sql`
- `summarize_table`
- `write_report`
- `notify_result`

## 一条最小执行链
1. 用户提出一个运营问题
2. 系统判断是即时问答还是后台任务
3. Agent 读取相关数据或执行 SQL
4. 将结果写回当前循环
5. 如果信息不足，继续追问或追加取数
6. 形成最终分析结论和建议
7. 若为长任务，则落任务记录与结果文件

## 第一版不要急着做的事
- 不要一开始就接太多外部系统
- 不要急着做复杂多 Agent 编排
- 不要先做花哨 UI
- 不要先追求全自动闭环

## 第一版成功标准
- 能稳定处理 3 类常见运营分析问题
- 任务状态可追踪
- 中断后能恢复
- 结果能落到固定位置并可回看
