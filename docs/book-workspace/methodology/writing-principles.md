# 写作原则与验收标准

更新时间：2026-04-01

本文件是对照三个高质量参考来源（tvytlx/ai-agent-deep-dive、zread.ai/instructkr/claude-code、微信公众号深度解读）做的方法论总结，作为本书所有章节的写作原则和验收标准。

---

## 一、差距诊断：我们的内容为什么平平无奇

对比之后，问题集中在三层：

**第一层：在描述架构，而不是揭示机制**

我们写的是"工具池采用先全集后过滤"——这是对代码结构的转述，读者看完知道"有这个东西"，但不知道"为什么这样设计、不这样会怎样"。

zread.ai 的写法是：`getSystemPrompt()` 不是文本，而是编排器——静态前缀适合 cache，动态后缀按会话条件注入。这一句话背后有一个具体的设计决策：**把 prompt 分成可缓存和不可缓存两段，是为了降低 API 成本**。这是机制，不是描述。

**第二层：没有挖到真正有价值的盲区**

tvytlx 挖的是读者自己看不到的东西：`AgentTool` 的 prompt 里写了"何时不该用 AgentTool"、`Verification Agent` 的核心是主动打破实现而不是复查——这些是反直觉的、有信息增量的发现。

我们写的内容，读者自己看一遍架构笔记就能得到，没有给出"你不看这篇就不会知道"的东西。

**第三层：章节没有观点，只有陈述**

好的技术文章有一个核心主张，然后用证据支撑它。zread.ai 的核心主张是：**Claude Code 是一个 Agent Operating System，不是 CLI 包装器**——然后用入口层、命令系统、工具层、prompt 架构逐一证明这个判断。

我们的章节是平铺的：这一节讲工具，下一节讲状态，再下一节讲恢复。没有一个贯穿全章的核心主张，读者读完不知道"这章想让我记住什么"。

## 二、高质量参考来源的方法论拆解

### zread.ai/instructkr/claude-code 的写法

**特点一：每个子系统都有精确的行号锚点**

不是"QueryEngine 管理会话"，而是：
> `submitMessage()` 在进入 Agent 循环之前执行确定性的设置流水线，步骤包括：权限包装（`QueryEngine.ts#L244-L271`）、系统提示词构建（`L288-L325`）、用户输入处理（`L410-L428`）、会话记录持久化（`L450-L463`）。

每个结论都有可验证的行号。读者可以自己去核对。

**特点二：揭示非显而易见的设计决策**

> `ProcessUserInputContext` 对象会被创建两次：一次在斜杠命令处理之前（带有可变的 `setMessages` 回调，以便 `/force-snip` 等命令可以重写历史记录），另一次在处理之后（带有空操作的 `setMessages`，因为除了循环之外不再期望进一步的变更）。

这不是"有两个对象"，而是解释了**为什么要创建两次**——第一次允许命令修改历史，第二次锁定历史防止意外修改。这是设计意图，不是代码描述。

**特点三：记录真实踩坑和防护机制的来历**

> `hasAttemptedReactiveCompact` 防护机制会在停止钩子重试循环中保留。之前在停止钩子阻断错误后将其重置为 `false` 曾导致无限循环：压缩 → 仍然过长 → 错误 → 停止钩子阻断 → 压缩 → … 烧掉数千次 API 调用。

这一段说明了一个 flag 的存在不是随意的，而是修复了一个真实的无限循环 bug。这种"这个设计背后有一个真实事故"的叙述，比任何架构描述都更有说服力。

**特点四：用表格呈现状态机的完整决策路径**

不是"有多种恢复机制"，而是：

| 恢复方式 | 触发条件 | 行为 |
|---------|---------|------|
| 上下文折叠排空 | 保留的提示词过长错误 + 存在已暂存的折叠 | 排空所有已暂存的上下文折叠并重试 |
| 反应式压缩 | 保留的提示词过长或媒体大小错误 | 派生压缩子 Agent 摘要历史记录 |
| Max-output-tokens 提升 | max-output-tokens 错误 + 默认上限生效 | 使用提升后的 64k token 限制重试 |
| Max-output-tokens 恢复 | max-output-tokens 错误 + 提升已耗尽 | 注入元消息指示模型从中途恢复 |

完整的决策路径，每条路径的触发条件和行为都是精确的。

### tvytlx/ai-agent-deep-dive 的写法

**特点一：用一个强主张统领全文**

> 这类成熟 Agent 产品的价值，不是一段 prompt，而是一整套把 prompt、tool、permission、agent、skill、plugin、hook、MCP、cache 和产品体验统一起来的 **Agent Operating System**。

这是一个可以被反驳的主张，不是一个无聊的陈述。读者会想知道：凭什么这么说？然后作者用后面的内容逐一证明。

**特点二：挖反直觉的发现**

> `AgentTool` 的 prompt 里明确写了"何时不该用 AgentTool"。

这是反直觉的——一个工具的 prompt 里写了"不要用我"的条件。这种发现让读者觉得"如果不看这篇，我永远不会注意到这个细节"。

**特点三：把 prompt 当作工程产物分析**

> `getSystemPrompt()` 里最核心的结构，是先构造静态部分，再加上动态部分。静态前缀更适合 cache，动态后缀按会话条件注入。

这不是"prompt 很长"，而是分析了 prompt 的工程结构：为什么要分静态和动态，分别服务什么目的。

## 三、写作原则（每章必须遵守）

### 原则一：每章必须有一个可被反驳的核心主张

不是"本章介绍工具系统"，而是"工具系统的设计说明了 Claude Code 把安全边界当成一等公民，而不是事后补丁"。

核心主张的检验标准：能不能被反驳？如果不能被反驳（比如"工具系统很重要"），就不是主张，是废话。

### 原则二：每个结论必须有精确的源码锚点

不是"QueryEngine 管理会话状态"，而是"QueryEngine 在 `L450-L463` 处，在 API 调用之前将用户消息写入会话记录——这意味着即使进程在请求中途被终止，`--resume` 也能正常工作"。

锚点的检验标准：读者能不能根据这个锚点自己去验证？如果不能，锚点不够精确。

### 原则三：必须揭示非显而易见的设计决策

每章至少要有一个"如果不看这篇，读者不会注意到"的发现。这个发现可以是：
- 一个 flag 的存在修复了一个真实 bug
- 一个函数被调用两次，两次的目的不同
- 一个工具的 prompt 里写了"不要用我"的条件
- 一个看起来简单的设计背后有复杂的权衡

### 原则四：用机制解释，不用结构描述

"先全集后过滤"是结构描述。"先全集后过滤的好处是：权限规则集中在一处，不散落在各个工具实现里，新增工具时不需要记得在每个地方加权限检查"——这是机制解释。

机制解释的检验标准：读者看完能不能回答"为什么这样设计，不这样会怎样"？

### 原则五：记录真实的边界情况和防护机制

每个防护机制的存在都有原因。`hasAttemptedReactiveCompact` 不是随意加的 flag，是修复了无限循环 bug。`migrateLegacyAttachmentTypes` 不是随意加的函数，是因为历史上改过数据格式。

这些"为什么存在"的解释，比"这个函数做了什么"更有价值。

### 原则六：完整的决策路径，不是部分列举

如果一个系统有 4 种恢复机制，就要把 4 种都写清楚，包括每种的触发条件和行为。不能只写"有多种恢复机制"，也不能只写其中 2 种。

## 四、验收标准（每章发布前必须通过）

### 第一组：基础验收（原有六条）

**标准一：核心主张测试**
能不能用一句话说出这章的核心主张？这个主张能不能被反驳？如果不能，重写。

**标准二：信息增量测试**
读者如果只看架构笔记，能不能得到这章的全部内容？如果能，这章没有信息增量，重写。

**标准三：锚点精确性测试**
每个关键结论是否有精确的源码锚点（文件名 + 函数名，最好有行号）？如果没有，补充或降级为推断。

**标准四：机制完整性测试**
对于每个"是什么"，是否解释了"为什么这样，不这样会怎样"？如果没有，补充机制解释。

**标准五：反直觉发现测试**
这章有没有至少一个"如果不看这篇，读者不会注意到"的发现？如果没有，说明挖得不够深。

**标准六：删减测试**
把这章的 20%-30% 删掉，读者的理解会不会受损？如果不会，说明有冗余，删掉那部分。

---

### 第二组：结构级优化验收

结构级优化要求每章在逻辑骨架上达到以下标准，而不只是语言层面的改写。

**标准七：第一性问题测试**
这章是否明确回答了"本章解决的第一性问题是什么"？第一性问题不是"介绍 X 系统"，而是"X 系统存在是为了解决什么根本矛盾"。

检验方法：把章节标题替换成"为什么需要 X"，看正文能不能直接回答这个问题。如果不能，说明章节没有从第一性问题出发，而是从结构描述出发。

示例（不合格）：第8章介绍任务系统的设计。
示例（合格）：第8章回答：为什么对话循环不足以承载长任务——以及任务对象化如何解决这个矛盾。

**标准八：隐含假设显式化测试**
每个架构决策背后都有隐含假设。这章是否把关键决策的隐含假设说出来了？

隐含假设的典型形式：
- "这个设计假设系统是单机运行的"
- "这个设计假设任务失败是偶发的，不是常态"
- "这个设计假设用户不会同时发起超过 N 个并发任务"

如果假设不成立，设计就会失效。把假设说出来，读者才能判断这个设计在自己的场景下是否适用。

**标准九：替代方案对比测试**
这章是否分析了至少一个替代方案，并说明了为什么没有选它？

替代方案分析的格式：
- 替代方案是什么
- 它在什么场景下更好
- Claude Code 为什么没有选它（成本、复杂度、场景不匹配等）

没有替代方案对比的设计分析，读者无法判断这个设计是"唯一正确答案"还是"在特定约束下的合理选择"。

**标准十：可迁移原则抽象测试**
这章是否从具体实现中抽象出了至少一条可迁移的设计原则？

可迁移原则的格式：
- 原则名称（一句话）
- 适用条件（什么场景下这条原则成立）
- 反例（什么情况下这条原则不适用）

示例：
- 原则：副作用操作应在独立的最小权限上下文中执行
- 适用条件：副作用操作的影响范围可以被明确界定
- 反例：如果副作用操作需要读取主上下文的大量状态，隔离的成本可能超过收益

**标准十一：技术债与边界条件测试**
这章是否指出了当前设计的技术债或边界条件？

技术债的典型形式：
- "这个方案在 X 场景下会成为瓶颈"
- "这个设计假设 Y，如果 Y 不成立，需要重新设计"
- "这里有一个已知的 Z 问题，当前用 W 绕过，但不是根本解决"

边界条件的典型形式：
- "当并发数超过 N 时，文件锁方案会退化"
- "当上下文超过 M token 时，压缩策略的语义保留率会下降"

### 第三组：模型级优化验收

模型级优化要求每章在认知模型层面达到以下标准。

**标准十二：心智模型测试**
读者读完这章，能不能在脑子里建立一个可以用来预测系统行为的心智模型？

心智模型的检验方法：给读者一个没有在章节中出现过的场景，看他们能不能用这章建立的模型推断出系统的行为。

示例：读完 ch07（query loop），读者应该能回答：如果模型在一轮中产生了 3 个工具调用，但第 2 个工具调用失败了，系统会怎么处理？

如果读者无法回答这类问题，说明章节只传递了知识，没有传递模型。

**标准十三：抽象层级一致性测试**
这章是否在同一个抽象层级上讨论问题，而不是在"架构设计"和"代码细节"之间跳跃？

不合格的例子：一段讨论"为什么需要任务系统"（架构层），下一段突然讨论"`.highwatermark` 文件的格式"（实现层），再下一段又回到"任务系统的设计原则"（架构层）。

合格的做法：先在架构层把问题说清楚，再下沉到实现层提供证据，最后回到架构层总结。层级切换要有明确的信号（"在实现层，这体现为……"）。

**标准十四：信息密度测试**
删除所有过渡句、铺垫句、总结重复句之后，章节的核心信息量是否减少？如果不减少，说明这些句子是冗余的，应该删除。

冗余句的典型形式：
- "这一点非常重要"（重要性应该由内容本身体现，不需要声明）
- "下面我们来看……"（直接进入内容）
- "综上所述，……"（如果前文逻辑清晰，不需要总结重复）
- "这个设计很有意思"（读者自己判断）

---

### 第四组：对抗级优化验收

对抗级优化要求每章能经受住"挑剔读者"的质疑。

**标准十五：强反驳测试**
这章的核心主张，能不能被一个合理的反驳推翻？如果能，说明主张不够严谨，需要加限定条件或修改主张。

强反驳的构造方法：找到一个真实存在的反例，看它是否推翻了主张。

示例：
- 主张："Claude Code 把连续执行当成一等公民"
- 强反驳："很多聊天工具也支持多轮对话，这不就是连续执行吗？"
- 应对：需要在章节中明确区分"多轮对话"和"任务连续执行"的本质差异，否则主张不成立

**标准十六：过度泛化测试**
这章的结论是否过度泛化？即，是否把 Claude Code 的特定场景约束下的设计，说成了普遍适用的真理？

过度泛化的典型形式：
- "Agent 系统都应该用文件系统存储任务状态"（实际上只在单机场景下合理）
- "先全集后过滤是工具管理的最佳实践"（实际上在工具数量极少时，这个模式的收益不明显）

合格的做法：每个结论都带上适用条件，让读者能判断这个结论在自己的场景下是否成立。

**标准十七：论据充分性测试**
这章的每个主张，是否有足够的论据支撑？论据的充分性标准：至少有一个源码锚点（直接证据）+ 一个机制解释（为什么这个证据支持这个主张）。

只有源码锚点没有机制解释：读者知道"有这个代码"，但不知道"为什么这个代码说明了主张"。
只有机制解释没有源码锚点：读者知道"理论上应该这样"，但不知道"实际上是不是这样"。

**标准十八：认知负荷测试**
读者在读完这章之后，需要记住的核心概念不超过 3 个。如果超过，说明章节试图在一章内解决太多问题，需要拆分或聚焦。

核心概念的定义：读者在读完这章一周后，还能清楚描述的概念。

## 五、需要补写的内容（当前缺口）

对照高质量参考来源，当前书稿有以下明显缺口：

**缺口一：prompt 系统没有专章**

`src/constants/prompts.ts` 的模块化装配架构是整个系统最有价值的部分之一，但当前书稿完全没有覆盖。需要新增一章专门分析：静态/动态分段的设计意图、各个 section 的注入条件、为什么这样设计而不是一个大字符串。

**缺口二：特性门控（feature flag）机制没有分析**

Bun 的 `feature()` 函数用于构建时死代码消除，这是一个非常有价值的工程决策——未激活的代码从产物中彻底剥离，保持生产构件精简。当前书稿完全没有提到这个机制。

**缺口三：多 Agent 协调器模式没有深入**

`coordinator/` 目录实现了 leader/worker 模式，工作节点通过邮箱系统将权限请求转发给主导节点。这是 Claude Code 支持真正多 Agent 协作的关键机制，当前书稿只是一笔带过。

**缺口四：现有章节缺乏行号级锚点**

当前书稿的源码引用只到文件名和函数名，没有行号。这让读者无法快速定位和验证。需要在修订时补充关键结论的行号锚点。

**缺口五：没有记录任何真实的 bug 修复或设计权衡**

`hasAttemptedReactiveCompact` 这类防护机制背后的真实事故，是最有说服力的内容。当前书稿完全没有这类叙述。

## 七、三级优化的操作定义

每章修订时，按以下三个层级依次检查：

### 结构级优化

目标：让章节的逻辑骨架能独立成立，不依赖读者的背景知识。

操作清单：
- [ ] 明确写出本章解决的第一性问题（不是"介绍 X"，而是"解决 Y 矛盾"）
- [ ] 把关键架构决策的隐含假设显式化（"这个设计假设……"）
- [ ] 分析至少一个替代方案，说明为什么没有选它
- [ ] 从具体实现中抽象出至少一条可迁移的设计原则
- [ ] 指出当前设计的技术债或边界条件

### 模型级优化

目标：让读者读完能建立可以预测系统行为的心智模型，而不只是记住知识点。

操作清单：
