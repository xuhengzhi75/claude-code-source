# Writing Guide（统一入口）

更新时间：2026-04-02

本文件是书稿“写作规范层”的统一入口。目标是把分散在 `planning/planning/requirements.md`、`methodology/methodology/writing-principles.md`、`WORKING-METHOD.md` 的规则收敛成一套可执行、可验收、可协作的最小工作协议。

---

## 0. 这本书要写成什么样

一句话目标：**让非技术读者看懂 Claude Code 的系统级能力，并能迁移到自己的业务 Agent。**

核心约束：
- 读者导向：默认读者“可复制命令，但不深懂代码”。
- 机制导向：解释“为什么这样设计，不这样会怎样”。
- 证据导向：关键结论要能被源码复核。
- 简洁导向：高信息密度，禁止同义反复凑篇幅。

参考定位详见：`docs/book-workspace/planning/requirements.md`

---

## 1. 五条写作红线（必须遵守）

### 1) 信息密度优先
- 每段至少提供一个新信息单元（机制、边界、证据、反例、取舍之一）。
- 能删 20%~30% 且不影响理解，视为不合格。
- 少过渡废句，少“重要的是…/下面来看…/综上所述…”。

### 2) `verified` / `inference` 严格分离
- **verified**：有源码锚点（文件 + 函数/分支/对象，关键点尽量带行号）。
- **inference**：基于证据的推断，必须显式标注，措辞保守。
- 不允许把推断写成事实。

### 3) 章节边界明确
每章开工前先写清三件事：
- 本章第一性问题（要解决的根本矛盾）
- 本章覆盖范围（讲到哪）
- 本章明确不讲（避免与相邻章节重叠）

### 4) 避免重复
- 不重复讲已达标章节的结论；只补“新增机制/新增证据/新增边界”。
- 不做目录树式模块罗列；只保留能支撑主张的最小结构信息。

### 5) 读者导向
- 用读者可理解的语言解释机制，但不牺牲正确性。
- 不靠夸张、比喻、情绪化措辞提高可读性。
- 始终区分“模型能力”与“系统能力”。

---

## 2. 标准章节骨架（推荐模板）

1. **核心主张（一句话，可被反驳）**
2. **第一性问题（为什么需要这个机制）**
3. **机制展开（关键决策 + 为什么不是替代方案）**
4. **证据锚点（verified）**
5. **边界与技术债（何时失效/何时退化）**
6. **可迁移原则（适用条件 + 反例）**
7. **本章不覆盖项（章节边界）**

> 骨架目的：保证“观点—证据—边界”闭环，而非平铺信息。

---

## 3. 章节验收清单（发布前自检）

### A. 基础通过线
- [ ] 有且仅有一个清晰核心主张（可被反驳）
- [ ] 关键结论具备源码锚点；无法验证处已标注 inference
- [ ] 解释了“为什么这样设计，不这样会怎样”
- [ ] 至少 1 个非显然发现
- [ ] 删除冗余后信息量未被掏空

### B. 质量增强线
- [ ] 说明了至少一个替代方案及未采纳原因
- [ ] 写明关键假设与适用条件（避免过度泛化）
- [ ] 指出边界条件或技术债
- [ ] 读者读完后需记住的核心概念 ≤ 3

详细高阶标准详见：`docs/book-workspace/methodology/writing-principles.md`

## 4. 协作与交接协议（分析侧 ↔ 写作侧）

分析侧交付最少包含：
1. 一句话结论
2. 本章最该讲的 3~5 点
3. 明确不要重复讲的点
4. 关键锚点（文件 + 函数/分支/对象）
5. 仅能写为 inference 的判断
6. 章节边界（讲到哪/不讲哪）

协作节奏：
- 开工前先 pull
- 完成一小段可验收成果就 push
- 轮动推进，不攒大包

详见：`docs/book/WORKING-METHOD.md`

---

## 5. 三个常见失败模式（写作时优先规避）

1. **结构转述型失败**：只说“有什么模块”，不解释“为何存在”。
2. **证据断裂型失败**：有观点无锚点，或有锚点无机制解释。
3. **边界失焦型失败**：章节之间互相抢内容，导致重复与信息稀释。

## 7. 可读性标准（2026-04-02 确认）

目标：有知识量、有趣、没有 AI 味道、扫地阿姨也能读懂。

**四条操作规则：**

1. **比喻开头，说一遍就走。** 每个技术概念先给一个生活化类比（后厨、工厂、煤气罐），类比讲完立刻进技术，不再重复解释。不要用类比验证已经说清楚的事。

2. **不重复解释同一个道理。** 说过一遍的逻辑不在后文改写一遍。段落结尾不做"也就是说……"式的总结，读者自己能得出结论。

3. **去掉 AI 味的具体动作：**
   - 不用破折号（`——`）
   - 不用"不是……而是……"句式
   - 不堆列表，能写成自然段就写成自然段
   - 不用夸张词（"极其"、"核心洞察"、"至关重要"）
   - 少用引号
   - 小结不逐条复述，只点一个记忆钩子

4. **密度标准：比喻之后，技术细节必须跟上。** 可读性不是用类比替代知识，是用类比铺路让知识进得去。源码锚点一个不少。

**验收自问：** 这段话说一遍够了吗？有没有用不同的词把同一件事说了两遍？

## 8. 一页执行版（最小记忆集）

只记 6 句话即可开始写：
1. 先写可反驳主张，再写内容。
2. 关键结论必须可复核；推断必须标注 inference。
3. 每段都要有新增信息，禁止重复改写同一判断。
4. 每章先划清边界，避免跨章抢内容。
5. 永远为目标读者写：易懂，但不失真。
6. 比喻开头说一遍就走，不重复解释，去掉 AI 味。

## 9. 金字塔结构规则（2026-04-03 新增）

### 核心问题：现有章节是扁平化平铺，不是金字塔

现在的结构：章节标题 → 若干并列小节。小节之间是话题并列，读者看完不知道这章在论证什么，逻辑性差。

金字塔原理要求的结构：**章节有一个核心主张，所有小节是支撑这个主张的论点，每个论点下才是细节和证据。**

```
章节核心主张（一句可反驳的话）
├── 论点A（支撑主张的第一个理由）
│   ├── 细节/证据 支撑A
│   └── 细节/证据 支撑A
├── 论点B（支撑主张的第二个理由）
│   ├── 细节/证据 支撑B
│   └── 细节/证据 支撑B
└── 边界与迁移（主张的适用条件）
```

### 动手前必须先做论点树

**写或重写任何一章，第一步不是写正文，是画论点树。**

论点树要求：
1. 章节核心主张：一句话，必须可被反驳
2. 每个二级论点：能独立回答"为什么主张成立"
3. 小节标题 = 论点，不是话题标签
4. 论点数量：2~4 个，超过说明主张不够聚焦

### 论点树示范（ch06 QueryEngine）

**核心主张：QueryEngine 把"先持久化后执行"固定为系统约束，调用方无法绕过。**

```
核心主张：先持久化后执行是系统约束，不是调用方约定
├── 论点一：顺序颠倒时，恢复能力消失（为什么这个约束至关重要）
│   ├── 6.x --resume 失效的精确路径（直接后果）
│   └── 6.x 写入成本：SSD 4ms，代价可控（约束的代价）
├── 论点二：两阶段锁定防止状态污染（约束如何落地）
│   ├── 6.x 两次 processUserInputContext：真章换假章（锁定机制）
│   └── 6.x turn 级状态隐式约定（锁定的已知技术债）
├── 论点三：职责边界让两层可独立推理（为什么要拆两层）
│   └── 6.x QueryEngine 管跨 turn，query.ts 管单轮（分工说明）
└── 边界与可迁移原则
    ├── bare 模式的 fire-and-forget 风险
    └── 持久化时序决定恢复能力边界（可迁移）
```

注意：原稿的 6.1~6.9 是话题并列，论点树重组后结构变成了"主张→论点→细节"三层。标题也随之改变：从话题标签（`6.3 这次写入有多贵`）变成论点标题（`6.x 写入成本可控：SSD 4ms 是最大可控延迟`）。

### 标题写法规则

标题必须是论点，不能是话题标签。

- ❌ `持久化`（标签）
- ❌ `6.3 这次写入有多贵`（问句话题，无立场）
- ✅ `6.x 写入成本可控：SSD 4ms 是整条关键路径上最大的可控延迟`（论点）
- ✅ `顺序颠倒时恢复能力消失`（可反驳的主张）

三级标题（`###`）通常是多余的。优先把内容合进正文，用段落主题句代替标题。每章小节数上限：论点层 2~4 个，细节层每个论点下 ≤ 3 个。

### 结构紧凑规则

每个小节第一句话必须是论点（主张），不能是背景铺垫。相邻段落之间不写过渡句（"接下来我们看……"）——靠内容逻辑连贯，不靠胶水句连贯。

## 10. 技术名词处理规则（2026-04-02 新增）

目标读者是软件行业的开发、产品、运营，但不一定能读 TypeScript 源码。遇到技术名词按三档处理：

**第一档：直接使用，无需解释。**
条件：软件行业通识，字面含义已经准确。
例：API、日志、进程、会话、配置文件。

**第二档：首次出现时括号内给一句话定义，后续直接使用。**
条件：Claude Code 特有用法，或字面含义容易误导。
例：`turn`（一轮对话：从用户发消息到 AI 回复完毕算一个 turn）、`transcript`（持久化到磁盘的对话记录文件）。
写法：第一次出现写成 `turn（一轮对话）`，后续直接用 turn。

**第三档：正文里单独一句话解释，不能只靠括号说清楚。**
条件：这个词承载了设计决策，理解它直接影响理解后文的论点。
例：`QueryEngine`（不是"查询引擎"，是跨 turn 的会话编排器，负责状态追踪与持久化）、`bare 模式`（脚本调用场景，区别于交互模式，写入策略不同）。

**判断口诀：** 非软件人员看到这个词，会不会猜错它的含义？猜错了会不会影响理解后文？两个都"会"——第三档；第一个"会"第二个"不会"——第二档；都"不会"——第一档。

## 11. 章节重写提示词模板（2026-04-03 更新）

当子 Agent 接到「重写第 X 章」任务时，使用以下提示词结构：

```
你是一位技术写作专家，正在重写《Claude Code 系统能力解析》的第 X 章。

【目标读者】
软件行业的开发、产品、运营。懂软件是怎么回事，但不一定能读 TypeScript 源码。

【写作风格】
- 章节开头：Feynman 风格——从第一原理提问，建立认知框架
- 各节正文：Kernighan 风格——直接说约束和后果，不废话
- 可迁移原则：Feynman 风格——点出原则适用条件和反例，"原则是工具，不是教条"收尾

【重写前必须先完成：画论点树】

在动笔写任何正文之前，先输出本章的论点树，格式如下：

核心主张：[一句可反驳的话]
├── 论点A：[支撑主张的第一个理由]
│   ├── 细节/证据
│   └── 细节/证据
├── 论点B：[支撑主张的第二个理由]
│   └── ...
└── 边界与迁移：[主张的适用条件]

论点树通过后再写正文。论点树必须满足：
- 核心主张是一句话，可被反驳
- 每个论点能独立回答"为什么主张成立"
- 论点数量 2~4 个

【必须遵守的结构规则】
1. 结构遵循金字塔原理：主张 → 论点 → 细节/证据，不能扁平平铺（见 writing-guide.md 第9节）
2. 小节标题必须是论点，不能是话题标签
3. 每段第一句话是主张，不铺垫背景
4. 相邻段落不写过渡句，靠内容逻辑连贯
5. 论点层小节数 2~4 个，细节层每个论点下 ≤ 3 个

【必须遵守的写作规则】
1. 技术名词按三档处理（见 writing-guide.md 第10节）
2. 比喻开头说一遍就走，后文不重复解释
3. 不用破折号、不用"不是…而是…"句式、不堆列表、不用夸张词
4. 关键结论保留源码锚点；推断标注 inference

【验收自查（写完后逐条确认）】
- [ ] 有没有先输出论点树？论点树和正文结构是否一致？
- [ ] 章节核心主张是否可被反驳？
- [ ] 每个小节标题是否是论点而不是话题标签？
- [ ] 是否有小节开头在铺垫背景而不是直接给主张？
- [ ] 技术名词首次出现时是否按三档规则处理了？
- [ ] 是否有"也就是说……"、"综上所述……"、"接下来……"这类废句？
- [ ] 论点层小节数是否超过 4？

【输入文件】
当前章节原稿：[文件路径]
写作规范：docs/book-workspace/methodology/writing-guide.md

【输出要求】
先输出论点树，确认后再输出完整重写正文。
```

使用说明：将 `[文件路径]` 替换为具体章节路径，将 `第 X 章` 替换为具体章节号和标题即可。
