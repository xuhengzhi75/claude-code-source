第1篇：为什么 Claude Code 不会"忘记"你让它做到一半的事

你有没有遇到过这种情况：

让 AI 帮你改一批文件，改到一半网络断了，或者你不小心关掉了窗口。重新打开，它完全不记得刚才做到哪了。你只能从头来。

这不是 AI 不够聪明，是它的设计方式决定的。

普通聊天工具的设计假设

ChatGPT、普通版 Claude 这类工具，设计时有一个隐含的假设：你的需求可以在一次对话里完成。

问一个问题，得到一个答案，对话结束。这个假设在问答场景下完全成立。

但当你让它做一件需要很多步骤的事——比如"帮我把这 30 个文件按新格式重新整理"——这个假设就开始出问题了。

它在内存里记着"现在做到第 7 个文件了"。进程一死，内存清空，这个记录消失。下次启动，它不知道自己做到哪了。

Claude Code 的不同之处

Claude Code 的设计假设是反过来的：进程随时可能死掉，这是正常情况，不是异常情况。

所以它在做每一件事之前，先把"我要做什么"写到磁盘上的一个文件里。

注意这个顺序：先写，再做。不是做完了再记录。

这个顺序的差别，决定了一件事：

如果进程在执行过程中被杀死，磁盘上已经有了"用户说了什么、做到哪一步了"的记录。下次启动，它可以从这个记录里找到上次的位置，继续往下做。

如果是"做完再记录"，进程死掉就意味着这一步的记录丢失，恢复只能从上一个完成的步骤开始。

一个具体的例子

你让 Claude Code 帮你重构一个项目，它开始工作。

做到一半，你的电脑死机了。

重启之后，你输入 --resume（继续上次的意思）。

它找到磁盘上的记录，知道你上次说了什么、做到哪一步了，从那个点继续。

这件事能成立，完全依赖"先写再做"这个顺序。如果顺序反过来，--resume 就是一句空话——它找不到任何可以恢复的记录。

记录用的是什么格式

Claude Code 把对话记录存成一个普通的文本文件，每条消息单独一行。

这个格式有一个关键特性：每次写入只是在文件末尾加一行，不需要重写整个文件。

这意味着：进程在任意时刻被杀死，已经写进去的内容都是完整的、可以读取的。不会出现"文件写到一半，格式损坏，读不了"的情况。

用数据库不是更专业吗？数据库的优势是支持复杂查询和并发写入，但 Claude Code 是本地单机工具，这些优势用不上。文本文件的优势是：出问题时，你直接打开文件就能看，不需要任何特殊工具。

这对你意味着什么

如果你在用 Claude Code 做一件需要很多步骤的事，中途出了问题，不要慌。

用 --resume 重新启动，它会从中断的地方继续，不需要你重新解释一遍你要做什么。

更重要的是，这背后有一个可以迁移的判断标准：一个 AI 工具是否值得信任去做长任务，取决于它在什么时候把状态写下来。

写在执行之前，中断可以恢复。写在执行之后，中断就是丢失。

下次你在选择用哪个 AI 工具做重要的长任务时，可以问一句：它会在做之前先记录吗？
