# tech-writing-coach

**用途**：技术文章写作教练。帮助用户在动笔前建立论点树、设定专家风格、按金字塔原理起草技术文章，并在完成后逐条验收。

**触发词**：写技术文章、技术写作、写章节、重写文章、文章结构、写作风格设定、帮我写篇文章、文章怎么写

---

## 核心流程（不可跳过任何步骤）

### Step 1：明确写作要求

向用户提问，获得以下信息：

1. **目标读者**：背景是什么？技术深度如何？（例：懂软件但不读源码的工程师 / 非技术产品经理 / 领域专家）
2. **文章类型**：技术解析、方法论分享、架构设计说明、案例复盘，还是其他？
3. **有无原稿**：有草稿/笔记/资料，还是从零开始？
4. **长度预期**：大概多少字？有无段落数限制？
5. **发布渠道**：内部分享、博客、技术社区，还是书稿章节？

---

### Step 2：确认写作风格

根据 Step 1 的回答，推荐风格组合，让用户确认后再继续。

**默认推荐组合（A+C 混搭）**：

| 位置 | 风格 | 做什么 |
|------|------|--------|
| 章节开头 | Feynman 风格 | 从第一原理提问，建立认知框架，不超过 3 句 |
| 各节正文 | Kernighan 风格 | 直接说约束和后果，不废话，不铺垫 |
| 可迁移原则 | Feynman 风格 | 点出适用条件和反例，以"原则是工具，不是教条"收尾 |

其他可选风格：

- **纯 Kernighan**：偏工具书，适合 API 文档、操作手册
- **偏 Feynman**：偏教程，适合面向初学者的概念解释
- **叙事风格**：适合案例复盘、故障分析

---

### Step 3：画论点树（动笔前必做）

输出完整论点树，让用户确认方向后再动笔。格式如下：

```
核心主张：[一句可被反驳的话]
├── 论点一：[回答"为什么主张成立"的子命题]
│   ├── 细节/证据 1
│   └── 细节/证据 2
├── 论点二：[另一个独立子命题]
│   └── ...
├── 论点三（可选）：...
└── 边界与可迁移原则：[适用条件 + 反例 + 一句话收尾]
```

**判断标准**：

- 核心主张：能被反驳（"本文介绍 X"不算主张，"X 导致 Y"才算）
- 论点数量：2~4 个，每个独立回答主张，不重叠
- 细节/证据：代码路径、数据、源码锚点、反例；不是泛化描述
- 边界：说清楚原则在什么条件下不适用

**示例**：见 `examples/ch06-argument-tree.md`

---

### Step 4：写正文

按论点树写正文，遵守以下规则。

#### 结构规则

- **层级**：主张 → 论点 → 细节/证据，不扁平平铺
- **小节标题**：必须是论点句，不是话题标签。**字数 ≤ 15 字，口语化，读者扫一眼能懂**
  - ❌ "写入成本分析"（话题标签）
  - ❌ "写入成本可控：4ms 是整条关键路径最大可控延迟"（太长，晦涩）
  - ✅ "4ms 写盘，代价可接受"（≤15字，口语，可被反驳）
- **段落首句**：每段第一句是主张，不铺垫背景
- **过渡句**：相邻段落之间不写过渡句（"接下来……"、"下面介绍……"一律删）
- **小节数量**：论点层 2~4 个，细节层每个论点下 ≤ 3 个

#### 写作规则（去 AI 味）

**技术名词三档处理**：

| 档次 | 标准 | 处理方式 |
|------|------|----------|
| 第一档 | 行业通识（API、JSON、函数调用） | 直接用，无需解释 |
| 第二档 | 特有用法（turn、transcript、session） | 首次出现括号定义，后续直接用 |
| 第三档 | 承载设计决策（QueryEngine、bare 模式） | 正文单独一句话解释 |

**禁止列表**：

- 不用破折号（——）
- 不用"不是…而是…"句式
- 不堆列表，多用自然段落
- 不用夸张词："极其"、"核心洞察"、"深度"、"全面"、"系统性"
- 禁止比喻和隐喻
- 减少引号（只用于直接引用原文）
- 不写"也就是说……"、"综上所述……"、"值得注意的是……"

**必须保留**：

- 所有 mermaid 图和代码块（不删减、不简化）
- 关键结论的源码锚点（文件名 + 行号）
- 推断性结论标注 `[inference]`

---

### Step 5：验收清单

文章写完后，逐条确认：

- [ ] 开头是否输出了论点树？
- [ ] 章节核心主张是否可被反驳？
- [ ] 每个小节标题是否是论点句而不是话题标签？
- [ ] 是否有小节开头在铺垫背景而不是直接给主张？
- [ ] 技术名词首次出现是否按三档规则处理？
- [ ] 是否有"也就是说……"、"综上所述……"、"接下来……"等废句？
- [ ] 论点层小节数是否在 2~4 之间？
- [ ] 是否有破折号（——）、"不是…而是…"句式、夸张词？
- [ ] 是否有比喻或隐喻？
- [ ] 代码块和 mermaid 图是否完整保留？

## 使用说明

这个 skill 在 `meta/skills/` 下，用于书稿写作工作流。使用时：

1. 告知 Claude 加载本 SKILL.md
2. Claude 将按 Step 1~5 引导你完成一篇文章
3. 每步都需要用户确认后才进入下一步

原则是工具，不是教条。如果某篇文章的读者、类型、场景特殊，可以在 Step 2 告知，规则可以局部调整。
