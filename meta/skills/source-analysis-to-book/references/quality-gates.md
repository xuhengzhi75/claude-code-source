# 质量验收标准

## 基础通过线（每章必须满足）

- [ ] 有且仅有一个清晰核心主张（可被反驳）
- [ ] 关键结论具备源码锚点；无法验证处已标注 inference
- [ ] 解释了"为什么这样设计，不这样会怎样"
- [ ] 至少 1 个非显然发现（读者不看这篇就不会注意到的细节）
- [ ] 删除冗余后信息量未被掏空

## 质量增强线

- [ ] 说明了至少一个替代方案及未采纳原因
- [ ] 写明关键假设与适用条件（避免过度泛化）
- [ ] 指出边界条件或技术债
- [ ] 读者读完后需记住的核心概念 ≤ 3

## 结构级验收

- [ ] 明确写出本章解决的第一性问题（不是"介绍 X"，而是"解决 Y 矛盾"）
- [ ] 把关键架构决策的隐含假设显式化（"这个设计假设……"）
- [ ] 分析至少一个替代方案，说明为什么没有选它
- [ ] 从具体实现中抽象出至少一条可迁移的设计原则
- [ ] 指出当前设计的技术债或边界条件

## 模型级验收

- [ ] 读者读完能建立可以预测系统行为的心智模型
- [ ] 抽象层级一致：架构层 → 实现层证据 → 架构层总结，层级切换有明确信号
- [ ] 删除所有冗余句（过渡句、铺垫句、重复总结句）
- [ ] 核心概念不超过 3 个

## 对抗级验收

- [ ] 构造一个强反驳，检验核心主张是否需要加限定条件
- [ ] 检查每个结论是否过度泛化，加上适用条件
- [ ] 确保每个主张有：源码锚点（直接证据）+ 机制解释（为什么这个证据支持主张）
- [ ] 读完后需要记住的核心概念不超过 3 个

---

## 整体项目验收

一个使用这个 Skill 的完整运行，至少满足：

**分析质量**：
- 结果通过控制流、状态和恢复逻辑解释系统，而不是通过目录列表
- 至少识别出一个核心系统矛盾或设计取舍
- 明确指出不可删除的组件
- 覆盖了边界情况和恢复路径，不只是 happy path

**证据质量**：
- verified 事实与 inference 推断严格分离
- 重要结论可以追溯到代码、注释、配置或调用链
- 在章节写作之前或同时更新了证据地图

**写作质量**：
- 每章或每节回答一个非显然问题
- 机制、约束和效果分开写
- 草稿不像模块目录或通用 AI 文章
- 信息密度足够高，删除 20%-30% 会明显影响理解

**续写质量**：
- 当任务跨多轮时，`project-status.md` 和状态快照已更新
- 另一个 Agent 应该能从保存的材料继续，而不需要重新读整个仓库
