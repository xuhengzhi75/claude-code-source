# 质量验收标准

## 基础通过线（每章必须满足）

- [ ] 有且仅有一个清晰核心主张（可被反驳）
- [ ] 关键结论具备源码锚点；无法验证处已标注 inference
- [ ] 解释了"为什么这样设计，不这样会怎样"
- [ ] 至少 1 个非显然发现（读者不看这篇就不会注意到的细节）
- [ ] 删除冗余后信息量未被掏空

## 质量增强线

- [ ] 说明了至少一个替代方案及未采纳原因
- [ ] 写明关键假设与适用条件（避免过度泛化）
- [ ] 指出边界条件或技术债
- [ ] 读者读完后需记住的核心概念 ≤ 3

---

## 结构级验收（每章发布前）

**标准一：核心主张测试**
能不能用一句话说出这章的核心主张？这个主张能不能被反驳？如果不能，重写。

**标准二：信息增量测试**
读者如果只看架构笔记，能不能得到这章的全部内容？如果能，这章没有信息增量，重写。

**标准三：锚点精确性测试**
每个关键结论是否有精确的源码锚点（文件名 + 函数名，最好有行号）？如果没有，补充或降级为推断。

**标准四：机制完整性测试**
对于每个"是什么"，是否解释了"为什么这样，不这样会怎样"？如果没有，补充机制解释。

**标准五：反直觉发现测试**
这章有没有至少一个"如果不看这篇，读者不会注意到"的发现？如果没有，说明挖得不够深。

**标准六：删减测试**
把这章的 20%-30% 删掉，读者的理解会不会受损？如果不会，说明有冗余，删掉那部分。

**标准七：第一性问题测试**
这章是否明确回答了"本章解决的第一性问题是什么"？第一性问题不是"介绍 X 系统"，而是"X 系统存在是为了解决什么根本矛盾"。

检验方法：把章节标题替换成"为什么需要 X"，看正文能不能直接回答这个问题。

**标准八：隐含假设显式化测试**
每个架构决策背后都有隐含假设。这章是否把关键决策的隐含假设说出来了？

隐含假设的典型形式："这个设计假设系统是单机运行的"、"这个设计假设任务失败是偶发的，不是常态"。

**标准九：替代方案对比测试**
这章是否分析了至少一个替代方案，并说明了为什么没有选它？

**标准十：可迁移原则抽象测试**
这章是否从具体实现中抽象出了至少一条可迁移的设计原则？格式：原则名称 + 适用条件 + 反例。

**标准十一：技术债与边界条件测试**
这章是否指出了当前设计的技术债或边界条件？

---

## 模型级验收

**标准十二：心智模型测试**
读者读完这章，能不能在脑子里建立一个可以用来预测系统行为的心智模型？

检验方法：给读者一个没有在章节中出现过的场景，看他们能不能用这章建立的模型推断出系统的行为。

**标准十三：抽象层级一致性测试**
这章是否在同一个抽象层级上讨论问题，而不是在"架构设计"和"代码细节"之间跳跃？

合格的做法：先在架构层把问题说清楚，再下沉到实现层提供证据，最后回到架构层总结。层级切换要有明确的信号（"在实现层，这体现为……"）。

**标准十四：信息密度测试**
删除所有过渡句、铺垫句、总结重复句之后，章节的核心信息量是否减少？如果不减少，说明这些句子是冗余的，应该删除。

冗余句的典型形式："这一点非常重要"、"下面我们来看……"、"综上所述，……"、"这个设计很有意思"。

---

## 对抗级验收

**标准十五：强反驳测试**
这章的核心主张，能不能被一个合理的反驳推翻？如果能，说明主张不够严谨，需要加限定条件或修改主张。

**标准十六：过度泛化测试**
这章的结论是否过度泛化？即，是否把特定场景约束下的设计，说成了普遍适用的真理？

合格的做法：每个结论都带上适用条件，让读者能判断这个结论在自己的场景下是否成立。

**标准十七：论据充分性测试**
这章的每个主张，是否有足够的论据支撑？论据的充分性标准：至少有一个源码锚点（直接证据）+ 一个机制解释（为什么这个证据支持这个主张）。

**标准十八：认知负荷测试**
读者在读完这章之后，需要记住的核心概念不超过 3 个。如果超过，说明章节试图在一章内解决太多问题，需要拆分或聚焦。

---

## 论点树验收（写作前）

在动笔写任何正文之前，论点树必须满足：

- [ ] 核心主张是一句话，可被反驳
- [ ] 每个论点能独立回答"为什么主张成立"
- [ ] 论点数量 2~4 个（超过说明主张不够聚焦）
- [ ] 小节标题 = 论点，不是话题标签

---

## 整体项目验收

一个使用这个 Skill 的完整运行，至少满足：

**分析质量**：
- 结果通过控制流、状态和恢复逻辑解释系统，而不是通过目录列表
- 至少识别出一个核心系统矛盾或设计取舍
- 明确指出不可删除的组件
- 覆盖了边界情况和恢复路径，不只是 happy path

**证据质量**：
- verified 事实与 inference 推断严格分离
- 重要结论可以追溯到代码、注释、配置或调用链
- 在章节写作之前或同时更新了证据地图

**写作质量**：
- 每章或每节回答一个非显然问题
- 机制、约束和效果分开写
- 草稿不像模块目录或通用 AI 文章
- 信息密度足够高，删除 20%-30% 会明显影响理解
- 无 AI 味词汇，无破折号，无"不是…而是…"句式

**续写质量**：
- 当任务跨多轮时，`project-status.md` 和状态快照已更新
- 另一个 Agent 应该能从保存的材料继续，而不需要重新读整个仓库
