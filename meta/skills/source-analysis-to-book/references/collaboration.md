# 多 Agent 协作规范

## 三方分工模型

| 角色 | 职责 | 主要工作区 |
|------|------|-----------|
| 源码分析侧 | 深读源码、产出证据材料、建立架构笔记、填写挖掘任务卡 | `book-workspace/architecture-notes/` |
| 写作侧 | 把证据材料转化为章节正文、识别内容薄弱点、起草挖掘任务卡 | `book/chapters/` |
| 验收侧 | 章节质量验收、对照验收标准逐条检查 | 全局 |

---

## 交接协议

### 分析侧 → 写作侧的交接格式

每次分析侧完成一个主题的深读，必须产出一份交接卡，包含：

```markdown
## 一句话结论
[这个主题的核心发现，一句话]

## 本章最该讲的 3~5 点
1. [点 1]
2. [点 2]
3. [点 3]

## 明确不要重复讲的点
- [已在其他章节覆盖的内容]

## 关键锚点
- `文件路径#L行号-行号`：[说明]

## 仅能写为 inference 的判断
- [推断内容]：[推断依据]

## 章节边界
- 讲到：[范围]
- 不讲：[范围]
```

### 写作侧 → 分析侧的挖掘任务卡

写作侧识别到某章内容薄弱时，起草挖掘任务卡（存放在 `book-workspace/excavation-tasks/ch**-excavation.md`）：

```markdown
## 章节：ch** [章节标题]

## 当前薄弱点
[描述当前内容的不足]

## 需要挖掘的问题
1. [具体问题 1]
2. [具体问题 2]

## 期望产出格式
- verified：有明确源码行号支撑
- inference：基于源码的合理推断，但无直接证据
填写时请附上：文件路径 + 行号范围 + 关键代码片段（可截取关键3-5行）
```

## 协作节奏

- 开工前先 pull
- 完成一小段可验收成果就 push
- 轮动推进，不攒大包
- 每次完成阶段性工作，立即更新状态文档

## Inbox 通信规范

多 Agent 通过 `docs/book-workspace/inbox/` 目录下的文件异步通信。每个角色有独立的收件箱文件（`for-[角色名].md`）。

### 消息格式

```markdown
---
发件人：[角色名]
时间：YYYY-MM-DD
主题：[简短描述]

[消息正文]

状态：待处理
---
```

### 状态管理规则（重要）

**任务完成后必须更新 inbox 状态字段**，将 `状态：待处理` 改为 `状态：已处理 ✅`，并在 commit message 里注明。

**原因**：如果不更新状态，下一个 agent 实例拉取代码后会看到"待处理"，可能重复处理同一条任务，造成重复工作甚至冲突。

**操作流程**：
1. 完成任务后，打开对应的 inbox 文件
2. 找到对应消息，将 `状态：待处理` 改为 `状态：已处理 ✅`
3. 将状态更新和任务产出一起 commit，commit message 格式：`chore(inbox): 标记 [任务名] 已完成`

## 安全规则

- 真实姓名不入仓库，用 GitHub 用户名或代号代替
- 内部账号、邮箱、API Key、Token 不入仓库
- 内部系统链接不入仓库
- 用 pre-commit hook 自动检测，检测规则存本地配置文件（不入仓库）

## 状态管理

任何 Agent 恢复上下文时，按这个顺序读：
1. `planning/project-status.md`（当前进度和下一步）
2. `status/status-index.md` + 最近一条 `status-*.md`（阶段快照）
3. `references/chapter-evidence-map.md`（章节覆盖情况）
4. 必要时再看具体的 `architecture-notes/` 或章节文件

**原则**：先读最小入口，不做全目录通读。先恢复状态，再继续修改。
