# 写作规则参考

## 五条写作红线（必须遵守）

### 1. 信息密度优先

- 每段至少提供一个新信息单元（机制、边界、证据、反例、取舍之一）
- 能删 20%~30% 且不影响理解，视为不合格
- 少过渡废句，少"重要的是…/下面来看…/综上所述…"

### 2. verified / inference 严格分离

- **verified**：有源码锚点（文件 + 函数/分支/对象，关键点尽量带行号）
- **inference**：基于证据的推断，必须显式标注，措辞保守
- 不允许把推断写成事实

### 3. 章节边界明确

每章开工前先写清三件事：
- 本章第一性问题（要解决的根本矛盾）
- 本章覆盖范围（讲到哪）
- 本章明确不讲（避免与相邻章节重叠）

### 4. 避免重复

- 不重复讲已达标章节的结论；只补"新增机制/新增证据/新增边界"
- 不做目录树式模块罗列；只保留能支撑主张的最小结构信息

### 5. 读者导向

- 用读者可理解的语言解释机制，但不牺牲正确性
- 不靠夸张、比喻、情绪化措辞提高可读性
- 始终区分"模型能力"与"系统能力"

---

## 标准章节骨架

每章按这个结构写：

1. **核心主张**（一句话，可被反驳）
2. **第一性问题**（为什么需要这个机制）
3. **机制展开**（关键决策 + 为什么不是替代方案）
4. **证据锚点**（verified，含文件路径和行号）
5. **边界与技术债**（何时失效/何时退化）
6. **可迁移原则**（适用条件 + 反例）
7. **本章不覆盖项**（章节边界）

## 金字塔结构规则（必须遵守）

### 核心问题：扁平平铺 vs 金字塔

扁平平铺：章节标题 → 若干并列小节。小节之间是话题并列，读者看完不知道这章在论证什么。

金字塔要求：**章节有一个核心主张，所有小节是支撑这个主张的论点，每个论点下才是细节和证据。**

```
章节核心主张（一句可反驳的话）
├── 论点A（支撑主张的第一个理由）
│   ├── 细节/证据 支撑A
│   └── 细节/证据 支撑A
├── 论点B（支撑主张的第二个理由）
│   └── ...
└── 边界与迁移（主张的适用条件）
```

### 动手前必须先画论点树

**写或重写任何一章，第一步不是写正文，是画论点树。**

论点树要求：
1. 章节核心主张：一句话，必须可被反驳
2. 每个二级论点：能独立回答"为什么主张成立"
3. 小节标题 = 论点，不是话题标签
4. 论点数量：2~4 个，超过说明主张不够聚焦

### 标题写法规则

标题必须是论点，不能是话题标签：

- ❌ `持久化`（标签）
- ❌ `这次写入有多贵`（问句话题，无立场）
- ✅ `写入成本可控：SSD 4ms 是整条关键路径上最大的可控延迟`（论点）
- ✅ `顺序颠倒时恢复能力消失`（可反驳的主张）

三级标题（`###`）通常是多余的。优先把内容合进正文，用段落主题句代替标题。每章小节数上限：论点层 2~4 个，细节层每个论点下 ≤ 3 个。

### 结构紧凑规则

每个小节第一句话必须是论点（主张），不能是背景铺垫。相邻段落之间不写过渡句——靠内容逻辑连贯，不靠胶水句连贯。

## 写作风格：Feynman + Kernighan

- **章节开头**：Feynman 风格——从第一原理提问，建立认知框架
- **各节正文**：Kernighan 风格——直接说约束和后果，不废话
- **可迁移原则**：Feynman 风格——点出原则适用条件和反例，"原则是工具，不是教条"收尾

---

## 去 AI 味规则（必须执行）

以下写法一律禁止：

- 不用破折号（`——`）
- 不用"不是……而是……"句式
- 不堆列表，能写成自然段就写成自然段
- 不用夸张词（"极其"、"核心洞察"、"至关重要"、"深入探讨"、"值得注意"、"不仅如此"）
- 少用引号
- 小结不逐条复述，只点一个记忆钩子
- 不写"也就是说……"、"综上所述……"、"接下来……"这类废句

**验收自问**：这段话说一遍够了吗？有没有用不同的词把同一件事说了两遍？

## 技术名词三档处理规则

目标读者是软件行业的开发、产品、运营，但不一定能读 TypeScript 源码。遇到技术名词按三档处理：

**第一档：直接使用，无需解释。**
条件：软件行业通识，字面含义已经准确。
例：API、日志、进程、会话、配置文件。

**第二档：首次出现时括号内给一句话定义，后续直接使用。**
条件：特有用法，或字面含义容易误导。
例：`turn`（一轮对话：从用户发消息到 AI 回复完毕算一个 turn）、`transcript`（持久化到磁盘的对话记录文件）。

**第三档：正文里单独一句话解释，不能只靠括号说清楚。**
条件：这个词承载了设计决策，理解它直接影响理解后文的论点。
例：`QueryEngine`（不是"查询引擎"，是跨 turn 的会话编排器，负责状态追踪与持久化）。

**判断口诀**：非软件人员看到这个词，会不会猜错它的含义？猜错了会不会影响理解后文？两个都"会"——第三档；第一个"会"第二个"不会"——第二档；都"不会"——第一档。

当 Agent 接到「重写第 X 章」任务时，使用以下提示词结构：

```

【写作风格】
- 章节开头：Feynman 风格——从第一原理提问，建立认知框架
- 各节正文：Kernighan 风格——直接说约束和后果，不废话
- 可迁移原则：Feynman 风格——点出原则适用条件和反例，"原则是工具，不是教条"收尾

【重写前必须先完成：画论点树】

在动笔写任何正文之前，先输出本章的论点树，格式如下：

核心主张：[一句可反驳的话]

└── 边界与迁移：[主张的适用条件]

论点树通过后再写正文。论点树必须满足：

【必须遵守的结构规则】

【必须遵守的写作规则】

- [ ] 是否有小节开头在铺垫背景而不是直接给主张？
- [ ] 技术名词首次出现时是否按三档规则处理了？
```

## 三个常见失败模式

### 1. 结构转述型失败

只说"有什么模块"，不解释"为何存在"。

表现：按目录顺序写，每节都像 README，信息正确但没有穿透力。

修正：改成"主矛盾 + 关键链路 + 设计取舍"结构。

### 2. 证据断裂型失败

有观点无锚点，或有锚点无机制解释。

表现：结论散在多个文档中，交接后容易丢上下文。

修正：为章节建立 evidence map，所有关键结论都能回指到证据。

### 3. 边界失焦型失败

章节之间互相抢内容，导致重复与信息稀释。

表现：章节像说明书分类，缺问题意识，缺推进感。

修正：每章先划清边界，避免跨章抢内容。

## 可复用的提示词方向

### 用于仓库初扫

> 不是按目录罗列，而是找这个系统的入口、执行链、状态载体、恢复机制、工具边界与不可删组件，并区分 verified facts 与 inference。

### 用于章节规划

> 给这套系统设计一组"非显然问题驱动"的章节，而不是功能说明书式目录。每章必须回答一个真正有张力的问题。

### 用于深读某个模块

> 分析这个模块在整个控制权转移链中的位置、上下游依赖、继续/终止/恢复条件、边界 case，以及它为什么不能被轻易删掉。

### 用于成稿检查

> 检查这章是否退化成模块介绍；如果是，重写为"矛盾—机制—约束—效果—非显然结论"的结构。
