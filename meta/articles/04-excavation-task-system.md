# 挖掘任务卡体系：从薄弱章节到 verified 证据

> 这篇文章沉淀自 `claude-code-source` 项目的实践。挖掘任务卡体系是这个项目在协作过程中自然演化出来的机制，解决了"章节内容薄弱但不知道缺什么"这个具体问题。

---

## 问题的起点

写作侧完成一章初稿后，经常遇到这种情况：逻辑完整，锚点也有，但读起来像"架构说明书"——每节都在描述"有什么"，缺少"让人拍大腿的发现"。读者看完能复述框架，但不会有"原来如此"的惊讶感。

这种薄弱不是逻辑问题，是素材问题。缺的是：反直觉的设计细节、真实的 bug 修复历史、边界情况的处理逻辑。

这些东西写作侧自己挖不出来——需要深入源码，追调用链，读注释，看 flag 的来历。这是源码分析侧的工作。

但问题是：写作侧说"这章不够好"，源码分析侧不知道该挖什么。

挖掘任务卡体系解决的就是这个问题：把"章节薄弱"这个模糊感受，转化成可执行的挖掘任务。

---

## 挖掘任务卡的格式

每张任务卡是一个 Markdown 文件，存放在 `docs/book-workspace/excavation-tasks/ch**-excavation.md`。

**文件头**：
```markdown
# ch** 挖掘任务卡：[章节标题]

**认领人：** startheart
**状态：** 待认领 / 进行中 / 已完成 ✅
**目标章节：** docs/book/chapters/ch**-*.md（当前 N 行）
**写作规范：** docs/book-workspace/methodology/writing-principles.md
```

**背景与现状**：描述当前章节的内容缺口是什么，为什么薄弱，改稿的瓶颈在哪里。

**挖掘任务**，分三级：

- 🔴 **必答**：写作侧无法在没有这些素材的情况下改文章。通常是 2~4 个具体问题，每个问题指向具体的源码文件和行号范围。
- 🟡 **加分项**：有这些素材，文章能从 7 分升到 9 分。
- 🟢 **意外发现区**：挖掘过程中发现的任何有趣细节，不限于任务卡里的问题。

**填写规范**：每条结论必须标注来源类型：
- `verified`：有明确源码行号支撑，附文件路径 + 行号范围 + 关键代码片段（3-5 行）
- `inference`：基于源码的合理推断，但无直接证据，需说明推断依据

## 一张真实任务卡的样子

以 ch11「为什么干不了真活」为例，写作侧识别到的缺口：

> ch11 现有内容逻辑通顺，但读起来像"架构说明书"——每节都有锚点，但缺少"让人拍大腿的发现"。读者看完能复述框架，但不会有"原来如此"的惊讶感。

写作侧起草的 🔴 必答问题：

**Q1：`errors.ts` 里那三种中止来源，最初是一种还是三种？**
现有章节提到 `isAbortError()` 统一判断三种来源（自定义 AbortError、DOMException、SDK 的 APIUserAbortError）。想知道的是：这三种来源是从一开始就分开处理的，还是随时间演进才分出来的？代码注释或 commit 历史里有没有线索？

**Q2：`hasAttemptedReactiveCompact` flag 是怎么引入的？**
ch11 提到这个 flag 修复了一个"压缩→仍过长→错误→压缩"的无限循环 bug。想知道的是：这个 bug 是不是真的发生过才加的补丁？代码注释里有没有更多描述？这个 flag 用在哪几个地方？

**Q3：`TurnInterruptionState` 的两种类型实际处理逻辑有什么不同？**
现有章节只提到"对应不同的恢复策略"，但没说具体怎么不同。想知道的是：恢复时这两种类型走的是什么不同路径？有没有一种比另一种更难处理？

注意这些问题的特点：每个问题都指向具体的文件和函数，都在追"为什么"而不是"是什么"，都在找反直觉的细节。

## 完整闭环流程

```
写作侧识别薄弱章节
        │
        ▼
起草挖掘任务卡（excavation-tasks/ch**-excavation.md）
  - 描述当前缺口
  - 列出具体挖掘问题（🔴/🟡/🟢 三级）
  - 指向具体源码文件和行号
        │
        ▼ commit + push
        │
        ▼
源码分析侧认领任务卡
  - 深挖源码
  - 把结论填入对应卡片
  - 每条结论标注 verified/inference
  - 附源码路径 + 行号 + 关键代码片段
        │
        ▼ commit + push
        │
        ▼
写作侧读取填好的卡片
  - 据此扩充/重写章节正文
  - 把 verified 结论转化为有张力的叙述
        │
        ▼ commit + push
        │
        ▼
验收侧验收
```

整个流程通过 git 仓库异步进行，不需要实时通信。每一步都有 commit 记录，任何时候都能回溯。

---

## 为什么这个机制有效

**它把模糊感受转化成了可执行任务**。"这章不够好"是模糊的，"`hasAttemptedReactiveCompact` flag 是怎么引入的，代码注释里有没有更多描述"是可执行的。

**它让两侧的工作对齐**。写作侧知道改稿需要什么素材，源码分析侧知道该挖什么。没有这个机制，两侧容易各自为战，产出无法对齐。

**它保留了挖掘过程的上下文**。任务卡里不只有问题，还有背景、现状、为什么这个问题重要。源码分析侧不需要重新理解章节，直接从任务卡出发。

**它强制区分 verified 和 inference**。填写规范要求每条结论标注来源类型，这个约束让写作侧在改稿时知道哪些结论可以直接引用，哪些需要保守表述。

## 任务卡的演进

这个项目里的任务卡经历了几个版本的演进：

**初始版本**：只有问题列表，没有背景描述，没有三级分类。源码分析侧不知道哪些问题最重要，容易把时间花在加分项上而忽略必答题。

**改进版本**：加入背景与现状描述，加入 🔴/🟡/🟢 三级分类，加入具体的文件路径和行号范围。这让源码分析侧能快速定位，也能判断优先级。

**成熟版本**：加入意外发现区（🟢），鼓励源码分析侧记录任何有趣的发现，不限于任务卡里的问题。这个区域产出了一些最有价值的内容——比如 `excluded-strings.txt` 的构建时检查机制，最初就是在意外发现区里记录的。

## 适用边界

挖掘任务卡体系适合：内容已有初稿但素材层薄弱、需要深入源码挖掘反直觉细节、有明确的分工（写作侧 + 源码分析侧）。

不适合：从零开始的章节（这时候需要的是架构笔记，不是挖掘任务卡）、内容问题是逻辑结构问题而不是素材问题（这时候需要的是重新画论点树）。
