# 多 Agent 协作写书：规范、踩坑与经验

> 这篇文章沉淀自 `claude-code-source` 项目的多 Agent 协作实践。三方分工（源码分析 / 写作 / 验收），133 次提交，约两天完成 19 章书稿。

---

## 一、为什么需要多 Agent 协作

单个 Agent 写一本源码解读书，面临两个根本限制：

**上下文窗口限制**：一个复杂仓库的源码分析材料，很快就会超出单个 Agent 的上下文窗口。如果把所有材料都塞进一个会话，后期的写作质量会明显下降。

**专业化分工**：源码分析和写作是两种不同的能力。源码分析需要深入追踪调用链、区分 verified 和 inference；写作需要把技术材料转化为有张力的叙述。让同一个 Agent 同时做这两件事，往往两件事都做不好。

解决方案：把工作拆分给多个专业化的 Agent，通过结构化的交接协议传递上下文。

---

## 二、三方分工模型

这次实践采用的分工模型：

| 角色 | 职责 | 工作区 |
|------|------|--------|
| 源码分析侧（startheart） | 深读源码、产出证据材料、建立架构笔记、填写挖掘任务卡 | `docs/book-workspace/architecture-notes/` |
| 写作侧（小许） | 把证据材料转化为章节正文、识别内容薄弱点、起草挖掘任务卡 | `docs/book/chapters/` |
| 验收侧（xuhengzhi75） | 章节质量验收、对照验收标准逐条检查 | 全局 |

**关键设计**：三方通过 git 仓库协作，不通过实时通信。每个角色的工作产出都以文件形式存在仓库里，任何时候都可以恢复上下文。

## 三、交接协议

多 Agent 协作最容易出问题的地方是交接。分析侧产出了大量材料，写作侧不知道从哪里下手；写作侧发现内容薄弱，分析侧不知道需要补充什么。

### 分析侧 → 写作侧的交接格式

每次分析侧完成一个主题的深读，必须产出一份交接卡，包含：

```markdown
## 一句话结论
[这个主题的核心发现，一句话]

## 本章最该讲的 3~5 点
1. [点 1]
2. [点 2]
3. [点 3]

## 明确不要重复讲的点
- [已在其他章节覆盖的内容]

## 关键锚点
- `文件路径#L行号-行号`：[说明]

## 仅能写为 inference 的判断
- [推断内容]：[推断依据]

## 章节边界
- 讲到：[范围]
- 不讲：[范围]
```

没有这个格式，写作侧拿到材料也不知道从哪里下手。

### 写作侧 → 分析侧的挖掘任务卡

写作侧识别到某章内容薄弱时，起草挖掘任务卡：

```markdown
## 章节：ch11 为什么干不了真活

## 当前薄弱点
[描述当前内容的不足]

## 需要挖掘的问题
1. [具体问题 1]
2. [具体问题 2]

## 期望产出格式
- verified：有明确源码行号支撑
- inference：基于源码的合理推断，但无直接证据
```

分析侧认领任务卡，深挖后把结论填入对应卡片，commit 提交。

## 四、提交规范

多 Agent 协作时，提交规范比单人写作重要得多。

### 格式

统一使用 `type(scope): purpose`：

```
docs(book): split manuscript from writing workspace
docs(chapters): tighten ch04-ch08 information density
docs(evidence): strengthen query engine anchors
docs(workflow): unify workspace navigation
chore(structure): move writing artifacts into book-workspace
docs(comments): annotate recovery and tool-system hotspots
```

### 核心原则：一次提交只做一类事

不要把多种性质的改动混在一个 commit 里。尤其不要混：
- 结构整理 + 正文重写
- 批量路径修复 + 内容判断修改
- 注释补充 + 脚本调整 + 文档重写

**原因**：多人协作时，混提会让 review 和回滚都变得困难。如果一个 commit 同时改了结构和内容，出问题时很难判断是哪部分导致的。

### 推荐提交流程

**小改动**：
1. 明确属于哪一类改动
2. 只改一类内容
3. 本地自查引用、结构与命名
4. 用 `type(scope): purpose` 提交
5. 立即 push，减少分叉

**大改动**：
1. 先更新状态/说明文档，定义目标与范围
2. 先提交结构性改动
3. 再提交内容性改动
4. 再提交证据/索引/路径修正
5. 每一类单独 commit
6. 每完成一段就 push，保持远端是最新协作基线

### 验收标准

一个 commit 如果不能快速回答下面三问，就不够好：
1. 这次改的是哪一类东西？
2. 为什么这类改动值得单独提交？
3. 如果出问题，能不能低成本单独回滚？

## 五、状态管理

多 Agent 协作最容易丢失的是"当前状态"。每个 Agent 的会话窗口都是有限的，如果没有显式的状态文档，每次恢复上下文都需要重新读大量材料。

### 状态文档体系

```
docs/book-workspace/
├── planning/
│   ├── project-status.md          # 总状态（最重要）
│   └── completed-coverage-and-writing-queue.md  # 覆盖率和写作队列
├── status/
│   ├── status-index.md            # 状态索引
│   └── status-YYYY-MM-DD.md       # 阶段快照
└── references/
    ├── chapter-evidence-map.md    # 章节证据映射
    └── conversation-log.md        # 关键用户要求与决策
```

### 恢复上下文的最小读取路径

任何 Agent 恢复上下文时，按这个顺序读：
1. `planning/project-status.md`（当前进度和下一步）
2. `status/status-index.md` + 最近一条 `status-*.md`（阶段快照）
3. `references/chapter-evidence-map.md`（章节覆盖情况）
4. 必要时再看具体的 `architecture-notes/` 或章节文件

**关键原则**：先读最小入口，不做全目录通读。先恢复状态，再继续修改。

## 六、安全规则

多 Agent 协作时，安全规则比单人写作更重要，因为任何一个 Agent 都可能不小心把敏感信息写入仓库。

### 不入仓库的内容

- 真实姓名（用 GitHub 用户名或代号代替）
- 内部账号、邮箱
- 内部系统链接
- API Key、Token

### 自动检测

用 pre-commit hook 自动检测敏感信息：

```bash
# scripts/hooks/pre-commit
# 检测真实姓名、内部账号等
```

检测逻辑存本地配置文件（不入仓库），避免把检测规则本身暴露出去。

## 七、踩坑汇总

### 坑 1：协作者称呼不统一

最初文档里混用了真实姓名和 GitHub 用户名，导致外部读者看到内部信息。

修正：统一用 GitHub 用户名（startheart / 小许 / xuhengzhi75），真实姓名不入仓库。

### 坑 2：分工边界不清晰

最初没有明确"谁负责什么"，导致两个 Agent 同时在改同一个文件，产生冲突。

修正：明确三方职责边界，每个角色有自己的主要工作区，跨区修改需要明确说明。

### 坑 3：没有挖掘任务卡体系

最初写作侧发现内容薄弱时，只是在文档里留了一个 TODO，分析侧不知道需要补充什么。

修正：建立挖掘任务卡体系（`docs/book-workspace/excavation-tasks/`），写作侧起草任务卡，分析侧认领填写，形成闭环。

### 坑 4：状态文档更新不及时

多个 Agent 并行工作时，`project-status.md` 经常落后于实际进度，导致新 Agent 恢复上下文时看到过时信息。

修正：每次完成一个阶段性工作，立即更新状态文档，作为 commit 的一部分提交。

### 坑 5：证据材料和章节正文混在一起

最初把架构分析笔记和章节正文都放在 `docs/book/` 里，导致读者目录里混入了大量写作过程材料。

修正：严格分离书本体（`docs/book/`）和写书工作区（`docs/book-workspace/`）。

### 坑 6：inbox 状态不更新导致重复处理

任务完成后，inbox 文件里的状态字段没有从"待处理"改为"已处理"。下一个 Agent 实例拉取代码后看到"待处理"，重复处理同一条任务，造成重复工作甚至冲突。

修正：**任务完成后必须更新 inbox 状态字段**，将 `状态：待处理` 改为 `状态：已处理 ✅`，并在 commit message 里注明（格式：`chore(inbox): 标记 [任务名] 已完成`）。这条规则必须写进协作规范，不能依赖 Agent 自觉。

### 坑 7：没有记录通信机制本身

多 Agent 协作建立了 inbox 通信机制，但没有把这个机制本身记录下来。新加入的 Agent 不知道有 inbox，也不知道怎么用，只能靠读代码猜。

修正：把通信机制的设计原则、消息格式、状态管理规则写成独立文档（`workflow/multi-agent-git-collab.md`），并在协作规范里引用。这份文档本身也是可复用的——下次搭建多 Agent 协作项目时，可以直接复用这套机制。

## 八、这套模型的适用边界

**适合**：
- 内容规模大（10 章以上），单个 Agent 上下文不够
- 有明确的分工（分析 / 写作 / 验收）
- 需要长周期推进（多天、多轮）
- 有 git 仓库作为协作基础

**不适合**：
- 简单的单章文章（分工成本大于收益）
- 没有 git 仓库的场景
- 需要实时协作（git 的异步协作模式有延迟）
