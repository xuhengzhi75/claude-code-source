# 多 Agent 协作写书：规范、踩坑与经验

> 这篇文章沉淀自 `claude-code-source` 项目的多 Agent 协作实践。三方分工（源码分析 / 写作 / 验收），133+ 次提交，20 章书稿，从初始的松散协作演进到有 inbox 通信层、挖掘任务卡体系和风格规范的完整协作系统。

---

## 为什么需要多 Agent 协作

单个 Agent 写一本源码解读书，面临两个根本限制。

**上下文窗口限制**：一个复杂仓库的源码分析材料，很快就会超出单个 Agent 的上下文窗口。如果把所有材料都塞进一个会话，后期的写作质量会明显下降。

**专业化分工**：源码分析和写作是两种不同的能力。源码分析需要深入追踪调用链、区分 verified 和 inference；写作需要把技术材料转化为有张力的叙述。让同一个 Agent 同时做这两件事，往往两件事都做不好。

解决方案：把工作拆分给多个专业化的 Agent，通过结构化的交接协议传递上下文。

---

## 三方分工模型

| 角色 | 职责 | 主要工作区 |
|------|------|-----------|
| 源码分析侧（startheart） | 深读源码、产出证据材料、建立架构笔记、填写挖掘任务卡 | `docs/book-workspace/architecture-notes/` |
| 写作侧（小许） | 把证据材料转化为章节正文、识别内容薄弱点、起草挖掘任务卡 | `docs/book/chapters/` |
| 验收侧（xuhengzhi75） | 章节质量验收、对照验收标准逐条检查、协调分工 | 全局 |

**关键设计**：三方通过 git 仓库协作，不通过实时通信。每个角色的工作产出都以文件形式存在仓库里，任何时候都可以恢复上下文。

---

## 协作系统的四个层次

这套协作系统不是一套流程，而是四个层次的叠加。

### 1. 沟通层：inbox 异步通信

`docs/book-workspace/inbox/` 是三方的异步通信中心。每个角色有一个专属收件箱文件（`for-小许.md`、`for-startheart.md`、`for-xuhengzhi75.md`），发消息就是往对方的文件里追加内容，push 后对方 pull 即可看到。

消息格式：
```
---
发件人：startheart
时间：2026-04-02
主题：权限系统交接卡已就绪

[消息正文]

状态：待处理 / 已处理
---
```

消息处理完后把状态改为"已处理"，不删除记录。这样任何时候都能回溯协作历史。

**为什么不用实时通信**：实时通信（微信、飞书）的问题是上下文不在仓库里，换一个 Agent 会话就丢失了。inbox 文件在仓库里，任何 Agent 都能读到完整的协作历史。

### 2. 任务层：挖掘任务卡

`docs/book-workspace/excavation-tasks/` 是写作侧和源码分析侧之间的任务交接区。

触发条件：写作侧发现某章内容薄弱——逻辑完整但素材层不够厚，缺少反直觉设计细节和 bug 修复历史。

任务卡格式：
- 背景与现状：当前章节的内容缺口是什么
- 挖掘任务：分三级（🔴 必答 / 🟡 加分项 / 🟢 意外发现区）
- 填写规范：每条结论标注 `verified`（有行号）或 `inference`（推断），附源码路径 + 行号 + 关键代码片段

这个机制把"章节薄弱"这个模糊感受转化成了可执行的挖掘任务，让源码分析侧知道该挖什么，让写作侧知道改稿的依据是什么。

### 3. 状态层：project-status + status 快照

`docs/book-workspace/planning/project-status.md` 是总状态文档，记录当前阶段、完成情况、下一步优先级。

`docs/book-workspace/status/` 存放阶段快照，每个重要里程碑留一份快照。

任何 Agent 恢复上下文时，按这个顺序读：`project-status.md` → `status-index.md` + 最近一条 `status-*.md` → `chapter-evidence-map.md`。先读最小入口，不做全目录通读。

### 4. 交付层：commit conventions + quality gates + handoff template

提交规范（`COMMIT-CONVENTIONS.md`）、质量验收标准（`methodology/writing-principles.md`）、交接模板（`workflow/analysis-handoff-template.md`）共同构成交付层。

这三个文件的作用是：让每次交付都有明确的格式和验收标准，不依赖口头约定。

## 交接协议

分析侧交付最少包含：一句话结论、本章最该讲的 3~5 点、明确不要重复讲的点、关键锚点（文件 + 函数/分支/对象）、仅能写为 inference 的判断、章节边界（讲到哪/不讲哪）。

没有这些，写作侧拿到材料也不知道从哪里下手。

---

## 提交规范

统一使用 `type(scope): purpose`：

```
docs(book): split manuscript from writing workspace
docs(chapters): tighten ch04-ch08 information density
docs(startheart): ch02 挖掘卡完成，三个必答任务全部 verified
feat(blog): 源码锚点改为 hover 展开面板
```

**一次提交只做一类事**，不要把章节重写 + 证据更新 + 结构整理混在一个 commit 里。

一个 commit 如果不能快速回答下面三问，就不够好：这次改的是哪一类东西？为什么这类改动值得单独提交？如果出问题，能不能低成本单独回滚？

## 角色演进：从松散分工到系统化协作

这套协作系统不是一开始就设计好的，而是在实践中逐步演进的。

**初始阶段**：三方分工存在，但边界模糊。没有 inbox，协作靠 commit message 传递信息。没有挖掘任务卡，章节薄弱时只能靠写作侧自己补。

**中期演进**：建立 inbox 通信层，解决了"信息传递不在仓库里"的问题。建立挖掘任务卡体系，解决了"章节薄弱但不知道缺什么"的问题。

**后期成熟**：引入 Feynman+Kernighan 写作风格，建立论点树前置流程，全章按新标准重写。把方法论抽成可复用 Skill，沉淀到 `meta/` 目录。

每次演进都是因为踩了坑，然后把解决方案固化成制度。

## 踩坑汇总

**坑 1：协作者称呼不统一**。最初文档里混用了真实姓名和 GitHub 用户名，导致外部读者看到内部信息。修正：统一用 GitHub 用户名，真实姓名不入仓库，用 pre-commit hook 自动检测。

**坑 2：分工边界不清晰**。最初没有明确"谁负责什么"，导致两个 Agent 同时在改同一个文件，产生冲突。修正：明确三方职责边界，每个角色有自己的主要工作区。

**坑 3：没有挖掘任务卡体系**。最初写作侧发现内容薄弱时，只是在文档里留了一个 TODO，分析侧不知道需要补充什么。修正：建立挖掘任务卡体系，写作侧起草任务卡，分析侧认领填写，形成闭环。

**坑 4：状态文档更新不及时**。多个 Agent 并行工作时，`project-status.md` 经常落后于实际进度，导致新 Agent 恢复上下文时看到过时信息。修正：每次完成一个阶段性工作，立即更新状态文档，作为 commit 的一部分提交。

**坑 5：写作风格不统一**。不同 Agent 写出来的章节风格差异很大，有的 AI 味重，有的过于学术。修正：引入 Feynman+Kernighan 风格规范，建立禁止列表，全章按新标准重写。

**坑 6：证据材料和章节正文混在一起**。最初把架构分析笔记和章节正文都放在 `docs/book/` 里，导致读者目录里混入了大量写作过程材料。修正：严格分离书本体（`docs/book/`）和写书工作区（`docs/book-workspace/`）。

---

## 安全规则

- 真实姓名不入仓库，用 GitHub 用户名或代号代替
- 内部账号、邮箱、API Key、Token 不入仓库
- 用 pre-commit hook 自动检测，检测规则存本地配置文件（不入仓库）
