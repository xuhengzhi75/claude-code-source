# 如何基于一个源码仓库构建一本解读书

> 这篇文章沉淀自 `claude-code-source` 项目的真实实践。从第一次提交到 20 章书稿上线，经历了三个阶段的方法论演进，133+ 次提交，三方协作（源码分析 / 写作 / 验收）。

---

## 为什么这件事比想象中难

拿到一个陌生仓库，想写一本解读书，最自然的冲动是：打开目录树，从第一个文件夹开始介绍。

这条路走不通。

按目录树写出来的东西叫"模块说明书"，不叫书。读者看完知道"有这些东西"，但不知道"为什么这样设计"，不知道"哪里是真正的难点"，也不知道"这套系统在解决什么根本矛盾"。

真正有价值的源码解读书，必须回答三个问题：这个系统在解决什么核心矛盾？它为什么非这样设计不可？哪些地方是真正的护城河，哪些是可以替换的外围配件？

这三个问题，靠读目录树是回答不了的。

---

## 三阶段工作流

### 阶段 A：仓库理解（不急着写章节）

第一步不是写，是建立全局定位。

具体做法：找入口（CLI 入口在哪里？daemon 在哪里？）、找主执行链（request → routing → orchestration → tools → state → output）、找状态载体（session、task、memory、checkpoint、recovery object 分别是什么）、找关键壁垒（tooling、runtime、安全边界、恢复机制、任务系统）、找异常闭环（失败、重试、续跑、超时、终止）。

这个阶段的产出不是章节，而是 `system-overview`、`runtime-structure`、`execution-flow-guide`、`task-recovery-map`。

**关键原则**：不按目录树写，按"控制权转移"写。请求从哪里进入，控制权交给了谁，哪些条件会继续/中断/恢复，哪些状态必须被保存，哪些组件一删系统就塌。

### 阶段 B：深读与证据整理

目标是把"理解"变成可引用材料。

对每个核心主题，建立证据卡，区分：`verified`（有源码锚点，文件 + 函数/分支/对象，关键点带行号）和 `inference`（基于证据的推断，必须显式标注，措辞保守）。同时记录 edge cases、design tradeoffs、why-it-matters。

这个阶段的产出：`architecture-notes/` 主题深读笔记、`chapter-evidence-map.md` 章节证据映射、`analysis-handoff-template.md` 交接模板。

**关键原则**：先做材料库，再写章节。不要在没有证据地图的情况下直接大篇幅写作。

### 阶段 C：成书写作

从材料库变成章节，不在写章时重新摸索源码。

每章的写作流程：先定本章的"第一性问题"（要解决的根本矛盾，不是"介绍 X 系统"）、从 evidence map 拉证据、先写论点骨架再补事实、每章加入 why now / design tradeoff / edge case / non-obvious conclusion、写完后检查是否退化成说明书。

## 写作风格：Feynman + Kernighan

这是这个项目在第三阶段引入的关键改进，把章节质量提升了一个量级。

**Feynman 风格**用在章节开头和可迁移原则部分：从第一原理提问，建立认知框架，不超过 3 句。不解释显然的东西，直接问"为什么这件事必须这样"。

**Kernighan 风格**用在各节正文：直接说约束和后果，不废话，不铺垫。每段第一句是主张，不是背景介绍。

两种风格的组合解决了一个具体问题：技术文章要么太学术（读者读不下去），要么太通俗（丢失精确性）。Feynman 开头建立认知框架，Kernighan 正文保持信息密度，两者互补。

**动笔前必做：画论点树**

论点树是动笔前的强制检查，格式：一个可被反驳的核心主张，2~4 个独立回答"主张为什么成立"的论点，每个论点下面挂证据或细节。

可被反驳是关键词。"本文介绍 QueryEngine 的工作原理"无法被反驳，它只是意图声明。"QueryEngine 把先持久化后执行固定为系统约束，调用方无法绕过"可以被反驳——这个张力才是文章存在的理由。

## 挖掘任务卡体系

这是这个项目在第二阶段发展出来的关键机制，解决了"章节内容薄弱但不知道缺什么"的问题。

**触发条件**：写作侧发现某章内容薄弱——逻辑完整但素材层不够厚，缺少反直觉设计细节和 bug 修复历史。

**流程**：
1. 写作侧起草挖掘任务卡（`excavation-tasks/ch**-excavation.md`），描述当前缺口，列出具体挖掘问题，分三级：🔴 必答 / 🟡 加分项 / 🟢 意外发现区
2. 源码分析侧认领任务卡，深挖源码后把结论填入对应卡片，每条结论标注 `verified`（有行号）或 `inference`（推断），附源码路径 + 行号 + 关键代码片段
3. 写作侧读取填好的卡片，据此扩充/重写章节正文
4. 验收侧验收

**为什么这个机制重要**：它把"章节薄弱"这个模糊感受转化成了可执行的挖掘任务，让源码分析侧知道该挖什么，让写作侧知道改稿的依据是什么。没有这个机制，两侧容易各自为战，产出无法对齐。

## 最容易踩的坑

**坑 1：模块罗列**。按目录顺序写，每节都像 README，信息正确但没有穿透力。修正：改成"主矛盾 + 关键链路 + 设计取舍"结构。

**坑 2：常识性总结**。说了很多"AI 代理需要状态/工具/记忆"这类正确废话，没回答"这个系统为什么非这样不可"。修正：逼自己写出不可删组件、替代方案为何不行、它真正难在哪里。

**坑 3：没有显式证据地图**。结论散在多个文档中，交接后容易丢上下文。修正：为章节建立 evidence map，任何时候都能从状态文档恢复上下文。

**坑 4：章节没有观点，只有陈述**。章节像说明书分类，缺问题意识，缺推进感。修正：每章先写问题句，再围绕问题组织证据。核心主张必须是可被反驳的判断。

**坑 5：AI 味重**。破折号滥用、"不是…而是…"句式、夸张词堆叠、比喻密集、过渡句泛滥。修正：用规则拦截，不靠感觉。建立禁止列表，逐条检查。

## 目录结构建议

```
project/
├── docs/
│   ├── book/
│   │   ├── README.md          # 读者入口
│   │   └── chapters/          # 章节正文
│   └── book-workspace/        # 写书工作区（不面向读者）
│       ├── planning/          # 提纲、排期、项目状态
│       ├── architecture-notes/ # 主题深读笔记
│       ├── references/        # 证据映射、交接模板
│       ├── methodology/       # 写作原则、方法论
│       ├── status/            # 阶段快照
│       ├── workflow/          # 工作方法、协作说明
│       ├── inbox/             # 三方异步通信
│       ├── excavation-tasks/  # 挖掘任务卡
│       └── skills/            # 可复用 skill 草案
├── blog/                      # 在线博客（可选）
├── meta/                      # 方法论沉淀（本目录）
└── COMMIT-CONVENTIONS.md      # 提交规范
```

**关键设计原则**：书本体（`docs/book/`）和写书工作区（`docs/book-workspace/`）严格分离。读者只需要看 `docs/book/`，写作者和 Agent 在 `docs/book-workspace/` 里工作。

---

## 质量验收标准

一章合格的源码解读，至少满足：有且仅有一个清晰核心主张（可被反驳）、关键结论具备源码锚点（无法验证处已标注 inference）、解释了"为什么这样设计，不这样会怎样"、至少 1 个非显然发现、删除冗余后信息量未被掏空。

---

## 给未来自己的七条规则

1. 先找控制权，再找模块。
2. 先写问题，再写答案。
3. 先证据，再判断。
4. 先机制，再价值。
5. 先边界，再主流程。
6. 先做材料库，再写章节。
7. 任何时候都不要把源码分析写成说明书。
