# 如何基于一个源码仓库构建一本解读书

> 这篇文章沉淀自 `claude-code-source` 项目的真实实践，时间跨度 2026-04-01 ~ 2026-04-02，133 次提交，19 章书稿，三方协作（源码分析 / 写作 / 验收）。

---

## 一、为什么这件事比想象中难

拿到一个陌生仓库，想写一本解读书，最自然的冲动是：打开目录树，从第一个文件夹开始介绍。

这条路走不通。

按目录树写出来的东西叫"模块说明书"，不叫书。读者看完知道"有这些东西"，但不知道"为什么这样设计"，不知道"哪里是真正的难点"，也不知道"这套系统在解决什么根本矛盾"。

真正有价值的源码解读书，必须回答三个问题：

1. 这个系统在解决什么核心矛盾？
2. 它为什么非这样设计不可？
3. 哪些地方是真正的护城河，哪些是可以替换的外围配件？

这三个问题，靠读目录树是回答不了的。

**仓库**：Claude Code 源码（已公开部分）

**目标**：写一本面向技术读者的系统级解读书，19 章，同时配套一个在线博客

**团队**：三方协作

- 验收侧（xuhengzhi75）：负责章节质量验收

## 三、核心方法：三阶段工作流

### 阶段 A：仓库理解（不急着写章节）

第一步不是写，是建立全局定位。

具体做法：
- 找入口：CLI 入口在哪里？daemon 在哪里？HTTP 接口在哪里？
- 找主执行链：request → routing → orchestration → tools → state → output
- 找状态载体：session、task、memory、checkpoint、recovery object 分别是什么
- 找关键壁垒：tooling、runtime、安全边界、恢复机制、任务系统
- 找异常闭环：失败、重试、续跑、超时、终止

这个阶段的产出不是章节，而是：
- `system-overview`：系统全局定位
- `runtime-structure`：运行时结构
- `execution-flow-guide`：执行链路图
- `task-recovery-map`：任务恢复路径

**关键原则**：不按目录树写，按"控制权转移"写。请求从哪里进入，控制权交给了谁，哪些条件会继续/中断/恢复，哪些状态必须被保存，哪些组件一删系统就塌。

### 阶段 B：深读与证据整理

目标是把"理解"变成可引用材料。

对每个核心主题（如 QueryEngine / QueryLoop / Task / Tooling / Recovery），建立：
- **verified facts**：有源码锚点（文件 + 函数/分支/对象，关键点带行号）
- **inference**：基于证据的推断，必须显式标注，措辞保守
- **edge cases**：边界情况和异常路径
- **design tradeoffs**：设计取舍和替代方案
- **why-it-matters**：为什么这个机制重要

这个阶段的产出：
- `architecture-notes/`：主题深读笔记
- `chapter-evidence-map.md`：章节证据映射
- `analysis-handoff-template.md`：交接模板

**关键原则**：先做材料库，再写章节。不要在没有证据地图的情况下直接大篇幅写作。

### 阶段 C：成书写作

从材料库变成章节，不在写章时重新摸索源码。

每章的写作流程：
1. 先定本章的"第一性问题"（要解决的根本矛盾，不是"介绍 X 系统"）
2. 从 evidence map 拉证据
3. 先写论点骨架，再补事实
4. 每章加入：why now / design tradeoff / edge case / non-obvious conclusion
5. 写完后检查是否退化成说明书

**标准章节骨架**：
1. 核心主张（一句话，可被反驳）
2. 第一性问题（为什么需要这个机制）
3. 机制展开（关键决策 + 为什么不是替代方案）
4. 证据锚点（verified）
5. 边界与技术债（何时失效/何时退化）
6. 可迁移原则（适用条件 + 反例）
7. 本章不覆盖项（章节边界）

## 四、最容易踩的坑

### 坑 1：模块罗列

表现：按目录顺序写，每节都像 README，信息正确但没有穿透力。

修正：改成"主矛盾 + 关键链路 + 设计取舍"结构。

### 坑 2：常识性总结

表现：说了很多"AI 代理需要状态/工具/记忆"这类正确废话，没回答"这个系统为什么非这样不可"。

修正：逼自己写出不可删组件、替代方案为何不行、它真正难在哪里。

### 坑 3：没有显式证据地图

表现：结论散在多个文档中，交接后容易丢上下文，多 Agent 协作时无法续写。

修正：为章节建立 evidence map / handoff template / status index，任何时候都能从状态文档恢复上下文。

### 坑 6：挖掘任务卡粒度不够

表现：写作侧发现内容薄弱，给分析侧的任务卡只写了"这章需要更多细节"，分析侧不知道从哪里下手，挖出来的东西还是不够用。

修正：挖掘任务卡必须做到三级分层——🔴 必答（写作侧无法在没有这些素材的情况下改文章）、🟡 加分项（有这些素材文章能从 7 分升到 9 分）、🟢 意外发现区（不在问题清单里的有趣东西）。每个问题必须指定目标文件和行号范围，让分析侧知道去哪里找。

这套分级设计的关键价值：🔴 保证最低可用性，🟡 驱动深度，🟢 给分析侧留空间发现真正有价值的反直觉细节——而这类细节往往是文章最有价值的部分。

### 坑 4：章节没有观点，只有陈述

表现：章节像说明书分类，缺问题意识，缺推进感。

修正：每章先写问题句，再围绕问题组织证据。核心主张必须是可被反驳的判断，不是无聊的陈述。

### 坑 5：推断伪装成事实

表现：把基于结构推断的判断写成了已验证的事实，读者无法区分。

修正：严格区分 `verified`（有源码锚点）和 `inference`（基于证据的推断），推断必须显式标注。

## 五、多 Agent 协作的关键规范

这次实践是三方协作，暴露了一些单人写作不会遇到的问题。

### 交接协议

分析侧交付最少包含：
1. 一句话结论
2. 本章最该讲的 3~5 点
3. 明确不要重复讲的点
4. 关键锚点（文件 + 函数/分支/对象）
5. 仅能写为 inference 的判断
6. 章节边界（讲到哪/不讲哪）

没有这些，写作侧拿到材料也不知道从哪里下手。

### 提交规范

统一使用 `type(scope): purpose` 格式：
- `docs(book): split manuscript from writing workspace`
- `docs(chapters): tighten ch04-ch08 information density`
- `docs(evidence): strengthen query engine anchors`

**一次提交只做一类事**，不要把章节重写 + 证据更新 + 结构整理混在一个 commit 里。

原因：多人协作时，混提会让 review 和回滚都变得困难。

### 协作节奏

- 开工前先 pull
- 完成一小段可验收成果就 push
- 轮动推进，不攒大包

### Inbox 通信机制

多 Agent 之间通过 `docs/book-workspace/inbox/` 目录下的文件异步通信，每个角色有独立的收件箱文件（`for-[角色名].md`）。这比 GitHub Issue 更轻量，比实时消息更持久，任何 Agent 实例拉取代码后都能看到完整的通信历史。

**最容易踩的坑**：任务完成后忘记更新 inbox 状态字段。如果不把 `状态：待处理` 改为 `状态：已处理 ✅`，下一个 Agent 实例拉取代码后会重复处理同一条任务。规则：**任务完成后必须更新状态，并在 commit message 里注明**。

完整的通信机制设计，见 `docs/book-workspace/workflow/multi-agent-git-collab.md`。

### 安全规则

真实姓名、内部账号等敏感信息不入仓库。用 pre-commit hook 自动检测，检测逻辑存本地配置文件，不入仓库。

## 六、目录结构建议

对任何"源码分析成书"项目，推荐统一使用：

```
project/
├── docs/
│   ├── book/
│   │   ├── README.md          # 读者入口
│   │   └── chapters/          # 章节正文
│   └── book-workspace/        # 写书工作区（不面向读者）
│       ├── planning/          # 提纲、排期、项目状态
│       ├── architecture-notes/ # 主题深读笔记
│       ├── references/        # 证据映射、交接模板
│       ├── methodology/       # 写作原则、方法论
│       ├── status/            # 阶段快照
│       ├── workflow/          # 工作方法、协作说明
│       └── skills/            # 可复用 skill 草案
├── blog/                      # 在线博客（可选）
└── COMMIT-CONVENTIONS.md      # 提交规范
```

**关键设计原则**：书本体（`docs/book/`）和写书工作区（`docs/book-workspace/`）严格分离。读者只需要看 `docs/book/`，写作者和 agent 在 `docs/book-workspace/` 里工作。

## 七、质量验收标准

一章合格的源码解读，至少满足：

**基础线**：
- 有且仅有一个清晰核心主张（可被反驳）
- 关键结论具备源码锚点；无法验证处已标注 inference
- 解释了"为什么这样设计，不这样会怎样"
- 至少 1 个非显然发现
- 删除冗余后信息量未被掏空

**质量增强线**：
- 说明了至少一个替代方案及未采纳原因
- 写明关键假设与适用条件
- 指出边界条件或技术债
- 读者读完后需记住的核心概念 ≤ 3

**对抗级**：
- 核心主张能经受强反驳
- 结论没有过度泛化
- 每个主张有：源码锚点（直接证据）+ 机制解释（为什么这个证据支持主张）

## 八、给未来自己的七条规则

1. 先找控制权，再找模块。
2. 先写问题，再写答案。
3. 先证据，再判断。
4. 先机制，再价值。
5. 先边界，再主流程。
6. 先做材料库，再写章节。
7. 任何时候都不要把源码分析写成说明书。

---

## 九、这套方法的适用边界

**适合**：
- 有一定规模的复杂系统（10 万行以上）
- 目标读者是技术人员
- 需要多轮续写，不是一次性输出
- 有多人或多 Agent 协作

**不适合**：
- 简单的库或工具（直接写 README 就够了）
- 目标是快速出一篇博客文章（这套流程太重）
- 没有源码访问权限（无法建立 verified 证据）

**不应承诺**：
- 一次性全自动写完整本高质量书
- 在没有源码证据的情况下输出深度结论
- 跳过人工判断直接生成最终权威内容
