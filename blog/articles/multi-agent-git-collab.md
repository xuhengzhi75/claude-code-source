# 让多个 AI Agent 通过 Git 仓库协作：一个正在运行的实验

> 本文记录的是一个真实运行中的实践：三个 AI Agent 实例，分工不同，通过 Git 仓库异步协作，共同完成一本技术书稿的写作。

---

## 问题

我们在写一本关于 Claude Code 源码架构的书。三个角色分工明确：

- **源码分析侧**：深挖源码，产出架构笔记和交接材料
- **写作侧**：把分析材料转化成章节正文
- **验收侧**：目录结构设计、内容验收、最终定稿

这三个角色分别由不同的 AI Agent 实例（运行在不同的 CatDesk 会话里）承担。问题来了：**它们没有共享内存，无法直接通信。**

怎么让三个独立的 Agent 实例协作起来？

---

## 直觉方案的问题

**GitHub Issue**：可以用，但摩擦点多。每次交接要手动开 issue，issue 和代码文件没有直接关联，通知延迟，关闭后上下文容易丢失。

**实时通信（大象/Slack）**：适合紧急沟通，但不适合任务交接。消息不持久，Agent 会话结束后无法回溯历史，无法携带结构化的代码片段和锚点。

**共享数据库/消息队列**：过重。为一个写作项目引入 Redis 或 MQ，杀鸡用牛刀。

---

## 我们的方案：Git 仓库即通信总线

Git 仓库天然具备通信系统需要的所有特性：

- **持久化**：消息不会因会话结束而丢失
- **版本历史**：所有通信有完整记录，可追溯
- **原子性**：commit 是原子操作，不会出现半写状态
- **推拉模型**：push/pull 即发送/接收

把这些特性用于 Agent 间通信，不需要任何额外基础设施。

### 两层通信结构

**第一层：inbox 文件（异步消息）**

每个角色有一个专属收件箱文件：

```
docs/book-workspace/inbox/
├── for-小许.md        ← 源码分析侧/验收侧 → 写作侧
├── for-startheart.md  ← 写作侧/验收侧 → 源码分析侧
└── for-xuhengzhi75.md ← 源码分析侧/写作侧 → 验收侧
```

消息格式简单：

```markdown
---
发件人：startheart
时间：2026-04-02
主题：权限系统交接卡已就绪

分析材料已产出，路径：architecture-notes/permission-system.md
核心内容：两阶段流水线、六种 PermissionMode、auto 模式分类器...

状态：待处理
---
```

处理完后把状态改为"已处理"，不删除——历史即文档。

**第二层：结构化任务卡（excavation-tasks）**

对于需要精确交接的任务，用结构化的任务卡文件：

```
docs/book-workspace/excavation-tasks/
├── ch11-excavation.md   ← 写作侧起草问题，源码分析侧填答案
├── ch12-excavation.md
└── ch13-excavation.md
```

任务卡按优先级分三级（🔴 必答 / 🟡 加分项 / 🟢 意外发现），每条结论标注 `verified`（有行号）或 `inference`（推断），附源码路径 + 行号 + 关键代码片段。

---

## 工作流

```
源码分析侧
  → 深挖源码，产出 architecture-notes/
  → 往 inbox/for-写作侧.md 追加消息
  → git commit + push

写作侧 pull
  → 读取 inbox，读取 architecture-notes/
  → 写章节正文
  → 往 inbox/for-验收侧.md 追加消息
  → git commit + push

验收侧 pull
  → 验收章节，调整目录
  → 往 inbox/for-源码分析侧.md 追加反馈
  → git commit + push
```

铁律只有两条：**完成即 push，开工前先 pull**。

---

## 这套机制解决了什么

**上下文持久化**：Agent 会话结束后，下次启动的新实例通过读 inbox + architecture-notes + workflow 文档，可以在几分钟内了解项目全貌并接续工作。不需要在 prompt 里塞大量背景信息。

**角色边界清晰**：每个文件有明确的生产者和消费者。源码分析侧不写章节正文，写作侧不深挖源码，验收侧不做具体实现。边界清晰，不重复作业。

**异步不阻塞**：三个角色各自独立运行，互不等待。源码分析侧在挖 ch13，写作侧在润色 ch07，验收侧在调整目录结构——同时进行，互不干扰。

**协作历史可追溯**：git log 就是完整的协作历史。什么时候谁产出了什么，什么时候谁消费了什么，一目了然。

---

## 抽象成通用模式

```
Agent A                   Git 仓库                   Agent B
  |                          |                          |
  |-- 写文件 + push -------> |                          |
  |                          | <------- pull + 读文件 --|
  |                          |                          |
  |                          | <-- 写文件 + push -------|
  | -- pull + 读文件 ------> |                          |
```

**文件即消息，commit 即发送，pull 即接收。仓库是共享内存，git 是通信协议。**

这个模式适合以下条件同时满足的场景：多个 AI Agent 实例需要协作，但运行在不同会话/进程里；任务是异步的，不需要实时响应；已经在用 Git 仓库管理项目产出；角色分工明确，每个角色有固定的输入和输出。

---

## 局限性

这套机制不适合需要实时交互的场景。如果 Agent A 必须等待 Agent B 的即时回复才能继续，push/pull 的延迟会成为瓶颈。这种情况下需要真正的消息队列或实时通信。

另外，这套机制依赖所有参与者遵守"完成即 push，开工前先 pull"的纪律。如果有人攒了大量改动才推，会产生合并冲突，破坏协作节奏。

---

## 现在的状态

这套机制在本项目里已经运行了几天。三个 Agent 实例通过 Git 仓库完成了 19 章书稿的写作、润色、源码分析和架构笔记的产出。inbox 文件里有完整的通信历史，excavation-tasks 里有结构化的任务交接记录。

它工作得出乎意料地好。主要原因可能是：Git 本来就是为异步协作设计的，只是通常用于人类协作。把它用于 Agent 协作，逻辑上是一致的。
