# 第8篇：它凭什么比普通 AI 工具更靠谱

你可能用过很多 AI 工具。

有些工具演示的时候很厉害，但真正用来做事的时候，总是在关键时刻掉链子。

Claude Code 不一样的地方，不是它的功能更多，而是它对"出错"这件事的处理方式不同。

---

## 演示和生产之间的距离

一个 AI 工具，能演示三步工具调用，看起来很厉害。

但真正能用来做事的工具，还需要回答：第二步超时了怎么办？网络断了怎么恢复？上下文太长了怎么处理？

这些问题，演示的时候不会出现。但在真实使用中，它们会出现。

**功能数量决定演示效果，错误处理密度决定系统能走多远。**

## 出错的时候，默认选更保守的那一边

Claude Code 的工具系统有一个设计原则：当不确定的时候，默认选更严格的限制。

比如，一个工具如果没有声明"我是只读的"，系统会默认它可能会修改文件，在某些场景下需要额外确认。

这会带来一点不便——用户可能多一次确认弹窗。但反过来，如果默认是"只读"，一个真正会修改文件的工具就可能在不该运行的时候悄悄跑起来，造成数据损坏。

数据损坏的 bug 很难发现，因为它在低并发的时候几乎不出现，只有在高并发的时候才暴露，而且很难复现。

**多一次弹窗，和数据损坏且难以复现，这两种代价不对等。** 所以选前者。

## 一个 flag，防止烧掉几十万次 API 调用

Claude Code 的代码里有一个布尔值，叫 `hasAttemptedReactiveCompact`。

它的作用是：记录"是否已经尝试过压缩上下文"。

为什么需要这个？因为曾经发生过这样的事故：

上下文太长了，触发压缩。压缩之后还是太长，再次触发压缩。再次压缩之后还是太长，再次触发……无限循环，一直在烧 API 费用。

代码注释里记录了这次事故的规模：1279 个会话里出现了这个问题，最极端的一个会话连续失败了 3272 次，全球每天因此浪费约 25 万次 API 调用。

解决方案是：压缩过一次之后，把这个 flag 设为 true。下次再遇到上下文太长，不再尝试压缩，而是让错误暴露出来。

**宁可明显失败，也不陷入无声的无限循环。**

## 崩溃恢复：先记录，再执行

Claude Code 在执行任何操作之前，会先把你的请求写入磁盘。

这样，如果执行过程中进程崩溃了，你的请求已经在磁盘上了。重启之后，系统知道上次做到哪里，可以从那个点继续。

但这里有一个细节：崩溃的情况不止一种。

有时候是你的请求发出去了，但 AI 还没开始回复，进程就崩了。有时候是 AI 已经开始回复了，回复到一半，进程崩了。

这两种情况的恢复方式不一样。Claude Code 会区分这两种情况，分别处理，而不是统一报错让你从头来。

## 权限隔离：后台任务不能干扰主流程

Claude Code 有一些后台任务，比如整理记忆、更新状态。

这些后台任务的权限被严格限制：它们只能做自己该做的事，不能访问其他工具，不能修改其他文件。

为什么要这么限制？因为如果后台任务能访问任意工具，它就可能意外触发一些操作，干扰你正在进行的主任务。

这种隔离不是靠约定（"大家都注意一点"），而是靠代码强制执行的。后台任务想做越权的事，系统直接拒绝，不给机会。

## 这些设计的共同点

回顾这几个例子，你会发现一个共同点：

它们都不是在功能上做加法，而是在出错路径上做加固。

演示 Agent 和能干活的 Agent 之间的差距，就是这些出错路径的处理密度。

一个工具能不能在真实环境里长期稳定运行，不取决于它能做多少事，而取决于它在出错的时候能不能优雅地处理，不让一个小问题变成大灾难。
