# 第9篇：用 AI 工具时，你真正在依赖什么

你在用 Claude Code 帮你做事。

你信任它，让它读你的文件，执行命令，修改代码。

但你有没有想过：你到底在依赖什么？

---

## 不是在依赖"AI 很聪明"

很多人用 AI 工具，依赖的是"AI 很聪明，不会犯错"。

这是一个危险的假设。

AI 会犯错。Claude Code 也会犯错。

真正值得依赖的，不是 AI 不犯错，而是**系统在 AI 犯错的时候，能不能把损失控制在可接受的范围内。**

## 权限系统：不是一个开关，是一条流水线

Claude Code 的权限系统，不是简单的"允许/拒绝"。

它分两个阶段：

第一阶段是"硬边界"：无论你选了什么模式，这些检查都会执行，都无法绕过。比如，`.git/` 目录的写保护，`npm publish` 这类危险操作的确认，这些是系统级的安全底线。

第二阶段才是"模式策略"：根据你选择的工作模式，决定其他操作是自动允许、询问你，还是直接拒绝。

这个设计的关键点是：**即使你选择了"跳过所有权限检查"的模式，第一阶段的安全底线仍然有效。** 你无法用任何模式绕过 `.git/` 写保护。

## 你在依赖的是：损失可控

当你让 Claude Code 执行一个命令，你在依赖的不是"它一定会做对"，而是：

如果它做错了，损失是可控的。

这体现在几个地方：

**操作前确认**：对于危险操作，系统会在执行前询问你。你有机会在损失发生之前叫停。

**权限隔离**：Claude Code 的后台任务只能做自己该做的事，不能越权操作其他文件或执行其他命令。一个任务出了问题，不会连带影响其他任务。

**状态可恢复**：任务状态存在文件里，进程崩溃后可以恢复。你不会因为一次崩溃就失去所有进度。

## 你在依赖的是：行为可预测

另一件你在依赖的事是：系统的行为是可预测的。

Claude Code 的权限规则是明确的，不是随机的。你知道哪些操作会被拦截，哪些会被允许，哪些会询问你。

这种可预测性很重要。如果系统的行为是随机的，你就无法建立信任，也无法放心地把重要任务交给它。

## 一个值得思考的问题

如果你在用一个 AI 工具，可以问自己几个问题：

这个工具在出错的时候，会怎么处理？是静默失败，还是明确报错？

这个工具的权限边界在哪里？它能访问什么，不能访问什么？

如果这个工具做了一件你不想要的事，你能撤销吗？

这些问题的答案，决定了你能在多大程度上信任这个工具，以及能把多重要的任务交给它。

## 积累的上下文是另一种依赖

还有一种依赖，很多人没有意识到：**你在依赖工具对你的了解。**

Claude Code 会在后台整理对话中的重要信息，保存到记忆文件里。用得越久，它对你的了解越深，帮助越精准。

这是一种好的依赖——它让工具越来越好用。

但这也意味着：如果你换了一个工具，这些积累的上下文就没了，需要重新建立。

在选择 AI 工具的时候，值得考虑：一年后，你在这个工具里积累的东西，能不能带走？
