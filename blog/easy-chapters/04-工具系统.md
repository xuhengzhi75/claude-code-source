# 第4篇：它的"手"和"脚"是什么

Claude Code 能帮你读文件、写代码、执行命令、搜索内容。

这些能力不是 AI 模型自带的。AI 模型本身只能生成文字，它没有办法直接操作你的电脑。

那它是怎么做到的？

---

## 工具：AI 的手脚

Claude Code 给 AI 模型配备了一套"工具"。每个工具对应一种具体的操作：读文件、写文件、执行命令、搜索代码……

当 AI 模型需要做某件事的时候，它不是直接去做，而是"调用"对应的工具，由工具去执行，再把结果告诉模型。

这就像一个指挥官和执行者的关系：模型负责决策（"我需要读一下这个文件"），工具负责执行（真正去读文件，把内容返回来）。

## 工具自带"说明书"

Claude Code 的工具设计有一个特别的地方：每个工具不只是一段执行代码，它还随身携带一套"说明书"，描述自己的安全属性。

比如：
- 这个工具是只读的，还是会修改文件？
- 这个工具的操作是可以撤销的，还是不可逆的？
- 这个工具可以和其他工具同时运行，还是必须排队？

这些信息不是由外部系统来判断的，而是工具自己声明的。

这样设计的好处是：不管以后增加多少工具，管理逻辑不会变得越来越复杂。每个工具自己负责说清楚自己的边界，调用方只需要读这些声明，不需要为每个工具单独写判断逻辑。

## 一个有意思的默认值设计

工具的安全声明有一个细节：**默认值选的是"更保守"的那一边。**

如果一个工具忘了声明"我是只读的"，系统会默认它可能会修改文件，在某些受限模式下就不让它运行。

反过来，如果默认是"只读"，那么一个真正会修改文件的工具，就可能在不该运行的时候悄悄跑起来，造成意外的修改。

两种遗漏的代价不一样：一个是"功能受限"，一个是"安全缺口"。所以默认选限制，想要更宽松的权限，必须主动声明。

这个逻辑在很多安全设计里都适用：**不确定的时候，选更保守的那一边。**

## 多个工具可以同时运行吗

可以，但有条件。

当 AI 模型同时发出多个工具调用时，Claude Code 会判断哪些可以并行，哪些必须排队。

判断规则很简单：如果所有正在运行的工具都是"并发安全"的，新来的工具也是安全的，那就可以同时跑。否则，等前面的跑完再说。

举个例子：模型同时要求"搜索代码"和"读文件"，这两个都是只读操作，可以同时进行。但如果还有一个"执行命令"（可能会修改文件），它就得等前两个完成后再运行。

这样做的原因是：如果一个命令正在修改文件，同时另一个工具在读这个文件，读到的可能是修改到一半的内容，结果不可预期。

## 工具在你看到之前就已经"定好了"

还有一个细节：Claude Code 在把工具列表发给 AI 模型之前，就已经完成了所有的筛选和排序。

被你的配置禁止的工具，模型根本看不到，不是"看得到但用不了"。

这个区别很重要。如果模型能看到一个工具但调用时被拒绝，它可能会反复尝试，浪费时间和资源。但如果这个工具从一开始就不在它的视野里，它就不会去尝试。
