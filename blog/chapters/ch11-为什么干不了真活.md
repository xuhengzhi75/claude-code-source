# 第11章 为什么很多 Agent 看起来像但干不了真活

演示场景和真实场景的差距，不是功能多少的差距，而是"边界情况处理密度"的差距。

演示任务通常有几个特点：任务边界清晰、输入格式规范、不需要中断恢复、不需要处理并发、失败了可以重来。真实工作场景则相反：任务边界模糊（用户说"帮我整理一下这个项目"，没有明确终止条件）、输入格式不规范（代码里有历史遗留的奇怪写法）、进行到一半可能被打断、多个任务可能同时进行、失败了需要从中间状态恢复。

很多 Agent 的设计隐含地假设了"任务会正常完成"。这个假设在演示场景下成立，在真实场景下不成立。

## 11.1 工具调用的错误处理

Claude Code 的工具调用有明确的错误处理层级。`src/utils/errors.ts#L12` 定义了 `AbortError`（用户主动中止），`errors.ts#L27` 的 `isAbortError()` 统一判断三种中止来源：自定义 `AbortError`、`DOMException`、SDK 的 `APIUserAbortError`。

`AbortError` 表示用户主动中止，不需要重试，直接清理状态。`src/utils/conversationRecovery.ts#L141` 定义的 `TurnInterruptionState` 区分了 `interrupted_prompt` 和 `interrupted_turn` 两种中断类型，对应不同的恢复策略。

很多 Agent 实现里，工具调用失败只有一种处理方式：报错、停止、让用户重试。这在演示里看起来没问题，因为演示任务通常不会触发这些边界情况。但在真实使用中，用户主动中止的任务不应该重试，网络超时的任务应该重试，这两类情况需要区分处理。

## 11.2 状态管理的一致性

`src/query.ts#L282` 的 `queryLoop()` 初始化了一个 `State` 对象，把所有跨迭代的可变状态集中在一处：`messages`、`toolUseContext`、`maxOutputTokensOverride`、`hasAttemptedReactiveCompact`、`turnCount` 等。注释里解释了为什么要用单一对象：每个恢复/继续分支可以原子替换所有循环变量，避免新增字段时遗漏更新。

这是显式状态管理的价值：任何时刻都能知道系统处于哪个状态，以及从这个状态能做什么操作。

很多 Agent 实现里，状态是隐式的：通过变量是否为 null、数组是否为空、某个 flag 是否为 true 来判断。这在简单场景下够用，但在并发、中断、恢复等复杂场景下，隐式状态很容易出现不一致。集中状态对象的问题是：随着系统演进，`State` 对象可能越来越大，包含越来越多的字段。如果不定期清理，`State` 会变成一个"什么都往里放"的垃圾桶，失去集中管理的意义。

## 11.3 上下文窗口的主动管理

`src/query.ts#L1089` 的注释解释了 `hasAttemptedReactiveCompact` 这个 flag 的来历：之前在停止钩子阻断错误后将其重置为 `false` 曾导致无限循环——压缩 → 仍然过长 → 错误 → 停止钩子阻断 → 压缩 → 无限循环，烧掉大量 API 调用。这个 flag 的存在不是随意的，是修复了一个真实的无限循环 bug。

被动等待的问题是：当 API 报 context window exceeded 时，当前轮次的工作已经失败，需要从上一个检查点恢复。如果没有检查点机制，就需要从头开始。主动管理需要知道当前 token 使用量，以及压缩的触发阈值。这两个信息都需要从 API 响应里提取，并在系统内维护。

## 11.4 工具权限的边界

`src/services/SessionMemory/sessionMemory.ts#L460` 的 `createMemoryFileCanUseTool()` 展示了细粒度权限的实现方式：不是"所有工具都可以用"，也不是"所有工具都需要确认"，而是"只有 Edit 操作且只针对特定文件才允许"。

权限管理的价值不只是安全，还有可预测性：用户知道 Agent 能做什么、不能做什么，不会出现 Agent 意外执行了用户没有预期的操作。细粒度权限需要对工具的风险等级有明确的分类，以及对用户意图的准确判断。这两件事都需要专门设计，不能靠"默认允许"或"默认拒绝"来替代。

## 11.5 长任务的进度可见性

[`src/query.ts#L1694`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/query.ts#L1694) 在 `BG_SESSIONS` 模式下，主循环会定期生成 task summary，让 `claude ps` 命令能看到后台任务的执行进度。这不是核心功能，但对真实使用体验影响很大。

如果任务是黑盒，用户只能等待，不知道是在正常执行还是已经卡住。进度可见性让用户知道任务在进行中，知道大概到哪一步了，知道如果出问题在哪里查看。

## 11.6 心智模型验证题

一个 Agent 在演示里能完成"帮我写一个 Python 脚本"这个任务，但在真实使用中频繁失败。最可能的原因是什么？

答案：演示任务的边界清晰（写一个脚本，完成即止），输入格式规范（用户提供了清晰的需求），不需要中断恢复，不需要处理并发。真实使用中，用户的需求可能模糊（"帮我整理一下这个项目"），输入可能包含不规范的代码，任务可能被打断，多个任务可能同时进行。这些情况在演示里不会触发，在真实使用中频繁出现，而 Agent 没有针对这些情况的处理机制。

## 11.7 本章小结

很多 Agent 在演示里好用，在真实工作中暴露问题，原因在于：错误处理不完整（[`errors.ts#L12`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/utils/errors.ts#L12)）、状态管理不一致（[`query.ts#L282`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/query.ts#L282)）、上下文窗口被动管理（[`query.ts#L1089`](https://github.com/xuhengzhi75/claude-code-source/blob/f82ac18334ec9ca9890b9294ea8efa72773fa97c/src/query.ts#L1089)）、工具权限粗粒度、长任务进度不可见。

演示和真实之间的距离，不是功能的距离，是对"出错"这件事的态度距离。演示假设一切顺利，真实要求一切都可能出错。
