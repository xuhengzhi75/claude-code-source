# 第17章 任务推进与运行时结构

`claude ps` 能实时显示后台任务的执行进度。这不是魔法，而是任务系统和主循环之间有明确的接缝。主循环执行过程中定期向任务系统报告进度，任务系统把进度写入文件，`claude ps` 读取文件展示。

没有这个接缝，任务是黑盒。有了接缝，长任务的执行状态才对外可见。

但任务系统的设计远不止"存个状态"。它实际上是一套多 Agent 协作的调度基础设施，处理了五类你不当心就会踩坑的问题：ID 重用、竞争认领、任务阻塞图、Agent 宕机后的任务归还、跨进程共享的任务空间。

## 17.1 任务的数据结构

[`tasks.ts#L76`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/utils/tasks.ts#L76) 的 `TaskSchema` 定义了任务的字段：

```typescript
z.object({
  id: z.string(),
  subject: z.string(),
  description: z.string(),
  activeForm: z.string().optional(), // 进行时形式，用于 spinner（如 "Running tests"）
  owner: z.string().optional(),       // agent ID
  status: TaskStatusSchema(),         // 'pending' | 'in_progress' | 'completed'
  blocks: z.array(z.string()),        // 本任务阻塞哪些任务
  blockedBy: z.array(z.string()),     // 哪些任务阻塞本任务
  metadata: z.record(z.string(), z.unknown()).optional(),
})
```

`activeForm` 这个字段值得单独说。它不是任务描述，而是进行时形式的动作短语——`"Running tests"` 而不是 `"run test suite"`。这个字段专门为 UI spinner 设计：当任务状态是 `in_progress` 时，UI 可以直接显示 `activeForm` 作为进度文案，不需要自己生成文案。

`blocks` 和 `blockedBy` 是双向阻塞图。`tasks.ts#L458` 的 `blockTask()` 同时更新两端：A blocks B 时，A 的 `blocks` 加入 B 的 ID，B 的 `blockedBy` 加入 A 的 ID。`deleteTask()` 删除任务时会级联清理，把该任务从所有其他任务的 `blocks` 和 `blockedBy` 里移除，防止悬挂引用。这是一个图的一致性问题，在多 Agent 并发创建/删除任务时如果不做级联清理，图会逐渐腐化。

## 17.2 ID 防重用：高水位标记

任务状态存在文件系统里，每个任务是 `<id>.json`。删除任务时，文件被删除，ID 就空出来了。如果下一个任务复用了这个 ID，历史记录里可能出现混淆：同一个 ID 对应过两个不同任务。

[`tasks.ts#L92`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/utils/tasks.ts#L92) 的高水位标记（`.highwatermark` 文件）解决这个问题。它记录曾经分配过的最大 ID。`createTask()` 在分配新 ID 时取"当前文件中最大 ID"和"高水位标记"的较大值，保证新 ID 总是递增。`deleteTask()` 在删除文件之前，先把被删任务的 ID 写入高水位标记（如果它比当前标记更大）。

高水位标记还在 `resetTaskList()` 里用到——多 Agent swarm 开始新一轮协作前需要清空任务池，`resetTaskList()` 删除所有任务文件，但先把当前最高 ID 写入高水位标记，保证新一轮的 ID 从上一轮结束的地方继续递增。

这是一个零依赖的方案：不用数据库，不用分布式 ID 生成器，一个文本文件解决问题。代价是每次分配 ID 都需要读高水位标记文件，但对于任务创建频率来说这个开销可以忽略。

## 17.3 两种锁粒度

任务系统有两种锁：任务级锁（锁单个任务文件）和任务列表级锁（锁整个任务池）。

大多数操作用任务级锁就够了。`updateTask()` 在修改一个任务时只锁这一个任务的文件，其他任务的并发修改不受影响。

但 `claimTask()` 需要更粗的锁。认领任务包含两步：检查 Agent 是否空闲，以及认领任务。这是典型的 check-then-act 场景——如果两步之间有时间窗口，两个 Agent 可能同时通过检查，同时认领同一个任务。

[`tasks.ts#L621`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/utils/tasks.ts#L621) 的 `claimTaskWithBusyCheck()` 用任务列表级锁把检查和认领原子化：

```typescript
async function claimTaskWithBusyCheck(...) {
  const lockPath = await ensureTaskListLockFile(taskListId)
  release = await lockfile.lock(lockPath, LOCK_OPTIONS)

  // 在同一把锁内：读所有任务、检查 agent 是否繁忙、检查任务是否可认领、认领任务
  // 中间不能释放锁
}
```

`ClaimTaskResult` 的失败理由有五种：`task_not_found`、`already_claimed`、`already_resolved`、`blocked`、`agent_busy`。

`agent_busy` 是其中最值得注意的一种。当一个 Agent 已经持有未完成的任务时，它被阻止认领新任务。`busyWithTasks` 字段返回它正在处理的任务 ID 列表，让调度方可以做出判断：等待这个 Agent 完成当前任务，还是把新任务分配给其他空闲 Agent。

任务级锁和列表级锁的区别不是技术细节，而是反映了两种不同的不变量：更新单个任务只需要保证这一个任务的原子性，而认领任务需要保证"Agent 状态"和"任务状态"的联合原子性。用错锁粒度，要么性能下降（用列表锁保护所有更新），要么竞争条件（用任务锁保护检查-认领）。

锁实现用的是文件锁（`.lock` 文件），不是数据库事务。文件锁的优势是零依赖、可以直接检查锁文件的存在；缺点是只在单机上有效。如果 Agent 分布在多台机器上，文件锁失效，需要改用 Redis 或数据库的分布式锁。

## 17.4 协作空间主键：谁的任务池

多 Agent 系统里，所有 Agent 必须操作同一个任务池，不能各自维护独立列表。任务池用目录区分：`~/.claude/tasks/<taskListId>/`。

[`tasks.ts#L199`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/utils/tasks.ts#L199) 的 `getTaskListId()` 决定 taskListId 是什么：

```typescript
export function getTaskListId(): string {
  if (process.env.CLAUDE_CODE_TASK_LIST_ID) {
    return process.env.CLAUDE_CODE_TASK_LIST_ID  // 显式指定
  }
  const teammateCtx = getTeammateContext()
  if (teammateCtx) {
    return teammateCtx.teamName  // 进程内 teammate 使用 leader 的 team name
  }
  return getTeamName() || leaderTeamName || getSessionId()
}
```

优先级有五级：显式环境变量 > in-process teammate 上下文 > 环境变量 `CLAUDE_CODE_TEAM_NAME` > leader 通过 `setLeaderTeamName()` 注册的名字 > session ID。

这个优先级设计解决了一个具体问题：tmux/iTerm2 模式下，外部 teammate 进程通过 `CLAUDE_CODE_TEAM_NAME` 环境变量知道自己属于哪个 team；进程内 teammate 通过 `teammateCtx.teamName` 知道；leader 自己调 `setLeaderTeamName()` 注册。三种启动方式最终都收敛到同一个 taskListId，写入同一个目录。

注意 `setLeaderTeamName()` 里有一个副作用：

```typescript
export function setLeaderTeamName(teamName: string): void {
  if (leaderTeamName === teamName) return
  leaderTeamName = teamName
  notifyTasksUpdated()  // 通知订阅者：任务池变了
}
```

leader 创建 team 时不只是改了一个变量，还触发了"tasks updated"事件。UI 订阅了这个事件——当 taskListId 变化时，UI 需要重新加载任务列表，因为它们现在看的是一个不同的目录。

## 17.5 Agent 宕机后的任务归还

Agent 宕机是多 Agent 系统的常见场景。一个 Agent 认领了任务，然后进程被杀，任务卡在 `in_progress` 状态，没有人完成它。

[`tasks.ts#L818`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/utils/tasks.ts#L818) 的 `unassignTeammateTasks()` 处理这种情况：

```typescript
export async function unassignTeammateTasks(
  teamName: string,
  teammateId: string,
  teammateName: string,
  reason: 'terminated' | 'shutdown',
): Promise<UnassignTasksResult>
```

当一个 teammate 被终止（`terminated`）或主动关闭（`shutdown`）时，调用方触发这个函数。它找出该 Agent 持有的所有未完成任务，把 `owner` 清空，状态重置为 `pending`，并生成一条通知消息：

```
<teammateName> was terminated. 2 task(s) were unassigned: #3 "Write unit tests", #7 "Update docs".
Use TaskList to check availability and TaskUpdate with owner to reassign them to idle teammates.
```

这条消息不是给人看的，而是注入给其他 Agent 的 task-notification，让它们知道有任务重新可用。`reason` 参数区分了 `terminated` 和 `shutdown`，生成的通知措辞不同（"was terminated" vs "has shut down"），方便人工排障时判断是异常退出还是正常退出。

对比来看：如果不做任务归还，宕机 Agent 的任务会永远卡在 `in_progress`，整个 swarm 陷入死锁——其他 Agent 看到任务有 owner，不去认领，但 owner 已经死了。任务归还机制把这个悬挂状态变成了可恢复的状态。

## 17.6 主循环与任务系统的接缝

主循环 `queryLoop()` 和任务系统之间有两处接缝，ch17 原版只提到了一处。

第一处（[`query.ts#L1577`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/query.ts#L1577)）：subagent 只处理发给自己的 task-notification，不处理其他 Agent 的。这是 Agent 隔离的关键——进程内多个 Agent 共享同一个命令队列，靠 `agentId` 字段区分"这条 task-notification 是给我的"。主线程处理 `agentId === undefined` 的命令，subagent 处理 `agentId === currentAgentId` 的命令。

第二处（[`query.ts#L1685`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/query.ts#L1685)）：在 `BG_SESSIONS` 模式下，主循环每次迭代结束时，调用 `taskSummaryModule.maybeGenerateTaskSummary()`，生成后台任务摘要写入文件，供 `claude ps` 读取。这个调用只发生在非 subagent 的主线程上（`!toolUseContext.agentId`），避免多个 Agent 都在生成摘要造成写入冲突。

这两处接缝都是隐式的——没有统一的事件总线或 TaskSystem 接口，而是在循环内直接调用。这个设计有代价：如果需要修改接缝逻辑（比如让 subagent 也能生成摘要），需要找到所有调用点逐一修改。一个更显式的替代方案是引入事件系统，让任务系统和主循环通过事件通信，但引入事件系统会增加另一种复杂度——事件顺序、重复触发、事件丢失的处理。Claude Code 选择了简单直接的耦合。

## 17.7 通知系统：不丢失、不重复

任务状态变化需要通知其他 Agent，让它们感知并响应。通知机制在 [`task/framework.ts`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/utils/task/framework.ts) 里：

`notifyTasksUpdated()` 处理进程内的同步通知（UI 刷新），用 try/catch 包住 `emit()`：

```typescript
export function notifyTasksUpdated(): void {
  try {
    tasksUpdated.emit()
  } catch {
    // Ignore listener errors — task mutations must not fail due to notification issues
  }
}
```

注释写得很清楚：任务写入操作不能因为通知失败而失败。任务持久化是核心操作，UI 刷新是副效应。如果调换优先级——通知失败导致任务写入回滚——系统会在 UI 有 bug 时整体不可用。

跨 Agent 的异步通知走 `enqueuePendingNotification()`，消息格式是 XML：

```xml
<task-notification>
  <task-id>3</task-id>
  <task-type>background_agent</task-type>
  <output-file>/tmp/tasks/3.output</output-file>
  <status>completed</status>
  <summary>Task "Write unit tests" completed successfully</summary>
</task-notification>
```

这条消息被注入下一次迭代的 tool results，让接收方 Agent 在下一轮循环里看到状态变化。XML 格式（而不是 JSON）是有意的选择——模型对 XML 标签的解析在训练数据里有足够的先例，比 JSON 更稳定。

## 17.8 运行时可观测性的两种实现

任务系统有两种可观测性机制，服务不同的消费方。

第一种：文件系统可见性。每个任务是 `<id>.json`，任务输出写入 `<id>.output`。出了问题可以直接 `cat` 任务文件，不需要查数据库或日志。[`diskOutput.ts`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/utils/task/diskOutput.ts) 的 `getTaskOutputDelta()` 用字节偏移读增量输出——每次只读新增的部分，不加载整个文件。对于可能持续写入的长任务，增量读取是必要的。

第二种：SDK 事件流。[`sdkProgress.ts`](https://github.com/xuhengzhi75/claude-code-source/blob/c68ee10/src/utils/task/sdkProgress.ts) 的 `emitTaskProgress()` 向 SDK 事件队列推送结构化进度：

```typescript
enqueueSdkEvent({
  type: 'system',
  subtype: 'task_progress',
  task_id: params.taskId,
  usage: {
    total_tokens: params.totalTokens,
    tool_uses: params.toolUses,
    duration_ms: Date.now() - params.startTime,
  },
  last_tool_name: params.lastToolName,
  summary: params.summary,
})
```

这个接口是为程序化消费设计的——外部集成（比如 CI 系统、监控平台）可以订阅 SDK 事件流，实时获取 Agent 的进度，包括已用 token 数、工具调用次数、执行时长。

两种机制的服务对象不同：文件系统可见性服务人工排障，SDK 事件流服务程序化集成。两者并存，互不依赖。

## 17.9 心智模型验证题

**题目一：** 如果 `LOCK_OPTIONS` 里 `retries` 从 30 改成 3，什么情况下系统会出问题？

答案：在 10+ Agent 并发抢占同一个任务时可能出问题。注释里写了这个数字的来源——每个 critical section 需要约 50-100ms，10个 Agent 排队最后一个需要约 900ms 等待。`retries=30` 给了约 2.6s 的总等待时间。改成 3 大约只有 315ms，10 个并发 Agent 的末尾几个可能等不到锁，返回 `task_not_found`（claimTask 的默认失败路径）。现象是任务被漏认领，Agent 空转。

**题目二：** `deleteTask()` 为什么在删除文件**之前**先更新高水位标记，而不是之后？

答案：如果先删文件，然后写高水位标记时进程崩溃，高水位标记没有更新，下一次 `createTask()` 可能分配到被删任务的 ID，导致 ID 重用。先写高水位标记，即使后续删除失败，ID 也不会被重用（高水位标记已经保护了这个 ID）。这是一个"宁可 ID 不用，不能 ID 重用"的权衡。

## 17.10 本章小结

任务系统是多 Agent 协作的调度基础设施，而不只是状态存储。核心设计点：

- **双向阻塞图**（`blocks` / `blockedBy`）：描述任务依赖关系，支持 DAG 级调度
- **高水位标记**：防止 ID 在删除/重置后被复用，保证历史可追溯
- **两种锁粒度**：任务级锁用于普通更新，列表级锁用于检查-认领原子化
- **协作空间主键**（`getTaskListId()`）：五级优先级保证不同启动方式的 Agent 汇聚到同一任务池
- **Agent 宕机归还**（`unassignTeammateTasks()`）：把悬挂状态变成可恢复状态
- **通知不阻塞写入**：任务持久化不因通知失败而失败

这六个设计点里，任何一个缺失，多 Agent 协作就会在某类边界情况下悄无声息地损坏数据或陷入死锁。
